# 教學型投影片 — 黃金守則與泛式

## § 0. 定位與使用方式

### 0.1 本文讀者

| 讀者 | 用途 |
|------|------|
| 課程講師 / 助教 | 設計 / 審閱每章投影片 |
| 教材設計師（ID / LXD） | 對齊 Bloom 分層、認知負荷理論到實際版面 |
| 知識付費作者 / 線上課創作者 | 建立可複製的製作 SOP |
| Agent-A（內容） / Agent-B（版面） | 生成內容時可引用的明確契約 |

**本文不是**：品牌簡報守則、純視覺設計指南、銷售簡報公式。這些屬顧問型守則或行銷領域。

### 0.2 與顧問型守則的差異（8 維度一句話對照）

| 維度 | 顧問型（決策） | 教學型（學習） |
|------|--------------|--------------|
| **目的** | 讓決策者在最短時間拿到結論 | 讓學員走過「不懂→懂→會用」的橋 |
| **標題** | 完整論述句（thesis statement） | 學習目標（可測量動詞） |
| **敘事弧** | Threat → Opportunity → Recommendation | Motivation → Concept → Application |
| **三層分層** | Data / Insight / Guidance | Concept / Mechanism / Application |
| **單一錨點** | 視覺最重處（尺寸 × 深色 × 留白） | 認知最重處（一張一概念） |
| **配色** | 1 主色 + 中性灰 | 1 主色 + 警示色（錯誤專用） + 成功色 |
| **密度** | 最小密度（能砍就砍） | 漸進密度（先鋪墊再加複雜度） |
| **互動** | 幾乎無（單向說服） | 每 5 張必有一張 ASK/CHECKPOINT |

### 0.3 與 SOP 工作流的接合

本守則完全相容於 `投影片生成_SOP_Agent工作流.md`：
- **Layer A（Prompt 約束）**：Agent-A 寫 `02_slides_design.md` 時改用教學動詞（定義/比較/實作），而非決策動詞（主張/建議/評估）
- **Layer B（泛式模板）**：Agent-B 寫 `03_layout_spec.json` 時從本文 §6 的 7 種教學原型挑選
- **Layer C（編輯檢查）**：Agent-E 執行 §9 checklist 取代（或補充）顧問型的 14 條驗收

未來課程在 `00_skeleton.yaml` 的 frontmatter 加 `track: teaching` 即走教學軌道；`track: consulting` 走顧問軌道；未標則預設教學。

## § 1. 教學投影片的本質

### 1.1 一句話

> **投影片 ≠ 資訊、≠ 內容、= 學習路徑。**
>
> 教學就像帶人過橋：你不能直接叫他跳過去，要一步一步鋪石頭。

### 1.2 三句口訣（五歲小孩版本）

**動機 → 概念 → 轉化**

- **動機**：為什麼要學（這頁不做，學員聽完就忘）
- **概念**：學什麼（這頁是本體）
- **轉化**：怎麼用（這頁不做，學員會說「我聽得懂但用不出來」）

### 1.3 顧問型 vs 教學型樣本對照

| — | 顧問型樣本 | 教學型樣本 |
|---|----------|-----------|
| 標題 | 「AI 投資報酬率已跨越關鍵臨界點」 | 「**本章結束後你能：** 寫出一段 Broadcasting 運算而不用 for 迴圈」 |
| 主圖 | 一張柱狀圖 + callout 錨點 | 一張 Input→Process→Output 三欄示意 |
| 三層 | Data（柱）/ Insight（+37pp 徽章）/ Guidance（紅圈：應加碼投資） | Concept（定義）/ Mechanism（shape 對齊規則）/ Application（範例一行 code） |
| 字型密度 | 最小（能砍就砍） | 中等（要足夠讓學員離線看也能懂） |
| 結尾 | 錨定建議（Call to Action） | 預告下一章 + 一題 recall |

### 1.4 為什麼 AI 產出的教學簡報總是「差了一口氣」

它只負責**內容生成**，缺**學習設計（Instructional Design）**。
本文就是要把「學習設計」寫成契約，讓 AI 能接住。

## § 2. 七步章節結構（Module-Level Arc）

教學的核心敘事弧對應顧問型的三幕 threat/opportunity/recommendation，但細分為 7 步：

### 2.1 七步結構

```
[1] Motivation      — 為什麼要學
[2] Problem         — 問題是什麼
[3] Concept         — 核心概念
[4] Mechanism       — 運作原理
[5] Example         — 範例
[6] Practice        — 練習
[7] Summary         — 收斂 + 下一章銜接
```

### 2.2 每步的投影片配比（以一章 16–20 張為基準）

| 步驟 | 張數 | 對應原型（見 §6） | 常見錯誤 |
|------|------|-----------------|---------|
| Motivation | 1–2 | MOTIVATION、ASK、SILENT | 略過這步，直接講定義 |
| Problem | 1 | ASK、SILENT、PROBLEM-FRAME | 問題太寬泛，沒有 input/output |
| Concept | 2–3 | CONCEPT-CARD、MATRIX | 一張塞 3 個概念 |
| Mechanism | 3–5 | MECHANISM-FLOW、TABLE、CHART | step 跳太快或無序 |
| Example | 3–5 | EXAMPLE-I/O、Code panel、PHOTO | 範例只有 code 沒 input/output |
| Practice | 1–2 | PRACTICE-PROMPT、CHECKPOINT | 練習擺最後（錯，應擺中間） |
| Summary | 1–2 | PYRAMID、SILENT | 只重述、沒連下一章 |

### 2.3 節奏口訣

> **概念 → 例子 → 問題 → 小結**（四拍循環）

每講完一個子概念，必走完一輪四拍。一章有 3–4 個子概念，就是 3–4 個四拍循環。

### 2.4 違規檢測（Agent-E 可機械掃描）

- ✗ 章節開頭 3 張內沒有 Motivation 或 Problem → 失格
- ✗ 連續 5 張沒有 Example 或 Practice → 「只講理論」警告
- ✗ 結尾沒有 Summary + 下一章預告 → 失格

## § 3. 十二條教學黃金守則（T1–T12）

> 守則用祈使句 + 一句話原因 + 失敗症狀 + 如何偵測四段式呈現。

### T1. 每張 slide 只講一個概念（One Concept Per Slide）

- **為何**：人類工作記憶一次只能處理 3–5 個 chunk，一張塞兩個概念就會把學員推出流程
- **失敗症狀**：學員看到第二頁時還在消化第一頁，開始當機
- **偵測**：標題能不能用一個名詞短語概括？能 → 通過；需要「與」或「以及」→ 不通過
- **相容**：對應顧問型 G1，但教學型反向——顧問型要「論述句」，教學型要「概念名稱」

### T2. 學習目標用可測量動詞（Bloom 動詞）

- **為何**：「了解 X」無法測量；「能在 5 分鐘內寫出 X」可以
- **動詞庫**：
  - Remember：**列出、說出、辨認、標示**
  - Understand：**解釋、比較、歸納、用自己的話說**
  - Apply：**寫出、實作、計算、畫出**
  - Analyze：**分辨、拆解、診斷、找出**
  - Evaluate：**評估、選擇、審查、判斷**
  - Create：**設計、建立、組合、產出**
- **失敗症狀**：章節結束後，學員不知道「我學到了什麼能用的東西」
- **偵測**：章節開頭第二張能不能用「本章結束後你能：_______」開頭，後面接一個具體動詞？

### T3. 動機先行（Motivation-First）

- **為何**：沒有動機的知識是噪音
- **形式**：任何新章節開頭必有 Motivation slide（痛點故事 / 真實案例 / 血淋淋反例）
- **失敗症狀**：學員整堂課聽完，問他「學這有什麼用」答不出來
- **偵測**：翻到章節第 1 張，3 秒內能說出「為什麼要學這個」嗎？

### T4. Chunking：每節 3–5 個資訊單位

- **為何**：Miller's Law（7±2）是聽覺；視覺閱讀更嚴，3–5 是上限
- **形式**：若一張 slide 需要列 7 個項目 → 拆成 2 張（4 + 3）或重新設計（轉成 2×2 matrix）
- **失敗症狀**：slide 變成「維基百科條目」
- **偵測**：一張 slide 的視覺單元（block）多於 5 個？→ 拆頁

### T5. 範例必配反例（Example + Anti-Pattern）

- **為何**：只給正面例子，學員無法界定概念邊界；看到錯誤才知道「什麼不是」
- **形式**：每個 EXAMPLE-I/O 頁之後，儘量配一張 PITFALL 頁（錯誤 vs 正確並陳）
- **失敗症狀**：學員會用，但一上手就犯老師沒示範過的錯
- **偵測**：章節中 EXAMPLE 頁的 80% 之後有無 PITFALL / 比較頁？

### T6. 練習題擺中間，不是結尾

- **為何**：結尾練習題 = 「下課作業」，學員當場不做；中間練習題 = 驅動下一節的張力
- **形式**：把章節切成 3–4 個小段，每段講完丟一題 CHECKPOINT，然後進入下一段
- **失敗症狀**：學員課後遇到第一個問題就卡住，因為從來沒在現場動手過
- **偵測**：章節中段（第 40–60% 位置）有無 PRACTICE-PROMPT？

### T7. 配色紀律：主色 + 警示色

- **為何**：顧問型用單色是為了權威；教學型需要「用顏色區分正誤」，因此必須引入警示色
- **色票**：
  - 主色：`#1B5E3F`（深綠，概念 / 正確 / 重點）
  - 警示色：`#C62828`（深紅，錯誤 / 陷阱 / 禁忌）
  - 成功色：`#2E7D32`（中綠，通過 / ✓）
  - 中性色：`#333333`（文字）、`#808080`（次要）、`#D3D3D3`（邊框）
- **紀律**：紅色只能用在錯誤範例或 Pitfall，不能當裝飾色
- **失敗症狀**：紅色氾濫，學員失去警示感

### T8. Code 只放關鍵行（≤ 20 行）

- **為何**：完整檔讓學員讀不下去；關鍵行+省略號可以聚焦概念
- **形式**：用 `# ...` 省略無關程式碼；關鍵行加底色或側線強調
- **失敗症狀**：40 行 code 塞滿一頁，字級 10pt，後排看不到
- **偵測**：code panel 字級 < 14pt → 失格；code 行數 > 20 → 警告

### T9. 圖文分工

- **為何**：人腦處理影像 vs 文字使用不同通道，同時給雙通道訊息學習效果最佳（Dual-Coding Theory）
- **形式**：左文 + 右圖（或上圖 + 下文），禁左右都放文字
- **失敗症狀**：兩欄文字 = 學員只讀左邊或只讀右邊
- **偵測**：slide 是否有任一「視覺元素」（圖、表、code、示意）？沒有 → 警告

### T10. 每 5 張必插一張檢核點

- **為何**：Spaced Repetition 輕量版；避免連續資訊流導致學員脫線
- **形式**：CHECKPOINT 可以是 3 題小測 / 一句話 recall / 舉手題
- **失敗症狀**：上完一章學員什麼都沒吸收
- **偵測**：章節中任意連續 6 張是否都無 ASK/CHECKPOINT？→ 警告

### T11. 類比要標明「失真處」

- **為何**：類比是教學利器，但類比永遠有失真；不標明就會讓學員把類比當本體
- **形式**：「這就像 X，**但不同的是** Y」；失真處用小字 + 灰色註腳
- **失敗症狀**：學員把 Python 類別當 C++ 類別用，踩坑
- **偵測**：類比頁是否有至少一行「but not like...」的說明？

### T12. 收尾三件套：What / Why / Next

- **為何**：沒有「Next」的結尾是斷點；Summary 頁必須與下一章的 Motivation 頁銜接
- **形式**：
  - What：本章的 3 個 takeaway（三句話）
  - Why：為什麼這 3 個是 takeaway（一句）
  - Next：下一章要解決的問題一句預告
- **失敗症狀**：章節結束後學員不知道下節課與本節有什麼關係
- **偵測**：最後一張有無「下一章」字樣或預告問題？

## § 4. 為什麼 AI 做不到（教學版）

AI 可以生出內容，但下列四件事需要人類判斷：

### 4.1 判斷觀眾的先備知識
- AI 無法知道「這批學員是否學過 OOP」
- 結果：過度解釋（對高程度學員）或跳太快（對低程度學員）
- 補救：人類在 skeleton 的 `audience.prerequisites` 明寫已會/未會

### 4.2 設計練習難度梯度
- AI 傾向給「中等難度」題目
- 教學需要 easy（建立信心）→ medium（遷移）→ hard（挑戰），而且是階梯式
- 補救：人類寫 practice 題目與預期作答時間

### 4.3 預測「哪裡容易誤解」
- AI 不知道「這個概念在臺灣常被誤解為 XX」
- 補救：人類在 skeleton 填 `common_misconceptions`，Agent-A 據此設計 Pitfall 頁

### 4.4 選對類比
- 類比要貼文化脈絡（講股市 vs 講夜市的受眾不同）
- 補救：人類在 skeleton 填 `audience_culture`（中小企業主 / 大學生 / 轉職者），Agent-A 從類比庫挑選

## § 5. 補救框架（三層機制）

### Layer A · Prompt 約束（給 Agent-A）

| 約束項 | 內容 |
|--------|------|
| 標題格式 | 禁命題句；用「概念名稱」或「學習目標：能 <動詞>」 |

| 動詞庫 | 只能從 §3-T2 Bloom 動詞庫挑選 |
| Pitfall 配額 | 每 3 張 EXAMPLE 至少配 1 張 PITFALL |

7 種教學專屬原型 + 11 種沿用顧問型原型 = 18 種可用。

T1–T12 × 12 條 checklist，講師過稿前自檢。

## § 6. 教學型投影片原型庫

### 6.1 沿用顧問型 11 種（簡述）

以下原型完全相容顧問型守則 §9，本文不重複定義，用途微調如下：

| 原型 | 教學場景用途 |
|------|------------|
| **CHART** | 呈現量化對比（如向量化 vs for 迴圈的 1000 萬筆效能） |
| **MATRIX** | 決策 / 選用矩陣（如 List / Dict / Set 選哪個） |
| **TABLE** | 語法對照、複雜度表 |
| **PHOTO** | 真實截圖（官方文件、IDE 畫面、錯誤訊息） |
| **VS** | 兩方案對比，搭配 PITFALL 的基礎 |
| **SILENT** | 停頓、強調一句話（如「資料是名詞、Python 是動詞」） |
| **ASK** | 過場提問（低認知成本） |
| **PYRAMID** | 章節收束三層金字塔 |
| **RISK-MITIGATION** | 教「怎麼避坑」的風險清單 |
| **TIMELINE** | 歷史演進（如 Python 發展史） |
| **QUOTE-CARD** | 經典引言（Dijkstra / Linus 等） |

### 6.2 教學專屬新增 7 種

#### ① MOTIVATION — 動機頁

- **用途**：章節開場；給學員「為什麼要學」的具體理由
- **組成**：
  - 上半：情境痛點（一張照片 / 真實錯誤截圖 / 故事）
  - 下半：誘餌問題（一句話問 + 留白答案）
- **版面骨架**（13.333 × 7.5 inch 16:9）：
  ```
  y=0.0–0.5  章節代碼 + 標題（小字）
  y=0.5–4.0  痛點情境圖或照片（寬 10 吋置中）
  y=4.2–5.5  大字誘餌問題（FONT_HERO 40pt）
  y=5.5–6.5  小字提示：「學完本章你就能 ___」
  y=6.8–7.5  footer
  ```
- **違規**：純文字、無情境、無誘餌問題 → 降級為 ASK

#### ② CONCEPT-CARD — 概念卡

- **用途**：核心定義頁（每個概念配一張）
- **組成**：
  - 標題：概念名稱（名詞短語）
  - 左：一句話定義（≤ 30 字）+ 一個反義詞對照
  - 右：一張示意圖
- **版面骨架**：
  ```
  y=0.0–0.8  概念名稱（FONT_TITLE 28pt，深綠）
  左欄 x=0.6–6.5,  y=1.2–6.5  一句話定義 + 反義對照
  右欄 x=6.8–12.7, y=1.2–6.5  示意圖 / 圖示
  y=6.8–7.5  footer
  ```
- **違規**：左右都放文字 → 違反 T9

#### ③ MECHANISM-FLOW — 原理流程

- **用途**：step-by-step 講解運作原理
- **組成**：3–5 個步驟方塊 + 箭頭連接 + 每方塊下一行小字註解
- **版面骨架**：
  ```
  y=0.0–0.8  標題（學習目標）
  y=1.5–4.5  水平 3–5 個方塊 + 箭頭
  y=4.8–6.5  每個方塊下的說明文字
  y=6.8–7.5  footer
  ```
- **規則**：箭頭方向一致（水平向右），不要 Z 形或折返

#### ④ EXAMPLE-I/O — 範例頁

- **用途**：展示實際運作（最重要的原型之一）
- **組成**：三欄 Input | Process | Output（或兩欄 Input → Output）
- **版面骨架**：
  ```
  y=0.0–0.8  「範例：<場景名稱>」
  三欄 x=0.6/4.5/8.4, 各寬 3.9 吋
    欄 1: Input（小資料集 / 參數）
    欄 2: Process（code 或流程圖）
    欄 3: Output（結果 / 螢幕截圖）
  箭頭 Input→Process→Output
  y=6.8–7.5  footer
  ```
- **違規**：只放 code 不附 Input/Output → 降級為 Code panel

#### ⑤ PRACTICE-PROMPT — 練習題

- **用途**：章節中段驅動學員動手
- **組成**：一個問題（大字）+ 答題空間或 Think-Pair-Share 提示
- **版面骨架**：
  ```
  y=0.0–0.8  「練習時間」+ 預計時長（如「3 分鐘」）
  y=1.5–4.0  問題（FONT_HERO 32pt，置中）
  y=4.5–5.5  答題提示（小字：「與同伴討論：__________」）
  y=6.0–6.5  參考答案隱藏線（演講時按鍵顯示；書面用腳註）
  y=6.8–7.5  footer
  ```
- **規則**：絕對不能在本頁直接給答案，留到下一張或演講時揭示

#### ⑥ PITFALL / ANTI-PATTERN — 陷阱頁

- **用途**：防呆；緊跟在 EXAMPLE 之後
- **組成**：左欄錯誤寫法（紅底或紅框 + ✗）| 右欄正確寫法（綠底或綠框 + ✓）+ 下方一句「為什麼」
- **版面骨架**：
  ```
  y=0.0–0.8  「常見錯誤：<錯誤名稱>」
  左欄 x=0.6–6.5, y=1.2–5.5  錯誤 code + 紅色 ✗
  右欄 x=6.8–12.7, y=1.2–5.5  正確 code + 綠色 ✓
  y=5.8–6.5  「為什麼：<一句話解釋>」
  y=6.8–7.5  footer
  ```
- **警示色使用**：紅 `#C62828`、綠 `#2E7D32`，**本頁是全課唯一可用紅色的頁型**

#### ⑦ CHECKPOINT — 檢核點

- **用途**：每 5 張插一張，驗收學員跟上了沒
- **組成**：3 題 recall（簡單）或 1 題 transfer（困難）
- **版面骨架**：
  ```
  y=0.0–0.8  「Check Point」+ 章節進度條（N/Total）
  y=1.2–6.5  題目區（列點 3 題或大字 1 題）
  y=6.8–7.5  footer
  ```
- **規則**：題目答案不在本頁；下一張或講師口頭揭示

### 6.3 已退役 / 不建議用於教學的顧問型原型

| 原型 | 原因 |
|------|------|
| EXEC-SUMMARY | 教學場景不需要「戰情室」總覽，學員需要的是敘事 |
| KPI-STRIP | 純數字對學員意義不大，除非該章就是教 KPI |
| FUNNEL | 漏斗概念偏行銷，教學少用 |

其餘顧問型原型都可在教學軌道使用。

## § 7. Bloom 分層與教學節奏

### 7.1 Bloom 分層 × 原型對應表

| Bloom 層級 | 教學原型 | 建議比例 | 驗證問題 |
|-----------|---------|---------|---------|
| Remember（記） | CONCEPT-CARD、TABLE、QUOTE-CARD | 20% | 學員能否列出 / 說出定義？ |
| Understand（懂） | MECHANISM-FLOW、CHART、ASK | 25% | 學員能否用自己的話解釋？ |
| Apply（用） | EXAMPLE-I/O、Code panel、PHOTO | 25% | 學員能否在新情境套用？ |
| Analyze（析） | VS、PITFALL、MATRIX | 15% | 學員能否拆解 / 比較？ |
| Evaluate / Create（評 / 造） | PRACTICE-PROMPT、CHECKPOINT、PYRAMID | 15% | 學員能否自主設計 / 判斷？ |

### 7.2 節奏口訣：四拍循環

```
概念（CONCEPT-CARD / MECHANISM-FLOW）
  ↓
例子（EXAMPLE-I/O）
  ↓
問題（ASK / PRACTICE-PROMPT / CHECKPOINT）
  ↓
小結（SILENT 一句話 / PYRAMID）
```

每 4–6 張完成一輪循環；一章 16–20 張 = 3–4 個循環。

### 7.3 違規檢測

- 連續 6 張都在 Remember / Understand 層 → 「只講理論」警告
- 章節內 Apply 層 < 15% → 「缺範例」警告
- 章節內無任何 Evaluate / Create → 「沒練習」失格

## § 8. 泛式套用優先級矩陣（章節類型 × 原型）

對應顧問型守則 §6 的「8 模組 × 3 守則」，教學型改為「章節類型 × 推薦原型組合」。

### 8.1 章節類型分類

| 類型 | 代表章節範例 | 原型組合 | Bloom 重心 |
|------|-------------|---------|-----------|
| **語法教學** | Ch03 Lambda / Comprehension | MOTIVATION + CONCEPT-CARD × 3 + EXAMPLE-I/O × 3 + PITFALL × 2 + PRACTICE | Apply 45% |
| **概念建構** | Ch04 OOP 核心 / Ch01 計概前導 | MOTIVATION + CONCEPT-CARD × 2 + MECHANISM-FLOW × 2 + EXAMPLE + CHECKPOINT | Understand 40% |
| **工具操作** | Ch08 Pandas / Ch07 NumPy | EXAMPLE-I/O × N + PITFALL × 2 + PRACTICE-PROMPT × 2 + 少量 CONCEPT-CARD | Apply 50% |
| **系統直覺** | Ch06 I/O 例外 / Ch05 OOP 繼承 | MECHANISM-FLOW × 3 + PITFALL × 2 + MATRIX + CHECKPOINT | Understand 35%, Apply 30% |
| **整合實戰** | Ch10 DataCleaner 合體 | EXAMPLE-I/O（完整 walkthrough）+ PITFALL + PRACTICE-PROMPT + PYRAMID 收束 | Create 30% |

### 8.2 套用範例

以 Ch03「Lambda 與特殊函式」（20 張）為例：

```
S1  MOTIVATION     — 為什麼寫過 for 還要學 Lambda
S2  CONCEPT-CARD   — Lambda = 匿名小函式
S3  MECHANISM-FLOW — lambda x: x*2 的語法拆解
S4  EXAMPLE-I/O    — prices → map(lambda) → discounted
S5  PITFALL        — Lambda 超過一行 = 該用 def
S6  CHECKPOINT     — 下列哪個該用 lambda？
S7  CONCEPT-CARD   — map / filter 三件套
S8  MECHANISM-FLOW — map(f, iterable) 流程
S9  EXAMPLE-I/O    — filter 過濾 + map 轉換
S10 PITFALL        — map 忘記 list() 包起
S11 ASK            — 如果用 Comprehension 寫會更短嗎？
S12 CONCEPT-CARD   — List Comprehension 句構
S13 MECHANISM-FLOW — [expr for x in ... if cond] 對應 for
S14 EXAMPLE-I/O    — 三種 Comprehension 對照
S15 PITFALL        — 巢狀超過兩層該拆
S16 PRACTICE-PROMPT— 用 comprehension 改寫這段
S17 CONCEPT-CARD   — Generator expression 差異
S18 EXAMPLE-I/O    — 記憶體對比
S19 VS             — 四兄弟同骨架（def / lambda / comp / gen）
S20 PYRAMID        — 收束：Lambda 不是炫技，是 Pandas apply 的前戲
```

## § 9. 編輯檢查 Checklist（12 條）

講師 / Agent-E 過稿前使用。勾選式：

### 結構層（T1–T6）

- [ ] **T1** 每張 slide 標題可用一個名詞短語概括（不含「與」「以及」）
- [ ] **T2** 章節第二張用「本章結束後你能：<動詞>」明確指出學習目標
- [ ] **T3** 章節第 1 張是 MOTIVATION 頁（3 秒內能說為何要學）
- [ ] **T4** 每張 slide 視覺單元 ≤ 5 個
- [ ] **T5** EXAMPLE 頁之後 80% 以上配有 PITFALL 或比較頁
- [ ] **T6** 章節中段（40–60% 位置）有 PRACTICE-PROMPT

### 視覺層（T7–T9）

- [ ] **T7** 紅色只出現在 PITFALL 頁，不當裝飾
- [ ] **T8** 所有 code panel 字級 ≥ 14pt、行數 ≤ 20
- [ ] **T9** 每張 slide 有至少一個視覺元素（圖 / 表 / code / 示意圖）；禁左右兩欄文字

### 節奏層（T10–T12）

- [ ] **T10** 章節內任意連續 6 張都有 ASK / CHECKPOINT 之一
- [ ] **T11** 若使用類比，附有「但不同的是 ___」的失真說明
- [ ] **T12** 最後一張含 What / Why / Next 三件套；Next 與下章開頭銜接

### 自動化補強

此 checklist 可直接作為 Agent-E 驗收 DSL：

```yaml
teaching_track_qa_rules:
  - rule: T1_one_concept_per_slide
    check: "title does not contain '與' or '以及' or '和'"
    severity: warning
  - rule: T3_motivation_first
    check: "first non-cover slide has prototype: MOTIVATION|ASK|SILENT"
    severity: fail
  - rule: T6_practice_in_middle
    check: "any slide in position 40%-60% has prototype: PRACTICE-PROMPT|CHECKPOINT"
    severity: warning
  - rule: T8_code_panel_font
    check: "all code panels font_size >= 14"
    severity: fail
  - rule: T10_checkpoint_cadence
    check: "no 6 consecutive slides without ASK|CHECKPOINT"
    severity: warning
  - rule: T12_summary_next
    check: "last slide contains keyword 下一章|Next|預告"
    severity: fail
```

## § 10. 視覺敘事控制框架（教學版 v1.0）

對應顧問型守則 §10（v1.2），教學版改造四個維度：

### 10.1 視線路徑（Attention Flow）

- 教學頁預設帶有**「疑問 → 解答」**的視線走向
- 通常是 F 形（左上問題 → 右側線索 → 下方結論）或 Z 形（標題 → 左例 → 右例 → 結論）
- MOTIVATION 頁刻意用大留白誘導眼睛先看痛點再看問題
- CONCEPT-CARD 強制左文右圖，訓練學員建立雙通道記憶

### 10.2 三層分層（Concept / Mechanism / Application）

對應顧問型 Data / Insight / Guidance，改為：

| 教學分層 | 顧問型對照 | 每張 slide 的要求 |
|---------|----------|-----------------|
| Concept（是什麼） | Data | 定義 / 名詞 / 圖示 |
| Mechanism（怎麼運作） | Insight | step-by-step / 流程 / 規則 |
| Application（能幹嘛） | Guidance | 範例 / 練習 / 實務場景 |

**違規**：資料型 slide 三層不全 → 學員只記得「有這個東西」，不會用。

### 10.3 單一認知焦點（One Concept Anchor）

- 對應顧問型「單一視覺錨點」，教學版改為「**一張一概念**」
- 不再看「尺寸 × 深色 × 留白」的視覺最重處，而是看「一張 slide 的核心概念能否用一個名詞說完」
- 違規：slide 講了兩個獨立概念 → 拆頁

### 10.4 敘事節奏（4-beat cadence）

如 §7.2，每 4–6 張完成「概念 → 例子 → 問題 → 小結」循環。
偏離檢測：連續 4 張沒出現 example → 違規；連續 6 張沒出現 practice → 違規。

### 11.2 字級規範

| 元素 | 大小 | 用途 |
|------|------|------|
| HERO | 40–44 pt | MOTIVATION / PRACTICE 大字問題 |
| TITLE | 28 pt | 一般標題 / 概念名稱 |
| SUBTITLE | 20 pt | 副標 |
| BODY | 14 pt | 正文（最小不低於此） |
| CODE | 16 pt | 程式碼（最小 14pt） |
| CAPTION | 12 pt | 圖說 |
| FOOTER | 10 pt | 頁碼 / 版權 |
| MICRO | 8 pt | 資料來源、類比失真說明 |
