# 沉浸式電子書 Design Guide

> 基於《第1章：程式到底是什麼？》的逆向工程分析

---

## 📚 核心理念

### 寫作定位
**不是教科書，是深度對話**

- 目標：與讀者建立智識上的連結，而非單向的知識傳遞
- 風格：像智慧的朋友在咖啡廳裡的深談
- 避免：制式化的教學框架、練習題、測驗

---

## 🎨 寫作風格原則

### 1. 敘事結構 (Narrative Structure)

#### 1.1 開場引入
```markdown
模式：問題 → 共鳴 → 承諾

範例：
> 「寫程式很難嗎？」
> 這是我最常被問到的問題。每次聽到，我總會微笑著回答：
> 「你知道嗎？你每天都在寫程式，只是用的不是電腦語言而已。」
```

**原則：**
- 從讀者最關心的問題開始
- 立即建立親近感
- 給予意外但令人安心的答案

#### 1.2 故事線貫穿
```markdown
結構：引入角色 → 展現困境 → 跟隨成長 → 見證改變

小美的故事線：
- 開頭：35歲行銷經理的日常與疑慮
- 中段：理解概念時的釋然
- 結尾：實際改變與成長
```

**原則：**
- 創造一個讀者能投射自己的角色
- 角色的疑慮=讀者的疑慮
- 角色的成長=讀者的可能性

### 2. 對話式寫作 (Conversational Tone)

#### 2.1 直接對話
```markdown
❌ 避免：「讀者應該理解...」
✅ 採用：「你會做蛋炒飯嗎？」「讓我問你...」「如果你也曾...」
```

#### 2.2 反問引導思考
```markdown
模式：提出問題 → 暫停思考 → 給出觀點

範例：
「買東西算找零需要微積分嗎？做菜調味需要線性代數嗎？」
（讓讀者自己得出答案）
「大部分的程式設計，用到的數學不會超過小學程度。」
```

#### 2.3 親密語氣標記
- 「讓我告訴你一個秘密」
- 「你知道嗎？」
- 「真的，就是這麼簡單」
- 「說穿了」

### 3. 概念解釋策略 (Concept Explanation)

#### 3.1 從熟悉到陌生
```markdown
順序：日常經驗 → 找出模式 → 連結到程式概念

範例：
蛋炒飯（熟悉）→ 步驟順序（模式）→ 程式邏輯（陌生）
```

#### 3.2 類比優於定義
```markdown
❌ 避免：「程式是一組指令的集合...」
✅ 採用：「就像食譜告訴你如何做出一道菜」
```

#### 3.3 重新定義框架
```markdown
模式：挑戰既有認知 → 提供新視角 → 解釋優勢

範例：
「電腦很聰明」→「電腦其實很笨」→「這種笨反而可靠」
```

### 4. 情感設計 (Emotional Design)

#### 4.1 共鳴點設置
```markdown
位置：每個段落開頭或轉折處
方法：描述讀者可能的心理狀態

範例：
「但隨即，一連串的疑慮湧上心頭...」
「新手最害怕看到紅色的錯誤訊息」
```

#### 4.2 鼓勵與肯定
```markdown
時機：在提出困難或挑戰後立即給予

範例：
問題：「35歲會不會太晚」
肯定：「年齡從來不是限制，反而可能是優勢」
```

#### 4.3 個人經驗分享
```markdown
用途：建立信任、展示可能性

範例：
「我記得我的第一個程式...那兩個單字出現時，我興奮得像個孩子」
```

### 5. 節奏控制 (Pacing)

#### 5.1 段落長度變化
```markdown
模式：長段落（深入說明）→ 短段落（強調）→ 中段落（過渡）

範例：
長：詳細解釋電腦為何需要精確指令
短：「這就是電腦。」
中：連結到下一個概念
```

#### 5.2 分隔符使用
```markdown
用途：給讀者喘息空間、標記主題轉換
頻率：每2-3個小節使用一次 ---
```

#### 5.3 留白的藝術
```markdown
單行強調：
「真的，就是這麼簡單。」
（前後空行）

思考暫停：
「如果你也曾有過這些想法，請讓我告訴你一個秘密——」
「這些都不是問題。真的。」
```

### 6. 視覺元素運用 (Visual Elements)

#### 6.1 引用框
```markdown
用途：標記重要對話、核心觀點
> 「寫程式很難嗎？」
```

#### 6.2 簡化的程式碼
```markdown
原則：
- 使用中文變數名增加親和力
- 偽代碼而非真實語法
- 重點在邏輯而非語法

範例：
如果現在是設定時間：
    響鈴
```

#### 6.3 列點但不濫用
```markdown
適用：
- 並列的特徵（3-5項）
- 步驟說明（不超過7步）

避免：
- 過長的列表
- 層層嵌套
```

## 📖 章節結構模板

### 標準章節流程

```markdown
1. 引子 (Hook) - 5%
   - 引發好奇的問題或故事
   - 直接觸及讀者痛點

2. 主角引入 (Character Introduction) - 10%
   - 建立可認同的角色
   - 展示角色的困境

3. 核心概念展開 (Core Concepts) - 40%
   - 3-4個主要觀點
   - 每個觀點用故事或類比支撐
   - 穿插角色的領悟時刻

4. 迷思破解 (Myth Busting) - 25%
   - 直面常見誤解
   - 提供反直覺的真相
   - 用實例和數據支持

5. 轉變見證 (Transformation) - 15%
   - 回到主角故事
   - 展示實際改變
   - 連結到讀者的可能性

6. 哲思收尾 (Philosophical Closure) - 5%
   - 昇華到更高層次
   - 留下思考空間
   - 溫暖的鼓勵
```

## ✍️ 文字風格指南

### 詞彙選擇

#### 優選詞彙
- 「你」而非「讀者」、「使用者」
- 「我」而非「筆者」、「作者」
- 「讓我們」而非「接下來將」
- 「試試看」而非「請嘗試」

#### 避免用語
- 學術術語（除非必要）
- 命令語氣（「必須」、「應該」）
- 過度謙虛（「個人認為」、「也許」）
- 教科書用語（「本章將討論」）

### 句式特徵

#### 短句的力量
```markdown
用於：強調、轉折、結論
範例：
「這就是電腦。」
「真的。」
「程式改變了什麼？」
```

#### 問句的運用
```markdown
類型：
1. 開放問句（引發思考）
   「如果我也會這個，生活會變成什麼樣子？」

2. 封閉問句（建立認同）
   「你會做蛋炒飯嗎？」

3. 反問句（挑戰認知）
   「如果寫程式需要天賦，那開車是不是也需要天賦？」
```

#### 平行結構
```markdown
用於加強記憶和節奏：

「不是神秘的咒語，
 不是天才的專利，
 而是一種表達邏輯的方式」
```

## 🎭 情境設計原則

### 場景選擇
- 優先日常場景（廚房、通勤、購物）
- 避免技術場景（伺服器房、程式競賽）
- 選擇有普遍性的經驗

### 細節描寫
```markdown
原則：具體但不冗長

範例：
「她看著鏡中的自己，牙刷停在半空中。」
（一個動作捕捉整個心理狀態）
```

### 時間跨度
```markdown
展示成長軌跡：
- 三個月後...（初步成果）
- 六個月後...（能力提升）
- 一年後...（質的改變）
```

## 📝 實施檢查清單

### 每節必備元素
- [ ] 一個引人入勝的開頭
- [ ] 至少一個生活化類比
- [ ] 主角或讀者的心理描寫
- [ ] 一個「原來如此」的轉折
- [ ] 溫暖鼓勵的語句

### 語氣檢查
- [ ] 是否像在對話而非講課？
- [ ] 是否有「我」和「你」的互動？
- [ ] 是否避免了命令和說教？
- [ ] 是否保持了謙遜但自信的語調？

### 節奏檢查
- [ ] 段落長度是否有變化？
- [ ] 是否有適當的停頓點？
- [ ] 資訊密度是否適中？
- [ ] 是否在難點後給予緩衝？

### 情感檢查
- [ ] 是否理解讀者的恐懼？
- [ ] 是否給予足夠的鼓勵？
- [ ] 是否建立了信任感？
- [ ] 是否留下希望和可能性？

## 🚫 常見陷阱

### 要避免的寫作模式

1. **過度教學化**
   - ❌ 明確的學習目標列表
   - ❌ 步驟化的操作指南
   - ❌ 練習題和作業

2. **過度簡化**
   - ❌ 幼稚化的比喻
   - ❌ 迴避真實的困難
   - ❌ 不切實際的承諾

3. **過度技術**
   - ❌ 專業術語堆砌
   - ❌ 細節過度展開
   - ❌ 忽略情感需求

4. **過度激勵**
   - ❌ 空洞的口號
   - ❌ 過度的成功故事
   - ❌ 忽視真實困難

## 💫 金句公式

### 創造記憶點

#### 公式一：對比反轉
```markdown
模板：不是A，而是B
範例：「不是我不夠聰明，而是需要學會用電腦能懂的方式說話」
```

#### 公式二：類比連結
```markdown
模板：A就像B
範例：「程式就像食譜，電腦就像廚師」
```

#### 公式三：哲理昇華
```markdown
模板：表象→本質
範例：「程式改變的，不只是工作效率，更是看待世界的方式」
```
