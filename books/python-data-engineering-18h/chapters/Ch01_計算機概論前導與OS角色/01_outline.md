## 一、章節定位

本章不教硬體組裝，而是要建立「**資料從哪裡來、在哪裡運算、存到哪裡去**」的宏觀心智模型。後面所有效能議題（為什麼 NumPy 快、為什麼大檔會卡、為什麼路徑跨平台會壞）都會回頭參照本章概念。

## 三、章節結構

### 1-1. 程式語言的演進與 Python 的定位（20 分鐘）
- 三代程式語言：機器碼 → 組合語言 → 高階語言
- 編譯 vs 直譯：執行速度 vs 開發速度的取捨
- Python 為何是「黏合語言」：底層 C 套件 + 高階語法
- **重點落地**：解釋為何後面 Ch7 NumPy 比純 Python 快 10–100 倍

### 1-2. 資料的基本單位：Bit、Byte 與編碼（15 分鐘）
- Bit / Byte 的關係，1 Byte = 8 Bit
- KB → MB → GB → TB → PB 的量級感
- 練習：估算 `1 億 × float64` 的記憶體佔用 ≈ 800 MB
- **重點落地**：建立「資料量」的空間敏感度

### 1-3. 馮紐曼架構與核心硬體元件（25 分鐘）
- 馮紐曼模型：CPU + Memory + I/O 的分工
- **CPU**：控制單元 + ALU；多核心是什麼
- **GPU**：為何 ML 需要它（上千核心適合矩陣平行）
- **RAM**：CPU 的工作區；資料必須先載入 RAM 才能處理
- **Storage**：SSD vs HDD；速度差距
- **重點落地**：引入 OOM（Out of Memory）概念

### 1-4. 程式執行原理與作業系統 OS（30 分鐘）
**OS 角色（資源協調者）**
- Python 沒有權限直接操作硬體，所有請求透過 OS 的 System Call

**OS 三大職責**
1. **記憶體管理**：OOM、Swap（拿硬碟假裝 RAM 導致系統卡死）
2. **檔案系統管理**：路徑跨平台問題（`\` vs `/`），引出 `pathlib`
3. **行程與執行緒排程**：Process vs Thread；GIL 前導

**I/O Bound vs CPU Bound**
- I/O Bound：爬蟲、讀大檔（瓶頸在等待）
- CPU Bound：複雜運算、特徵工程（瓶頸在計算）
- 對應策略：multiprocessing / async / 向量化

**Linux 補充（5 分鐘）**
- 業界資料庫、雲端、排程器幾乎 100% 跑在 Linux
- 鼓勵未來學會 CLI 執行 .py 腳本

## 四、收尾流程圖：Python 程式的一生

```
寫下 df = pd.read_csv('data.csv')
        ↓
Python 直譯器轉為 Bytecode
        ↓
向 OS 發出 System Call 請求讀檔
        ↓
OS 檔案系統定位檔案 → I/O 控制器讀取
        ↓
資料載入 RAM（OS 記憶體管理）
        ↓
CPU 開始運算（OS 排程管理）
```
