## S6 — MATRIX · Bit / Byte / 量級

🖼️ 畫面：2×3 矩陣，列出資料容量階梯。最後一格深綠底：結論。

📣 畫面上的字：
- 標題：資料量的階梯：每一級都是 1024 倍，你要有量級感
- Bit：0 或 1，1 個電位
- Byte：8 Bit，可表 256 種狀態（一個 ASCII 字元）
- KB/MB：文件、一張圖、幾秒音訊
- GB：一部電影、100 萬筆資料
- TB/PB：公司資料湖、搜尋引擎 index
- **結論（深綠底）：估算資料量 → 選工具 → 估記憶體 → 避免 OOM**

🎙️ 講者這時說：
「量級感比精確數字重要。你看到一個 CSV 5 GB，要能立刻反射『這個不能整包塞進 8 GB 機器』。這個反射，Ch2 的 generator、Ch6 的 OOM 都會用到。」

## S9 — VS · CPU vs GPU

🖼️ 畫面：左右兩欄，左上角大照片佔位（CPU 晶片特寫）vs 右上角大照片佔位（GPU 晶片特寫）。中間 VS 與深綠膠囊「10-50× FP32 吞吐」。下方摘要。

📣 畫面上的字：
左欄 CPU：
- 少數強者（8–16 核）
- 單核時脈高（~3-5 GHz）
- 擅長：複雜分支邏輯
- 典型場景：OS、Web、一般程式
- 照片佔位：Intel/AMD CPU 晶片特寫

右欄 GPU：
- 多數平庸者（數千 CUDA 核）
- 單核時脈低（~1-2 GHz）
- 擅長：同構大量平行
- 典型場景：矩陣、渲染、神經網路
- 照片佔位：NVIDIA GPU 晶片特寫

摘要：矩陣運算走 GPU 不是流行，是因為它能被切成上千份同時做。

🎙️ 講者這時說：
「CPU 是讓十個不同工作同時進行的多工老將；GPU 是讓一萬個完全一樣的小活同時做的裝配線。你訓練神經網路，99% 都是『同一種乘法做十億次』——這就是為什麼 GPU 快，不是因為它神，是因為你的工作剛好是它設計來做的那種。」

## S10 — FLOW · 儲存金字塔

🖼️ 畫面：5 節點橫向流程鏈：Register → L1/L2 Cache → RAM → SSD → HDD。每節點下標延遲與容量。節點 3（RAM）深綠實底強調。

📣 畫面上的字：

- Register（CPU 內）· <1 ns · B 量級

- RAM（**highlight**）· ~100 ns · GB 量級 · 資料處理主戰場

- 下方摘要：上下相差 10^7 倍——這就是為什麼『讀檔』永遠是程式最慢的一步。

## S12 — PYRAMID · OS 三大職責

🖼️ 畫面：金字塔堆疊（4 層不含底座）：頂層 Scheduler、中上 File System、中下 Memory Manager、底層「Hardware（CPU/RAM/Disk）」。左右兩條縱骨：權限控管、Interrupt 處理。下方反向論述框。

📣 畫面上的字：
- 標題：OS 三大職責：記憶體、檔案、行程——數據工程的三座隱形山
- 頂層 Scheduler：誰先跑、跑多久、何時切換（Process / Thread / GIL）
- File System：檔案怎麼組織、路徑怎麼解析、權限誰管
- Memory Manager：誰的 RAM 誰的 Swap、OOM 發生時殺誰
- Hardware：底層實體資源
- 縱骨：權限控管 · Interrupt 處理
- 反向論述框（底部）：不懂 OS，你的 Python 程式永遠只是在黑盒上跳舞。

🎙️ 講者這時說：
「這三件事剛好對應後面三章：Memory Manager → Ch6 的 OOM；File System → Ch5 的 pathlib；Scheduler → Ch9 的 GIL 與 multiprocess。今天你不用背細節，你要把這張圖釘在腦袋裡，後面三章回來對號入座。」

## S13 — RISK · OOM 與 Swap

🖼️ 畫面：左欄「Risk（OOM 來源）」，右欄「Mitigation（你的對策）」，雙框對稱。下方結語一行。

📣 畫面上的字：
左欄（真風險）：
- 一次 `pd.read_csv` 大檔：整包進 RAM
- `groupby` 後 pivot：中介表格指數膨脹
- DataFrame 鏈式 `.copy()`：同一份資料在 RAM 複製 5 份
- Swap 啟動：效能掉 100–1000 倍，系統進入半死

右欄（課程設計）：
- 用 `chunksize=` 或 Polars/Dask 分塊
- 先 filter 再 groupby；用 `observed=True`
- 用 `inplace=True` 或明確 view；別讓 copy 鏈無限長
- 監看 `free -h` / `psutil.virtual_memory()`，提早退場

摘要：OOM 不是意外，是能估算、能預防的必然——前提是你有量級感（S7）。

🎙️ 講者這時說：
「這張的關鍵不是背技巧，是建立一個反射：寫完 `pd.read_csv(...)` 下一秒就問自己『它幾 GB、我有幾 GB』。這個反射，救你 80% 的 OOM。」

## S14 — MATRIX · pathlib 的救贖

🖼️ 畫面：2×2 矩陣，左上/右上對比壞寫法與好寫法；右下深綠實底結論格。

📣 畫面上的字：
- 標題：跨平台路徑小坑：Windows \ vs POSIX /——為什麼要用 pathlib
- (1,1) 壞寫法：`path = folder + "\\" + name`（Windows 硬編、Linux 爆炸）
- (1,2) 好寫法：`path = Path(folder) / name`（OS 自動選分隔符）
- (2,1) 壞寫法：`open("data/2024/q1/sales.csv")`（相對路徑、看執行位置）
- (2,2) **結論（深綠底）：用 pathlib 不是品味問題，是跨平台可移植性的最低保險。**

🎙️ 講者這時說：
「你現在可能覺得『字串拼路徑我用了十年沒出事』——那是因為你一直在同一台機器上。當你的程式從你的 Mac 上傳到 Linux 雲端，第一個爆的就是這裡。pathlib 一行解決、零成本。」

## S15 — VS · I/O Bound vs CPU Bound

🖼️ 畫面：左右對比欄，左「I/O Bound」vs 右「CPU Bound」。中間深綠 VS 與膠囊「選錯策略 = 白做工」。下方摘要。

📣 畫面上的字：
左欄 I/O Bound（瓶頸在等）：
- 症狀：CPU 很閒、在等網路/硬碟
- 典型：爬蟲、API 呼叫、讀大檔、資料庫
- 對策：async / asyncio / threading
- 為什麼：GIL 不影響 I/O，等待可重疊

右欄 CPU Bound（瓶頸在算）：
- 症狀：CPU 100%、程式跑不動
- 典型：特徵工程、矩陣運算、模型訓練
- 對策：multiprocessing / 向量化 / GPU
- 為什麼：GIL 限制 thread，得開新進程

摘要：搞錯類型，你開 100 條 thread 也跑不快——診斷清楚再選工具。

🎙️ 講者這時說：
「這張是 Ch9 的前導，也是所有併發討論的定盤星。你寫一段程式覺得慢，先問：我慢在等、還是慢在算？答對這題，Python 就三把武器任你挑：async、thread、process。」
