### A1 — 你以為在學工具，其實在學模型的底層語言

**核心主張：** 所有 ML/DL 框架的核心計算都建立在多維陣列操作上，NumPy 是理解這一切的最短路徑。

**講師講解要點：**

- 打開任何 PyTorch 或 TensorFlow 的教學，第一步幾乎都是 `import numpy as np`。為什麼？因為 tensor 的概念與操作幾乎直接繼承自 NumPy。
- 線性迴歸的預測公式 `y = Xw + b`，X 是特徵矩陣、w 是權重向量。沒有矩陣概念，你就只能背公式，無法理解為什麼這樣計算。
- NumPy 不是「讓你算快一點」的工具，而是讓你用正確的語言描述計算問題的工具。快只是副作用。
- 2024 年的 AI 工程師每天用的概念：batch size、feature dimension、embedding dimension，全部是 ndarray 的 shape 概念。
- 學 NumPy 的終點不是 NumPy，而是 PyTorch、JAX、TensorFlow 都變得可讀。

**視覺建議：** 一張橫向對照表，左邊是高中數學的向量 / 矩陣符號，右邊是對應的 NumPy 程式碼。中間畫箭頭，標示「你已經懂這個，只差語法」。

**轉場：** 「但 Python 不是本來就有 list 嗎？為什麼需要 ndarray？下一張說清楚。」

### A2 — ndarray：為科學計算設計的資料結構

**核心主張：** list 是通用容器，可以放任何東西；ndarray 是同質性的數值陣列，這個限制帶來根本性的效能與語義優勢。

**講師講解要點：**

- Python list 可以放 `[1, "hello", None, [2, 3]]`，這種彈性在記憶體上付出代價：每個元素是一個獨立的 Python 物件，有額外的類型標記、引用計數、記憶體位址。
- ndarray 只放一種型別（例如 float64），所有元素緊密排列在連續記憶體空間中，CPU 的快取命中率大幅提升。
- 效能差距在小資料看不出來，但在 100 萬筆資料時，list 的迭代可能慢 50-100 倍。
- `np.array([1,2,3])` vs `[1,2,3]`：外表相似，底層天壤之別。前者是 C 語言陣列包了一層 Python 介面，後者是 Python 物件指標的集合。
- 關鍵：ndarray 讓你把一個「對整個陣列的操作」送進去，而不是對每個元素分別操作。這就是向量化的基礎。

**視覺建議：** 記憶體示意圖。左邊畫 Python list：一排格子，每格指向分散的物件（含型別標記、值、引用計數）。右邊畫 ndarray：一排緊密排列的數值格子，直接是值，沒有額外開銷。在下方加一行：「同樣 100 萬個 float，ndarray 用的記憶體約是 list 的 1/8」。

**轉場：** 「有了 ndarray，接下來最重要的三個概念：shape、axis、dtype。理解這三個，你就能看懂任何框架的錯誤訊息。」

### A3 — shape / axis / dtype：三個你必須看懂的概念

**核心主張：** shape 描述尺寸、axis 描述操作方向、dtype 描述元素型別。這三個概念是所有後續操作的語言基礎。

**講師講解要點：**

- **shape**：`arr.shape` 回傳一個 tuple，例如 `(100, 28, 28)` 表示 100 張 28x28 的灰階影像。shape 不只是「大小」，它是資料的意義標記。在 ML 裡，`(batch_size, features)` 是你最常看到的 2D shape。
- **axis**：`np.sum(arr, axis=0)` 是沿著第 0 軸加總，對矩陣來說就是對每一列加總得到一行向量。`axis=1` 則是對每一行加總。很多初學者在這裡卡住，關鍵記憶法：axis 是你要「壓縮掉」的那個維度。
- **dtype**：`float64` 精度高但佔記憶體，`float32` 是深度學習的標準選擇（GPU 的原生精度），`int8` 用於模型量化。型別不只影響記憶體，也影響計算精度。
- 常見錯誤：兩個 ndarray 做矩陣乘法前 shape 不匹配，最常見的 shape mismatch 有三種，逐一示範。
- 使用 `.reshape()` 與 `.T` 進行 shape 變換，理解「資料沒有動，只是視角換了」。

**視覺建議：** 三個並排的方塊示意圖。第一個：一個 3D 立方體，標示三個軸的方向（axis 0 = 深度、axis 1 = 行、axis 2 = 列）。第二個：一個 2D 矩陣，展示 `sum(axis=0)` 與 `sum(axis=1)` 的結果方向。第三個：dtype 精度對照表（float64 / float32 / int32 / int8 + 記憶體占用）。

**轉場：** 「理解 shape 和 axis 之後，你會發現 NumPy 最神奇的能力：broadcasting。它能讓不同 shape 的陣列直接相加，不需要迴圈。」

### A4 — Broadcasting：讓資料自己跑起來

**核心主張：** Broadcasting 是 NumPy 的核心魔法，允許不同 shape 的陣列在特定規則下進行操作，消滅大量重複的迴圈程式碼。

**講師講解要點：**

- 最簡單的案例：`arr + 5`，一個陣列加一個純量。你沒有寫迴圈，NumPy 自動把 5「廣播」到每個元素。這是 broadcasting 的最基本形式。
- 更複雜的案例：一個 `(3, 4)` 的矩陣加一個 `(1, 4)` 的向量，向量會沿著第 0 軸廣播，等效於把向量複製三份再相加。但 NumPy 實際上不複製記憶體，只是在計算時虛擬地廣播。
- Broadcasting 規則（從右往左對齊）：如果兩個維度相同，可以直接操作；如果一個維度是 1，可以廣播；如果兩個維度不同且都不是 1，報錯。
- 實際應用：特徵標準化。每個特徵減去均值、除以標準差，用 broadcasting 一行搞定：`(X - X.mean(axis=0)) / X.std(axis=0)`，不需要任何迴圈。
- 注意事項：broadcasting 的錯誤訊息是 `operands could not be broadcast together with shapes (3,4) (4,)`，看到這個，先對齊兩個 shape 從右到左，找出不相容的維度。

**視覺建議：** 動態圖示（靜態版）：左邊一個 `(3, 4)` 的藍色矩陣，右邊一個 `(1, 4)` 的橙色向量，用虛線箭頭表示橙色向量「複製」三份覆蓋到藍色矩陣上，最後變成一個綠色的 `(3, 4)` 結果矩陣。底部加一個 broadcasting 規則的簡表。

**轉場：** 「Broadcasting 讓你不用寫迴圈，但為什麼不用迴圈這件事如此重要？下一張說向量化思維。」

### A5 — 向量化思維：不是快，是對的

**核心主張：** 用 for 迴圈處理陣列操作不只是效能問題，而是用了錯誤的抽象層次描述問題。向量化是讓程式碼的語義與數學意圖對齊。

**講師講解要點：**

- 以計算兩個向量的點積為例，展示三個版本：Python for 迴圈版、NumPy 向量化版、`np.dot()` 版。效能差距可達 100 倍，但更重要的是：哪一個最清楚說明「這是點積運算」？
- 向量化迫使你用「整個陣列」來思考，而不是「每個元素」。這和 ML 框架的思維一致：你不操作單一樣本，你操作一個 batch。
- 常見的向量化替換：`np.where()` 替換條件迴圈、`np.cumsum()` 替換累加迴圈、`np.apply_along_axis()` 處理逐行操作（但要謹慎，它並非完全向量化）。
- 識別「可以向量化」的信號：如果你的 for 迴圈在遍歷 ndarray 的每個元素做相同操作，幾乎都可以向量化。
- 反例：有時迴圈是必要的，例如遞歸依賴（後一個值依賴前一個值的計算結果），這種情況向量化不適用。不要盲目去除所有迴圈。

**視覺建議：** 左右對比程式碼塊。左邊：10 行的 for 迴圈版計算向量點積。右邊：1 行的 `np.dot(a, b)`。底部加時間比較條形圖（模擬數據）：for 迴圈 2500ms vs NumPy 5ms（100 萬維向量）。在右邊下方加一行文字：「更短、更清晰、更接近數學原意」。

**轉場：** 「向量化讓你能用 NumPy 寫出接近數學的程式。但 NumPy 在 ML 工作流裡究竟扮演什麼角色？」

### A6 — NumPy 在 ML/DL 的實際角色

**核心主張：** NumPy 是 ML 工作流的共通貨幣，從特徵矩陣到模型權重，都以陣列形式流通。

**講師講解要點：**

- 一個典型的 ML 流程：載入資料 → 預處理 → 建立特徵矩陣 X（shape: `(n_samples, n_features)`）→ 建立標籤向量 y（shape: `(n_samples,)`）→ 送入模型訓練。X 和 y 幾乎永遠是 NumPy ndarray 或與其相容的格式。
- scikit-learn 的所有估計器（Estimator）都接受 NumPy 陣列作為輸入。你寫 `clf.fit(X, y)` 時，X 就是一個 `(n_samples, n_features)` 的 ndarray。
- PyTorch 的 tensor 可以直接從 NumPy 陣列轉換：`torch.from_numpy(arr)`，且在 CPU 上共享記憶體（zero-copy）。這不是巧合，PyTorch 的設計刻意與 NumPy 保持介面一致性。
- 深度學習的核心是矩陣乘法：每一層的前向傳播就是 `output = activation(W @ input + b)`，W 是權重矩陣、@ 是 NumPy 的矩陣乘法運算符。
- 工程意義：學會 NumPy 之後，你看 ML 框架的原始碼不再是天書。`(batch_size, seq_len, hidden_dim)` 這種三維 tensor 的 shape，就是 ndarray 概念的直接延伸。

### A7 — 里程碑：NumPy 2.0 — 2006 年以來第一個 major release

**核心主張：** NumPy 2.0（2024 年發布）是 18 年來第一個 major version 升級，代表科學計算 Python 生態在成熟後仍保持演進的能力。

**講師講解要點：**

- NumPy 1.0 於 2006 年發布。此後 18 年都以 1.x 的形式發展，每個版本都保持嚴格的向後相容性。2024 年的 2.0 是一個明確信號：生態願意接受破壞性變更來提升品質。
- NumPy 2.0 的三個重要改動：（1）清理了大量已棄用的 API，移除長達 10 年的「deprecated 但沒刪」函式；（2）提升 Python 類型標註的覆蓋率，讓 IDE 靜態分析更準確；（3）字串類型的大幅改進，加入 `StringDType`，讓字串陣列效能大幅提升。
- 對學習者的實際意義：如果你搜尋到的教學使用了 `np.bool`、`np.int`、`np.float` 等裸型別名稱，那是 NumPy 1.x 的語法。在 2.0 中這些已被移除，應改用 Python 原生型別 `bool`、`int`、`float` 或 NumPy 特定型別如 `np.float64`。
- 工具有版本意識是工程師的基本素養。不要只學「如何用」，也要有能力識別「這是哪個版本的用法」。
- 這個里程碑也代表 NumPy 社群在主動清理技術債，是開源生態健康的表現。

**視覺建議：** 一條時間軸，從 2006 年的 NumPy 1.0 到 2024 年的 NumPy 2.0，中間標示幾個重要的 1.x 里程碑。右側用醒目框格標示 2.0 的三個主要改動。底部加一個「版本遷移提示」的小框：常見的 1.x → 2.0 API 變更對照表（3 個例子即可）。

### B1 — DataFrame：分析世界的工作台

**核心主張：** DataFrame 不是試算表軟體的替代品，而是一個帶有型別系統、索引機制與豐富操作 API 的計算物件。

**講師講解要點：**

- DataFrame 的核心結構：列（rows）與欄（columns）。每個欄是一個 Series（一維帶標籤陣列），每個 Series 有獨立的 dtype。這與 ndarray 不同：ndarray 整個只能有一個 dtype，DataFrame 每欄可以不同。
- Index 的概念很多人忽略：index 不是一個欄位，它是列的「身份證」。預設是 0, 1, 2... 但可以是日期、字串、多層級索引。很多初學者因為不理解 index 而在 merge/join 時遇到意想不到的結果。
- DataFrame 是「帶標籤的矩陣」，這個說法幫助你理解它和 NumPy 的關係：DataFrame 在分析時用欄名操作，在計算時底層是 ndarray（或 Arrow array）。
- 建立 DataFrame 的三種常見方式：從 dict 建立、從 CSV/Excel 讀入、從 SQL 查詢結果轉換。實務上最常用的是讀檔。
- DataFrame 是 pandas 的核心物件，也是資料分析工作流的起點與終點。原始資料進來是 DataFrame，分析結果匯出也是 DataFrame（然後轉 CSV 或存進資料庫）。

**視覺建議：** 一個 DataFrame 的解剖圖：左側標出 index 欄（帶標籤）、頂部標出 columns（欄名）、每欄下方標出 dtype、每個 Series 用不同顏色框起。旁邊附上與 Excel 試算表的對比說明，強調「DataFrame 的欄有型別，Excel 的格沒有」。

**轉場：** 「有了 DataFrame，第一件事是什麼？讀進資料，然後看一眼手上的材料。」

### B2 — 讀進來、看一眼、查型別：每個專案都從這裡開始

**核心主張：** 資料分析的第一步不是計算，是偵察。在對資料一無所知時，有一套標準的「初次見面流程」。

**講師講解要點：**

- 五個必備的初始偵察函式：`df.head()`（看前 5 筆）、`df.info()`（欄名、型別、非空計數）、`df.describe()`（數值型欄位的統計摘要）、`df.shape`（幾列幾欄）、`df.dtypes`（每欄的型別）。這是每個資料分析專案的前五行程式碼。
- `df.info()` 是最重要的那一個：它同時告訴你欄名、型別、非空數量（從而反推缺失數量）、記憶體使用量。一眼看完你就知道有沒有嚴重缺失、有沒有型別不對（例如日期被讀成 object）。
- 型別推斷陷阱：pandas 讀 CSV 時會猜型別。常見的問題：含有「NT$1,200」的金額欄被讀成 object 而非 float、含有「2023-01-15」的日期欄被讀成 object 而非 datetime64、含有整數的欄位因為有一個缺失值而變成 float64（因為 NaN 是 float）。
- `df.value_counts()` 用於類別型欄位的分布快覽，是 `describe()` 對非數值欄位的補充。
- 培養「資料第一眼的直覺」：看完 `info()` 後，腦子裡要有一張評估表：缺失值多嗎？有沒有型別異常？欄位名稱有沒有空格或特殊字元？樣本量夠嗎？

**視覺建議：** 一份模擬的 `df.info()` 輸出截圖，用不同顏色標注關鍵資訊：欄名（藍）、dtype（橙）、非空計數（綠，並標示哪些有缺失）、記憶體使用（紅框）。旁邊附上「初次見面流程」的 checklist：五個函式依序列出，每個旁邊標注「告訴你什麼」。

### B3 — 篩選、排序、聚合、轉換：分析師每天在做的四件事

**核心主張：** 掌握 boolean indexing、sort_values、agg 與 apply 這四個操作，就能解決 80% 的資料操作需求。

**講師講解要點：**

- **篩選（Filtering）**：`df[df['age'] > 30]`，這是 boolean indexing，方括號裡是一個 True/False 的 Series。組合條件用 `&` 和 `|`（不是 `and` 和 `or`），每個條件必須用括號包住：`df[(df['age'] > 30) & (df['city'] == 'Taipei')]`。
- **排序（Sorting）**：`df.sort_values('salary', ascending=False)` 按薪資降序排列。`sort_values` 預設回傳新的 DataFrame（不改動原始資料），這是 pandas 的一般設計原則：大部分操作回傳新物件。
- **聚合（Aggregation）**：`df['salary'].mean()`、`df.agg({'salary': ['mean', 'std'], 'age': 'median'})`。聚合是把多個值壓縮成一個或幾個摘要值。
- **轉換（Transform）**：`df['salary_log'] = np.log(df['salary'])`、`df['full_name'] = df['first'] + ' ' + df['last']`。轉換是建立新欄位，不改變 DataFrame 的形狀。
- 操作鏈（Method Chaining）：`df.query(...).sort_values(...).head(10)` 是許多資深 pandas 用戶偏好的風格，讓資料流水線的每一步都可讀。但在 pandas 2.0 的 Copy-on-Write 背景下，要理解鏈式操作的記憶體語意。

**視覺建議：** 四個並排的操作示意卡，每張卡包含：操作名稱、一行程式碼示例、輸入表格示意 → 輸出表格示意（用箭頭連接）。讓學生一眼看到每個操作「改變了什麼」。

**轉場：** 「四個基礎操作之後，我們看一個很多人學了語法卻沒有學會思維的操作：groupby。」

### B4 — groupby 不是語法，是業務切片的思維工具

**核心主張：** groupby 的本質是「split-apply-combine」三步驟，它把資料問題轉換成業務問題：按什麼維度切、對每個切片做什麼、再把結果合回來。

**講師講解要點：**

- groupby 的三步驟：**Split**（按一個或多個欄位把 DataFrame 切成若干組）、**Apply**（對每組套用一個函式，通常是 `sum`、`mean`、`count`、`agg`）、**Combine**（把每組的結果合併成一個新的 DataFrame）。
- 業務問題直接翻譯為 groupby 語法：「各部門的平均薪資」→ `df.groupby('department')['salary'].mean()`；「各地區、各產品類別的銷售總額」→ `df.groupby(['region', 'category'])['revenue'].sum()`。
- `groupby` + `agg`：對不同欄位套用不同函式。`df.groupby('region').agg({'revenue': 'sum', 'order_count': 'count', 'unit_price': 'mean'})`。一次查詢，多個指標。
- `transform` vs `agg` 的區別：`agg` 壓縮維度（輸出比輸入小）；`transform` 保持維度（輸出與輸入等長）。例如計算每個員工的薪資佔部門總薪資的比例，需要用 `transform` 把部門總薪資對齊回每一列。
- 從業務分析的視角：在商業智慧（BI）裡，幾乎所有的「按 X 看 Y 指標」都可以用 groupby 表達。掌握 groupby 就是掌握了把資料表翻譯成業務洞察的能力。

**視覺建議：** 一張三步驟示意圖。左邊：一個混合的銷售資料表（含 region、category、revenue 欄）。中間：被虛線切成三個顏色不同的子表（Split，按 region 分組）。右邊：三個子表各自計算 sum（Apply），然後合併成最終結果表（Combine）。整個圖標記 split-apply-combine 三個步驟。

**轉場：** 「做完聚合分析，你可能發現有些格子是空的、有些數字看起來很奇怪。這就引出資料清理的核心課題。」

### B5 — 缺失值與離群值：資料要像資料，才能餵模型

**核心主張：** 沒有清理過的資料是毒，不是原料。ML 模型對「不乾淨」的資料毫無抵抗力，而資料清理是讓模型能運作的前提。

**講師講解要點：**

- **缺失值的三種來源**：（1）資料收集時就沒有（例如問卷某題空白）；（2）資料合併時的欄位不對齊（JOIN 後的 NaN）；（3）系統故障或資料管道問題（sensor 未回傳值）。來源不同，處理策略不同。
- **缺失值的四種處理策略**：刪除（`dropna()`，適合缺失比例極低且隨機缺失）；填補固定值（`fillna(0)` 或 `fillna('Unknown')`，適合知道「無值就等於零/未知」的情況）；統計填補（`fillna(df.mean())`，適合數值型且分布接近常態）；模型填補（用其他特徵預測缺失值，適合缺失比例高且重要的欄位）。
- **離群值的識別**：IQR 方法（Q3 - Q1 的 1.5 倍外為離群）和 Z-score 方法（超過 3 個標準差為離群）。用 pandas 的 `.quantile()` 和 NumPy 的 `np.std()` 搭配實作。
- **離群值的處理**：不一定要刪除。有時離群值是真實的（業績最頂尖的 1% 員工不是異常），有時是錯誤的（年齡 999 歲是輸入錯誤）。必須結合業務理解判斷。
- **清理的目標**：讓每一欄的型別正確、缺失值的比例和模式已知、極端值的存在是有意識的選擇，而不是無意識的遺漏。這是資料能「被信任」的最低門檻。

**視覺建議：** 一張資料清理前後對照表：左邊是一個「髒」的 DataFrame（有 NaN、型別錯誤、疑似離群的數字），右邊是清理後的版本，每個修改位置用顏色標注。底部附一個決策樹：缺失值 → 缺失比例 > 50%？→ 是/否分叉，引導到不同的處理策略。

**轉場：** 「資料清理完，我們來看 pandas 這個工具本身的演進。」

### B6 — 里程碑：pandas 2.0 — Arrow backend、Copy-on-Write 與記憶體革命

**核心主張：** pandas 2.0 是架構層級的升級，Copy-on-Write 解決了長達十年的語意不一致問題，Arrow backend 帶來大幅度的記憶體效率提升。

**講師講解要點：**

- **Copy-on-Write（CoW）**：pandas 1.x 最臭名昭著的問題之一是「鏈式賦值」的不確定行為。`df[df['age'] > 30]['salary'] = 0`，這行程式碼在 1.x 裡可能修改原始 df，也可能不修改，取決於 pandas 內部是否做了複製，結果是 undefined behavior。pandas 2.0 引入 CoW：任何操作回傳的結果都是獨立的（但複製發生在寫入時，不是讀取時），徹底消滅這個歧義。
- **Arrow backend**：pandas 2.0 允許使用 Apache Arrow 作為底層儲存格式（之前預設是 NumPy）。Arrow 格式支援 nullable integer（原生 NaN，而非用 float 來表示 int 的 NaN）、字串效率大幅提升、記憶體使用量在典型資料集上可減少 30-50%。
- 對學習者的實際意義：讀取資料時加入 `dtype_backend='arrow'` 參數，或在建立 DataFrame 時指定，可以直接享受記憶體優化。但 API 兼容性需要確認，部分老舊 API 在 Arrow 後端行為略有不同。
- **背後的故事**：CoW 的提案在 GitHub 討論了數年，有大量反對聲音（擔心效能和遷移成本），最終被合入是社群對程式語意正確性的集體共識。這是一個「做對的事比做快的事更重要」的工程決策案例。
- 版本遷移意識：如果你維護舊的 pandas 程式碼，CoW 可能改變某些操作的語意，需要用 `pd.options.mode.copy_on_write = True` 先在 1.x 環境下測試相容性。

**視覺建議：** 兩個並排的程式碼對比框。左框標題「pandas 1.x 的語意地雷」，展示一段使用鏈式賦值的程式碼，下方顯示「修改有效？無效？取決於…」的警告。右框標題「pandas 2.0 Copy-on-Write」，展示正確的寫法（`.loc` 或直接賦值），下方顯示「語意清晰，結果可預期」。底部附一個 Arrow backend 的記憶體節省示意圖（柱狀比較）。

**轉場：** 「pandas 很強，但有邊界。最後一張投影片，我們談工具規模感。」

### B7 — 工具規模感：知道什麼時候換工具，和知道怎麼用工具一樣重要

**核心主張：** pandas 是小到中型資料的最佳選擇，但知道它的邊界、以及邊界之後有什麼工具，是工程師成熟度的標誌。

**講師講解要點：**

- **pandas 的適用範圍**：記憶體內（in-memory）操作，資料集在幾百 MB 到幾 GB 之間（取決於機器記憶體）。速度對大多數分析任務足夠，生態最成熟，教學資源最豐富。在 Python 資料分析的語境下，pandas 是預設選擇，也是你要最熟練的工具。
- **Polars**：用 Rust 撰寫的 DataFrame 庫，API 設計上借鑒 pandas 但更嚴格（沒有 index、強制 lazy evaluation 選項）。在單機上比 pandas 快 5-20 倍是常見的結果，Arrow 原生支援，適合需要高效能但資料仍在單機的場景。2023-2024 年快速崛起。
- **DuckDB**：內嵌式 SQL 分析引擎，可以直接查詢 CSV、Parquet、pandas DataFrame，不需要資料庫伺服器。適合數 GB 到數十 GB 的資料，SQL 介面對 BI 背景的人非常友善，與 pandas 互操作性好。
- **Apache Spark（PySpark）**：分散式計算框架，適合 TB 級以上的資料，需要集群環境。學習成本高，但是資料工程師和大數據工程師的標配工具。PySpark DataFrame API 設計上與 pandas 相似，降低了遷移學習成本。
- **選擇框架**：`資料大小 < 1GB` → pandas；`1GB–100GB 且在單機` → Polars 或 DuckDB；`> 100GB 或需要分散式` → Spark。這不是嚴格規則，而是起點直覺。

**視覺建議：** 一張工具規模尺，水平軸是資料量（從 MB 到 PB），上方標示不同工具的適用範圍（用彩色橫條表示）：pandas（綠，MB 到幾 GB）、Polars/DuckDB（橙，GB 到數十 GB）、Spark（藍，數十 GB 到 PB）。每個工具下方加 1-2 個關鍵特徵描述。最底部加一行：「本課程專注 pandas，但你應該知道邊界在哪裡。」

## 九、常見學習障礙與對應說明

| 障礙 | 常見表現 | 建議說明方式 |
|------|----------|--------------|
| axis 方向混淆 | `sum(axis=0)` 結果 shape 搞錯 | 記憶法：axis 是你要「壓掉」的那個維度 |
| broadcasting 報錯 | 看到 shape mismatch 不知道哪邊錯 | 從右到左對齊兩個 shape，找第一個不相容的維度 |
| index vs column 混淆 | 用 `df[0]` 取第一列失敗 | index 是列的標籤，不是欄名；取列要用 `.iloc[0]` 或 `.loc[label]` |
| boolean indexing 語法錯誤 | 用 `and` 而非 `&` | Python 的 `and` 不支援 Series 的逐元素操作，需要 `&` |
| groupby 結果的 index 問題 | groupby 後 index 變成分組欄位 | 加上 `.reset_index()` 把分組欄位還原為普通欄位 |
| Copy-on-Write 行為 | 修改了 slice 但原始 df 沒變 | 這是 CoW 的正確行為，要修改原始 df 必須直接操作 df |

## 十、與前後模組的連接

**來自 M2 的技能：**  
M2 建立了 list comprehension 和函式定義的基礎。在 M3 中，我們把 list comprehension 的思維「升維」到向量化操作，把函式概念應用在 `apply` 和 `agg` 中。

**通往 M4 的橋樑：**  
M3 的輸出——一個清理後的 DataFrame——正是 M4 資料視覺化的直接輸入。`matplotlib` 和 `seaborn` 都可以直接從 DataFrame 的欄位繪圖，pandas 的 `df.plot()` 是最快的視覺化捷徑。

**通往 M5 的橋樑：**  
M3 中建立的特徵矩陣 X 和標籤向量 y 的概念，是 M5 scikit-learn 前導的直接前置知識。`X = df[feature_cols].to_numpy()` 這一行，就是 pandas 和 scikit-learn 的銜接點。

**通往 ML 的大橋：**  
整個 M3 的核心輸出是「可信任的、型別正確的、缺失值已處理的特徵矩陣」。這就是為什麼 M3 是本課程最長的模組。模型的好壞，80% 取決於你在 M3 做了什麼。
