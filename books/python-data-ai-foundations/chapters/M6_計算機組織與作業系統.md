完成本模組後，學生能夠：

1. 用自己的話解釋 CPU、GPU、RAM、儲存裝置在 AI 訓練流程中各扮演什麼角色

6. 能夠從錯誤訊息（OOM、Permission Denied、FileNotFoundError）反推系統層的根因

### 投影片 3：CPU — 指令執行者

**核心訊息：** CPU 是通用指令執行核心，負責控制流程與邏輯判斷，但核心數有限，不適合大量平行數值運算。

**講師台詞要點：**
- CPU 的強項：執行複雜邏輯、控制流（if/for/函式呼叫）、管理記憶體位址、調度 OS 工作
- CPU 的弱點：核心數有限（4 到 128 核），無法同時執行數千個浮點運算
- 在 AI 訓練中 CPU 做什麼：讀取資料、做前處理、控制訓練迴圈、呼叫 GPU、儲存結果
- 類比：CPU 像一個高度彈性的工程師，什麼都能做，但一次只能同時處理幾件事

**視覺建議：** CPU 晶片示意圖，標出「幾個核心 (4/8/32)」與「控制單元 / ALU / Cache」。旁邊對應 Python 程式碼片段（for 迴圈、if 判斷）。

**過渡語：** 「CPU 把資料拿過來，放在哪裡？RAM。」

### 投影片 4：RAM — 工作桌，不是倉庫

**核心訊息：** RAM 是執行中程式的暫存空間，比磁碟快幾千倍，但容量有限且斷電即失，資料移動有隱性成本。

**講師台詞要點：**
- 速度層級：CPU cache (ns) >> RAM (ns-μs) >> SSD (μs-ms) >> HDD (ms)。讓學生記住量級差，不用背數字
- 為什麼大 DataFrame 危險：一個 float64 DataFrame 100 萬列 × 100 欄 = 約 800 MB；複製一次就再多 800 MB，做幾個 groupby 中間結果疊加很快就超出可用 RAM
- OOM 不是程式 bug，是你放上工作桌的東西太多了
- Python 有 GC（垃圾回收），但不是即時的；del 不代表立刻釋放

**視覺建議：** 工作桌比喻圖：桌面（RAM）、抽屜（Cache）、倉庫（磁碟）。桌面上堆了幾個 DataFrame 物件，標示大小，桌面快要溢出。

**過渡語：** 「RAM 是大家共用的，但 GPU 有自己的記憶體，而且它的算法完全不一樣。」

---

### 投影片 5：GPU — 為什麼深度學習愛它？

**核心訊息：** GPU 有數千個小核心，天生適合矩陣/張量平行運算，這正是深度學習的核心計算結構。

**講師台詞要點：**
- CPU vs GPU 對比：CPU 幾個強核心做複雜邏輯；GPU 幾千個弱核心做同樣的簡單運算（加法、乘法）
- 深度學習的本質是大量矩陣乘法（線性層、卷積、注意力機制），這種「相同運算重複幾百萬次」的模式正好命中 GPU 強項
- PyTorch 的 tensor 設計：在 CPU 上是普通陣列；`.to('cuda')` 把資料搬到 GPU，後續運算在 GPU 上自動平行
- GPU 有自己的 VRAM（視訊記憶體），與 RAM 分開；`CUDA out of memory` 代表 VRAM 滿了，不是 RAM

**視覺建議：** CPU（4 大格）vs GPU（數千小格）的對比圖，旁邊放一個矩陣乘法示意圖，箭頭指向 GPU 的小格子同時被填滿。底部一行 PyTorch 程式碼：`tensor.to('cuda')`。

**過渡語：** 「GPU 很強，但資料還在磁碟上；把資料送上 GPU 的這段路，就是訓練效能的關鍵戰場。」

### 投影片 6：資料移動有成本 — DataLoader 的底層邏輯

**核心訊息：** AI 訓練的速度瓶頸常在資料搬運（disk→RAM→GPU），而非運算本身；DataLoader 的設計就是在解決這個搬運問題。

**講師台詞要點：**
- 訓練迴圈的資料路徑：磁碟 → `DataLoader` 讀取 → CPU 前處理 → `.to('cuda')` 搬到 GPU → 前向傳播 → 反向傳播 → 梯度更新
- `num_workers` 參數：讓多個子 process 同時從磁碟讀資料，讓 GPU 不用等 CPU
- `pin_memory=True`：把 RAM 中的 tensor 鎖定在特定位址，讓 CPU→GPU 的傳輸走更快的 DMA 通道
- batch size 的取捨：太小 → GPU 利用率低；太大 → VRAM 爆掉
- 類比：GPU 是工廠流水線，DataLoader 是物料補給車。流水線再快，補給跟不上就是等待

**視覺建議：** 時間軸圖，X 軸是訓練步驟，Y 軸分成「GPU 計算中」與「GPU 等待資料」，對比 `num_workers=0` vs `num_workers=4` 的時間比例差異。底部標出 PyTorch 官方文件所列三大核心支柱：tensor、autograd、DataLoader。

**過渡語：** 「現在我們對硬體有感覺了。讓我們做一個小練習，把這個流程親手畫一遍。」

### 投影片 7（練習 A）：分析一次訓練的資料搬運路徑

**核心訊息：** 練習把一段 PyTorch 訓練程式碼對應到硬體旅程，強化心智模型。

**練習說明：**

給定以下簡化的 PyTorch 訓練片段，請學生逐行標注「這行程式碼觸發了什麼硬體動作」：

```python
# 練習 A：標注每一行的硬體動作
dataset = ImageFolder('/data/train', transform=transforms)      # (1)
loader  = DataLoader(dataset, batch_size=32, num_workers=4)     # (2)

model   = ResNet18().to('cuda')                                 # (3)

for images, labels in loader:                                   # (4)
    images = images.to('cuda')                                  # (5)
    outputs = model(images)                                     # (6)
    loss = criterion(outputs, labels.to('cuda'))                # (7)
    loss.backward()                                             # (8)
    optimizer.step()                                            # (9)
```

**參考標注答案（講師逐行帶）：**

| 行號 | 硬體動作 |
|------|---------|
| (1) | 建立索引，磁碟 inode 被讀取，但圖片尚未載入 RAM |
| (2) | 建立 4 個子 process，準備從磁碟平行讀取 |
| (3) | 模型權重初始化在 RAM，`.to('cuda')` 複製到 VRAM |
| (4) | 子 process 從磁碟讀 32 張圖，解碼後存入 RAM |
| (5) | 32 張圖從 RAM 複製到 GPU VRAM（PCIe 傳輸） |
| (6) | GPU 執行矩陣運算，結果留在 VRAM |
| (7) | loss 計算在 VRAM；`.to('cuda')` 把 labels 從 RAM 搬到 VRAM |
| (8) | GPU 計算反向梯度，全程在 VRAM |
| (9) | GPU 更新權重，全程在 VRAM |

**時間：** 15 分鐘（獨立思考 8 分鐘 + 講師帶答案 7 分鐘）

### 投影片 8：切換 — 從硬體到作業系統

**核心訊息：** 硬體本身是被動的；是 OS 決定誰在什麼時候用多少硬體資源。AI 工程師遇到的很多「奇怪問題」，根因在 OS 層。

**講師台詞要點：**
- 快速回顧 Part A：硬體是資源，資料搬運有成本，GPU 是平行計算核心
- 引入 Part B 的問題：「硬體是鐵板，但怎麼決定你的 Python 程式可以用多少 RAM？磁碟上哪個路徑是你的？同時跑兩個訓練腳本，CPU 時間怎麼分？」
- 這些問題的答案都是：作業系統
- Red Hat 文件描述 OS 三大核心功能：process scheduling、memory management、file systems — 這三個正是今天 Part B 的主軸

**視覺建議：** 兩層架構圖：上層是你的 Python 程式；下層是 OS（process / memory / file system）；最底層是硬體（CPU / RAM / 磁碟）。OS 層有雙向箭頭，暗示它「翻譯」上層需求，「管理」下層資源。

**過渡語：** 「OS 的第一個工作：管理執行單元。Process 和 Thread，你分得清楚嗎？」

### 投影片 9：Process 與 Thread — 執行的兩種粒度

**核心訊息：** Process 是獨立執行的程式實例（有自己的記憶體空間），Thread 是 process 內部的輕量執行單元（共用記憶體）；Python 的 GIL 讓多執行緒無法真正平行執行 CPU 密集任務。

**講師台詞要點：**
- Process：「你開啟一個 Jupyter kernel，就是一個 process。兩個 Jupyter 不共用變數，因為記憶體空間隔開了」
- Thread：「同一個 process 裡，一個 thread 讀檔、另一個 thread 做計算，共用記憶體但要搶 CPU 時間」
- Python GIL（Global Interpreter Lock）：CPython 在同一時間只讓一個 thread 執行 Python bytecode，所以 CPU 密集任務用 `threading` 不會變快，要用 `multiprocessing`
- DataLoader `num_workers` 用的是 `multiprocessing`，不是 `threading`，這才能真正平行讀檔
- 重要問題：「如果兩個訓練腳本同時跑，它們是兩個 process，OS 要幫它們排程 CPU 時間，所以兩個加起來不會等於兩倍速」

**視覺建議：** 並排對比圖。左側：兩個分離的 process 方塊，各有自己的 RAM 區域。右側：一個 process 方塊內有多個 thread，共用 RAM 區域，但 GIL 像一把鎖圖示標在旁邊。

**過渡語：** 「Thread 共用記憶體。記憶體管理不好，就會看到最常見的 AI 錯誤之一：OOM。」

---

### 投影片 10：記憶體管理 — OOM 為什麼發生？

**核心訊息：** OOM 是 OS 的記憶體分配失敗，有兩個不同戰場（RAM 的 OOM 與 GPU VRAM 的 OOM），診斷方式不同；大型 DataFrame 操作是 RAM OOM 的高發區。

**講師台詞要點：**
- 兩種 OOM 要分清：`MemoryError` 是 CPU RAM 滿了；`CUDA out of memory` 是 GPU VRAM 滿了，解決方式不同
- RAM OOM 高發場景：
  - `pd.concat` 多個大 DataFrame（每次建立新物件）
  - `.copy()` 大表格（pandas 預設某些操作會觸發隱性複製）
  - 訓練資料一次全部載入不分 batch
- VRAM OOM 高發場景：batch size 太大；模型太大；中間梯度沒清除（忘了 `optimizer.zero_grad()`）
- Python GC 不即時：即使 `del df`，記憶體不一定馬上釋放；`gc.collect()` 可以強制觸發，但不保證
- 診斷工具：`htop`（CPU RAM 使用）、`nvidia-smi`（GPU VRAM 使用）

**視覺建議：** 雙欄對照表。左欄：RAM OOM（原因 + 症狀 + 解法）。右欄：VRAM OOM（原因 + 症狀 + 解法）。底部加一行 `nvidia-smi` 輸出截圖，標示 VRAM 使用量欄位。

**過渡語：** 「記憶體問題解決了，下一個常見的卡關：路徑問題。File system 不只是「放檔案的地方」。」

---

### 投影片 11：File System — 路徑不只是字串

**核心訊息：** File system 定義了檔案的組織方式、路徑的解析規則、掛載點與權限，理解它才能在不同環境（本機/伺服器/Docker）間無縫切換。

**講師台詞要點：**
- 路徑的本質：不是「字串」，是 OS 對 inode（真實儲存位址）的一層命名映射；同一個檔案可以有多個路徑（hard link）
- 絕對路徑 vs 相對路徑：`/data/train/images` vs `./train/images`。在伺服器上腳本的「當前目錄」可能跟你想的不一樣
- 掛載點（mount point）：在 Linux 上，外部硬碟、網路儲存（NFS/S3）、Docker volume 都是「掛載」進檔案系統樹的，路徑形式和本地路徑一樣，但底下是不同裝置
- 權限三要素：owner / group / others × read / write / execute（rwxrwxrwx）；`Permission denied` 的根因
- Python 實作建議：用 `pathlib.Path` 而非字串拼接；用 `Path.exists()` 做防禦性檢查；部署到 Linux 時用 `/` 分隔符不用 `\`

**視覺建議：** 樹狀 file system 圖：`/` 根目錄展開，標出 `/home`、`/data`（掛載標示）、`/tmp`。旁邊一個 Python 程式碼片段，用 `pathlib.Path` 操作路徑並做 `exists()` 檢查。

**過渡語：** 「路徑搞清楚了，但還有一個更難排查的問題：在我電腦上可以跑，到伺服器就掛了。」

---

### 投影片 12：環境差異 — 為什麼「在我機器上沒問題」？

**核心訊息：** 「在我機器上可以」是 AI 工程師的高頻災難，根因在 OS 差異、權限模型、套件版本與環境變數；Docker 和虛擬環境的本質是複製執行環境。

**講師台詞要點：**
- 四大根因：
  1. OS 差異（Windows `\` vs Linux `/`；Windows 沒有 `fork()`，`multiprocessing` 行為不同）
  2. 權限差異（本機 admin，伺服器普通用戶；`/data` 可能需要 sudo 才能讀）
  3. 套件版本差異（本機 `torch==2.1`，伺服器 `torch==1.9`，API 不同）
  4. 環境變數差異（`CUDA_VISIBLE_DEVICES`、`DATA_PATH`、`HOME`）
- 解法方向：
  - `venv` / `conda`：隔離 Python 套件版本（M5 有講，這裡快速複習）
  - Docker：把 OS + 套件 + 環境變數一起打包，做到可複製的執行環境
  - `os.environ.get()` 讀環境變數，不要硬編碼路徑
- 為什麼 Linux/Docker 在 AI 工程師的工作流中幾乎不可避免：雲端 GPU 伺服器、MLOps pipeline、模型部署都以 Linux container 為標準

**視覺建議：** 左右對比圖。左側：「本機」（Windows，admin 權限，PyTorch 2.1，硬編碼路徑）。右側：「伺服器」（Linux，普通用戶，PyTorch 1.9，不同路徑）。中間一個「崩潰」圖示。底部：Docker 把整個執行環境框起來，標示「可複製」。

### 投影片 13（練習 B）：診斷一個 OOM 場景

**核心訊息：** 練習從錯誤訊息出發，逆向推理系統層根因，並提出解法。

**練習說明：**

以下是一段程式碼與對應的錯誤訊息，請學生回答三個問題：

```python
# 場景：你正在訓練一個圖像分類模型
import torch
from torch.utils.data import DataLoader
from torchvision.datasets import ImageFolder
import torchvision.transforms as transforms

dataset = ImageFolder('/data/train', transform=transforms.ToTensor())
loader  = DataLoader(dataset, batch_size=512)   # batch size 設很大

model = MyModel().to('cuda')

for images, labels in loader:
    images = images.to('cuda')
    outputs = model(images)
    loss = criterion(outputs, labels.to('cuda'))
    loss.backward()
    # 忘了加 optimizer.zero_grad()
    optimizer.step()
```

**錯誤訊息：** `RuntimeError: CUDA out of memory. Tried to allocate 2.50 GiB`

**診斷問題（小組討論，5 分鐘）：**

1. 這個 OOM 發生在 RAM 還是 VRAM？怎麼判斷？
2. 列出至少兩個可能導致這個錯誤的原因（從程式碼中找）
3. 如果你是工程師，你會先改哪一行？為什麼？

**參考答案（講師帶）：**

| 問題 | 答案 |
|------|------|
| OOM 位置 | VRAM（`CUDA out of memory` 是 GPU 記憶體，不是 RAM） |
| 可能原因 1 | `batch_size=512` 太大，一次搬太多張圖到 VRAM |
| 可能原因 2 | 缺少 `optimizer.zero_grad()`，梯度在每個 step 疊加累積，佔用越來越多 VRAM |
| 優先修改 | 先加 `optimizer.zero_grad()`（這是 bug）；再降低 `batch_size` |

**時間：** 15 分鐘（小組討論 8 分鐘 + 講師帶答案 7 分鐘）

**核心訊息：** 硬體與 OS 不是「底層細節」，而是 AI 應用效能與穩定性的決定因素；系統視角讓你從「能跑」進化到「能上線」。

**講師台詞要點：**
- 快速回顧：用六個問題把今天的概念串成一條線
  1. 「為什麼訓練吃 GPU？」→ 因為深度學習本質是矩陣乘法，GPU 平行核心天生適合
  2. 「為什麼大 batch 導致 OOM？」→ 因為 VRAM 是有限的，batch 越大，單次佔用越多
  3. 「為什麼檔案讀取慢會拖垮訓練？」→ 磁碟 → RAM → GPU 的搬運鏈，哪段慢，GPU 就在等
  4. 「為什麼數值陣列比 Python list 快？」→ NumPy array 是連續記憶體，CPU cache 命中率高；Python list 是指標陣列，每次都要跳記憶體
  5. 「為什麼 DataLoader 的 num_workers 能加速？」→ 多 process 同時從磁碟讀取，讓 GPU 不用等 CPU
  6. 「為什麼在伺服器上跑不動？」→ OS 差異、權限、環境變數、套件版本

- 重申核心金句：**"Systems turn local scripts into scalable applications."**

### 常見學生困惑點

1. **「GPU 記憶體和 RAM 是什麼關係？」** → 強調它們是物理上分離的，CPU 程式看不到 GPU VRAM，反之亦然；`.to('cuda')` 是實際的資料複製，不是「指向」
2. **「Python GIL 那 multiprocessing 是怎麼繞過的？」** → 多個 process 各自有獨立的 Python 直譯器，各自有一把 GIL，彼此不衝突
3. **「Docker 學起來很複雜，一定要學嗎？」** → 這堂課只要理解「它的目的是複製執行環境」，不要求學生現在學 Docker 操作；M7 之後在 MLOps 路線才會深入
