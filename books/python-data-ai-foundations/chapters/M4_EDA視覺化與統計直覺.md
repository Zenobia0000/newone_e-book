## 課程定位

**所屬課程**：2026 Python 數據分析與 AI 工程基礎：從資料思維到系統底盤（24 小時）
**模組編號**：Module 4／共 8 個模組

**模組核心主張**：
圖表是思考工具，不是裝飾品；統計是判斷工具，不是公式集。
EDA 的終點不是一張漂亮圖，而是一個能被驗證的業務假設。

> "Statistics turns observations into confidence."

完成本模組後，學員能夠：

1. 說明四種圖表任務類型（分佈、比較、關聯、時序），並為給定問題選擇正確圖形

5. 解釋假設檢定的邏輯：「這個差異可靠嗎？」而不是「這個差異存在嗎？」
6. 將一個業務問題翻譯成可被統計回答的統計問題

### Slide 1：圖表不是裝飾，是思考的外掛

**核心訊息**：圖表不是用來讓報告看起來專業，而是用來幫你發現資料裡你肉眼看不到的結構。

**講師講解要點**：
- 大多數人學視覺化的動機是「讓報告好看」，但這是本末倒置——好圖的標準是「它讓你發現了什麼」
- 試著用純數字表回答「這兩組的差異大嗎？」，再換成圖，感受資訊密度的差異
- Anscombe's Quartet：四組統計數字完全相同的資料集，但一畫出來結構完全不同，這就是圖表的力量
- EDA 中的圖是給自己看的草稿，不是給老闆看的成品——探索圖和報告圖是兩件事
- 重要觀念：圖表的目的是「問問題」，而不是「給答案」

**視覺建議**：
左右對比圖——左側是一張密密麻麻的數字表格，右側是同一份資料的散佈圖，散佈圖中明顯可見離群點與趨勢。底部標注：「哪一個讓你更快看出問題？」

**銜接下一張**：
「既然圖表是思考工具，那思考什麼問題就需要什麼圖。接下來我們看四種最常見的圖表任務。」

### Slide 2：四種圖表任務類型

**核心訊息**：選圖的起點是問題，不是套件。先問「我想知道什麼」，再選工具。

**講師講解要點**：
- **分佈圖**（Distribution）：這筆數值的形狀是什麼？集中還是分散？有沒有雙峰？工具：histogram, KDE, boxplot
- **比較圖**（Comparison）：A 和 B 哪個比較大？有沒有統計上的差異？工具：bar chart, grouped boxplot
- **關聯圖**（Relationship）：X 變動時，Y 怎麼變？有沒有相關性？工具：scatter plot, heatmap, pairplot
- **時序圖**（Time Series）：這個指標隨時間如何演變？有沒有季節性或趨勢？工具：line chart, area chart
- 常見錯誤示範：想問「A 和 B 哪個比較大」卻畫了 pie chart，問題和圖型不匹配導致洞察失焦

**視覺建議**：
2x2 卡片格，每格一個任務類型，包含任務名稱、代表問題範例、縮圖示意，以及適合的圖形工具列表。使用一致的色彩系統區分四種類型。

**銜接下一張**：
「知道有哪些圖，但還不夠——更根本的問題是：什麼樣的資料問題值得畫圖？」

### Slide 3：什麼是好的資料問題？

**核心訊息**：圖來自問題，不是來自資料集。沒有問題，圖只是雜訊。

**講師講解要點**：
- 壞問題範例：「我想看看這份資料長什麼樣。」→ 沒有焦點，會畫出 20 張沒有結論的圖
- 好問題的三個條件：**可觀察**（能在資料裡找到對應的欄位）、**可比較**（有比較基準）、**可驗證**（結果能幫助做決定）
- 好問題範例：「新用戶的首週購買金額，是否比老用戶低？」→ 可觀察（有購買金額欄位）、可比較（新 vs 老）、可驗證（結果影響行銷策略）
- 問題從哪裡來？業務痛點、異常警示、目標對齊——不是「看到什麼欄位就問什麼問題」
- 練習思維轉換：給定一個情境，學員要先寫下問題，再決定要畫哪種圖

**視覺建議**：
三欄對比表：左欄「壞問題」（模糊、無方向），中欄「好問題三要素」（可觀察、可比較、可驗證），右欄「好問題範例」。底部加一個流程箭頭：業務情境 → 問題 → 圖表任務類型 → 選圖工具。

**銜接下一張**：
「有了問題，怎麼系統性地探索一份新資料？我們有一條標準路徑。」

### Slide 4：EDA 標準路徑

**核心訊息**：EDA 不是隨機亂看，而是從全貌到細節、從異常到關係、最後收斂到假設的結構化過程。

**講師講解要點**：
- **第一步：看全貌** — `df.info()`, `df.describe()`, 缺值比例，資料型態是否正確。目的是建立資料的「地圖」
- **第二步：抓異常** — 極端值在哪裡？分佈是否有意外的峰值？這些異常是資料品質問題還是業務訊號？
- **第三步：找關係** — 哪些欄位之間有相關性？是正相關還是負相關？有沒有看起來隨機其實有規律的群集？
- **第四步：形成假設** — 基於以上發現，提出可以被後續分析或實驗驗證的假設。EDA 的輸出是假設，不是結論
- 重要提醒：EDA 過程是迭代的——發現關係之後可能回頭重新看異常，這很正常

**視覺建議**：
垂直四步流程圖，每步驟有：步驟名稱、主要工具/方法、代表產出物（圖或表的縮圖）、一個「此步常見錯誤」的警示框。箭頭連接各步，並有一個回頭迭代的虛線箭頭強調非線性。

**銜接下一張**：
「走完這四步，我們手上有了一些發現。接下來的問題是：怎麼把這些發現變成一個有說服力的故事？」

### Slide 5：如何用一頁圖表講業務故事

**核心訊息**：圖表本身不說話，說話的是圖表背後的詮釋結構：觀察 → 洞察 → 建議。

**講師講解要點**：
- 大多數分析師的圖表只停在「觀察」層：「這個月銷售額下降了 15%。」這是描述，不是洞察
- **觀察（Observation）**：資料告訴我們什麼事實？客觀陳述，有數字支撐
- **洞察（Insight）**：為什麼會這樣？背後的機制或模式是什麼？這需要跨欄位比較或領域知識
- **建議（Recommendation）**：基於這個洞察，我們應該做什麼？這讓分析有商業價值
- 一頁紙的力量：練習把三段文字 + 一張主圖 + 一個標題句壓縮在同一頁，強迫自己聚焦

**視覺建議**：
示範一頁「業務故事圖」樣板，包含：大標題（洞察句，不是「銷售分析」），主圖（標注關鍵數據點），三段文字框（觀察、洞察、建議），底部標注資料來源與時間範圍。用紅色虛線框標示每個區域的用途。

**銜接下一張**：
「理論說完了。現在輪到你，用真實資料做一次完整的 EDA 故事簡報。」

### Slide 6：【練習 A】三段洞察簡報

**核心訊息**：EDA 的能力只能從實作中建立，不能從聽講中建立。

**練習說明**：

**資料集**：零售電商訂單資料（提供清洗後的 CSV，約 5000 筆）

**欄位包含**：訂單日期、用戶ID、商品分類、購買金額、是否首購、城市

**任務說明**：
1. 依照 EDA 四步路徑探索資料（看全貌 → 抓異常 → 找關係 → 形成假設）
2. 從探索結果中，選出三個你認為最有業務價值的發現
3. 為每個發現，產出一張圖 + 一段「觀察 → 洞察 → 建議」文字
4. 最終輸出：三頁業務故事簡報（Markdown + 圖片）

**評估標準**：
- 圖表選型是否符合問題類型
- 洞察層是否超越觀察層（有解釋，不只是描述）
- 建議是否具體且可行動

**時間**：30 分鐘實作，5 分鐘小組分享

**講師引導要點**：
- 提醒學員先寫問題再畫圖，不要打開資料就開始畫
- 引導觀察：「你畫這張圖想問什麼問題？這張圖有沒有回答到你的問題？」
- 常見卡關點：沒有洞察只有觀察 → 追問「為什麼會這樣？」

### Slide 7：為什麼分析需要統計？

**核心訊息**：沒有統計，你只是在描述現象，不是在判斷現象。描述說「有差異」，統計說「這個差異可靠嗎」。

**講師講解要點**：
- 情境題：你做了一個新的 App 引導流程，A 版轉換率 12%，B 版轉換率 14%。B 版比較好嗎？
- 純描述的回答：「B 版高了 2 個百分點。」這是觀察，不是判斷
- 統計的回答：「在樣本數 n=500 的情況下，這 2% 的差異在 95% 信賴水準下無法排除是隨機誤差造成的。」這是判斷
- 統計的本質是：在資料永遠不完整的情況下，用嚴謹的方式量化我們對判斷的信心
- 不用統計的代價：做了很多「看起來有效」但其實只是噪聲的決策，浪費資源

**視覺建議**：
左右對比：左側「純描述的世界」——一張柱狀圖，A=12%, B=14%，下方標注「看起來 B 贏了」。右側「有統計的世界」——同一張圖，加上誤差棒（error bar），誤差棒重疊，下方標注「差異在信心範圍內無法確認」。

**銜接下一張**：
「要理解統計的判斷力，必須先建立幾個核心概念的直覺：母體、樣本、均值、變異數。」

### Slide 8：母體、樣本、均值、變異數

**核心訊息**：統計不是計算器，是不確定性的語言。每個數字背後都有一個關於「我們能知道多少」的假設。

**講師講解要點**：
- **母體（Population）**：你真正想了解的全集——所有用戶、所有訂單、所有未來可能的實驗結果。我們幾乎永遠無法取得完整母體
- **樣本（Sample）**：我們實際拿到的那一部分。分析的對象是樣本，但判斷的目標是母體
- **均值（Mean）**：樣本的「重心」。它告訴你中心在哪裡，但不告訴你資料有多分散
- **變異數（Variance）**：資料離重心的平均距離的平方。它告訴你均值有多「代表性」——變異數大，均值就不可靠
- 直覺類比：均值是說「平均工資 10 萬」，變異數是說「有人 3 萬有人 50 萬」——同樣的均值，變異數不同，意義完全不同

**視覺建議**：
四個方塊的概念圖：母體（大圓，虛線邊界，「未知全集」）、樣本（小圓，實線，「可觀察部分」）、均值（圓點標在分佈中心）、變異數（雙箭頭標示資料散佈範圍）。旁邊加一個反例：「平均薪資的陷阱」——兩個薪資分佈，均值相同但形狀完全不同。

**銜接下一張**：
「變異數帶我們進入分佈的概念——為什麼看分佈比看均值更接近真相？」

### Slide 9：分佈與抽樣直覺

**核心訊息**：均值是一個點，分佈是完整的故事。抽樣讓我們從故事片段推斷整本書。

**講師講解要點**：
- 為什麼看分佈？均值可以被少數極端值拉偏；分佈讓你看到「大多數人在哪裡」而不只是「中心在哪裡」
- **常態分佈（Normal Distribution）**：為什麼它無所不在？中央極限定理的直覺版本：大量獨立隨機事件相加，結果趨向常態。身高、測量誤差、考試分數都有這個特性
- **偏態分佈**：收入、房價、用戶停留時間——多數人集中在低端，少數極高值拉高均值。這種資料用均值會嚴重誤導
- **抽樣的直覺**：從同一個母體多次抽樣，每次的樣本均值都不一樣——這個「均值的分佈」就是抽樣分佈，是假設檢定的基礎
- 關鍵問題：「我觀察到的這個均值，有多大可能只是抽樣運氣造成的？」

**視覺建議**：
三欄圖解：第一欄常態分佈示意（對稱鐘形，標注均值=中位數=眾數）；第二欄右偏分佈（收入分佈，標注均值被拉向右側，而中位數更能代表「一般人」）；第三欄抽樣示意（從同一個大圓多次抽小圓，每次的 x-bar 不同，這些 x-bar 形成一個新的分佈）。

**銜接下一張**：
「抽樣帶來不確定性。假設檢定就是把這個不確定性量化、然後用它做決定的工具。」

### Slide 10：假設檢定在做什麼？

**核心訊息**：假設檢定問的不是「有沒有差異」，而是「這個差異是不是可靠到足以讓我行動」。

**講師講解要點**：
- **H0（虛無假設）**：預設立場——「沒有差異，你看到的差異只是抽樣噪聲」。統計的保守主義：除非證據夠強，否則維持 H0
- **H1（對立假設）**：你想證明的事——「有差異，而且這個差異是真實的」
- **p-value**：如果 H0 是真的，你觀察到這麼大（或更大）差異的機率。p=0.03 意思是：「如果根本沒有差異，我有 3% 的機率看到這麼大的差異。」
- **顯著水準（α）**：你設定的容忍門檻，通常是 0.05。p < α → 拒絕 H0 → 接受有差異的結論
- 常見誤解：p < 0.05 不代表差異很大，只代表差異是真實的。差異大不大是 effect size 的問題

**視覺建議**：
流程圖：情境描述 → 設定 H0/H1 → 計算統計量 → 計算 p-value → 與 α 比較 → 做決定。旁邊加一個「法庭類比」：H0 = 無罪推定，p-value = 反對無罪的證據強度，α = 定罪門檻。

**銜接下一張**：
「假設檢定在業務中最常見的應用場景就是 A/B Test。讓我們從資料分析師的角度看它的本質。」

### Slide 11：A/B Test 的統計本質

**核心訊息**：A/B Test 是用統計把產品決策從「感覺上比較好」變成「數據上可以確認更好」。

**講師講解要點**：
- A/B Test 的商業動機：每天有無數個「我覺得這個改版會更好」的直覺，統計讓我們用數據驗證而不是用猜測下注
- **從資料分析師的角度**，A/B Test 的本質是：在隨機分組的條件下，測量兩組指標差異，並用假設檢定判斷差異是否超過隨機噪聲
- **實驗設計的關鍵**：樣本數夠嗎？分組真的隨機嗎？測試時間夠長嗎（避免時間偏誤）？
- **統計顯著 vs 業務顯著**：轉換率從 10% 提升到 10.1%，統計上顯著（因為樣本數百萬），但 0.1% 的提升值得為它改版嗎？這是業務判斷，不是統計判斷
- A/B Test 的局限：它能告訴你「哪個比較好」，但不能告訴你「為什麼比較好」——那需要定性研究

**視覺建議**：
A/B Test 全流程圖：用戶流量 → 隨機分組（50/50）→ A 組（舊版）、B 組（新版）→ 收集指標 → 假設檢定 → 決策。重點標注三個判斷點：分組是否隨機、樣本數是否足夠（統計力 power）、差異是否具業務意義。底部加一個「決策矩陣」：統計顯著 + 業務顯著 → 上線；統計顯著 + 業務不顯著 → 再評估；統計不顯著 → 延長測試或放棄。

**銜接下一張**：
「現在輪到你把抽象的統計概念和具體的業務情境接起來。」

### Slide 12：【練習 B】業務問題翻譯機

**核心訊息**：統計的實用價值在於，它能讓你把「我覺得」的判斷換成「資料顯示，在 X% 信心水準下」的判斷。

**練習說明**：

**任務**：將以下三個業務問題，翻譯成可被統計回答的統計問題，並指出需要哪些資料、用什麼方法

| # | 業務問題 | 翻譯目標 |
|---|---------|---------|
| 1 | 「新的 App 推播設計讓用戶更願意打開嗎？」 | 翻譯成 A/B Test 假設檢定設計 |
| 2 | 「週末的平均訂單金額真的比平日高嗎？」 | 翻譯成雙樣本 t 檢定設計 |
| 3 | 「我們的用戶留存率有沒有在最近三個月改善？」 | 翻譯成時序比較 + 信賴區間設計 |

**翻譯格式模板**：
```
H0（虛無假設）：___
H1（對立假設）：___
需要的資料：___
比較的指標：___
判斷方法：___
樣本數考量：___
```

**時間**：25 分鐘個人實作，10 分鐘講師講解標準答案

**講師引導要點**：
- 重點不是答案正確，而是能說清楚「H0 是什麼」和「需要什麼資料來拒絕它」
- 常見錯誤：H0 和 H1 寫反了（把想證明的事寫成 H0）
- 延伸討論：同一個業務問題，不同的翻譯方式會導致不同的實驗設計，沒有唯一正確答案

## 模組總結

### 本模組的兩條主線

**視覺化主線**：圖表是思考工具 → 問題決定圖形選擇 → EDA 是結構化探索過程 → 洞察需要翻譯成業務語言

**統計主線**：描述不等於判斷 → 不確定性需要被量化 → 分佈比均值更完整 → 假設檢定是可靠差異的守門人

### 兩條主線的交匯點

EDA 的終點是形成假設；統計是驗證假設的工具。好的資料分析師，是能在這兩條主線之間流暢切換的人——用圖表發現問題，用統計確認問題，用業務語言解釋問題。

### 下一步：M5 特徵工程

本模組建立的統計直覺（分佈、相關性、異常值）將直接應用在 M5 的特徵工程——我們需要理解資料的統計性質，才能知道什麼樣的特徵轉換對模型有幫助。
