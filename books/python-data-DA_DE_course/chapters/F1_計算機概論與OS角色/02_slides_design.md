## F4 · CONCEPT-CARD — 廚房三角色（類比主軸）

- 🖼️ 畫面：左欄文字三角色定義 + 右欄廚房示意圖（真圖佔位）
- 📣 畫面上的字：
  - CPU — 廚師（真正做事、越多核＝越多廚師）
  - RAM — 工作檯（料要放檯面上才能切、檯面決定一次做多少菜）
  - Storage — 倉庫（容量大但遠、廚師看不到倉庫）
  - 記憶點：料不搬到工作檯，廚師碰不到 → 這就是「讀檔」在做的事
- 🎙️ 講者這時說：「我不要你背 CPU、RAM、Storage 這三個字——我要你記住一個廚房：廚師、工作檯、倉庫。廚師是真正做事的人；工作檯是他切菜的地方；倉庫在遠處放著所有原料。廚師看不到倉庫，要料就得叫人搬上來。這就是你電腦每一秒都在發生的事。」

## F5 · EXAMPLE-I/O — read_csv 的七步旅程

- 🖼️ 畫面：三欄 Input | Process | Output，中間箭頭
- 📣 畫面上的字：
  - Input：`df = pd.read_csv('sales.csv')`
  - Process：Python 直譯器 → OS System Call → 檔案系統 → I/O 控制器 → RAM
  - Output：DataFrame 躺在 RAM 裡，CPU 才能開始做菜
- 🎙️ 講者這時說：「你只寫了一行 read_csv，可是背後 OS 幫你跑了七步。為什麼要給你看這張？因為等下你看到任何 I/O 操作都該想起這七步——每一步都可能是瓶頸。」

## F9 · CONCEPT-CARD — RAM 工作檯 vs Storage 倉庫

- 🖼️ 畫面：左右對比 vs_two_col + 中間 1000× delta 徽章
- 📣 畫面上的字：RAM 奈秒級、Storage 毫秒級、差距 1000×
- 🎙️ 講者這時說：「為什麼一定要先搬 RAM？因為 CPU 根本看不到硬碟。硬碟距離 CPU 的距離，如果工作檯是你面前的桌子，倉庫就在隔壁棟三樓。差一千倍。這解釋了為什麼 read_csv 會吃滿 RAM——它在做搬家。」

## F10 · PITFALL (P1) — OOM

- 🖼️ 畫面：左紅錯 / 右綠對 / 底部 why 綠條 / 最底銜接 S2 提示
- 📣 畫面上的字：錯：`pd.read_csv('big_5gb.csv')` MemoryError；對：`chunksize=100_000` + `dtype={'id':'int32'}`；→ 銜接 S2
- 🎙️ 講者這時說：「這是你上工第一週一定會踩的坑。五 GB CSV 直接讀，系統會先沉默十分鐘，然後你要嘛 OOM、要嘛電腦當機到你硬按 power。秘密是那十分鐘系統在觸發 Swap——拿硬碟假裝 RAM，慢一千倍。S2 會教你怎麼分批讀，我先種下這顆種子。」

## F11 · CONCEPT-CARD — I/O 為什麼慢 100×

- 🖼️ 畫面：左右對比 ns vs ms + 100-1000× delta
- 📣 畫面上的字：CPU 1 ns 一次加法 · SSD 0.1 ms 一次讀檔 · 類比：廚師切菜 1 秒 vs 倉庫叫貨 5 分鐘
- 🎙️ 講者這時說：「這張要記一輩子。算加法、算乘法，CPU 一秒幾十億次——快到你感覺不到。可是讀檔一秒只能幾萬次，慢到你會等。所以你的 Pandas 卡住，九成是在等讀檔不是在等計算。S1 我們會講為什麼『計算』那一層也可以被 Python 拖慢——但本質還是這張圖。」

## F13 · PITFALL (P2) — 路徑跨平台

## F14 · PRACTICE — 估算你自己的筆電

- 🖼️ 畫面：白底四步驟列點 / 底部銜接 S2
- 📣 畫面上的字：①RAM 幾 GB ②安全額度=RAM×0.3 ③一筆=欄數×8B ④安全筆數=額度÷筆大小；例：16GB × 50 欄 ≈ 1200 萬列

## F15 · CONCEPT-CARD — I/O vs CPU Bound

- 🖼️ 畫面：左右對比 vs_two_col
- 📣 畫面上的字：I/O Bound → async/threading · CPU Bound → 向量化/multiprocessing；判斷法：看工作管理員
- 🎙️ 講者這時說：「選錯武器就是白工。判斷法很簡單：打開工作管理員，CPU 飆滿就是 CPU Bound，網路或硬碟燈狂閃就是 I/O Bound。兩種的加速武器完全不同，別用錯。」
