## 2. 考題類型分析

### 類型一：概念辨析（約 25%）
No Code vs Low Code 差異、ChatGPT vs API、工具定位。
- 典型形式：「以下關於 No Code 的描述何者正確？」
- 破解：記住核心定義差異，特別是 No Code = 零程式碼 vs Low Code = 少量程式碼。

### 類型二：工具配對（約 30%）
給場景選工具、給工具選場景。
- 典型形式：「某公司想要...，最適合使用以下哪個工具？」
- 破解：用四象限地圖（文字/圖片/程式/流程）先定類別，再選具體工具。

### 類型三：提示工程判斷（約 20%）
辨認提示技巧（Zero-shot/Few-shot/CoT）、判斷提示好壞。
- 典型形式：「以下哪段提示使用了 Few-shot 技巧？」
- 破解：有範例 = Few-shot、要求推理 = CoT、直接問 = Zero-shot。

### 類型四：RAG 相關（約 15%）
RAG 流程、RAG vs 微調選型。
- 典型形式：「以下何者最適合使用 RAG？」
- 破解：知識會更新 → RAG。行為要改變 → 微調。

### 類型五：限制與風險（約 10%）
GenAI 工具的幻覺、偏見、隱私風險。
- 典型形式：「使用 GenAI 工具時應注意以下哪項風險？」
- 破解：幻覺（亂編）、偏見（訓練資料偏差）、隱私（機密資料洩露）三大風險。

## 3. 模擬試題（10 道）

### 題目 1 — No Code vs Low Code（L121-3）

**關於 No Code 與 Low Code 開發方式，以下敘述何者正確？**

(A) No Code 和 Low Code 的差異僅在於使用介面不同，技術本質完全相同
(B) No Code 平台完全不需要撰寫任何程式碼，Low Code 平台以視覺化為主但可搭配少量程式碼
(C) Low Code 平台完全不需要撰寫程式碼，比 No Code 更適合非技術人員
(D) No Code 的客製化能力通常高於 Low Code

**答案：B**

**詳解**：
- (A) 兩者不只是介面不同，核心差異在於是否需要寫程式碼，以及目標用戶不同。
- (B) 正確。No Code = 零程式碼（拖拉完成），Low Code = 少量程式碼（視覺化為主+程式碼擴充）。
- (C) 方向反了。Low Code 需要少量程式碼，No Code 才是完全不需要。No Code 才更適合非技術人員。
- (D) 反了。Low Code 因為可以寫少量程式碼，客製化能力通常高於 No Code。

**考點**：No Code = 零程式碼 vs Low Code = 少量程式碼。這是最基本也最高頻的考點。

### 題目 2 — 工具定位（L122-3）

**關於 Microsoft Copilot Studio，以下敘述何者最正確？**

(A) Copilot Studio 是一個 IDE 外掛，用來輔助工程師撰寫程式碼
(B) Copilot Studio 是一個 Low Code 平台，用來建立聊天機器人和工作流程
(C) Copilot Studio 是 ChatGPT 的微軟版本，功能完全相同
(D) Copilot Studio 是一個圖片生成工具，類似 Midjourney

**答案：B**

**詳解**：
- (A) 描述的是 GitHub Copilot，不是 Copilot Studio。
- (B) 正確。Copilot Studio 是微軟的 Low Code 平台，主要用途是建立客製化聊天機器人和工作流程自動化。
- (C) Copilot Studio 和 ChatGPT 定位完全不同。ChatGPT 是通用對話 AI，Copilot Studio 是流程自動化平台。
- (D) Copilot Studio 不做圖片生成。

**考點**：Copilot 家族辨別——GitHub Copilot 寫程式 vs Copilot Studio 做流程。

### 題目 3 — ChatGPT vs API（L122-1）

**某新創公司想在自己的客戶服務 App 中嵌入 AI 問答功能。以下哪種方式最合適？**

(A) 直接在 App 中嵌入 ChatGPT 的網頁連結
(B) 使用 OpenAI API 進行程式整合
(C) 安裝 GitHub Copilot
(D) 使用 Midjourney

**答案：B**

**詳解**：
- (A) 嵌入網頁連結不是真正的整合，用戶體驗差且無法客製化。
- (B) 正確。OpenAI API 是面向開發者的程式介面，可以嵌入自己的產品中，控制 system prompt、參數等。
- (C) GitHub Copilot 是程式碼自動補全工具，不是用來建客服功能的。
- (D) Midjourney 是圖片生成工具，與客服問答無關。

**考點**：ChatGPT = 終端用戶產品 vs OpenAI API = 開發者程式介面。「嵌入自己的產品」= 需要 API。

### 題目 4 — 提示工程技巧辨認（L122-5）

**以下提示使用了哪種提示工程技巧？**

```
請參考以下範例進行翻譯：
範例 1：I love programming → 我喜歡寫程式
範例 2：The weather is nice → 今天天氣真好
現在請翻譯：Machine learning is fascinating
```

(A) Zero-shot
(B) Few-shot
(C) Chain-of-Thought
(D) RAG

**答案：B**

**詳解**：
- (A) Zero-shot 是不給範例直接問，但此提示給了兩個範例。
- (B) 正確。提供了 2 個輸入→輸出的範例，讓 AI 學習翻譯的風格和格式，然後處理新的輸入。這就是 Few-shot。
- (C) Chain-of-Thought 是要求 AI 一步步推理，此提示沒有要求推理過程。
- (D) RAG 是從知識庫檢索文件來增強生成，與提示技巧不同。

**考點**：有範例 = Few-shot。這是最直覺的判斷方式。

### 題目 5 — 提示工程好壞判斷（L122-6）

**以下哪個提示的設計品質最佳？**

(A)「幫我寫一封信」
(B)「寫一封專業的信，要好一點」
(C)「你是一位人資經理，請撰寫一封 200 字的錄取通知信，語氣正式友善，需包含到職日期和薪資資訊」
(D)「快點幫我寫信，越長越好」

**答案：C**

**詳解**：
- (A) 太簡略，沒有指定角色、格式、任何具體要求。
- (B) 略好但「專業」和「好一點」太模糊，AI 無法判斷標準。
- (C) 正確。包含 RTFC 四要素：角色（人資經理）、任務（撰寫錄取通知信）、格式（200 字+正式友善語氣）、限制（需包含到職日期和薪資）。
- (D) 「越長越好」是反指標，好提示追求精準不是字數。

**考點**：好提示 = 有 RTFC 結構（角色/任務/格式/限制），不是字數多。

### 題目 6 — RAG 原理（L122-7）

**關於 RAG（Retrieval-Augmented Generation）技術，以下敘述何者正確？**

(A) RAG 是一種全新的大型語言模型
(B) RAG 透過修改 LLM 的模型權重來增加新知識
(C) RAG 在生成回答前先從外部知識庫檢索相關資訊，以提高回答的準確性
(D) RAG 技術不需要任何外部資料來源

**答案：C**

**詳解**：
- (A) RAG 不是模型，是一種架構模式（方法），結合檢索和生成。
- (B) 修改模型權重是微調（Fine-tuning）的做法，RAG 不改模型。
- (C) 正確。RAG 的流程：先從知識庫檢索相關文件 → 把文件和問題結合 → 餵給 LLM 生成回答。
- (D) RAG 的核心就是需要外部知識庫作為檢索來源。

**考點**：RAG = 架構模式（不是模型）+ 不改模型權重 + 需要外部知識庫。

### 題目 7 — RAG vs 微調（L122-8）

**某律師事務所希望建立一個 AI 系統，能即時回答關於最新法規條文的問題（法規每月更新）。最適合的技術方案是？**

(A) 使用通用的 ChatGPT 直接回答
(B) 對 LLM 進行微調（Fine-tuning），將所有法規寫入模型
(C) 使用 RAG，建立法規條文知識庫並即時檢索
(D) 不使用 AI，改用關鍵字搜尋

**答案：C**

**詳解**：
- (A) 通用 ChatGPT 有知識截止日問題，無法回答最新法規。且可能產生幻覺。
- (B) 微調可以學法律知識，但法規每月更新就要每月重新微調，成本太高且不即時。
- (C) 正確。法規每月更新 → 知識會變動 → RAG 最適合。只需更新知識庫就好，不用重新訓練模型。
- (D) 關鍵字搜尋缺乏語意理解能力，無法像 LLM 那樣理解問題並組織回答。

**考點**：知識會頻繁更新 → RAG（更新知識庫即可）。微調需重新訓練，不適合頻繁更新。

### 題目 8 — 工具場景配對（L122-2/L122-4）

**以下 GenAI 工具與適用場景的配對，何者「不正確」？**

(A) Midjourney — 根據文字描述生成產品概念圖
(B) GitHub Copilot — 在 IDE 中自動補全程式碼
(C) Copilot Studio — 在 VS Code 中自動補全程式碼
(D) Gemini — 分析長篇文件並摘要重點

**答案：C**

**詳解**：
- (A) 正確配對。Midjourney 的核心功能就是文字→圖片生成。
- (B) 正確配對。GitHub Copilot 是 IDE 外掛，自動補全程式碼。
- (C) 不正確。在 VS Code 中自動補全程式碼的是 GitHub Copilot（或 VS Code Copilot），不是 Copilot Studio。Copilot Studio 是 Low Code 流程自動化平台。
- (D) 正確配對。Gemini 擅長長上下文處理和文件分析。

**考點**：Copilot 家族辨別。GitHub Copilot = 寫程式。Copilot Studio = 流程自動化（不寫程式）。

### 題目 9 — GenAI 限制（L122-10）

**企業在使用 GenAI 工具時，以下哪項「不是」需要注意的風險？**

(A) AI 可能生成看似正確但實際錯誤的內容（幻覺）
(B) AI 的回答可能反映訓練資料中的偏見
(C) 員工可能將公司機密資料輸入到公開的 AI 服務中
(D) AI 工具的處理速度太慢，無法應用於企業場景

**答案：D**

**詳解**：
- (A) 幻覺是 GenAI 最主要的風險之一。LLM 會自信地生成錯誤資訊。
- (B) 偏見風險確實存在。如果訓練資料有偏差，AI 的輸出也會有偏差。
- (C) 隱私/機密洩露是企業使用 GenAI 的重大風險。
- (D) 不正確。現代 GenAI 工具的處理速度通常很快，這不是主要風險。主要風險是準確性、偏見和隱私。

**考點**：GenAI 三大風險 = 幻覺 + 偏見 + 隱私。速度不是主要問題。

### 題目 10 — 綜合判斷（L121+L122）

**某中型企業的 IT 部門想建立以下四個 AI 應用，請問配對何者最不恰當？**

(A) 行銷部門的社群貼文自動生成 → ChatGPT + 人工審核
(B) 公司內部知識管理問答系統 → RAG 架構 + 公司文件知識庫
(C) 非技術人員建立簡單的請假 App → No Code 平台（如 Airtable）
(D) 設計部門生成產品廣告圖片 → GitHub Copilot

**答案：D**

**詳解**：
- (A) 恰當。ChatGPT 擅長文字生成，加上人工審核確保品質。
- (B) 恰當。公司內部文件 + 問答需求 = RAG 的典型場景。
- (C) 恰當。非技術人員 + 簡單 App = No Code 的典型場景。
- (D) 不恰當。GitHub Copilot 是程式碼自動補全工具，不是圖片生成工具。生成產品廣告圖片應該用 Midjourney 或 DALL-E。

**考點**：工具與場景的配對。GitHub Copilot 是寫程式的，不是做圖的。

### 考前 30 分鐘速記清單

1. **No Code = 零程式碼**，**Low Code = 少量程式碼**
2. **Copilot 家族**：GitHub Copilot 寫程式 / Copilot Studio 做流程 / M365 Copilot 辦公
3. **四象限工具地圖**：文字(ChatGPT/Gemini) / 圖片(Midjourney/DALL-E) / 程式(Copilot/Cursor) / 流程(Copilot Studio)
4. **提示工程 RTFC**：Role-Task-Format-Constraint
5. **三種提示技巧**：Zero-shot(直接問) / Few-shot(給範例) / CoT(一步步推理)
6. **RAG 三步驟**：Retrieve → Augment → Generate
7. **RAG vs 微調**：知識更新→RAG / 行為改變→微調
8. **GenAI 三大風險**：幻覺/偏見/隱私

### 易錯陷阱提醒

- 「No Code = Low Code」→ 錯，差在有無程式碼
- 「Copilot Studio 是寫程式工具」→ 錯，是 Low Code 流程自動化
- 「提示越長越好」→ 錯，要有結構不是有字數
- 「RAG 是一種模型」→ 錯，是架構模式/方法
- 「ChatGPT = OpenAI API」→ 錯，產品 vs 程式介面
- 「GenAI 可以完全取代人工」→ 錯，需要人工審核（幻覺風險）
- 「語音轉文字用 ChatGPT」→ 錯，需要 ASR 工具（如 Whisper）
