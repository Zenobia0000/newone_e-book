## S4 · CONCEPT-CARD — No Code：完全不寫程式的開發方式

- 🖼️ 畫面：上方定義框 + 下方左右優劣對比
- 📣 畫面上的字：
  - 定義：No Code = 用拖拉視覺化介面完成應用開發，完全不需要寫程式碼
  - 目標用戶：非技術人員（業務、行銷、管理者）
  - 代表平台：Bubble（Web App）、Airtable（資料庫）、Zapier（自動化串接）
  - 左優勢：上手快 / 成本低 / 迭代快 / 不需技術人員
  - 右限制：客製化程度低 / 複雜邏輯難實現 / 平台綁定（vendor lock-in）/ 效能受限
- 🎙️ 講者這時說：「No Code 就像 Canva 對設計師的意義——你不需要會 Photoshop 也能做出不錯的圖。但如果你要做超客製化的設計，Canva 就不夠用了。No Code 的限制就是它的天花板比較低，複雜需求搞不定。」

## S5 · CONCEPT-CARD — Low Code：少量程式碼的加速開發

- 🖼️ 畫面：上方定義框 + 下方左右優劣對比
- 📣 畫面上的字：
  - 定義：Low Code = 以視覺化開發為主，必要時可寫少量程式碼擴充功能
  - 目標用戶：有基礎技術能力的開發者（公民開發者 Citizen Developer）
  - 代表平台：Microsoft Power Platform、OutSystems、Mendix
  - 左優勢：比傳統開發快 3-5 倍 / 有一定客製化能力 / 企業級整合 / 適合內部應用
  - 右限制：仍需基礎技術能力 / 複雜整合有挑戰 / 平台相依性 / 學習曲線中等
- 🎙️ 講者這時說：「Low Code 的定位是『加速器』——它不是讓你不寫程式，是讓你少寫。視覺化拖拉搞定八成，剩下兩成複雜邏輯用少量程式碼補。Power Platform 裡的 Power Apps、Power Automate 就是典型代表。」

## S6 · PITFALL (P1) — No Code 和 Low Code 不是同義詞

- 🖼️ 畫面：左紅錯 / 右綠對 / 底部對比表
- 📣 畫面上的字：
  - 錯：「No Code 和 Low Code 基本上是同一種東西」
  - 對：「No Code 完全不寫程式碼；Low Code 以視覺化為主但需要少量程式碼」
  - 對比表：

  | | No Code | Low Code |
  |--|---------|----------|
  | 程式碼 | 零 | 少量 |
  | 目標用戶 | 非技術人員 | 有基礎技術者 |
  | 客製化 | 低 | 中 |
  | 複雜度上限 | 低 | 中高 |

- 🎙️ 講者這時說：「這題幾乎必考。記住一個差異就好：No Code 是零程式碼，Low Code 是少量程式碼。然後延伸：零程式碼所以非技術人員能用，但客製化低；少量程式碼所以需要一點技術底，但能做更複雜的東西。」

## S13 · CONCEPT-CARD — 提示工程：RTFC 四要素

- 🖼️ 畫面：中央四格框架 + 下方範例
- 📣 畫面上的字：
  - R = Role 角色：「你是一位...」— 設定 AI 的角色和專業背景
  - T = Task 任務：「請你...」— 明確說明要 AI 做什麼
  - F = Format 格式：「以...格式呈現」— 指定輸出格式（條列/表格/段落）
  - C = Constraint 限制：「注意.../不要...」— 設定邊界和限制條件
  - 範例：「你是一位資深行銷專家（R），請撰寫一篇 500 字的產品介紹（T），使用條列式重點（F），避免使用專業術語（C）。」
- 🎙️ 講者這時說：「RTFC 四個字母就是你的提示工程武器。寫提示之前先想：我要 AI 扮演什麼角色？做什麼任務？用什麼格式？有什麼限制？四個都有，提示就不會差。考題會給你兩段提示問哪個好——有 RTFC 結構的那個就是答案。」

## S14 · PITFALL (P2) — 提示工程不是把問題打長一點

- 🖼️ 畫面：上方壞提示（紅框）/ 下方好提示（綠框）
- 📣 畫面上的字：
  - 壞提示（紅）：「幫我寫一篇文章，要很好很專業很詳細，內容要豐富，字數要多一點，語氣要正式。」→ 含糊、沒有具體指示
  - 好提示（綠）：「你是一位資深科技記者（R）。請撰寫一篇 800 字的文章介紹 RAG 技術（T），包含三個段落：定義、流程、應用場景（F）。目標讀者是非技術背景的管理者，避免使用程式碼和數學公式（C）。」→ 結構清晰、具體可執行
  - 底部：「好的提示不是字多，是有結構。」
- 🎙️ 講者這時說：「壞提示的問題不是太短，是太模糊。『很好很專業很詳細』——你覺得 AI 知道你的『好』是什麼標準嗎？好提示像寫工作說明書：角色明確、任務具體、格式指定、限制清楚。考試看到兩個提示比較題，找有 RTFC 結構的那個。」

## S15 · CONCEPT-CARD — 提示工程三大技巧：Zero-shot / Few-shot / CoT

- 🖼️ 畫面：三欄技巧卡 + 每欄一個範例片段
- 📣 畫面上的字：
  - 左：Zero-shot — 不給範例直接問 · 「請將以下句子翻譯成英文：...」· 適合簡單明確的任務
  - 中：Few-shot — 給 2-3 個範例 · 「範例一：...→... / 範例二：...→... / 現在請處理：...」· 適合需要特定格式或風格的任務
  - 右：Chain-of-Thought (CoT) — 要求一步步推理 · 「請一步步思考：...」· 適合推理、計算、邏輯判斷
  - 底部：「Zero-shot 最快 / Few-shot 最穩 / CoT 最適合推理。考題考你能辨認哪段提示用了哪種技巧。」
- 🎙️ 講者這時說：「這三個名詞是提示工程的核心術語，考試一定出。Zero-shot 就是裸問——不給範例直接問。Few-shot 就是給幾個範例讓 AI 學格式。CoT 就是叫 AI 一步步想。考題形式通常是給你一段提示問你這用了什麼技巧，或者問你某個場景該用什麼技巧。」

## S16 · PRACTICE-PROMPT — 判斷提示好壞

- 🖼️ 畫面：三組提示對比（A/B 兩欄）
- 📣 畫面上的字：
  - 組一：A「幫我寫一封信」vs B「你是一位人資主管，請撰寫一封 300 字的新人歡迎信，語氣親切正式，包含報到日期和注意事項」→ B 好（有 RTFC）
  - 組二：A「這個數學題怎麼解？」vs B「請一步步解這個數學題：...首先計算...然後...」→ B 好（用了 CoT）
  - 組三：A「翻譯以下內容」vs B「範例：Hello→你好 / Thank you→謝謝 / 請翻譯：Goodbye」→ B 好（用了 Few-shot）
  - 底部：「好提示的判斷標準：有結構、有具體性、有適當的技巧」
- 🎙️ 講者這時說：「三十秒自己判斷。都不難對吧？考試的提示比較題也是這個難度。有結構的贏沒結構的，有範例的贏沒範例的，要求推理的贏直接要答案的。」

## S17 · CONCEPT-CARD — RAG：檢索增強生成

- 🖼️ 畫面：RAG 三步驟流程圖（真圖佔位）
- 📣 畫面上的字：
  - 痛點：LLM 的兩大問題 — 知識過時（訓練截止日後的事不知道）+ 幻覺（不知道就亂編）
  - RAG 流程三步驟：
    - Step 1 Retrieve 檢索：用戶提問 → 從知識庫/文件庫搜尋相關段落
    - Step 2 Augment 增強：把檢索到的文件段落和用戶問題組合成增強提示
    - Step 3 Generate 生成：LLM 根據增強提示生成有依據的回答
  - 類比：「RAG = 帶小抄考試。知識庫是小抄，LLM 是考生。考生不用全背，帶著小抄就能答對。」
- 🎙️ 講者這時說：「RAG 解決的是 LLM 最大的兩個痛點。第一，ChatGPT 的知識有截止日，2023 年以後的事它不知道。第二，它不知道的東西會亂編（幻覺）。RAG 的解法很直覺——你問問題時，先幫你從資料庫裡找到相關的資料，再把資料和問題一起餵給 LLM。這樣 LLM 就有依據了，不用亂猜。」

## S18 · CONCEPT-CARD — RAG vs 微調 (Fine-tuning)

- 🖼️ 畫面：左右對比 vs_two_col
- 📣 畫面上的字：
  - 左：RAG 檢索增強生成 — 不改模型，外掛知識庫 · 知識可即時更新 · 成本低（不用重新訓練）· 適合：知識會變動的場景（法規/內部文件/即時資訊）· 類比：帶小抄考試
  - 右：微調 Fine-tuning — 改模型權重 · 學會新風格/領域/行為 · 成本高（需要訓練資料+GPU）· 適合：需要改變模型核心行為的場景（特定語氣/專業術語/特殊格式）· 類比：讀完整本書再考
  - 底部：「知識會更新 → RAG。行為要改變 → 微調。」
- 🎙️ 講者這時說：「這是高頻考題。最簡單的判斷：如果你的知識會常常更新（法規、公司文件、新聞），用 RAG——因為你只要更新知識庫就好，不用重新訓練模型。如果你要讓 AI 用特定風格說話或學會特殊領域的術語，用微調——因為你要改變模型的行為。」

## S19 · CHECKPOINT — RAG vs 微調場景判斷

- 🖼️ 畫面：白底四題列表 + RAG/微調 標記
- 📣 畫面上的字：
  - 場景 A：建立公司內部文件問答系統，文件每月更新 → RAG（知識常更新）

  - 場景 C：法律事務所需要 AI 查詢最新法規判例 → RAG（即時檢索最新資訊）
  - 場景 D：醫療 AI 需要學會使用醫學專業術語和診斷邏輯 → 微調（學習專業領域知識和行為）
