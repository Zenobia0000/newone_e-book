## 3. 關鍵教學點

### Part A：AI 導入評估與規劃（L212）

#### 需求分析四步法
- **痛點識別**：現有流程的瓶頸在哪？人工成本、錯誤率、時效性。
- **可行性評估**：有沒有足夠的資料？技術是否成熟？團隊能力是否匹配？
- **ROI 估算**：導入成本（開發+維護+基礎設施）vs 預期效益（效率提升+成本節省+營收增長）。
- **優先級排序**：高影響+高可行性的先做；低影響+低可行性的不做。
- **考試要點**：ROI 計算方式、可行性評估的標準指標。

#### 技術方案選擇
- **自建模型**：客製化高、成本高、需要專業團隊、資料不外流。
- **採購現成方案**：快速導入、成本可控、但客製化受限。
- **AutoML 平台**：降低技術門檻、加速原型驗證、但黑箱風險（Q5 考點）。
- **考試陷阱**：AutoML 不等於不需要人——仍需要領域專家定義問題和驗證結果。

#### 目標設置與資源分配
- **SMART 原則**：Specific / Measurable / Achievable / Relevant / Time-bound。
- **資源分配**：人力（DS/DE/PM）、算力（GPU/雲端預算）、資料（標註成本）、時間。
- **考試要點**：識別資源分配不合理的情境（如人力全壓在建模、忽略數據準備）。

### Part B：數據準備與模型選擇（L213）

#### 數據收集與清洗
- **收集方法**：內部系統擷取、外部採購、爬蟲、標註平台、合成資料。
- **資料品質維度**：完整性、一致性、準確性、時效性、代表性。
- **清洗步驟**：缺值處理（刪除/填補/插值）→ 異常值偵測（IQR/Z-score）→ 重複資料去除 → 格式統一。
- **考試要點**：選擇缺值處理方式的依據（MCAR/MAR/MNAR）。

#### 特徵工程
- **數值特徵**：標準化（Z-score）、正規化（Min-Max）、對數轉換、多項式特徵。
- **類別特徵**：One-Hot Encoding、Label Encoding、Target Encoding。
- **特徵選擇**：Filter（相關係數/卡方）、Wrapper（RFE）、Embedded（L1 正則化）。
- **考試要點**：何時該用標準化 vs 正規化、One-Hot 的維度爆炸問題。

#### 模型選擇
- **監督式**：迴歸（線性/Ridge/Lasso）、分類（SVM/決策樹/隨機森林/XGBoost/神經網路）。
- **非監督式**：聚類（K-means/DBSCAN）、降維（PCA/t-SNE）。
- **深度學習**：CNN（影像）、RNN/LSTM（序列）、Transformer（NLP/多模態）。
- **選擇依據**：資料量、特徵維度、可解釋性需求、推論速度限制、標註成本。

### Part C：風險識別與管理（L212）

#### 風險四象限
- **技術風險**：模型精度不足、過擬合、資料漂移（data drift）、概念漂移（concept drift）。
- **資安風險**：供應鏈攻擊（Q9 考點）、對抗攻擊（adversarial attack）、資料投毒（data poisoning）、模型竊取。
- **倫理風險**：偏見與歧視（bias）、不透明決策（black box）、隱私侵犯。
- **法規風險**：個資法/GDPR 合規、AI 法規（EU AI Act）、行業特定法規。

#### 供應鏈攻擊（Q9 深入）
- **定義**：攻擊者透過第三方元件（預訓練模型、套件、資料集）植入惡意程式碼。
- **防範**：驗證模型來源（Model Card/Data Card）、使用可信任儲存庫、定期安全掃描、SBOM（軟體物料清單）。

#### 負責任 AI 框架
- **公平性**（Fairness）：避免對特定群體的歧視。
- **可解釋性**（Explainability）：決策過程可被理解。
- **透明性**（Transparency）：訓練資料和方法公開。
- **問責性**（Accountability）：出錯時有明確的責任歸屬。
- **隱私保護**（Privacy）：最小資料收集原則、匿名化、差分隱私。

### Part D：系統集成與部署（L213）

#### 系統架構設計
- **雲端部署**：彈性擴展、按用量計費、AWS SageMaker / GCP Vertex AI / Azure ML。
- **地端部署**：資料不出場、延遲低、但維護成本高。
- **Edge 部署**：模型壓縮（quantization/pruning/distillation）、TensorRT、ONNX Runtime。
- **混合架構**：訓練在雲端、推論在地端/Edge。

#### 模型部署技術
- **容器化**：Docker 封裝模型+依賴 → Kubernetes 編排 → 自動擴縮容。
- **API 設計**：REST API / gRPC、版本控制（/v1/predict）、認證授權。
- **CI/CD for ML**：模型版本管理（MLflow/DVC）→ 自動測試 → 自動部署 → 監控回饋。
- **考試要點**：容器化 vs 無容器化的優劣比較。

#### 系統測試驗證（Q15 考點）
- **單元測試**：測試單一模組功能正確性。
- **整合測試**：測試子系統之間的介面與資料流（Q15 優先驗證）。
- **壓力測試**：模擬高併發場景，驗證系統承載能力。
- **A/B 測試**：新舊版本對比，驗證效能提升是否顯著。
- **關鍵訊息**：整合測試是系統整合後「第一個」要做的驗證——確認子系統串接正確，再做其他測試。

#### 效能監控與模型更新
- **監控指標**：推論延遲、吞吐量、錯誤率、資料漂移偵測、模型效能衰退。
- **更新策略**：定時重訓、觸發式重訓（指標低於閾值）、線上學習。
- **穩定性與可用性**：SLA 設計、負載均衡、故障轉移（failover）、灰度發布。

## 4. 演練設計（30 min）

### 情境題

> 你是某製造業公司的 AI 專案負責人，老闆要你導入「AI 瑕疵檢測系統」替代人工目檢。請設計：
> 1. 需求分析（痛點、可行性、ROI 估算要素）
> 2. 技術方案選擇（自建 / 採購 / AutoML，說明理由）
> 3. 風險評估（列出至少 3 類風險 + 對應管控措施）
> 4. 部署架構建議（雲端 / 地端 / Edge，說明理由）
> 5. 系統測試計畫（至少包含 3 種測試方法）

### 分組方式

- 3-4 人一組，15 分鐘討論 + 撰寫，10 分鐘報告，5 分鐘講師補充。
- 講師補充重點：製造業場景常見的 Edge 部署需求、即時推論的延遲要求、品管標準對模型精度的要求。

## 5. 考試常見陷阱

1. **AutoML 考點**：AutoML「自動化」的範圍是特徵選擇+模型選擇+超參數調整，不包含問題定義和資料收集——選項若說「完全自動化不需人工」就是錯的。
2. **供應鏈攻擊**：不是只有程式碼會被攻擊——預訓練模型、第三方資料集、Python 套件都可能被植入後門。
3. **整合測試**：Q15 的考點——系統整合後「最先」要驗證的是子系統之間的介面正確性，不是效能或安全。
4. **部署架構**：考試喜歡問「哪個場景適合 Edge 部署」——答案通常是「需要低延遲+離線運行+資料不可上傳」。
5. **資料漂移**：容易混淆 data drift（輸入分布改變）和 concept drift（輸入與輸出關係改變），考試會區分。

## 8. 常見 Q&A

- **Q：AutoML 和 MLOps 差在哪？**
  A：AutoML 是「自動化建模過程」（選特徵、選模型、調參），MLOps 是「自動化模型的部署和運維」（CI/CD、監控、重訓）。兩者解決不同階段的問題。
- **Q：Edge 部署的模型精度會下降嗎？**
  A：通常會。因為模型壓縮（量化、剪枝、蒸餾）會犧牲一些精度換取速度和體積。考試若問「Edge 部署的代價」，答案包含精度下降和模型更新不便。
- **Q：供應鏈攻擊怎麼考？**
  A：通常是情境題——描述一個使用第三方預訓練模型的場景，問「可能面臨什麼風險」或「應採取什麼防範措施」。記住 SBOM（軟體物料清單）這個關鍵字。
