## S02 · ASK · 起

- 一句話：大部分 AI 專案不是死在技術，是死在規劃。
- 講者腳本：快速調查：有人經歷過 AI 專案上線的嗎？有人經歷過 AI 專案做到一半被砍的嗎？87% 的 AI 專案沒能從 PoC 進到 production。原因不是模型不行，是需求沒搞清楚、資料沒準備好、風險沒評估、部署沒規劃。

## S04 · MECHANISM_FLOW · 起

- 視覺：水平五階段流程圖——「需求分析 → 方案設計 → 數據準備 → 模型建構 → 系統部署」。每階段下方掛 2-3 個關鍵活動，上方標對應 L212/L213。流程尾端加「監控更新」迴圈箭頭。
- 一句話：AI 導入不是線性的——是有回饋迴圈的流程。
- 講者腳本：這五個階段就是 AI 專案的生命週期。考試不會問你「第三步是什麼」，但會用情境題測你知不知道某個問題屬於哪個階段。注意最後的監控更新——上線不是終點，是另一個循環的起點。

## S05 · TABLE · 承

- 視覺：editorial_table 四列——痛點識別 / 可行性評估 / ROI 估算 / 優先級排序。每列有「做什麼」「怎麼做」「考試怎麼考」三欄。
- 一句話：需求分析不是聊天，是有結構的四步流程。
- 講者腳本：痛點要量化——「人工目檢錯誤率 15%」比「品質不穩定」有用。可行性要看三件事：有沒有資料、技術成不成熟、團隊行不行。ROI 要算導入成本和預期效益。優先級用影響力 x 可行性矩陣排序。考試喜歡問「以下哪個最適合先導入 AI」。

## S06 · VS · 承

- 視覺：三欄對比——左「自建模型：客製化高/成本高/需專業團隊」、中「採購方案：快速導入/成本可控/客製受限」、右「AutoML：降低門檻/加速驗證/黑箱風險」。底部標示 Q5 考點。
- 一句話：三條路各有適用場景，考試考的是你的「選擇依據」。
- 講者腳本：Q5 考 AutoML 趨勢——AutoML 自動化的是特徵選擇、模型選擇、超參數調整，但問題定義和資料收集還是需要人。選項若說「AutoML 完全取代 ML 工程師」就是錯的。記住：AutoML 降低了建模門檻，但沒有降低規劃門檻。

## S07 · PITFALL · 承

- 視覺：vs_two_col——左「錯：只看 accuracy=95% 就選模型」紅叉；右「對：同時比較 accuracy / inference time / cost / interpretability」綠勾。中間加一個表格小圖暗示多維度比較。
- 一句話：準確率是必要條件，不是充分條件。
- 講者腳本：最常見的考試陷阱：題目說「模型 A 準確率最高，應選擇模型 A」——但如果模型 A 推論需要 10 秒，而場景要求即時回應，那就不能選。模型選擇永遠是多維度的：精度、速度、成本、可解釋性、部署限制。

## S08 · TABLE · 承

- 視覺：editorial_table——橫軸：模型類型（線性迴歸/決策樹/隨機森林/XGBoost/SVM/神經網路/CNN/Transformer）；縱軸：精度/速度/可解釋性/資料需求/適用場景。用星號評級。
- 一句話：這張表記熟，考試情境題直接對號入座。
- 講者腳本：考試會給你一個場景，問你該選哪個模型。關鍵判斷：資料量小 + 需要可解釋性 → 決策樹/線性模型。資料量大 + 精度優先 → XGBoost/神經網路。影像 → CNN。序列/文本 → Transformer。這張表不用死背，理解每個模型的強項和弱項就夠了。

## S09 · MECHANISM_FLOW · 承

- 視覺：垂直流程圖——「數據收集 → 探索性分析(EDA) → 清洗處理 → 特徵工程 → 特徵選擇」。每步右側掛具體方法（如清洗：缺值處理/異常值偵測/去重/格式統一）。
- 一句話：數據準備佔專案 60-80% 的時間，考試也佔不少題。
- 講者腳本：數據準備流程：先收集、再 EDA 看資料長什麼樣、然後清洗、接著做特徵工程、最後特徵選擇。考試重點：缺值處理方式的選擇依據（MCAR 可以直接刪、MAR 用模型填補、MNAR 需要額外調查），還有標準化 vs 正規化的差異。

## S10 · PITFALL · 承

- 視覺：vs_two_col——左「錯：拿到資料直接訓練，accuracy 怎麼都上不去」；右「對：先做 EDA 發現 30% 缺值 + 類別不平衡，處理後 accuracy 提升 15%」。
- 一句話：垃圾進垃圾出——不做 EDA 就是盲人開車。
- 講者腳本：真實案例：某團隊拿到 10 萬筆資料直接訓練，accuracy 只有 60%。做完 EDA 才發現：30% 的欄位有缺值、標籤有 95:5 的不平衡、還有 5% 的重複資料。處理完之後，同一個模型 accuracy 到 85%。考試不會問你怎麼寫 code，但會問你「下一步該做什麼」——答案通常是 EDA。

## S11 · SILENT · 轉

- 視覺：深綠滿版：「規劃做完了，數據準備好了，接下來是兩個考試重災區：風險管理 + 系統部署。」
- 一句話：過渡——從規劃進入風險和執行面。
- 講者腳本：前面講的是「怎麼做對的事」，接下來講的是「怎麼防止事情出錯」和「怎麼把模型真正上線」。這兩塊加起來佔 L212+L213 接近一半的考題比重。

## S12 · MATRIX · 轉

- 視覺：2x2 矩陣——橫軸「發生機率（低/高）」，縱軸「影響程度（低/高）」。四格填入：技術風險（高機率/中影響）、資安風險（中機率/高影響）、倫理風險（中機率/高影響）、法規風險（低機率/高影響）。每格列 2-3 個具體風險。
- 一句話：風險管理不是全部都防，是按優先級分配資源。
- 講者腳本：考試喜歡問「以下哪個風險最應該優先處理」——用這個矩陣判斷：高影響的優先，同影響看機率。資安風險和倫理風險通常是高影響，因為出事會上新聞。技術風險機率最高但通常可以透過重訓修復。

## S13 · TABLE · 轉

- 視覺：editorial_table 三列——供應鏈攻擊 / 對抗攻擊 / 資料投毒。每列有「攻擊方式」「攻擊目標」「防範措施」「考試出法」四欄。Q9 標記在供應鏈攻擊列。
- 一句話：資安三大威脅——記住名詞和防範關鍵字就能答題。
- 講者腳本：Q9 考供應鏈攻擊：攻擊者不直接攻擊你的系統，而是在你使用的第三方元件裡埋雷——預訓練模型、Python 套件、開源資料集都可能中招。防範關鍵字：SBOM（軟體物料清單）、可信任儲存庫、安全掃描。對抗攻擊是在輸入加微小擾動騙模型。資料投毒是在訓練資料裡混入惡意樣本。

## S14 · VS · 轉

- 視覺：三欄對比——「雲端：彈性/按量付費/資料上傳」vs「地端：低延遲/資料不出場/維護成本高」vs「Edge：超低延遲/離線運行/模型受限」。底部加判斷流程：資料敏感？→ 地端。需要離線？→ Edge。其他 → 雲端。
- 一句話：部署架構沒有最好的，只有最適合場景的。
- 講者腳本：考試最愛考的情境：「工廠產線需要即時瑕疵檢測，且不能連網」——答案是 Edge 部署。「醫療影像分析，資料不可上傳雲端」——答案是地端部署。「新創公司快速上線 MVP」——答案是雲端部署。記住這個決策樹就能答大部分題。

## S15 · MECHANISM_FLOW · 轉

- 視覺：水平流程圖——「開發環境 → Docker 封裝 → K8s 編排 → CI/CD 管線 → 生產環境 → 監控告警」。每個節點配圖標和一句說明。上方標示 MLOps 概念。
- 一句話：模型部署不是 copy 檔案，是一條完整的工程管線。
- 講者腳本：容器化部署的考試重點：Docker 解決「我的電腦上跑得動」的問題——把模型和所有依賴打包成映像檔。Kubernetes 解決「一台機器不夠用」的問題——自動擴縮容。CI/CD 解決「每次更新都要手動」的問題——程式碼推送後自動測試、自動部署。考試問「容器化的主要優勢」，答案是環境一致性和可移植性。

## S16 · TABLE · 轉

- 視覺：editorial_table 四列——單元測試 / 整合測試 / 壓力測試 / A/B 測試。每列有「測什麼」「怎麼測」「何時做」「常見考法」四欄。Q15 標記在整合測試列。
- 一句話：Q15 考的就是「系統整合後第一件事該驗什麼」——答案是介面正確性。
- 講者腳本：Q15 直接考：系統整合測試最優先驗證的是什麼？答案是子系統之間的介面和資料流是否正確——不是效能、不是安全、不是使用者體驗。邏輯：如果介面都串不通，測效能沒有意義。單元測試在整合之前做，壓力測試在整合測試通過之後做，A/B 測試在上線時做。記住這個順序。

## S17 · PITFALL · 轉

- 視覺：vs_two_col——左「錯：每個子系統各自通過測試 → 直接上線 → 使用者回報錯誤」；右「對：子系統各自通過 → 整合測試驗證介面 → 壓力測試 → 灰度發布 → 全量上線」。
- 一句話：子系統各自 OK 不代表合起來 OK——這是整合測試存在的理由。
- 講者腳本：考試最常見的陷阱：選項說「所有模組通過單元測試即可上線」——這是錯的。差的是整合測試。真實案例：推論服務 OK、前端 OK，但兩邊對 JSON 格式的定義不一致，串起來就爆了。整合測試就是測這個。

## S18 · PRACTICE_PROMPT · 合

- 視覺：情境題居中——「為製造業 AI 瑕疵檢測系統設計完整導入規劃」。下方五格：需求分析 / 方案選擇 / 風險評估 / 部署架構 / 測試計畫。每格留填寫區。
- 一句話：不寫下來就不算學會——用本章框架解一道完整的情境題。
- 講者腳本：這題包含了 L212 和 L213 所有考點。需求分析：人工目檢的痛點是什麼？方案選擇：CNN + Edge 部署？還是雲端 API？風險：模型漂移、對抗攻擊、產線停機。部署：Edge（即時+離線需求）。測試：先整合測試驗介面，再壓力測試驗產線速度。五分鐘，現在動筆。

## S19 · PYRAMID · 合

- 視覺：五層金字塔，由底到頂——需求分析（最寬）/ 數據準備 / 模型選擇 / 風險管理 / 系統部署（最窄）。每層標 1 個考試關鍵字。頂端標「成功上線」。
- 一句話：底層沒做好，上面全部白搭——這是 Ch07 的核心邏輯。
- 講者腳本：需求沒搞清楚 → 數據收集方向錯。數據沒準備好 → 模型訓練白費。模型選錯 → 部署困難。風險沒管 → 上線出事。每一層都是下一層的基礎。考試情境題通常會在某一層設定錯誤，問你「哪裡有問題」——用這個金字塔由下往上檢查。
