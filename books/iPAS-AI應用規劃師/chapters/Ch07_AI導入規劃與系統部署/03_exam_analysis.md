## 二、考題類型分析

### 類型 1：概念辨析題（約 30%）

- 考法：給定義或描述，問是什麼技術/方法/概念。
- 範例：「在 AI 專案中，自動化特徵選擇、模型選擇和超參數調整的技術稱為？」→ AutoML。
- 對策：名詞定義要精確，特別注意容易混淆的概念（AutoML vs MLOps, data drift vs concept drift）。

### 類型 2：情境判斷題（約 40%）

- 考法：描述一個場景，問最適合的方案/最優先的步驟/最可能的風險。
- 範例：「某工廠需要在產線上即時檢測瑕疵，且生產線無法連接外網，最適合的部署方式？」→ Edge 部署。
- 對策：建立判斷框架（如部署決策樹、風險優先級矩陣），而非死記答案。

### 類型 3：流程排序題（約 20%）

- 考法：列出多個步驟，問正確順序或下一步。
- 範例：「系統整合後，最應優先進行的驗證是？」→ 整合測試（介面正確性）。
- 對策：記住核心流程的順序（需求→數據→建模→部署；單元→整合→壓力→A/B）。

### 類型 4：風險識別題（約 10%）

- 考法：描述一個操作方式，問存在什麼風險或應如何防範。
- 範例：「團隊直接下載 Hugging Face 上不明來源的預訓練模型使用，可能面臨的風險？」→ 供應鏈攻擊。
- 對策：記住三大資安威脅的攻擊方式和防範措施。

## 三、模擬試題（10 題）

### 題 1（概念辨析 / L212）

**關於 AutoML，以下敘述何者正確？**

(A) AutoML 可以完全取代 ML 工程師，無需人工介入
(B) AutoML 自動化的範圍包含問題定義、資料收集到模型部署的全流程
(C) AutoML 主要自動化特徵選擇、模型選擇與超參數調整，但問題定義仍需人工
(D) AutoML 只適用於結構化資料，無法處理影像或文本

**答案：C**

**詳解**：AutoML 自動化的是建模過程中的「選特徵、選模型、調參」，但問題定義（決定要解什麼問題）、資料收集（決定用什麼資料）、結果驗證（判斷商業價值）仍需要領域專家和人工判斷。(A) 誇大了 AutoML 的能力。(B) 問題定義和資料收集不在 AutoML 範疇。(D) 現代 AutoML 平台（如 Google AutoML Vision）已支援影像和文本。

### 題 2（情境判斷 / L212）

**某電商公司想導入 AI 推薦系統，目前有三個候選方案。評估時最不應該只看的單一指標是？**

(A) 推薦準確率
(B) 系統推論延遲
(C) 導入與維護成本
(D) 模型可解釋性

**答案：A**

**詳解**：題目問「最不應該只看的單一指標」——意思是不能「只看」這一個就做決定。準確率是最容易被單獨拿來比較的指標，但選擇 AI 方案需要多維度評估（精度、延遲、成本、可解釋性等）。只看準確率忽略了推論延遲（影響使用者體驗）、成本（影響 ROI）、可解釋性（影響除錯和合規）。四個選項都很重要，但 (A) 是最常被誤用為唯一標準的。

### 題 3（情境判斷 / L213）

**某智慧工廠需要在產線上進行即時瑕疵檢測（延遲 <50ms），且生產設備無法連接外部網路。最適合的部署方式是？**

(A) 全雲端部署，使用 AWS SageMaker
(B) 地端伺服器部署，搭配內網 API
(C) Edge 部署，將壓縮模型部署在產線端設備
(D) 混合部署，訓練在雲端、推論走雲端 API

**答案：C**

**詳解**：關鍵條件有二——即時（延遲 <50ms）和離線（無法連網）。(A)(D) 需要網路連接，不符合離線需求。(B) 地端伺服器可以離線，但從設備到伺服器的傳輸仍有延遲，且單一伺服器要服務多條產線可能成為瓶頸。(C) Edge 部署將模型直接跑在產線端設備，延遲最低且可離線運行。代價是需要模型壓縮（量化/剪枝），精度可能略降。

### 題 4（風險識別 / L212）

**開發團隊從公開平台下載了一個預訓練的物件偵測模型，未經驗證直接用於產品中。此做法最主要面臨的資安風險是？**

(A) 對抗攻擊（Adversarial Attack）
(B) 資料投毒（Data Poisoning）
(C) 供應鏈攻擊（Supply Chain Attack）
(D) 模型逆向工程（Model Inversion）

**答案：C**

**詳解**：供應鏈攻擊是指攻擊者透過第三方元件（預訓練模型、開源套件、公開資料集）植入惡意程式碼或後門。直接使用未經驗證的預訓練模型，可能載入含有後門的權重，導致在特定輸入下產生錯誤行為。(A) 是在推論時透過微小擾動欺騙模型。(B) 是在「訓練資料」中混入惡意樣本，但題目是下載已訓練好的模型。(D) 是試圖從模型推回訓練資料，方向不同。防範措施：驗證模型來源、檢查 Model Card、使用可信任儲存庫、建立 SBOM。

### 題 5（流程排序 / L213）

**AI 系統整合完成後，應最優先進行的驗證是？**

(A) 效能壓力測試，確認系統能承受高併發
(B) 安全滲透測試，確認系統無漏洞
(C) 整合測試，確認子系統之間的介面與資料流正確
(D) 使用者驗收測試，確認功能符合需求

**答案：C**

**詳解**：系統整合後的「第一步」是確認各子系統之間的介面能正確串接、資料流能正確傳遞。如果介面都不通，做壓力測試 (A) 沒有意義、做安全測試 (B) 基礎不穩、做使用者驗收 (D) 更是太早。正確順序：整合測試 → 壓力測試 → 安全測試 → 使用者驗收。這是 Q15 的直接考點。

### 題 6（概念辨析 / L213）

**關於模型部署的容器化（Containerization），以下敘述何者錯誤？**

(A) Docker 可將模型與所有依賴封裝，確保環境一致性
(B) Kubernetes 用於容器的編排與自動擴縮容
(C) 容器化部署的主要優勢是模型精度會提升
(D) 容器化有助於實現 CI/CD 的自動化部署流程

**答案：C**

**詳解**：容器化解決的是部署環境問題（環境一致性、可移植性、擴縮容），與模型精度完全無關。(A) 正確，Docker 的核心價值就是「打包環境」。(B) 正確，K8s 負責容器編排。(D) 正確，容器化配合 CI/CD 可以實現自動測試和部署。記住：容器化是部署工程的最佳實踐，不會影響模型本身的性能。

### 題 7（情境判斷 / L212）

**AI 專案的需求分析階段，以下哪項最不屬於可行性評估的考量？**

(A) 是否有足夠且高品質的訓練資料
(B) 相關 AI 技術是否已成熟
(C) 團隊是否具備必要的技術能力
(D) 競爭對手是否已經導入類似技術

**答案：D**

**詳解**：可行性評估關注的是「我們能不能做到」——資料可得性 (A)、技術成熟度 (B)、團隊能力 (C) 都直接影響專案是否做得出來。(D) 競爭對手的狀況是「要不要做」的商業策略考量，不是「能不能做」的可行性問題。可行性評估的三大支柱：資料、技術、團隊。

### 題 8（概念辨析 / L212）

**以下何者最能描述「負責任 AI」（Responsible AI）框架中「可解釋性」的意義？**

(A) AI 系統的原始碼必須完全公開
(B) AI 系統的決策過程能被相關利害關係人理解
(C) AI 系統的訓練資料必須完全透明
(D) AI 系統的預測結果必須 100% 正確

**答案：B**

**詳解**：可解釋性（Explainability）是指 AI 的決策過程能被理解——使用者、管理者、監管者能知道「為什麼 AI 做出這個決定」。(A) 是開源的概念，不等於可解釋。(C) 更接近「透明性」（Transparency）。(D) 是準確性的要求，與可解釋性無關。可解釋性的實踐方法：SHAP、LIME、attention visualization、decision path。

### 題 9（情境判斷 / L213）

**團隊發現已上線的 AI 模型準確率逐月下降，但訓練資料和模型本身未更動。最可能的原因是？**

(A) 模型過擬合（Overfitting）
(B) 資料漂移（Data Drift）
(C) 超參數設定錯誤
(D) 訓練資料標註錯誤

**答案：B**

**詳解**：關鍵條件：「逐月下降」+「訓練資料和模型未更動」。這表示問題不在模型本身，而是真實世界的資料分布改變了——即資料漂移（Data Drift）。例如，用去年的消費資料訓練的推薦模型，今年消費者偏好變了，準確率自然下降。(A) 過擬合在訓練階段就能發現。(C)(D) 模型未更動就不會突然出現這些問題。對策：持續監控輸入資料分布、設定漂移偵測閾值、觸發式重訓。注意區分 data drift（輸入分布變）和 concept drift（輸入與輸出關係變）。

### 題 10（綜合情境 / L212 + L213）

**某醫療機構要導入 AI 輔助診斷系統，以下規劃步驟中，最不合理的是？**

(A) 先進行需求分析，確認臨床痛點與預期效益
(B) 評估資料可得性，確認有足夠的標註醫學影像
(C) 選擇最新的大型語言模型作為核心技術，以追求最高準確率
(D) 設計風險管理計畫，涵蓋病患隱私保護與 AI 決策可解釋性

**答案：C**

**詳解**：醫療影像輔助診斷的核心技術應是 CNN 或 Vision Transformer，而非大型語言模型（LLM）。「選擇最新的」和「追求最高準確率」犯了兩個錯誤：(1) 模型類型不匹配任務需求，(2) 醫療場景需要同時考慮精度、可解釋性、合規性，不能只追求準確率。(A)(B)(D) 都是合理的規劃步驟。此題測試的是「依場景選擇合適技術」的能力——不是越新越好，是越匹配越好。

## 四、備考策略

### 高頻考點優先級

1. **必背**：AutoML 定義與限制、供應鏈攻擊防範、整合測試優先性、部署架構選擇依據。
2. **熟悉**：需求分析四步法、風險四象限、模型選擇多維度、容器化優勢。
3. **了解**：SMART 原則、MLOps 概念、負責任 AI 五原則、SLA 設計。

### 答題技巧

1. **情境題看關鍵條件**：延遲要求 → 考部署架構。第三方元件 → 考供應鏈攻擊。系統整合後 → 考測試順序。
2. **排除法**：看到「完全」「一定」「只要」等絕對用語，通常是錯的選項。
3. **多維度思考**：任何選項只提一個維度（如只說準確率），通常不是最佳答案。
4. **流程題看順序**：需求→數據→建模→部署；單元→整合→壓力→驗收。順序錯就是錯。

### 易混淆概念

| 概念 A | 概念 B | 區分方式 |
|---|---|---|
| AutoML | MLOps | AutoML 自動化建模，MLOps 自動化部署運維 |
| Data Drift | Concept Drift | Data Drift 輸入分布變，Concept Drift 輸入輸出關係變 |
| Rollback | Fallback | Rollback 退回上一版，Fallback 走備用路徑 |
| 供應鏈攻擊 | 對抗攻擊 | 供應鏈攻擊透過第三方元件，對抗攻擊透過輸入擾動 |
| 標準化 | 正規化 | 標準化用均值和標準差（Z-score），正規化用最小最大值（Min-Max） |
