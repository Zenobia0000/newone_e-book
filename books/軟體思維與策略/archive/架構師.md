# 系統設計：架構師武僧 Agent (Architect Warrior Monk Agents)

> **版本**：v0.1 (MVP Planning)
> **來源**：《軟體架構精通 - 架構師武僧寶典》
> **目標**：打造一個「虛擬架構委員會」，引導用戶走完架構設計七步流程，產出可執行的架構文件。

---

## 1. 核心設計哲學 (Core Philosophy)

本系統遵循以下三大原則，確保產出不僅是聊天記錄，而是工程資產：

1.  **資料結構優先 (Data-First)**
    *   Agent 之間不傳遞自然語言，而是傳遞結構化的 `ArchitectureContext` (JSON/YAML)。
    *   確保上下文在多個 Agent 之間無損傳遞。

2.  **詢問式引導 (Socratic Method)**
    *   Agent 不直接給答案，而是透過《寶典》中的關鍵問題（如「這能賺錢嗎？」、「P99 延遲要求是多少？」）來逼問使用者。
    *   目標是填補資訊缺口，而非單純生成文字。

3.  **文件即代碼 (Docs as Code)**
    *   所有產出皆為 Markdown / Mermaid / OpenAPI / ADR 標準格式。
    *   產出物可直接進入 Git 版本控制，作為專案的 Single Source of Truth。

## 2. 系統流程圖 (System Flow)

這是一個線性但可迭代的 Pipeline，對應《架構師武僧寶典》的 **七步架構設計流程**。

```mermaid
graph TD
    User((用戶)) --> Orchestrator[🧙‍♂️ 總架構師 (Orchestrator)]
    
    Orchestrator --> Step1[🕵️ 需求分析師 Agent]
    Step1 --> |PRD & NFRs| Step2[🧠 領域建模師 Agent]
    Step2 --> |Domain Model| Step3[⚖️ 技術策略師 Agent]
    Step3 --> |Tech Stack & TCO| Step4[🏗️ 系統設計師 Agent]
    Step4 --> |C4 & API| Step5[🛡️ 風險評估師 Agent]
    Step5 --> |Risk Report| Step6[👨‍💻 技術主管 Agent]
    Step6 --> |Scaffold & Guidelines| Step7[🦅 演進守護者 Agent]
    Step7 --> |Observability| Docs[📚 最終架構白皮書]

    subgraph "Shared Context (State)"
        req[需求矩陣]
        const[約束條件]
        decisions[ADR 決策]
    end
```

## 3. 節點詳細設計 (Node Specifications)

每個節點都是一個專門的 Agent，擁有獨立的 System Prompt 和輸入/輸出規範。

### Node 1: 需求分析師 (The Analyst)
*   **對應章節**：第 2 章 (需求分析) & 第 9 章 (約束條件)
*   **核心任務**：將模糊願景轉化為量化指標。
*   **關鍵提問 (Input Strategy)**：
    *   **商業目標 (Why)**：這個系統如何賺錢或省錢？
    *   **約束條件 (Constraints)**：預算？Deadline？團隊技能樹？
    *   **NFR 量化**：拒絕形容詞（很快），要求數字（P99 < 200ms）。
    *   **預期規模**：DAU？資料增長量？
*   **輸出產物 (Output)**：
    *   📄 `01_PRD.md` (產品需求文件)
    *   📊 `02_NFR_Matrix.csv` (量化的非功能需求矩陣)
    *   📝 `03_Constraints.json` (時間、預算、人力限制)

### Node 2: 領域建模師 (The Modeler)
*   **對應章節**：第 3 章 (概念設計)
*   **核心任務**：劃分邊界，識別核心實體 (DDD)。
*   **關鍵提問 (Input Strategy)**：
    *   有哪些核心名詞？(Ubiquitous Language)
    *   業務流程的邊界在哪裡？誰負責什麼？
*   **輸出產物 (Output)**：
    *   📐 `04_Domain_Model.mermaid` (類別圖/ER圖)
    *   🗺️ `05_Context_Map.mermaid` (Bounded Contexts)

### Node 3: 技術策略師 (The Strategist)
*   **對應章節**：第 4 章 (技術選型)
*   **核心任務**：基於 ROI 和 TCO 選擇技術堆疊。
*   **關鍵提問 (Input Strategy)**：
    *   團隊最熟哪種語言？(Node/Python/Java/Go)
    *   是一致性重要 (ACID) 還是可用性重要 (BASE)？
    *   雲端預算上限？
*   **輸出產物 (Output)**：
    *   📋 `06_Tech_Stack.md` (前端/後端/DB/Infra 選型清單)
    *   💰 `07_TCO_Analysis.md` (預估成本分析)
    *   📜 `ADR-001-Tech-Stack-Selection.md` (架構決策記錄)

### Node 4: 系統設計師 (The Architect)
*   **對應章節**：第 6 章 (組件架構) & 第 7 章 (宏觀視角)
*   **核心任務**：繪製藍圖，定義接口。
*   **關鍵提問 (Input Strategy)**：
    *   是否需要微服務？(Check: 團隊規模 > 20人？)
    *   通訊模式？(REST vs gRPC vs Queue)
*   **輸出產物 (Output)**：
    *   🏗️ `08_System_Architecture_C4.mermaid` (C4 模型圖)
    *   🔌 `09_API_Spec_Draft.yaml` (OpenAPI 初稿)
    *   🌊 `10_Data_Flow.mermaid` (關鍵業務數據流圖)

### Node 5: 風險評估師 (The Validator)
*   **對應章節**：第 5 章 (品質屬性)
*   **核心任務**：攻擊架構，找出弱點。
*   **關鍵提問 (Input Strategy)**：
    *   如果 DB 掛了怎麼辦？
    *   如果流量突然暴增 10 倍？
    *   單點故障 (SPOF) 在哪裡？
*   **輸出產物 (Output)**：
    *   ⚠️ `11_Risk_Assessment.md` (風險與緩解措施)
    *   🔥 `12_Failure_Mode_Analysis.md` (故障模式分析)

### Node 6: 技術主管 (The Tech Lead)
*   **對應章節**：第 3 章 (實施指導)
*   **核心任務**：讓工程師能開工。
*   **關鍵提問 (Input Strategy)**：
    *   Git Flow 策略？
    *   Linter/Formatter 規範？
*   **輸出產物 (Output)**：
    *   📏 `13_Development_Guidelines.md` (開發規範)
    *   📂 `14_Project_Structure_Tree.txt` (建議的資料夾結構)
    *   🚀 `15_Scaffold_Command.sh` (初始化專案的指令建議)

### Node 7: 演進守護者 (The Guardian)
*   **對應章節**：第 5 章 (可觀察性) & 第 8 章 (進階模式)
*   **核心任務**：確保系統上線後可維護。
*   **輸出產物 (Output)**：
    *   🔭 `16_Observability_Plan.md` (Logging/Metrics/Tracing 策略)
    *   🛣️ `17_Evolution_Roadmap.md` (MVP 之後的演進路徑)

## 4. 資料結構定義 (Data Structure)

這是 Agent 之間傳遞的 `State` 定義 (Pseudo-Python Pydantic Model)：

```python
class ArchitectureContext:
    # 1. 需求 (Node 1)
    project_name: str
    business_goals: List[str]
    constraints: Dict[str, Any] # budget, timeline, team_skill
    nfr_metrics: Dict[str, str] # {"latency": "200ms", "availability": "99.9%"}
    
    # 2. 領域 (Node 2)
    entities: List[str]
    bounded_contexts: List[str]
    
    # 3. 技術決策 (Node 3)
    tech_stack: Dict[str, str] # {"backend": "Python/FastAPI", "db": "PostgreSQL"}
    architecture_style: str # "Monolith", "Microservices", "Serverless"
    adr_records: List[Dict]
    
    # 4. 設計 (Node 4)
    c4_containers: List[Dict]
    api_endpoints: List[str]
    
    # 5. 輸出檔案路徑
    generated_files: List[str]
```

## 5. 實作計畫 (Implementation Plan)

### Phase 1: MVP (核心三巨頭)
*   **目標**：最小可行性流程，產出基本的架構圖與 PRD。
*   **成員**：
    1.  **Orchestrator** (路由與狀態管理)
    2.  **Node 1: 需求分析師** (確保方向正確)
    3.  **Node 4: 系統設計師** (產出 C4 圖與技術棧，暫時吸收 Node 2/3 功能)

### Phase 2: Enhanced (決策與風險)
*   **目標**：增加深度與穩定性。
*   **成員**：
    *   加入 **Node 3: 技術策略師** (產出 ADR)
    *   加入 **Node 5: 風險評估師** (進行架構攻擊模擬)

### Phase 3: Complete (落地與運維)
*   **目標**：全流程覆蓋，直接生成代碼腳手架。
*   **成員**：
    *   加入 **Node 2, 6, 7** (補完領域建模、實施指導與演進規劃)
