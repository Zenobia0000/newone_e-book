# M7：架構模式與決策

## 系統設計與架構思維 — 第七模組（共 24 小時課程）

**時數：** 3 小時
**主線：** B（架構設計線）
**前置模組：** M6 系統設計思維（RESHADED 框架、擴展性模式、資料庫選型）

---

## 模組定位

M6 教你「怎麼分析一個系統要什麼」；M7 教你「怎麼做出有依據的架構決策並把它記下來」。

這是架構師工作的核心：面對真實的技術選擇、有限的時間與資源、相互矛盾的品質要求，做出可以被解釋、可以被追溯、可以被後人理解的決定。

本模組不講空泛的「最佳實踐」。所有模式都附帶「適用條件」與「代價」，因為沒有萬用架構，只有適合當下約束的架構。

**模組金句：**
> "Architecture is the set of decisions you wish you could get right early — and the art of making them explicit."

---

## 學習目標（7 項）

完成本模組後，學生能夠：

1. **用 TCO 框架做技術選型**，而非跟風技術熱度。
2. **識別並分析品質屬性的取捨**，理解性能、可用性、可擴展性、安全性、可維護性之間的張力。
3. **使用 C4 模型繪製系統架構圖**，在不同抽象層次清晰溝通架構。
4. **選擇適當的 API 風格**（REST / GraphQL / gRPC），說明各自的適用場景。
5. **解釋 CAP 定理與一致性模型**，知道在分散式系統中為什麼「強一致性 + 高可用」無法同時成立。
6. **運用微服務 vs. 單體架構決策框架**，避免過早或過晚的服務拆分。
7. **說明事件驅動架構、CQRS、Saga 模式的用途**，並撰寫 Architecture Decision Records（ADRs）記錄決策。

## 核心概念清單

學完本模組，以下概念你要能用自己的話解釋：

- TCO（Total Cost of Ownership）的四個成本維度
- 品質屬性取捨三角（鐵三角）
- C4 Model：Context / Container / Component / Code 四層
- REST、GraphQL、gRPC 的核心差異
- CAP 定理：Consistency、Availability、Partition Tolerance
- BASE vs. ACID：最終一致性的含義
- 一致性模型梯度：強一致 / 線性一致 / 最終一致
- 複製策略：主從複製、多主複製、無主複製
- 共識演算法：Raft / Paxos 的角色（不要求推導，要知道解決什麼問題）
- 單體架構（Monolith）的三種形態
- 微服務的邊界定義原則：DDD 的 Bounded Context
- 服務拆分的代價：分散式系統的八個謬誤
- 事件驅動架構（EDA）：事件作為系統整合媒介
- Event Sourcing：狀態 = 事件序列的重播
- CQRS：讀模型與寫模型分離
- Saga 模式：跨服務交易的補償邏輯
- ADR（Architecture Decision Record）的五段結構

### Slide 2 — 技術選型的陷阱：從眾 vs. TCO

**核心訊息：**
技術選型應該用 TCO（總擁有成本）分析，而不是跟風熱門技術。

**談話要點：**

**TCO 的四個成本維度：**

1. **採購 / 初始成本**：授權費、雲端費用、初期開發工時。對開源技術來說，這通常最低，也最容易被誤認為「免費」。

2. **學習與訓練成本**：團隊學習曲線有多陡？Kubernetes 很強大，但如果只有一個人懂，那是負債不是資產。規則：技術的稀缺程度 = 招募風險。

3. **運營與維護成本**：這個系統跑起來之後，每個月要花多少工時監控、調校、升級？自建 Kafka 叢集的運營成本，比用 Confluent Cloud 貴多少？

4. **替換成本**：如果三年後要換掉，代價是什麼？技術鎖定（vendor lock-in）是真實的風險。選擇開放標準 vs. 專有服務，要把替換成本算進去。

**反例（血淚教訓模式）：**
- 「MongoDB 不需要 schema，超方便！」→ 三年後資料品質災難，移轉成本極高。
- 「Kubernetes 是業界標準！」→ 三個人的新創，DevOps 工程師只有一人，cluster 崩潰沒人能修。
- 「Rust 效能超好！」→ 找不到會 Rust 的工程師，功能開發延誤六個月。

**選型決策矩陣：**
```
             低 TCO                高 TCO
好技術   [目標象限]              [要評估是否值得]
壞技術   [炒作受害者]            [應該避免]
```

**視覺建議：**
兩欄對比：左欄「從眾選型流程」（看 Hacker News → 覺得很酷 → 採用），右欄「TCO 選型流程」（四維度分析表格）。底部加一個實際案例對比表（PostgreSQL vs. MongoDB for OLTP）。

**銜接語：**
「選完技術後，下一個挑戰是：你要同時滿足哪些品質要求？這裡有個根本性的問題。」

### Slide 3 — 品質屬性取捨三角

**核心訊息：**
系統品質屬性之間存在根本性的取捨，架構設計的本質是在約束下做出明確的取捨選擇。

**談話要點：**

**五大品質屬性（Quality Attributes）：**

1. **性能（Performance）**：延遲（Latency）和吞吐量（Throughput）。典型目標：P99 < 200ms，10,000 QPS。
2. **可用性（Availability）**：系統在預期時間內正常運作的比例。99.9%（三個九）= 每年停機 8.7 小時；99.99%（四個九）= 每年停機 52 分鐘。
3. **可擴展性（Scalability）**：當負載增加時，系統能否透過增加資源線性擴展？水平擴展（加機器）vs. 垂直擴展（加規格）。
4. **安全性（Security）**：機密性（Confidentiality）、完整性（Integrity）、可用性（Availability）—— CIA 三角。
5. **可維護性（Maintainability）**：程式碼和系統能被理解、修改、測試的難易度。MTTR（平均修復時間）是關鍵指標。

**核心取捨關係：**

```
          [性能]
           /\
          /  \
         /    \
[可用性] -------- [可擴展性]
         \    /
          \  /
           \/
      [可維護性]
```

典型取捨：
- 性能 vs. 安全性：加密每次都有延遲代價
- 可用性 vs. 一致性：（這是 CAP 定理的預告）
- 可擴展性 vs. 可維護性：微服務擴展好但複雜度高
- 性能 vs. 可維護性：過度優化的程式碼難以理解

**實際案例 — 電商訂單系統：**
- 強一致性（下單後庫存立即更新）→ 犧牲可用性（資料庫鎖定）
- 高可用性（99.99%）→ 最終一致性（庫存可能短暫不準）
- 低延遲（< 100ms）→ 犧牲部分一致性（快取可能過時）

**關鍵教訓：** 架構師不是「讓每個品質屬性都最大化」，而是「在給定的業務約束下，明確選擇犧牲哪一個」。

**視覺建議：**
以五邊形雷達圖呈現，三個不同系統（電商、醫療記錄、遊戲排行榜）各有不同的品質屬性輪廓，直觀呈現取捨差異。

**銜接語：**
「知道要什麼之後，怎麼跟別人溝通你的架構？這就是 C4 模型的用途。」

### Slide 4 — C4 模型：四層說清系統

**核心訊息：**
C4 模型提供四個抽象層次，讓不同受眾（CTO、開發者、DevOps）都能從適合自己的層次理解系統。

**談話要點：**

**四個抽象層次：**

**Level 1 — System Context（系統情境圖）**
- 受眾：所有人，包括非技術利害關係人
- 內容：你的系統（一個黑盒子）與外部使用者、外部系統的關係
- 問題：「這個系統是做什麼的？誰用它？它跟哪些外部系統互動？」
- 例：電商系統 ← 顧客、賣家；→ 支付閘道、物流 API、Email 服務

**Level 2 — Container（容器圖）**
- 受眾：開發團隊、架構師
- 內容：系統由哪些可部署的單元組成（Web App、API、資料庫、Message Queue）
- 注意：這裡的「容器」不是 Docker，是「可獨立部署的執行環境」
- 問題：「技術棧是什麼？資料怎麼在各容器之間流動？」

**Level 3 — Component（元件圖）**
- 受眾：開發者
- 內容：某個容器內部的主要模組（Controller、Service、Repository）
- 問題：「這個服務的內部結構是什麼？哪些模組負責什麼職責？」

**Level 4 — Code（程式碼圖）**
- 受眾：需要深入實作的開發者
- 內容：類別圖、方法呼叫關係
- 實務上：這層通常由 IDE 自動生成，不需要手動維護

**為什麼 C4 比 UML 更實用：**
- 不需要學習複雜的符號系統
- 每一層圖對應明確的問題和受眾
- 容易維護（只更新需要的層次）

**視覺建議：**
四張並排的小圖，從 Level 1 到 Level 4 逐步放大，箭頭顯示「鑽入」關係。以電商系統為例，每層都有具體的盒子和連線。

**銜接語：**
「畫完架構圖，下一個具體決策：系統之間、服務之間怎麼溝通？API 怎麼設計？」

### Slide 5 — API 風格選擇：REST / GraphQL / gRPC

**核心訊息：**
REST、GraphQL、gRPC 各有適用場景，沒有哪個永遠更好。選擇取決於通訊模式、效能需求和客戶端多樣性。

**談話要點：**

**REST（Representational State Transfer）**
- 核心概念：資源（Resource）+ HTTP 動詞（GET / POST / PUT / DELETE）
- 優點：無狀態、可快取、廣泛的工具鏈支援、容易被人類閱讀
- 缺點：Over-fetching（回傳太多資料）、Under-fetching（需要多次請求）、缺乏強型別
- 適合：公開 API、Web 應用後端、需要廣泛相容性的場景
- 最佳實踐：版本化（/api/v1/），HATEOAS，OpenAPI 文件

**GraphQL**
- 核心概念：客戶端指定「我需要什麼欄位」，伺服器按需回傳
- 優點：解決 Over/Under-fetching、強型別 Schema、單一端點
- 缺點：N+1 查詢問題（需要 DataLoader）、快取困難、複雜查詢可能拖垮後端
- 適合：複雜前端需求（多 UI 共用同一 API）、行動應用（頻寬敏感）、BFF（Backend for Frontend）模式
- 典型誤用：用 GraphQL 做內部服務間通訊（gRPC 更適合）

**gRPC（Google Remote Procedure Call）**
- 核心概念：用 Protocol Buffers 定義服務介面，生成強型別客戶端/伺服端程式碼
- 優點：高效率（二進位序列化）、強型別、支援雙向串流、自動生成客戶端程式碼
- 缺點：瀏覽器支援有限（需要 grpc-web）、Protocol Buffers 不如 JSON 易讀、除錯工具較少
- 適合：微服務間通訊、效能敏感場景、多語言服務（強型別合約）、串流場景

**決策流程：**
```
客戶端多樣（Web + Mobile + 第三方）？
  → YES：REST（公開 API）或 GraphQL（複雜前端需求）
  → NO：是否是服務間通訊？
       → YES：gRPC（效能優先）
       → NO：REST（最通用）
```

**視覺建議：**
三欄比較表，欄位包含：序列化格式、型別安全、瀏覽器支援、適用場景、缺點。底部加決策流程圖。

**銜接語：**
「API 決定了系統如何溝通。但在分散式系統中，溝通本身就帶來一個根本性的問題：資料一致性。」

### Slide 6 — 分散式系統的第一課：CAP 定理

**核心訊息：**
在發生網路分區（Partition）時，分散式系統必須在一致性（Consistency）和可用性（Availability）之間做出選擇——不可能兩者都保全。

**談話要點：**

**三個屬性的定義：**

- **一致性（Consistency）**：所有節點在同一時刻看到相同的資料。更精確地說：每次讀取要麼返回最新寫入的值，要麼返回錯誤。
- **可用性（Availability）**：每個請求都能收到回應（不管這個回應是否包含最新資料）。
- **分區容忍性（Partition Tolerance）**：即使網路中斷導致節點無法通訊，系統仍能繼續運作。

**關鍵洞察：**
在真實的分散式系統中，網路分區是不可避免的（網路會斷）。因此，你沒有能力「捨棄 P」。真正的選擇是：**當分區發生時，你要 C 還是要 A？**

**CP 系統（選 C，犧牲 A）：**
- 例：HBase、Zookeeper、etcd
- 行為：網路分區時，系統拒絕服務（返回錯誤），確保資料一致
- 適合：金融交易、分散式鎖、配置管理——「資料錯誤比服務中斷更不可接受」

**AP 系統（選 A，犧牲 C）：**
- 例：Cassandra、DynamoDB、CouchDB
- 行為：網路分區時，系統繼續服務，但不同節點可能返回不同版本的資料
- 適合：社群媒體、購物車、DNS——「服務中斷比資料暫時不一致更不可接受」

**ACID vs. BASE：**
- ACID（關聯式資料庫）：Atomicity、Consistency、Isolation、Durability → 強一致
- BASE（NoSQL 傾向）：Basically Available、Soft State、Eventually Consistent → 最終一致

**實際案例解析：**
Amazon 的購物車選擇 AP：當網路故障時，寧可讓兩個節點各自接受「加入購物車」操作，最後合併（可能有重複）。因為「購物車服務無法使用」比「購物車有重複商品」造成更大的業務損失。

**常見誤解澄清：**
CAP 定理描述的是「最壞情況下的選擇」，不是說 CP 系統平常不可用，或 AP 系統平常不一致。正常網路條件下，兩者都能表現良好。

**視覺建議：**
三角形圖：三個頂點 C、A、P，中間標示「三選二」的三條邊。旁邊用實際資料庫品牌標示在各自的位置（Zookeeper → CP，Cassandra → AP，MySQL 單機 → CA 但無法分散式）。

**銜接語：**
「CAP 定理告訴我們選擇的代價，但實際系統中，一致性不是非黑即白的——它有梯度。」

### Slide 7 — 一致性模型梯度與實務選擇

**核心訊息：**
一致性不是布林值，而是一個光譜。了解不同一致性模型的語意，才能選擇符合業務需求的系統。

**談話要點：**

**一致性模型從強到弱：**

**1. 線性一致性（Linearizability / Strong Consistency）**
- 語意：一旦寫入成功，所有後續讀取都看到最新值
- 類比：你存了 100 元，馬上去另一個 ATM 提款，餘額一定反映那 100 元
- 代價：延遲高，需要跨節點協調
- 用途：金融帳戶、分散式鎖

**2. 順序一致性（Sequential Consistency）**
- 語意：所有節點看到的操作順序一致，但不要求與實際發生時間完全對應
- 比線性一致性寬鬆，但仍保留「全序」

**3. 因果一致性（Causal Consistency）**
- 語意：有因果關係的操作（A 在 B 之前）在所有節點以相同順序看到。無因果關係的操作可以任意順序
- 類比：你在社群媒體發文，然後評論——評論一定在貼文之後出現
- 比順序一致性更容易實作，且直覺上合理

**4. 最終一致性（Eventual Consistency）**
- 語意：如果沒有新的更新，最終所有節點會收斂到相同值。但「最終」可能是幾毫秒到幾分鐘
- 代價：讀取可能返回舊值（stale read）
- 用途：DNS、購物車、社群媒體 feed

**5. 讀寫一致性（Read-Your-Own-Writes Consistency）**
- 語意：你寫入後，你自己的讀取一定看到最新值。其他人可能看到舊值
- 實務常見：很多應用最低需要這個保證

**複製策略選型：**

| 策略 | 一致性 | 可用性 | 延遲 | 適用場景 |
|------|--------|--------|------|---------|
| 單主同步複製 | 強 | 中 | 高 | 金融交易 |
| 單主非同步複製 | 弱 | 高 | 低 | 讀多寫少應用 |
| 多主複製 | 最終 | 極高 | 低 | 多資料中心寫入 |
| 無主複製（Quorum） | 可調 | 高 | 中 | Cassandra 模式 |

**Quorum 讀寫的直覺：**
若 n 個副本，寫入 w 個確認，讀取 r 個副本：
只要 `w + r > n`，就能保證讀到最新寫入。
- n=3, w=2, r=2：「三個副本，寫入需要 2 確認，讀取查詢 2 個，必有交集」

**視覺建議：**
垂直梯度條：從上（強一致）到下（弱一致），每個層級標注技術名稱、代價、典型應用。旁邊用時序圖說明「線性一致 vs. 最終一致」的讀取行為差異。

**銜接語：**
「了解了一致性的取捨，我們來面對另一個最常被問錯的架構問題：要不要拆微服務？」

### Slide 8 — 單體 vs. 微服務：選錯的代價

**核心訊息：**
微服務不是升級版的單體架構，它解決了特定的規模問題，但引入了分散式系統的全部複雜性。大多數系統應該從「模組化單體」開始。

**談話要點：**

**單體架構的三種形態：**

1. **大泥球（Big Ball of Mud）**：沒有明確模組邊界，任何程式碼可以呼叫任何程式碼，修改一個地方到處爆炸。這不是好的單體，這是失敗的單體。

2. **模組化單體（Modular Monolith）**：在一個部署單元內，有清晰的模組邊界（明確的 API，不直接存取其他模組的資料庫）。改一個模組不影響其他模組。這是大多數新系統的正確起點。

3. **微核心架構（Microkernel）**：核心系統 + 外掛（Plugin）機制，適合工具類產品（IDE、CMS）。

**微服務的真實定義：**
微服務不是「很小的服務」，而是「有自己的資料儲存、可獨立部署、由獨立團隊擁有的服務」。

真正的微服務邊界原則：**DDD 的 Bounded Context**。每個服務都是一個業務能力的自給自足實現。

**分散式系統的八個謬誤（Fallacies of Distributed Computing）：**
1. 網路是可靠的
2. 延遲為零
3. 頻寬是無限的
4. 網路是安全的
5. 拓撲不會改變
6. 只有一個管理員
7. 傳輸成本為零
8. 網路是同質的

**選擇微服務的前提條件（全部要滿足）：**
- 系統的不同部分有**真實存在的不同擴展需求**（訂單模組需要 100 台機器，評論模組只需要 2 台）
- 團隊規模大到**康威定律**生效（多個獨立團隊，避免跨團隊協調成本）
- 有能力建設**微服務基礎設施**：服務發現、分散式追蹤、CI/CD、容器化
- 業務邊界**已經足夠清晰**（用模組化單體運行過一段時間）

**決策矩陣：**

| 情境 | 建議 |
|------|------|
| 新產品 / 新創 | 模組化單體，先把業務搞清楚 |
| 5-20 人工程團隊 | 模組化單體 + 少數獨立服務（如認證服務） |
| 50+ 人工程團隊，多個業務線 | 評估微服務（按 Bounded Context 拆分） |
| 單一服務效能瓶頸 | 先優化，再考慮抽出單一瓶頸服務 |

**反面教材：**
「微服務驅動的過度設計」：5 人新創，Day 1 就做 20 個微服務，每個功能要改 5 個 repo，部署需要 30 分鐘。問題不是微服務本身，是太早採用。

**視覺建議：**
左右對比圖：左側「模組化單體」（一個矩形，內部有清晰的模組邊界和箭頭）；右側「微服務」（多個矩形，中間有網路、訊息佇列、服務發現等基礎設施，複雜性明顯）。底部加一個決策樹。

**銜接語：**
「拆成微服務後，服務之間怎麼協作？同步 HTTP 呼叫只是其中一種方式，事件驅動是另一種更鬆耦合的選擇。」

### Slide 9 — 事件驅動架構：讓系統解耦

**核心訊息：**
事件驅動架構（EDA）用「事件發生了」取代「請你幫我做某事」，讓服務之間的耦合降到最低。

**談話要點：**

**事件的本質：**
事件是「已經發生的事實」的不可變記錄。`OrderPlaced`、`PaymentCompleted`、`InventoryReserved` 都是事件。

事件 vs. 命令（Command）：
- 命令：「請你做某事」（CreateOrder）— 發送者期待特定接收者執行特定動作，有直接耦合
- 事件：「某事已發生」（OrderPlaced）— 發送者不關心誰在聽、誰在處理

**EDA 的三種模式：**

1. **Event Notification（事件通知）**
   - 事件很小，只通知「有事情發生」
   - 接收方需要自己查詢詳細資訊
   - 優點：最低耦合；缺點：接收方需要額外查詢

2. **Event-Carried State Transfer（事件攜帶狀態轉移）**
   - 事件包含足夠的資訊，接收方不需要回頭查詢
   - `OrderPlaced { orderId, userId, items, totalPrice, timestamp }`
   - 優點：接收方自給自足；缺點：事件大，可能有資料重複

3. **Event Sourcing（事件溯源）**
   - 事件不只是通知，它就是唯一的資料來源（詳見 Slide 10）

**訊息中介軟體選型：**

| 工具 | 特性 | 適用場景 |
|------|------|---------|
| Apache Kafka | 高吞吐量、事件持久化、消費者群組、回放能力 | 事件溯源、資料串流管線 |
| RabbitMQ | 靈活路由、多種交換模式、AMQP 協定 | 工作佇列、RPC-over-messaging |
| AWS SNS/SQS | 全託管、與 AWS 生態深度整合 | 雲端原生應用 |
| Redis Pub/Sub | 超低延遲，但不持久化 | 即時通知（可接受丟失） |

**EDA 的代價（不要忽略）：**
- 除錯困難：事件流難以追蹤，需要分散式追蹤工具（Jaeger、Zipkin）
- 最終一致性：事件非同步處理，不能保證即時一致
- 事件演化：事件 Schema 變更需要向後兼容，這比 API 版本化更難
- 冪等性（Idempotency）：事件可能被重複投遞，消費者必須能安全地處理重複事件

**視覺建議：**
三個服務圖（Order Service、Payment Service、Inventory Service）透過 Kafka 解耦。左側顯示「同步呼叫」版本（三個服務直接連線），右側顯示「事件驅動」版本（全部透過 Kafka 中介）。對比箭頭密度。

**銜接語：**
「事件驅動帶來解耦，但如果我們把事件作為資料的唯一儲存媒介，會發生什麼？這就是 Event Sourcing 和 CQRS 的核心思想。」

### Slide 10 — CQRS + Event Sourcing：讀寫分離的極致

**核心訊息：**
CQRS 分離讀寫模型以優化各自的需求；Event Sourcing 用事件序列作為系統的真相來源，兩者常結合使用但各自獨立。

**談話要點：**

**CQRS（Command Query Responsibility Segregation）**

傳統架構：用同一個資料模型處理讀取（查詢）和寫入（命令），但這兩種操作的需求往往相反：
- 寫入需要：正規化（避免資料重複）、交易保證、一致性約束
- 讀取需要：效能（快速回應）、彈性（按不同維度組合資料）、可快取

CQRS 的解法：
- **Command Side（寫側）**：負責處理業務邏輯和狀態變更，更新規範化的寫入模型
- **Query Side（讀側）**：維護一個或多個針對查詢優化的「讀模型」（Materialized View）

同步方式：寫側發出事件，讀側訂閱事件並更新讀模型。兩側最終一致。

**好處：**
- 讀模型可以任意優化（反正規化、搜尋索引、圖資料庫）
- 寫模型和讀模型可以獨立擴展
- 複雜查詢不影響寫入效能

**代價：**
- 系統複雜度顯著增加
- 需要處理讀側的最終一致性（使用者剛寫入，立刻讀取可能看到舊資料）
- 不適合簡單 CRUD 應用

---

**Event Sourcing（事件溯源）**

傳統方式：資料庫儲存「當前狀態」（帳戶餘額 = 1000）。
Event Sourcing 方式：資料庫儲存「狀態變更事件」（存款 500、提款 200、轉帳 300）。當前狀態 = 重播所有事件。

**核心洞察：** 事件序列是唯一的真相（Source of Truth）。當前狀態只是事件的「投影（Projection）」。

**帶來的能力：**
- **完整稽核日誌**：每個狀態變更都有完整記錄，無法修改歷史
- **時間旅行（Time Travel）**：重播到任意時間點的狀態（「三個月前的帳戶餘額是多少？」）
- **事件回放**：修復 bug 後，可以用正確邏輯重新處理歷史事件
- **多種投影**：同一批事件可以產生不同維度的讀模型

**代價：**
- 讀取當前狀態需要重播事件（用 Snapshot 優化）
- 事件 Schema 的演化是永久性的（不能修改歷史）
- 思維模式轉換困難

**CQRS + Event Sourcing 的典型組合：**

```
[使用者] → Command → [寫側：驗證並發出事件]
                              ↓
                     [Event Store（Kafka/EventStoreDB）]
                              ↓
              [讀側投影處理器：更新讀模型]
                              ↓
[使用者] ← Query ←  [讀模型：資料庫/搜尋引擎/快取]
```

**適用場景：** 金融系統（需要完整稽核）、電商訂單（需要狀態機追蹤）、需要高效能讀取且寫入複雜的系統。

**視覺建議：**
左側傳統 CRUD 流程（單一資料庫），右側 CQRS + Event Sourcing 完整流程圖（Command → Event Store → 多個讀模型投影器 → 多種讀模型資料庫）。用顏色區分寫側（藍）和讀側（綠）。

**銜接語：**
「但還有一個問題沒解決：在微服務世界裡，跨多個服務的交易怎麼辦？資料庫 ACID 不跨服務邊界。」

### Slide 11 — Saga 模式：跨服務交易怎麼辦

**核心訊息：**
Saga 模式用「一系列本地交易 + 補償操作」取代跨服務的分散式交易，在最終一致性的前提下實現跨服務業務完整性。

**談話要點：**

**問題背景：**
在微服務架構中，每個服務有自己的資料庫。「下訂單」這個操作涉及：
1. 訂單服務：建立訂單記錄
2. 庫存服務：扣減庫存
3. 支付服務：扣款
4. 物流服務：建立出貨單

如果步驟 3 失敗，如何回滾步驟 1 和 2？跨服務沒有共用的 ACID 交易。

**為什麼不用分散式交易（2PC / XA）：**
- 效能差：需要全局鎖，所有參與者都被鎖住直到交易完成
- 可用性差：任何一個參與者失敗，整個交易被卡住
- 實務上，大多數現代系統不採用 2PC

**Saga 的核心思想：**
把一個跨服務的業務流程拆成一系列「本地交易（Local Transaction）」，每個本地交易在自己的服務內完成後，發出事件觸發下一步。如果某步失敗，觸發之前所有步驟的「補償操作（Compensating Transaction）」。

**補償操作 ≠ 回滾：**
補償是新的業務操作，語意上「撤銷」之前的效果。例如：付款失敗，補償是退款；庫存扣減的補償是庫存歸還。

**兩種 Saga 實作方式：**

**1. 協調式 Saga（Choreography-based）**
- 沒有中央協調者，每個服務訂閱事件、處理後發出下一個事件
- 優點：服務完全解耦，沒有單點故障
- 缺點：流程邏輯分散，難以追蹤整體狀態；當服務增多，事件依賴關係難以管理

```
OrderService → [OrderCreated] → InventoryService → [InventoryReserved] → PaymentService
                                                                             ↓ [PaymentFailed]
InventoryService ← [InventoryReleased] ← OrderService ← [OrderCancelled] ←┘
```

**2. 編排式 Saga（Orchestration-based）**
- 一個中央 Saga Orchestrator 負責協調所有步驟
- 優點：流程邏輯集中，容易追蹤、測試和修改
- 缺點：Orchestrator 是中央依賴，需要確保其高可用

```
[Saga Orchestrator]
    → 呼叫 OrderService.CreateOrder()
    → 呼叫 InventoryService.ReserveInventory()
    → 呼叫 PaymentService.ProcessPayment() [失敗]
    → 呼叫 InventoryService.ReleaseInventory() [補償]
    → 呼叫 OrderService.CancelOrder() [補償]
```

**選擇建議：**
- 簡單流程（3 步以內）：協調式
- 複雜流程（需要分支、重試、超時）：編排式

**實作工具：**
- AWS Step Functions（托管 Orchestrator）
- Temporal.io（開源工作流引擎）
- Apache Camel（複雜整合路由）

**視覺建議：**
上方：「失敗場景」流程圖，顯示正常路徑（綠色）和補償路徑（紅色），清晰標示每一步的補償操作。下方：協調式 vs. 編排式的架構對比圖。

**銜接語：**
「我們討論了這麼多模式和決策。但架構決策最大的問題不是做決定，而是三年後沒有人記得為什麼做這個決定。ADR 解決這個問題。」

### Slide 12 — ADR：把決策理由寫下來

**核心訊息：**
Architecture Decision Records（ADRs）把架構決策的「為什麼」變成可搜尋、可追溯的文字記錄，讓未來的團隊成員不必重複踩過同樣的坑。

**談話要點：**

**為什麼需要 ADR：**
典型場景：新工程師加入，問「為什麼我們用 Kafka 不用 RabbitMQ？」。沒有 ADR 的回答：「呃，當初的架構師這麼決定的，他已經離職了。」有 ADR 的回答：「ADR-007，兩年前因為需要事件回放能力，Kafka 的持久化日誌是關鍵需求。」

ADR 的核心價值：
1. 防止「集體失憶」——決策理由不要只在人的腦子裡
2. 避免「輪迴式決策」——防止三年後重新討論同樣已討論過的問題
3. 幫助新人理解系統——不只知道「是什麼」，還知道「為什麼」
4. 迫使決策者把想法寫清楚——寫的過程本身就是澄清思路的過程

**ADR 的五段結構（Nygard 格式）：**

```markdown
# ADR-[編號]：[決策標題]

## 日期
YYYY-MM-DD

## 狀態
[Proposed / Accepted / Deprecated / Superseded by ADR-XXX]

## 情境（Context）
描述面臨的問題是什麼，為什麼需要做這個決定。
包含約束條件（技術、時間、團隊能力）和背景資訊。

## 決策（Decision）
我們決定做什麼。用主動語氣的完整句子。
「我們將使用 PostgreSQL 作為主要資料庫。」

## 後果（Consequences）
這個決定帶來什麼結果：
- 好的後果：[列出正面影響]
- 壞的後果：[誠實列出代價和限制]
- 需要後續行動的項目：[例如：需要設定備份策略]
```

**ADR 的最佳實踐：**
- 存放在版本控制中（和程式碼放在一起，`docs/adr/` 資料夾）
- 編號連續，永不刪除（過期的標記為 Deprecated，不刪除）
- 每個 ADR 只記錄一個決策
- 保持簡短（一頁以內）
- 用「當時的約束」來記錄，不要事後修改

**ADR 的常見錯誤：**
- 只記錄決定，不記錄被考慮過的替代方案
- 事後才寫，失去了「記錄當時思考過程」的價值
- 寫成技術文件，而非決策記錄（ADR 解釋「為什麼」，不是「怎麼做」）

**工具與格式：**
- `adr-tools`：命令列工具，自動管理編號
- MADR（Markdown Architecture Decision Records）：更詳細的格式
- Confluence / Notion：若團隊已有 Wiki，也可整合

**視覺建議：**
一個完整的 ADR 範本截圖（選擇資料庫的 ADR 範例），每個段落用顏色標注：灰色（情境）、藍色（決策）、綠色（好的後果）、橘色（壞的後果）。旁邊加一個文件夾結構，顯示 `docs/adr/` 如何組織。

**銜接語：**
「理論都講完了。現在讓我們把這些東西用起來——設計一個電商系統，並寫幾份 ADR。」

### 練習一：電商系統架構設計（90 分鐘）

**練習目標：**
綜合應用本模組的所有概念，設計一個電商平台的核心架構，並做出有根據的技術決策。

**情境設定：**

你是一家剛成立的電商新創的首席架構師。以下是你的約束條件：

- **團隊**：8 位工程師（4 後端、2 前端、1 DevOps、1 你）
- **時間線**：MVP 要在 3 個月內上線
- **預期規模**：初期 1,000 日活，12 個月後預期 50,000 日活
- **技術偏好**：團隊熟悉 Python（Django）和 PostgreSQL，有人用過 Redis，沒有人用過 Kafka 或微服務
- **預算**：有限，優先使用 AWS 管理服務

**系統功能範圍：**
- 使用者註冊 / 登入
- 商品瀏覽和搜尋
- 購物車
- 下訂單（包含庫存確認和支付）
- 訂單狀態追蹤
- 商家後台（上架商品、查看訂單）

**任務一：C4 Level 1 + Level 2 圖（30 分鐘）**

1. 畫出 System Context 圖：識別出所有外部使用者（顧客、商家、管理員）和外部系統（支付閘道、Email / SMS 通知、物流 API）

### 練習二：架構評審——找出問題（30 分鐘，可選）

**情境設定：**

以下是一個初創公司的架構決策記錄，請找出其中的問題並提出改進建議：

**公司背景：**
3 個月前成立，2 位工程師，正在開發一個食譜分享社群平台，預期初期使用者 500 人。

**他們的架構決策清單：**
1. 使用 15 個微服務（使用者服務、食譜服務、評論服務、按讚服務、標籤服務、搜尋服務、通知服務、媒體上傳服務、認證服務、推薦服務、食材服務、分類服務、關注服務、動態消息服務、報告服務）
2. 每個服務都有獨立的 PostgreSQL 資料庫
3. 使用 Kubernetes 管理所有服務
4. 跨服務交易使用 2PC（Two-Phase Commit）
5. API 全部用 GraphQL
6. 用 Kafka 做服務間通訊
7. 用 Event Sourcing 儲存所有狀態

**任務：**
1. 找出至少 3 個明顯的架構問題，說明「為什麼這是問題」
2. 針對這個團隊的真實約束條件，提出更合理的架構方向
3. 如果你是這個公司的技術顧問，你的第一句話會說什麼？

**預期答案方向（講師參考）：**
- 2 人團隊不應該有 15 個微服務，運維複雜度會摧毀開發速度
- Kubernetes 對 2 人團隊是過重的基礎設施
- 2PC 在現代分散式系統中幾乎不被採用
- Event Sourcing 不應該用在所有狀態，只用在需要稽核日誌的核心業務
- 正確方向：模組化單體 + PostgreSQL + Redis 快取 + 簡單佇列

### 常見學生疑問與回答

**Q：CAP 定理說要在 C 和 A 之間選一個，但 AWS RDS 說它又一致又高可用，這是矛盾的嗎？**

A：不矛盾。CAP 定理講的是「當網路分區發生時」的選擇。AWS RDS Multi-AZ 在正常情況下兩個都給你；但在 AZ 之間網路中斷時，它選擇了 C（主節點的寫入被接受，備節點在同步完成前不提升為主）。你通常遇到的不是 CAP 極端情況，而是日常的延遲 vs. 一致性取捨。

**Q：微服務一定比單體差嗎？**

A：不是「差」，是「代價不同」。微服務解決了「大型組織、多個獨立團隊、不同擴展需求」的問題。如果你沒有這些問題，你就在沒有問題的地方引入了分散式系統的複雜性。正確問題是：「我的真實瓶頸是什麼？微服務解決了這個瓶頸嗎？」

**Q：CQRS 是不是一定要搭配 Event Sourcing？**

A：不是。CQRS 只是說「讀模型和寫模型分開」，你可以用任何方式同步兩側。Event Sourcing 是說「用事件序列作為唯一的真相來源」。它們很常一起用，但各自是獨立的模式。一個系統可以只用 CQRS（用資料庫觸發器或應用層同步讀模型），不用 Event Sourcing。

**Q：ADR 應該在決策之前還是之後寫？**

A：理想是在決策過程中或決策時寫。「情境」段落在決策前就能寫，「決策」和「後果」在做出決定後補完。如果你在做決定時就強迫自己寫 ADR，那個「後果：壞的」段落會迫使你誠實面對代價，這本身就是很有價值的思考過程。

**Q：Saga 模式的補償操作是否等同於資料庫回滾？**

A：不是。資料庫回滾是「好像從來沒發生過」，而 Saga 的補償是「新的業務操作，語意上撤銷」。例如：「退款」不是「從來沒有付款」，而是「付了，然後又退了」。這對稽核日誌來說很重要——所有動作都留有記錄。

### 教學難點與建議

**難點 1：CAP 定理**
學生常誤解為「平常的系統設計」問題，但 CAP 只在網路分區（P）發生時才有意義。建議用具體故障場景（資料中心之間的網路斷線）來說明，不要用抽象的「一致性 vs. 可用性」。

**難點 2：Event Sourcing 的思維轉換**
從「儲存當前狀態」轉換到「儲存事件序列」需要一個具體例子。銀行帳戶是最好的類比：帳戶餘額不是直接儲存的，而是所有交易的總和。學生一旦理解這個例子，其他場景就容易了。

**難點 3：Saga 的補償 vs. 回滾**
很多學生混淆兩者。建議用「退款流程」做類比，強調「補償是新的業務操作，不是時間倒退」。

**難點 4：微服務邊界**
學生常問「多小才算微服務」。答案是：以 DDD Bounded Context 為邊界，而不是以程式碼行數或 API 數量。一個服務可以有 50 個 API，只要它們都屬於同一個業務能力。

## 模組總結

**核心訊息回顧：**

1. 技術選型用 TCO，不用熱度——算清楚學習成本、運維成本、替換成本
2. 品質屬性有根本性的取捨——你的工作是明確選擇犧牲哪個，而不是假裝全都要
3. C4 模型讓不同受眾都能理解架構——不同層次的圖解決不同的溝通問題
4. API 風格取決於通訊場景——REST（公開 API）、GraphQL（複雜前端）、gRPC（服務間高效通訊）
5. CAP 定理是分散式系統的基本法則——網路分區時，一致性和可用性只能選一個
6. 微服務解決的是「組織規模」問題，不是「技術複雜度」問題——大多數系統應從模組化單體開始
7. 事件驅動、CQRS、Saga 是工具，不是目標——用在真實問題存在的地方
8. ADR 是架構師最重要的輸出之一——決策會忘記，文件不會

**一句話帶走：**
> 好的架構不是選最炫的技術，而是在真實的約束下做出明確的取捨——然後把理由寫下來，讓三年後的你不必重新發明輪子。
