## 模組定位

這是「工程基礎線」的最後一塊拼圖。

M4 告訴你硬體長什麼樣子：CPU、GPU、RAM、Disk 是哪些零件、做什麼事。M5 要回答下一個問題：**誰在管這些零件？** 答案是作業系統。

當你跑 Python 程式時，你從來不直接碰 CPU 或 RAM。是 OS 幫你分配記憶體、排程執行時間、管理檔案、隔離程序。OS 是你的程式與硬體之間的隱形中間人。

這個模組從真實錯誤訊息出發，倒推 OS 原理：

- `MemoryError` / `CUDA out of memory` → 記憶體管理
- `Permission denied` → 檔案系統權限
- `"Works on my machine"` → 環境差異與容器化
- 訓練瓶頸、DataLoader 卡住 → Process/Thread 與 I/O 排程

**學完 M5，你能回答：** 一段 Python 訓練迴圈，從按下 Enter 到 GPU 開始算，OS 在背後做了哪些事？

### S2：Process 與 Thread——你的程式不是孤獨的

**版面類型：** 概念圖解（左圖右文）

**核心比喻：**

> Process 是一個工廠。Thread 是工廠裡的工人。
> 兩個工廠有獨立的土地、機器、倉庫。同一個工廠的工人共用廠房和倉庫。

**Process 生命週期圖（狀態機）：**

```
         fork()
           |
           v
   +-------+-------+
   |      NEW      |
   +-------+-------+
           |  OS 接受
           v
   +-------+-------+
   |     READY     | <─────────────────────┐
   +-------+-------+                       |
           |  CPU 排程                     |
           v                               |
   +-------+-------+   I/O 或等待資源      |
   |    RUNNING    | ─────────────────> +--+------+
   +-------+-------+                   | WAITING  |
           |  執行完畢                  +--+------+
           v                               |  I/O 完成
   +-------+-------+                       |
   |  TERMINATED   |   <───────────────────┘
   +---------------+
```

**Process vs Thread 對比表：**

| 維度 | Process | Thread |
|------|---------|--------|
| 記憶體空間 | 獨立（隔離） | 共用（同一 Process） |
| 建立成本 | 高（複製整個地址空間） | 低（只建執行堆疊） |
| 通訊方式 | IPC（Pipe / Socket / 共享記憶體） | 直接讀寫共享變數（需要加鎖） |
| 崩潰影響 | 不影響其他 Process | 可能拖垮整個 Process |
| Python 範例 | `multiprocessing.Process` | `threading.Thread` |

**DataLoader 的實際案例：**

```python
# 為什麼 num_workers > 0 時用 multiprocessing 而非 threading？
from torch.utils.data import DataLoader

# num_workers=4 → 開 4 個子程序（Process），不是 Thread
loader = DataLoader(dataset, batch_size=32, num_workers=4)

# 因為：
# 1. Python GIL 讓多 Thread 無法真正並行 CPU 計算
# 2. 每個 worker 需要獨立執行 Python 解釋器
# 3. 資料預處理是 CPU-bound，需要真正的多核並行
```

**關鍵結論：** Thread 共享記憶體所以通訊快，但 Python GIL 讓它無法真正並行。需要真正並行時，用 Process。

### S3：Python GIL 的真相——為什麼多執行緒快不起來

**版面類型：** 深度解析（圖示 + 程式碼 + 對比）

**GIL 定義：**

> **Global Interpreter Lock（全域解釋器鎖）**
> CPython 的一把鎖，確保任何時刻只有一個 Thread 在執行 Python bytecode。

**為什麼要有 GIL？**

Python 的記憶體管理用 Reference Count。如果多個 Thread 同時修改同一物件的 ref count，會產生 race condition，導致物件提早被釋放或記憶體洩漏。GIL 是最簡單的解法：乾脆不讓兩個 Thread 同時跑。

**GIL 的實際影響：**

```
CPU-bound 任務（矩陣計算、JSON 解析）：
  Thread 1 拿到 GIL → 執行 → 釋放 GIL
  Thread 2 等待     → 拿到 GIL → 執行 → 釋放 GIL
  結果：2 個 Thread 比 1 個還慢（有切換開銷）

I/O-bound 任務（等網路、等硬碟）：
  Thread 1 開始 I/O → 釋放 GIL（I/O 時不需要執行 Python bytecode）
  Thread 2 趁機拿到 GIL → 做自己的事
  結果：多 Thread 有效，I/O 可以重疊
```

**三種並行方案的選擇矩陣：**

| 情境 | 推薦方案 | 原因 |
|------|---------|------|
| CPU-bound（數值計算、圖像處理） | `multiprocessing` | 繞過 GIL，真正多核 |
| I/O-bound（爬蟲、資料庫查詢） | `threading` 或 `asyncio` | GIL 在 I/O 等待時會釋放 |
| 非同步 I/O（高並發網路服務） | `asyncio` | 單執行緒事件迴圈，零切換開銷 |
| ML 訓練資料預處理 | `multiprocessing`（DataLoader） | CPU-bound 的影像解碼/增強 |

**GIL 的例外：** NumPy、PyTorch 的 C 擴展在執行期間會釋放 GIL，所以純 tensor 計算是真正並行的。GIL 只鎖 Python bytecode。

### S4：CPU 排程——時間不是你說了算

**版面類型：** 機制說明（時間軸圖 + 說明）

**核心問題：** 8 個程序同時想用 CPU，OS 怎麼決定誰先跑？

**三種主要排程演算法：**

**1. Round-Robin（循環輪替）**

```
時間槽（Time Quantum）= 10ms

程序 A: ████░░░░░░░░░░░░████░░░░
程序 B: ░░░░████░░░░░░░░░░░░████
程序 C: ░░░░░░░░████░░░░░░░░░░░░

每個程序輪流跑 10ms，然後被換下去
```

特性：公平、沒有餓死問題、回應時間可預測。缺點：Context Switch 有開銷。

**2. Priority Scheduling（優先權排程）**

```
高優先權（系統程序）：██████████████████
中優先權（互動程序）：░░░░░░████████
低優先權（背景程序）：░░░░░░░░░░░░░░░░████

問題：低優先權程序可能永遠等不到 CPU（Starvation）
解法：Aging — 等太久自動升高優先權
```

**3. Preemptive vs Non-preemptive**

| 類型 | 說明 | 現代 OS |
|------|------|--------|
| Non-preemptive | 程序自願放棄 CPU 才換人 | 舊式系統 |
| Preemptive | OS 強制奪回 CPU（時間到、更高優先權任務來了） | Linux、Windows、macOS |

**對 ML 訓練的啟示：**

```
你的訓練腳本在 OS 眼中只是一個普通的 Process。
當系統同時跑很多東西時（瀏覽器、Jupyter、後台服務），
你的訓練程序也會被搶走 CPU 時間。

解法：
  nice -n -20 python train.py   # 提高優先權（需要 root）
  taskset -c 0,1,2,3 python train.py  # 綁定特定 CPU 核心
```

**CPU Burst vs I/O Burst：**

```
大部分程式不是「一直在算」，而是「算一下，等一下，再算」

程序生命週期：
  CPU Burst → I/O Burst → CPU Burst → I/O Burst → ...

I/O-bound 程序（Web server、爬蟲）：短 CPU Burst，長 I/O Burst
CPU-bound 程序（訓練模型、影像處理）：長 CPU Burst，短 I/O Burst

OS 排程器需要區分這兩種，才能有效利用資源
```

### S5：記憶體管理——Virtual Memory 與 OOM 的兩種死法

**版面類型：** 對比分析（上半原理，下半兩種 OOM 對比）

**Virtual Memory 的核心概念：**

每個 Process 都以為自己有一整塊連續的記憶體（Virtual Address Space）。OS 的 Memory Management Unit（MMU）把虛擬地址翻譯成實際的實體記憶體地址。

```
Process A 的視角：               實際 RAM：
┌─────────────────┐             ┌─────────────────┐
│ 0x0000 - 程式碼  │  ──MMU──>  │ 實體頁面 #1042   │
│ 0x1000 - 堆疊    │            │ 實體頁面 #0237   │
│ 0x2000 - Heap   │            │ 實體頁面 #1891   │
│ ...             │            │ ...              │
└─────────────────┘             └─────────────────┘

Process B 也以為自己有 0x0000 開始的空間
但它的 0x0000 對應到不同的實體頁面 → 程序之間互相隔離
```

**Paging（分頁）機制：**

- 實體記憶體被切成固定大小的「頁框」（Page Frame），通常 4KB
- 虛擬記憶體被切成同樣大小的「頁」（Page）
- Page Table 記錄每個虛擬頁對應哪個實體頁框
- 如果一個頁不在 RAM 裡（在 Swap 或還沒載入），就會觸發 **Page Fault**，OS 把它從磁碟載入 RAM

**兩種 OOM 的根本差異：**

```
CPU RAM OOM（MemoryError）               GPU VRAM CUDA OOM
═══════════════════════════             ═══════════════════════════
位置：系統 RAM（由 OS 管理）              位置：GPU 專屬顯示記憶體（VRAM）
管理者：Linux OOM Killer                 管理者：CUDA 驅動 / PyTorch
觸發：process 使用超過可用 RAM            觸發：GPU tensor 超過 VRAM 容量
錯誤訊息：MemoryError                    錯誤訊息：RuntimeError: CUDA out of memory
OS 反應：殺掉記憶體用量最多的 Process     反應：拋出例外，GPU 狀態可能殘留
解法：減少資料量、分批處理、增加 Swap     解法：減小 batch size、gradient checkpointing
```

**Linux OOM Killer 的運作邏輯：**

```
當系統 RAM 耗盡時，Linux 不會直接崩潰。
OOM Killer 計算每個 Process 的「oom_score」：

oom_score 越高 = 越容易被殺
計算因素：記憶體用量、執行時間、是否是系統程序

# 查看某個程序的 oom_score
cat /proc/$(pgrep python)/oom_score

# 保護重要程序（-1000 = 不會被殺）
echo -1000 > /proc/$(pgrep sshd)/oom_score_adj
```

**GPU VRAM 管理實戰：**

```python
import torch

# 查看 GPU 記憶體使用
print(torch.cuda.memory_allocated() / 1024**3, "GB 已用")
print(torch.cuda.memory_reserved() / 1024**3, "GB 已保留")

# CUDA OOM 後，PyTorch 不會自動釋放快取
torch.cuda.empty_cache()  # 釋放 PyTorch 的記憶體快取

# 防止梯度佔用記憶體（推理時）
with torch.no_grad():
    output = model(input)

# Gradient Checkpointing：用計算換記憶體
from torch.utils.checkpoint import checkpoint
output = checkpoint(model_block, input)
```

### S6：Python 垃圾回收——你的物件什麼時候消失

**版面類型：** 底層機制（流程圖 + 程式碼驗證）

**兩層機制：**

**第一層：Reference Counting（引用計數）**

```python
import sys

a = [1, 2, 3]         # ref count = 1
b = a                 # ref count = 2
print(sys.getrefcount(a))  # 3（傳入 getrefcount 本身也算一次）

del b                 # ref count = 1
del a                 # ref count = 0 → 立即釋放記憶體
```

引用計數的問題：**循環引用**

```python
# 這兩個物件永遠不會被 Reference Counting 釋放
a = {}
b = {}
a['ref'] = b   # a 指向 b
b['ref'] = a   # b 指向 a

del a
del b
# a 和 b 的 ref count 都是 1，但沒有任何外部變數指向它們
# → 記憶體洩漏（如果沒有第二層）
```

**第二層：Cyclic GC（週期性垃圾回收）**

```
CPython 的分代回收（Generational GC）：

Generation 0（新物件）：頻繁掃描
Generation 1（存活過一次掃描）：較少掃描
Generation 2（長期存活物件）：很少掃描

邏輯：越老的物件越不可能是垃圾
Python 定期掃描 Gen 0，找出循環引用 → 一起釋放
```

```python
import gc

# 手動觸發垃圾回收
gc.collect()

# 查看三個世代的統計
print(gc.get_count())   # (gen0物件數, gen1物件數, gen2物件數)

# 在記憶體敏感場景，手動管理
gc.disable()   # 禁用 GC（需要手動呼叫 gc.collect()）
# ... 大量建立/刪除物件的程式碼 ...
gc.collect()
gc.enable()
```

**實踐建議：**

| 場景 | 建議 |
|------|------|
| 大型 DataFrame 不再需要 | `del df; gc.collect()` |
| 訓練迴圈中的中間 tensor | 用 `with torch.no_grad()` 避免梯度圖積累 |
| 長時間服務，擔心記憶體洩漏 | 定期用 `tracemalloc` 追蹤記憶體快照 |
| 大量物件的批次處理 | 用 generator 而非 list comprehension |

### S7：IPC——程序怎麼說話

**版面類型：** 三種機制比較（圖示 + 場景說明）

**背景：** Process 之間的記憶體是隔離的。要讓兩個 Process 交換資料，需要 IPC（Inter-Process Communication）。

**三種主要 IPC 機制：**

**1. Pipe（管線）**

```
Process A ──[寫端]──► [Pipe Buffer] ──[讀端]──► Process B
                        （OS Kernel）
```

```python
import multiprocessing as mp

parent_conn, child_conn = mp.Pipe()

def worker(conn):
    data = conn.recv()
    conn.send(data * 2)
    conn.close()

p = mp.Process(target=worker, args=(child_conn,))
p.start()

parent_conn.send(42)
print(parent_conn.recv())   # 84
p.join()
```

特性：簡單、單向或雙向、適合小量資料的串流傳遞。

**2. Shared Memory（共享記憶體）**

```
Process A ─┐
           ├──► [共享記憶體區塊] ◄──┤
Process B ─┘                       └─ Process C
```

```python
from multiprocessing import shared_memory
import numpy as np

# 建立共享記憶體區塊
shm = shared_memory.SharedMemory(create=True, size=1024*1024)

# Process A：寫入大型陣列
arr_a = np.ndarray((256, 256), dtype=np.float32, buffer=shm.buf)
arr_a[:] = some_image_data

# Process B：直接讀取（零複製）
arr_b = np.ndarray((256, 256), dtype=np.float32, buffer=shm.buf)
process_image(arr_b)   # 不需要複製資料

shm.close()
shm.unlink()   # 釋放共享記憶體
```

特性：效能最高（零複製）、適合大量資料。需要自行處理同步（Lock）。

**3. Socket（網路 Socket）**

```
Process A ──► [Socket] ──► [Network Stack] ──► [Socket] ──► Process B
                                                              （可以在不同機器）
```

特性：跨機器通訊、適合分散式系統。開銷最大（需要序列化 + 網路堆疊）。

**選擇矩陣：**

| 需求 | 推薦 IPC |
|------|---------|
| 父子程序串流傳遞資料 | Pipe |
| 多個 Worker 共享大型陣列（影像批次） | Shared Memory |
| 跨機器通訊 | Socket |
| 同機器的服務間通訊 | Unix Domain Socket 或 Shared Memory |

### S8：檔案系統——Inode、路徑、權限的真實含義

**版面類型：** 結構圖解（三個子主題分塊）

**Inode：檔案的身份證**

```
你以為「檔案名稱」就是檔案。其實不是。

目錄（Directory）                 Inode（真正的檔案）
┌─────────────────────┐          ┌────────────────────────┐
│ 名稱       Inode 號碼│          │ Inode #1042            │
│ ─────────────────── │          │ ─────────────────────  │
│ train.py   → #1042  │──────►   │ 擁有者: zenobia        │
│ model.pt   → #2891  │          │ 群組: staff            │
│ data/      → #0037  │          │ 權限: rwxr-xr-x        │
└─────────────────────┘          │ 大小: 2.4 KB           │
                                  │ 修改時間: 2026-04-12   │
                                  │ 資料區塊位址: [...]    │
                                  └────────────────────────┘

Inode 儲存「關於檔案的一切」，除了檔案名稱。
檔案名稱只是一個指向 Inode 的「連結」（Hard Link）。
```

**Path as Object（pathlib 的設計哲學）**

```python
from pathlib import Path

# 舊方式（字串拼接，跨平台有問題）
import os
path = os.path.join("data", "raw", "train.csv")   # 字串

# 新方式（Path 物件，跨平台安全）
path = Path("data") / "raw" / "train.csv"          # 物件

# Path 物件的優雅操作
path.exists()             # 檔案存在嗎？
path.is_file()            # 是檔案嗎？
path.stem                 # "train"（不含副檔名）
path.suffix               # ".csv"
path.parent               # Path("data/raw")
path.with_suffix(".parquet")  # 替換副檔名

# 讀取檔案（自動處理編碼）
content = path.read_text(encoding="utf-8")
```

**權限模型：`rwxr-xr-x` 的解讀**

```
rwxr-xr-x  zenobia  staff  2456  train.py
│││││││││
│││└┤└┤└┤
│││ │ │ └── Others（其他人）：r-x = 可讀、不可寫、可執行
│││ │ └──── Group（所屬群組）：r-x = 可讀、不可寫、可執行
```

### S10：容器化——namespace + cgroup = Docker 的靈魂

**版面類型：** 原理揭密（概念拆解 + 對比圖）

**問題根源：「在我電腦上能跑」**

```
你的 MacBook：
  Python 3.11
  numpy 1.26
  CUDA 不存在（CPU only）
  /usr/local/lib 裡有一堆舊版套件

生產伺服器：
  Python 3.9
  numpy 1.23（版本不同！）
  CUDA 11.8
  /usr/local/lib 的套件完全不同

結果：你的程式碼只在你電腦能跑。
```

**Docker 的解法：把環境打包進去**

```
Image（映像）= 作業系統 + 套件 + 程式碼 的靜態快照
Container（容器）= 從 Image 跑起來的執行實例

類比：
  Image ≈ 程式的安裝檔（.exe / .dmg）
  Container ≈ 跑起來的程式視窗
```

**Docker 的底層兩根支柱：**

**支柱 1：Linux Namespace（隔離）**

```
Namespace 類型       隔離什麼
────────────────    ──────────────────────────────
PID Namespace       容器裡的 PID 1 是 init，看不到 host 的其他程序
Network Namespace   容器有自己的網路介面、IP、Port 空間
Mount Namespace     容器有自己的檔案系統視圖（看不到 host 的根目錄）
User Namespace      容器裡的 root ≠ host 的 root
IPC Namespace       隔離 shared memory 和 semaphore

結果：容器「以為」自己是獨立的系統，但其實共享 host 的 OS kernel
```

**支柱 2：cgroup（資源限制）**

```bash
# 限制容器只能用 2 個 CPU 和 4GB RAM
docker run \
  --cpus="2.0" \
  --memory="4g" \
  myapp

# 底層實際上是這樣（Linux cgroup）：
# /sys/fs/cgroup/memory/docker/<container_id>/memory.limit_in_bytes
```

**Namespace 隔離 + cgroup 限制 = 容器的本質**

容器不是虛擬機（VM）。VM 模擬整個硬體，容器只是用 Linux Namespace 和 cgroup 做出的「隔離程序」。容器共享 host 的 kernel，所以比 VM 輕量得多。

**Dockerfile 基本結構：**

```dockerfile
# 1. 選擇基礎映像（包含 Python 3.11 的輕量 Linux）
FROM python:3.11-slim

# 2. 設定工作目錄
WORKDIR /app

# 3. 複製並安裝依賴（先複製 requirements，利用 layer cache）
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 4. 複製程式碼
COPY src/ ./src/

# 5. 定義啟動指令
CMD ["python", "src/train.py"]
```

```bash
# 建構 Image
docker build -t myapp:v1 .

# 執行 Container
docker run --rm \
  -v $(pwd)/data:/app/data \   # 掛載資料目錄
  --gpus all \                  # 使用 GPU
  myapp:v1

# 查看執行中的容器
docker ps

# 進入容器 shell（debug 用）
docker exec -it <container_id> bash
```

**為什麼容器解決了「在我電腦上能跑」的問題：**

```
你 build 的 Image 包含：
  ✓ 指定版本的 Python
  ✓ 精確的套件版本（pip freeze > requirements.txt）
  ✓ 所有系統依賴
  ✓ 環境變數設定

任何人只要 docker pull myapp:v1，就能跑出完全一樣的環境。
不管是你的 MacBook、同事的 Linux 工作站，還是雲端伺服器。
```

### S11：網路基礎——Port、Socket、TCP/IP 三分鐘版

**版面類型：** 銜接橋梁（精簡圖解 + 類比）

**為什麼這裡要講網路基礎？**

M6（系統設計思維）會討論 API、微服務、負載均衡。這些全建立在網路通訊上。這一頁是「剛好夠用」的最小知識集。

**TCP/IP 四層模型（從上到下）：**

```
Layer 4: Application（應用層）
  HTTP、HTTPS、WebSocket、gRPC
  你的程式直接互動的層

Layer 3: Transport（傳輸層）
  TCP（可靠、有序、連線導向）
  UDP（快速、無序、無連線）

Layer 2: Internet（網路層）
  IP 位址（IPv4 / IPv6）
  資料包路由

Layer 1: Link（連接層）
  實體傳輸（乙太網路、Wi-Fi）
  MAC 位址
```

**Port 的概念：**

```
一台主機只有一個 IP，但可以同時跑很多服務。
Port 是門號碼。

IP 位址 = 房子的地址
Port = 房間的號碼

常用 Port：
  22    → SSH（遠端登入）
  80    → HTTP
  443   → HTTPS
  5432  → PostgreSQL
  6379  → Redis
  8080  → 開發用 Web server（約定俗成）
```

**Socket 作為通訊端點：**

```python
import socket

# TCP Server
server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server.bind(("0.0.0.0", 8888))   # 監聽 0.0.0.0（所有介面）的 8888 Port
server.listen(5)
conn, addr = server.accept()
data = conn.recv(1024)

# TCP Client
client = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
client.connect(("127.0.0.1", 8888))
client.send(b"Hello")
```

Socket = (IP 位址, Port) 組合。一個 Socket 就是一個通訊端點。

**TCP 三次握手（簡版）：**

```
Client              Server
  │   SYN (我想連)    │
  │──────────────────►│
  │  SYN-ACK (好的)   │
  │◄──────────────────│
  │    ACK (收到)     │
  │──────────────────►│
  │                   │
  │   [開始傳資料]    │
```

三次握手保證雙方都準備好了，這是 TCP「可靠」的代價。UDP 沒有握手，所以更快但不保證送達。

**下一步（M6 預告）：**
系統設計思維會在這個基礎上討論 HTTP API、資料庫連線、負載均衡、快取伺服器——全都是「哪個程序在哪個 IP 的哪個 Port 上提供什麼服務」的問題。

### S12：本模組總結——OS 作為隱形管理者

**版面類型：** 整合頁（全圖概覽 + 回應 S1 的四個問題）

**OS 的角色總結：**

```
                        ┌─────────────────────────────┐
                        │      你的 Python 程式        │
                        └──────────────┬──────────────┘
                                       │
              ┌────────────────────────▼───────────────────────┐
              │                  作業系統                       │
              │                                                 │
              │  ┌──────────┐  ┌──────────┐  ┌──────────────┐  │
              │  │Process/  │  │ 記憶體    │  │  檔案系統    │  │
              │  │Thread    │  │ 管理      │  │  + 權限      │  │
              │  │排程      │  │ (VM/Paging│  │  (Inode/rwx) │  │
              │  └──────────┘  └──────────┘  └──────────────┘  │
              │                                                 │
              │  ┌──────────┐  ┌──────────┐  ┌──────────────┐  │
              │  │  IPC     │  │  網路    │  │  信號/服務   │  │
              │  │(Pipe/SHM)│  │(TCP/Port)│  │  (systemd)   │  │
              │  └──────────┘  └──────────┘  └──────────────┘  │
              └────────────────────────┬───────────────────────┘
                                       │
              ┌────────────────────────▼───────────────────────┐
              │              硬體（M4 的內容）                   │
              │    CPU Cores    RAM    SSD    GPU    Network    │
              └─────────────────────────────────────────────────┘
```

**回應 S1 的四個問題：**

| 錯誤訊息 | 根因 | 解法 |
|---------|------|------|
| `MemoryError` / `CUDA out of memory` | RAM 或 VRAM 被 OS/CUDA 拒絕分配 | 減少 batch size、`torch.no_grad()`、`gc.collect()` |
| `Permission denied` | 檔案系統的 rwx 權限不足 | `chmod`、`chown`、確認使用者身份 |
| "Works on my machine" | 環境差異（Python 版本、套件、路徑） | Docker 容器化，打包整個環境 |
| GPU 使用率只有 30% | DataLoader 是 I/O 瓶頸，GIL 限制多執行緒 | `num_workers` multiprocessing，或 pin_memory |

**M5 → M6 的橋接：**

M5 讓你理解一台機器上的 OS 如何管理資源。M6 要問：**當一台機器不夠用時，怎麼辦？** 那就是系統設計的起點——多台機器、負載均衡、分散式資料庫、快取叢集。

### 練習一：OS 偵探——從指令行診斷系統狀態

**目標：** 用 Linux 指令診斷一個「訓練腳本跑很慢」的場景。

**情境設定：** 你收到回報，一台訓練伺服器的 Python 訓練腳本比預期慢 3 倍。你需要從 OS 層面找出原因。

**任務清單：**

```bash
# 步驟 1：查看系統整體狀況
# Q: CPU 使用率是多少？哪個程序在用？
top -bn1 | head -20

# 步驟 2：查看記憶體狀況
# Q: 有沒有使用 Swap？（Swap 使用代表 RAM 不足）
free -h

# 步驟 3：找出訓練程序
# Q: 訓練程序的 PID 是多少？它的狀態是 R（Running）還是 D（Waiting for I/O）？
ps aux | grep train.py
ps -p <PID> -o pid,stat,cmd,%cpu,%mem,etime

# 步驟 4：查看磁碟 I/O
# Q: 磁碟讀寫是不是瓶頸？
iostat -x 1 5   # 每秒刷新，共 5 次（需安裝 sysstat）

# 步驟 5：查看 GPU 狀態（如果有 NVIDIA GPU）
nvidia-smi
watch -n 1 nvidia-smi   # 每秒更新
```

**診斷問題範例腳本：**

```python
# slow_training_sim.py
# 這個腳本刻意製造不同類型的瓶頸，用指令行找出是哪一種

import time
import numpy as np
from pathlib import Path

BOTTLENECK = "io"  # 改成 "cpu" 或 "memory" 看不同症狀

def cpu_bottleneck():
    """CPU 瓶頸：大量矩陣計算"""
    for _ in range(100):
        a = np.random.rand(1000, 1000)
        b = np.random.rand(1000, 1000)
        c = a @ b   # 矩陣乘法

def io_bottleneck():
    """I/O 瓶頸：頻繁讀寫小檔案"""
    for i in range(1000):
        path = Path(f"/tmp/test_{i}.bin")
        path.write_bytes(np.random.bytes(1024))
        data = path.read_bytes()
        path.unlink()

def memory_bottleneck():
    """記憶體瓶頸：建立大量物件不釋放"""
    accumulator = []
    for i in range(10000):
        accumulator.append(np.random.rand(1000))  # 刻意不清理
        if i % 1000 == 0:
            print(f"Step {i}, accumulated {len(accumulator)} arrays")

if BOTTLENECK == "cpu":
    cpu_bottleneck()
elif BOTTLENECK == "io":
    io_bottleneck()
elif BOTTLENECK == "memory":
    memory_bottleneck()
```

**預期學習：** 學生應能從 `top` 的輸出區分 CPU-bound vs I/O-bound，理解 `ps` 的 STAT 欄位（`R` / `S` / `D`），並能用 `free -h` 判斷是否有記憶體壓力。

### 常見學生困惑與解答

**Q1：Thread 共享記憶體不是比 Process 更有效率嗎？為什麼 Python 要用 multiprocessing？**

A：Thread 共享記憶體本身是更有效率的（零複製開銷）。問題不是效率，是 GIL。Python 的 GIL 讓同一時刻只有一個 Thread 能跑 Python bytecode。對 CPU-bound 任務，多 Thread 等於輪流跑，沒有加速效果。`multiprocessing` 給每個 Process 一個獨立的 Python 解釋器（獨立的 GIL），才能真正並行。代價是資料傳遞需要序列化（pickle）。

**Q2：Virtual Memory 和真實 RAM 有什麼差別？虛擬記憶體是「假的」嗎？**

A：不是假的，是對應關係。Virtual Memory 是每個程序「以為自己有的」連續地址空間。OS 的 MMU 負責把虛擬地址對應到實體 RAM 的地址。虛擬記憶體的好處：1) 程序隔離（程序 A 無法讀程序 B 的記憶體）；2) 每個程序都以為自己有完整的地址空間，簡化程式設計；3) 可以超量使用（Swap：把暫時不用的頁面放到磁碟）。

**Q3：Docker 和 VM 到底有什麼不同？為什麼 Docker 更輕量？**

A：VM 用 Hypervisor 模擬整個硬體，每個 VM 都跑完整的 OS kernel（幾百 MB 到 GB 級）。容器不模擬硬體，只用 Linux Namespace 和 cgroup 做程序隔離，共享 host 的 OS kernel。所以容器啟動時間是秒級，VM 是分鐘級；容器 Image 通常幾 MB 到幾十 MB，VM Image 幾 GB。代價：容器的隔離性比 VM 弱（共享 kernel 意味著 kernel 漏洞可以逃逸容器）。

**Q4：SIGTERM 和 SIGKILL 什麼時候該用哪個？**

A：先用 SIGTERM（`kill -15`）。給程序機會清理資源、寫完 checkpoint、關閉資料庫連線。如果程序沒有回應（或者你知道它沒有實作 SIGTERM handler），再用 SIGKILL（`kill -9`）強制終止。對訓練腳本，正確做法是在 Python 程式裡寫 `signal.signal(signal.SIGTERM, cleanup_handler)`，讓它在收到 SIGTERM 時儲存 checkpoint 再退出。

## 本模組核心心智模型

學完這個模組，你應該有這個心智模型：

**每次你的 Python 程式執行一個操作，背後有一個隱形的管理者在做決策：**

```
你寫：data = open("file.csv").read()

OS 在做：
  1. 接到 open() 系統呼叫
  2. 查 inode 表，確認 file.csv 的存在和位置
  3. 確認你的程序有權限讀取（rwx 檢查）
  4. 分配一個 File Descriptor 給你的程序
  5. 排程 I/O 操作（你的程序進入 WAITING 狀態）
  6. 磁碟讀完後，OS 把資料複製到你的 Virtual Memory 空間
  7. 你的程序回到 READY → RUNNING 狀態
  8. read() 返回，你以為只是「讀了一個檔案」

你看到的是一行程式碼。OS 做了七個步驟。
```

這就是為什麼「懂 OS」讓你有不同的工程師視角：你知道「一行程式碼背後發生了什麼」。
