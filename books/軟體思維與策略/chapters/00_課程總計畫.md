# 系統設計與架構思維：從軟體工程到 AI 時代的架構師之路

## 24 小時線上課程完整計畫

---

## 壹、課程核心主張

本課程是「2026 Python 數據分析與 AI 基礎」（Course 1）的進階續篇。Course 1 教你用 Python 做資料分析；本課程教你理解程式如何被組織、系統如何運作、架構如何設計。

學完本課程，你不只會寫程式，更會知道：程式碼背後的記憶體在做什麼、CPU 和 GPU 怎麼分工、作業系統怎麼管理資源、系統怎麼從單機走向分散式、架構決策如何影響整個產品的命運。

**一句話定位：**
從「會寫 Python」升級到「懂軟體工程、能設計系統、有架構師視角」的 24 小時進階課。

## 肆、為什麼需要這門課？

### 1. 因為 AI 應用最後都會回到軟體工程問題
模型只是冰山一角。資料管線、API 封裝、狀態管理、部署流程、效能調校——這些全是軟體工程問題。

### 2. 因為「會寫程式」和「會做工程」之間有一條鴻溝
腳本能跑一次就算完成；系統要能在不同環境重現、被別人維護、承受真實流量。

### 3. 因為架構決策的影響比任何一行程式碼都大
選錯資料庫、忽略 CAP 定理、沒做負載分流——這些錯誤不是改幾行 code 能修的。

### 4. 因為 AI 時代的工程師需要更完整的視角
當 AI 能自動生成程式碼，工程師的價值在於：理解系統、設計架構、做出正確的技術決策。

## 捌、教學敘事節奏 — 三幕結構

### 第一幕：理解底層（M1-M5）
從軟體的本質開始，一路建立 OOP 結構能力、Python 工程能力、硬體直覺、OS 直覺。結束時，學生能理解「一段程式跑起來的時候，整台電腦在做什麼」。

### 第二幕：學會設計（M6-M7）
從工程實踐跳到系統設計。學生開始用架構師的眼光看問題：不只想「怎麼寫」，而是想「怎麼設計才能撐住」。

### 第三幕：看見全貌（M8）
把所有東西整合。用真實案例驗證，補上軟實力與職涯視角，讓學生帶著完整的工程師世界觀離開。

## 玖、投影片風格原則

### 原則 1：每個底層概念都要有應用場景
不講孤立的理論。Stack/Heap 要連到「為什麼遞迴太深會 stack overflow」；process/thread 要連到「為什麼 DataLoader 用 multiprocessing」。

### 原則 2：從症狀倒推，不從教科書正序
不是「先講 CPU pipeline 再講為什麼訓練慢」，而是「訓練為什麼慢？因為 GPU 在等資料。為什麼在等？因為 disk -> RAM -> GPU 有搬運成本。」

### 原則 3：用圖說話
程式 -> CPU -> RAM -> Disk 流程圖、class/object 關係圖、process/thread 生命週期圖、系統架構圖、C4 model 分層圖。

### 原則 4：高層視角頁金句
- "Hardware is finite and rigid; software is infinite and malleable."
- "OOP is not academic gymnastics — it's how systems survive."
- "Systems turn local scripts into scalable applications."
- "Architecture is the set of decisions you wish you could get right early."
