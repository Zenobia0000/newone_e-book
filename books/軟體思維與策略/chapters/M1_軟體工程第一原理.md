### 開場定位說明

Course 1 教你用 Python 解決資料分析問題。M1 問的是一個更根本的問題：**你知道你在做什麼嗎？**

不是語法層面的「知道」——而是「這段程式在記憶體裡做了什麼？它屬於軟體工程的哪個環節？我該如何有系統地思考這個問題？」

M1 是整門課程的基礎地基。沒有這層認知，後面的 OOP、計算機組織、系統設計都會變成零散的技能點，而不是一個連貫的工程師世界觀。

## 二、學習目標

完成本模組後，學員能夠：

1. **用神經系統類比解釋硬體與軟體的本質差異**，說明為何「軟體是無限可塑的」這個命題不只是口號，而是有物理根據的工程洞察。

2. **複述 SWEBOK 六大核心知識領域**，並能將自己在 Course 1 做過的事情（寫腳本、除錯、跑分析）正確對應到軟體工程的哪個環節。

3. **應用第一原理思考法**分析一個自己熟悉的技術問題，說明為何這種思考方式優於類比式學習。

4. **區分 Stack 與 Heap 記憶體的運作機制**，解釋 LIFO 模型、Stack Frame、動態配置的概念，以及這些概念如何影響變數作用域與函數行為。

5. **在 Python Tutor 中追蹤一段含有遞迴或物件操作的 Python 程式碼**，準確指出哪些變數在 Stack 上、哪些物件在 Heap 上。

6. **解釋 Stack Overflow 錯誤的根本成因**，以及無限遞迴為何必然崩潰——不只是背錯誤名稱，而是從記憶體模型推導出來。

7. **說明理解 Stack/Heap 如何賦予「跨語言搬家」的能力**，以傳值 vs. 傳址為例，說明這個底層觀念在 Python、Java、Go 中是共通的。

8. **為自己的 Course 2 學習旅程建立一張心智地圖**，知道 M1 到 M8 各模組解決什麼問題、它們之間的邏輯關係是什麼。

### Slide 1：開場——你已經寫了程式，現在來理解它

**核心訊息**
Course 1 讓你學會「用 Python 做事」。本課程的目標是讓你理解「你在做什麼」。

**講師講解要點**

1. **從學員的已知出發。** 請學員回想在 Course 1 寫過最複雜的一段程式。問：「你知道那段程式跑起來的時候，電腦記憶體裡在發生什麼事嗎？」多數人不知道——這正是本課程的起點。

2. **「會用」和「理解」是兩個不同的認知層次。** 會用 pandas 的 `groupby` 是工具層；知道 `groupby` 在記憶體裡建立了一個怎樣的資料結構、為什麼大資料集會慢、如何優化——這是工程層。

3. **這不是要推翻 Course 1，而是在它上面建樓。** 所有 Course 1 學到的 Python 技能，在 Course 2 都會得到「底層解釋」。你不是在重學，是在看清楚你已經知道的東西背後的機制。

4. **本課程的定位：從「資料分析師」往「軟體工程師」或「資料工程師」轉型的第一步。** 這不是職稱的差異，而是思考問題的維度不同——工程師看到一個問題，會問「系統層面怎麼設計」，而不只是「這行程式碼怎麼寫」。

5. **今天結束後你會得到什麼。** 一張完整的軟體工程知識地圖（SWEBOK）、一個解釋程式行為的底層模型（Stack/Heap）、一種思考問題的方式（第一原理）——這三件事會是整門課的認知骨架。

**視覺建議**
左側：一段簡短的 pandas 程式碼截圖（學員熟悉的感覺）。右側：同一段程式碼旁邊出現 Stack 示意圖、記憶體位址、「你知道這在做什麼嗎？」的問號。對比感要強烈。

**過場語**
「在搞懂記憶體之前，我們先回答一個更根本的問題：軟體到底是什麼？」

### Slide 2：軟體是什麼？（不是你以為的答案）

**核心訊息**
軟體不是程式語言，不是語法，不是 `.py` 檔案。軟體是對計算資源的邏輯控制指令集。

**講師講解要點**

1. **拆解「軟體」這個詞的常見誤解。** 多數人把軟體等同於「程式碼」或「App」。這只是表象。軟體的本質是：一組指令，告訴物理硬體如何分配資源（時間、空間、I/O）來完成特定任務。

2. **用一個思想實驗說明。** 假設你把同一份 Python 程式碼，分別跑在 4GB RAM 的筆電和 512GB RAM 的伺服器上，程式碼沒有變，但行為會不同（一個 OOM crash，一個輕鬆跑完）。這說明：軟體是和硬體互動的，它不是孤立存在的文字。

3. **軟體的層次：不只是你寫的那一層。** 從底往上：韌體（Firmware）→ 作業系統（OS）→ 執行環境（Runtime/VM）→ 標準函式庫 → 你的應用程式碼。你在 Course 1 寫的 Python 腳本，站在這整個軟體堆疊的最頂端。

4. **「語法」是介面，不是本質。** Python 的 `for` 迴圈、Java 的 `for` 迴圈、Go 的 `for range`——語法不同，但底層都是：修改計數器、比較邊界、跳轉指令。語法是給人看的，機器執行的是指令序列。

5. **這個定義帶來什麼實際影響？** 當你把軟體定義為「對資源的邏輯控制」，你看待效能問題的方式就會改變：慢不是因為「程式碼寫得爛」，而是因為「對資源的控制方式不夠好」——這才是工程師的診斷視角。

**視覺建議**
軟體堆疊圖：從底部的 CPU / RAM / Storage 硬體層，往上依序標示 Firmware → OS Kernel → Python Runtime → Standard Library → 你的程式碼。用不同顏色區分層次，最頂端的「你的程式碼」用高亮色標示，旁邊加注「你目前站在這裡」。

**過場語**
「既然軟體是對物理資源的邏輯控制，那我們就必須理解軟體和硬體之間的根本關係。」

### Slide 3：硬體與軟體的二元性

**核心訊息**
硬體是有限且剛性的；軟體是無限且可塑的。這不是比喻，而是工程現實中最重要的設計約束。

**講師講解要點**

1. **硬體的「有限性」是物理法則，不可協商。** 你的 RAM 是 16GB 就是 16GB，CPU 的時脈速度有物理上限，網路頻寬是固定的。這些約束不會因為你的程式碼「寫得很好」就消失。工程師的工作，就是在這些硬性約束內找到最好的軟體方案。

2. **硬體的「剛性」意味著什麼？** 硬體一旦出廠，功能就固定了。你無法透過軟體讓一顆 CPU 多出一個核心，也無法讓 1TB 的 SSD 變成 2TB。硬體是實體的，物理上的修改需要人工和金錢。

3. **軟體的「無限性」展示了人類智識的力量。** 同樣的硬體，透過不同的軟體，可以是音樂播放器、科學計算機、語言模型推論引擎、即時通訊工具。硬體定義天花板，軟體決定在天花板以下你能走多高。

4. **軟體的「可塑性」是工程師最強大的武器。** 你可以在不更換任何硬體的情況下，讓系統效能提升 10 倍（換一個更好的演算法）、讓系統功能完全改變（更新版本）、讓系統支援 10 倍的流量（水平擴展）。這些都是純粹的軟體決策。

5. **二元性的工程意涵。** 因為硬體有限，工程師需要學會「省著用」——理解記憶體、CPU、I/O 的成本。因為軟體可塑，工程師有責任「用得好」——不浪費資源，同時盡可能發揮硬體的潛能。課程後面的每個模組，都是在這個框架下展開的。

**視覺建議**
左右對比圖。左側「硬體」：一張伺服器機架照片，下方列出固定數字（8 Core CPU / 64GB RAM / 1TB SSD）。右側「軟體」：同樣的硬體，但旁邊有多個軟體功能圖示（Web Server / ML Model / Database / Game），用箭頭表示「同一硬體，無數軟體可能性」。底部一行大字：**"Hardware defines the ceiling. Software decides how high you go."**

**過場語**
「這個二元性用一個你已經很熟悉的類比來說，其實就更清楚了。」

### Slide 4：神經系統類比——意識如何運行在物質上

**核心訊息**
硬體是神經系統（物理載體）；軟體是心智與意識（邏輯編碼）。這個類比不只是比喻，它揭示了軟體的本質。

**講師講解要點**

1. **為什麼需要這個類比？** 因為軟體是「看不見的」，用物理類比幫助大腦建立直觀模型。神經科學與控制論早在電腦出現之前，就在研究「邏輯控制如何運行在物理結構上」這個問題——硬體/軟體二元性其實是一個古老的哲學問題的現代工程版本。

2. **對應關係的細節。** CPU ≈ 大腦皮質（運算、決策）；RAM ≈ 工作記憶（短期存取，斷電清空）；SSD/HDD ≈ 長期記憶（持久儲存）；神經突觸的電化學訊號 ≈ CPU 的二進制電子訊號；DNA 的指令編碼 ≈ 軟體的程式碼邏輯。

3. **「有意識的行為需要物質載體，但不等於物質。」** 一具沒有意識的大腦能做什麼？什麼都做不了。一台沒有軟體的電腦能做什麼？什麼都做不了。但「意識」不是「神經元」，「軟體」也不是「電晶體」——它是運行在物質上的資訊模式。

4. **這個類比的限制——以免學員過度類比。** 大腦沒有「記憶體上限」（或上限非常模糊）；電腦的 RAM 是精確量化的。大腦的「程式碼」不能像軟體一樣任意修改；電腦軟體可以。類比是認知工具，不是精確模型——當類比失效的地方，才是工程有趣的地方。

5. **從類比回到工程洞察。** 當你理解到「你的程式碼實際上是在指揮物理資源」，你就不會再只問「這行程式碼怎麼寫」，而是會問「這行程式碼會消耗多少 RAM？會讓 CPU 忙多久？會觸發多少 I/O？」——這就是工程師和腳本寫手的思維差距。

**視覺建議**
雙欄對比圖，左側是人腦解剖圖（標示大腦皮質、海馬體、神經突觸），右側是電腦組成圖（標示 CPU、RAM、SSD、I/O 介面）。用虛線連接對應部位。底部：一行引用文字 *"Just as DNA encodes the instructions for a biological organism, software provides the logical code that controls hardware operations."*

**過場語**
「現在我們知道軟體是什麼了。下一個問題是：軟體工程是什麼？它比『寫程式』大多少？」

### Slide 5：SWEBOK——軟體工程的完整版圖

**核心訊息**
「寫程式」只是軟體工程的一個環節（Construction）。SWEBOK 告訴你完整的知識版圖長什麼樣。

**講師講解要點**

1. **為什麼要介紹 SWEBOK？** 因為大多數學習者——包括很多工作多年的工程師——都把軟體工程等同於「寫程式」。這個認知偏差導致兩個問題：一是職涯天花板（只會寫，不會設計、測試、管理）；二是對自己知識盲點沒有自覺（不知道自己不知道什麼）。

2. **IEEE 的定義。** SWEBOK（Software Engineering Body of Knowledge）是 IEEE 電腦學會編纂的軟體工程知識標準。它的定義：「將系統化、規範化、可度量的方法應用於軟體的開發、運行和維護」。關鍵詞：系統化（不是臨時湊合）、規範化（有標準）、可度量（能評估品質）。

3. **六大核心知識領域一一說明。**
   - **需求（Requirements）**：做什麼？確認你在解決真正的問題，而不是臆想的問題。許多專案失敗不是技術爛，是做出了沒人要的功能。
   - **設計（Design）**：怎麼做？系統架構、模組邊界、介面定義——這是程式碼之前的藍圖。沒有設計就開始寫程式，等於沒看地圖就開始蓋房子。
   - **建構（Construction）**：把設計變成程式碼——這是大家最熟悉的部分，但它只佔工程的一段，不是全部。
   - **測試（Testing）**：驗證你的程式碼做了它應該做的事。測試不是「寫完再說」，是工程品質的護城河。
   - **維護（Maintenance）**：現實是，工程師 80% 的時間在讀和修改現有程式碼，不是在寫全新功能。維護能力是被低估的核心技能。
   - **管理（Management）**：版本控制、敏捷開發、Sprint 規劃——軟體是團隊運動，管理讓多人協作成為可能。

4. **SWEBOK 是本課程的隱形骨架。** M1 建立本質觀；M2–M3 深入 Construction；M4–M5 解釋支撐一切的底層（計組 + OS）；M6–M7 處理 Design + Requirements；M8 整合全部並加入 Management 視角。

5. **對學員的挑戰性問題。** 「你上次做資料分析專案，有沒有寫測試？有沒有考慮維護性？有沒有分析清楚需求再動手？」多數人都沒有——這不是批評，這是本課程存在的理由。

**視覺建議**

### Slide 7：第一原理思考法

**核心訊息**
類比式學習讓你快速入門，卻讓你被工具鎖死。第一原理思考讓你理解本質，使你能跨越工具的更迭。

**講師講解要點**

1. **什麼是類比式學習（Analogy-based Learning）？** 「React 很像 Vue，我學過 Vue，所以 React 我也很快就會。」這種學習方式短期有效，因為你能快速建立熟悉感。但問題是：你只學會了「表面相似的部分」，當它們的差異點成為關鍵問題時，你就不知道為什麼了。

2. **什麼是第一原理思考？** Elon Musk 的描述：「把問題拆解到最基本的事實，然後從零開始重建理解，而不是依賴已有的類比或慣例。」應用到軟體學習：不問「這個框架怎麼用」，而問「這個框架解決了什麼物理限制？為什麼這個解法是合理的？」

3. **具體例子——以學習 React 為例（對照類比法和第一原理法）。**
   - 類比法：「React 很像 jQuery，但更有結構。」
   - 第一原理法：「網頁的本質是 DOM 樹。頻繁直接操作 DOM 代價高（觸發 reflow / repaint）。所以需要一個最小化 DOM 操作次數的機制。Virtual DOM + Diffing 演算法就是這個機制。React 是這個機制的一個實作。」第一原理版本讓你知道為什麼 React 存在，而且未來當 Diff 演算法被更好的方案取代時，你也能理解新方案為什麼更好。

4. **應用到本課程的具體做法。** 每次遇到一個新概念，問三個問題：(1) 這個概念解決了什麼物理或邏輯上的問題？(2) 如果沒有這個概念，最壞的情況是什麼？(3) 這個概念的最簡版本（最本質的形式）是什麼？這三個問題能幫你過濾「記憶」和「理解」的差距。

5. **第一原理思考的代價。** 它比類比式學習慢。第一次用這種方式學習一個概念，你可能要花三倍時間。但掌握了本質後，你跨語言、跨框架的速度會遠快於類比式學習者。這是投資，不是浪費時間。

**視覺建議**
左側「類比式學習路徑」：一條看似平坦但不斷分叉的路，每個分叉代表一個「新框架/新語言」，學習者要一條條路重走。右側「第一原理學習路徑」：一條先往下挖的路（挖到底層原理），然後從這個深根往上長出多條分支，代表可以快速遷移到不同語言和框架。底部：**"類比式學習讓你入門，第一原理讓你精通。"**

### Slide 8：Stack——秩序與速度的殿堂

**核心訊息**
Stack 是管理函數執行環境的 LIFO 記憶體區域。理解它，你就理解了作用域、執行順序、和遞迴的本質。

**講師講解要點**

1. **Stack 的物理直覺。** 想像一疊盤子：你只能放在最上面，也只能從最上面拿。這就是 LIFO（Last-In, First-Out）。電腦的 Stack 記憶體遵循同樣的原則——函數呼叫時 Push，函數返回時 Pop。這個簡單規則，讓 CPU 能以極快的速度管理函數的進出。

2. **Stack Frame 的組成。** 每次函數被呼叫，CPU 會在 Stack 頂端建立一個 Stack Frame（執行環境區塊）。這個區塊包含：該函數的所有區域變數（local variables）、傳入的參數（parameters）、函數執行完後要跳回去的位址（return address）。當函數返回，這個 Frame 被 Pop，它占用的記憶體瞬間釋放。

3. **Stack 為什麼快？** 因為 CPU 有一個專門的暫存器叫 Stack Pointer（SP），它永遠指向 Stack 的頂端。存取 Stack 頂端的資料，CPU 不需要任何搜尋或計算，直接讀取——速度接近硬體極限。加上 Stack 的記憶體是連續的，CPU 快取（Cache）的命中率極高。

4. **Stack 與作用域的關係。** 「為什麼函數 A 裡定義的變數，在函數 B 裡讀不到？」因為每個函數有自己的 Stack Frame，而且函數 A 的 Frame 在 A 返回後就被 Pop 掉了。變數的「作用域」（scope），在底層對應的就是 Stack Frame 的「生命週期」。這是跨所有語言都成立的物理規律。

5. **Stack 的限制。** Stack 的空間通常只有幾 MB（具體視 OS 和語言設定而定）。這意味著你不能在 Stack 上存放很大的資料（例如一個 1GB 的陣列），也意味著函數呼叫的層數是有上限的。這個上限，引出了下一個重要概念：Stack Overflow。

**視覺建議**
動態式示意圖（可分步驟展示）：初始狀態是空的 Stack；呼叫 `main()` 後，Push 一個 Frame；`main()` 呼叫 `process(data)` 後，再 Push 一個 Frame；`process()` 返回後，Pop 最頂端的 Frame；最終 `main()` 返回，Stack 清空。每個 Frame 內部顯示變數名稱和值。右側同步顯示程式碼，高亮目前執行到的那一行。

**過場語**
「Stack 讓我們理解了有序的部分。現在來看無序的那半邊：Heap。」

### Slide 9：Heap——自由與混沌的荒原

**核心訊息**
Heap 是動態配置物件的記憶體區域。傳址、副作用、垃圾回收——Python 物件行為最讓初學者困惑的地方，根源都在這裡。

**講師講解要點**

1. **為什麼需要 Heap？** 有些資料你在寫程式時不知道它會多大（使用者輸入的字串、從資料庫讀出的陣列），有些資料你需要它在多個函數呼叫之間存活（一個物件被好幾個函數共享）。這些情況下，Stack 的 LIFO 模型就不夠用了——你需要一個可以在執行期間隨時申請、隨時釋放的記憶體區域。這就是 Heap。

2. **Heap 的運作模型。** 程式向 OS 申請一塊記憶體（例如 Python 建立一個 list）→ OS 在 Heap 的某個位置分配空間 → 把這塊空間的「地址」（記憶體位址）交給程式 → 程式透過這個地址（引用 / reference）來存取和修改這塊資料。關鍵點：Stack Frame 裡存的不是物件本身，而是「物件在 Heap 上的地址」。

3. **傳址 vs. 傳值——Python 學員最常犯的 Bug 根源。** 在 Python 中，整數、字串等基本型別通常是傳值（或表現得像傳值）；list、dict、自訂物件等是傳址（引用）。當你把一個 list 傳給函數，函數收到的是 Heap 上那個 list 的地址，不是複本。所以在函數內部 `append()` 了東西，原始的 list 也跟著改變——這叫「副作用（Side Effect）」，是 Bug 的高發地帶。

4. **Garbage Collection（GC）的工作。** 在 C/C++ 中，你需要手動 `free()` Heap 上不再用的記憶體，否則會「記憶體洩漏（Memory Leak）」。Python、Java、Go 都有自動 GC：執行環境定期掃描 Heap，找出「沒有任何引用指向它」的物件，自動回收其記憶體。GC 的存在讓程式碼更安全，但 GC 本身會消耗 CPU 時間——這是自動記憶體管理的代價。

5. **Heap 和 Stack 的分工總結。** Stack 管「函數執行環境」——有序、快速、自動清理、空間小。Heap 管「物件和資料」——自由、彈性、需要 GC 或手動管理、空間大但存取較慢。理解這個分工，你就能預測任何一段程式碼的記憶體行為。

**視覺建議**
左側是整齊的 Stack（方框整齊堆疊，用箭頭標示 LIFO Push/Pop）。右側是散落的 Heap（不規則的彩色方塊散布在大空間裡，有些標記「GC 待回收」，有些用箭頭從 Stack Frame 裡的「變數」指向 Heap 上的物件）。中間用一條虛線分隔兩個區域，標題分別是「Stack：秩序」和「Heap：自由」。

**過場語**
「現在我們同時理解了 Stack 和 Heap，就能解釋兩個在 Python 學習中常見、但往往沒有得到好解釋的現象。」

### Slide 10：Stack Overflow 與傳址副作用——兩個 Bug 的物理根源

**核心訊息**
Stack Overflow 不只是一個網站名稱；傳址副作用不只是「Python 的怪行為」。它們都有清晰的記憶體模型解釋。

**講師講解要點**

1. **Stack Overflow 的完整故事。** 無終止條件（或終止條件永遠不被觸發）的遞迴函數，每次呼叫自己都會 Push 一個新的 Stack Frame。Stack 的空間有限（幾 MB），最終空間耗盡，OS 強制終止程式，報出 `RecursionError: maximum recursion depth exceeded`（Python 版本）或 `StackOverflowError`（Java 版本）。預防方法：(1) 確保遞迴有正確的終止條件；(2) 考慮用迭代（iteration）替換深層遞迴；(3) Python 可用 `sys.setrecursionlimit()` 調整上限，但這是治標不治本。

2. **一個讓學員頓悟的 Python 程式碼例子。** 展示以下程式碼，讓學員預測輸出：
   ```python
   def modify_list(lst):
       lst.append(99)

   my_list = [1, 2, 3]
   modify_list(my_list)
   print(my_list)  # [1, 2, 3, 99] <- 為什麼？
   ```
   解釋：`my_list` 在 Stack 上存了一個「指向 Heap 上陣列的地址」。`modify_list` 收到的是同一個地址，所以修改的是同一個 Heap 物件。

3. **對比「基本型別的傳值行為」。** 展示對比：
   ```python
   def modify_int(n):
       n = n + 1

   x = 10
   modify_int(x)
   print(x)  # 10 <- 沒有改變，為什麼？
   ```
   解釋：整數 `10` 在 Python 是不可變物件（Immutable）。`n = n + 1` 在 Heap 上建立了一個新的整數物件 `11`，並讓 `n` 指向新物件，不影響原始的 `x` 指向的 `10`。

4. **實務影響：如何避免傳址副作用？** 方法一：傳入前做複本 `modify_list(my_list.copy())`；方法二：在函數內部明確建立複本；方法三：使用不可變資料結構（tuple 替代 list）。知道問題的物理根源，就能選擇正確的修復方法。

5. **連結到函數式編程的概念（預告 M2）。** 避免副作用是「純函數（Pure Function）」的核心要求——同樣的輸入永遠給出同樣的輸出，不修改任何外部狀態。這個概念在 M2 的 OOP 與函數式設計中會更詳細展開。

**視覺建議**
上半部：Stack Overflow 動畫示意——Stack Frame 不斷堆疊，最終畫面上出現一個紅色的「爆炸」效果，標示「記憶體耗盡」。下半部：傳址示意圖——Stack Frame 裡的 `my_list` 變數有一個箭頭指向 Heap 上的 `[1, 2, 3]`；函數 `modify_list` 的 Stack Frame 裡的 `lst` 變數也有箭頭指向同一個 Heap 物件，`lst.append(99)` 之後，Heap 上的物件變成 `[1, 2, 3, 99]`，而 `my_list` 的箭頭仍然指向同一個物件。

### 常見學員困惑點與處理方式

**1. 「Heap 和 Stack 是在 RAM 裡的不同位置嗎？」**
是的。RAM（實體記憶體）在邏輯上被 OS 分成多個區段。從低位址到高位址大致是：程式碼段（Code Segment）→ 靜態資料段（Data Segment）→ Heap（往高位址增長）→ ...空白區間... → Stack（往低位址增長）。Stack 和 Heap 是同一塊 RAM 的不同邏輯區域，共享同一個物理空間，但從兩端往中間增長，理論上若兩者都增長得很大，可能互相碰撞——但在現代 OS 有虛擬記憶體保護的情況下，通常會先觸發 Stack Overflow 或 OOM（Out of Memory）。

**2. 「Python 沒有指標（Pointer），為什麼說 list 是用引用（Reference）存取？」**
Python 確實沒有 C 式的「裸指標」，但 Python 的所有物件都是透過引用（reference）來操作的。在 CPython 實作中，每個 Python 物件在記憶體中都有一個 `ob_refcnt`（引用計數器），GC 用這個計數來決定何時回收。當你寫 `a = [1, 2, 3]`，`a` 這個名稱實際上是一個「引用」，指向 Heap 上的 list 物件。Python 的「無指標」是語法層面的安全設計，底層仍然是地址操作。

**3. 「第一原理思考聽起來很慢，有什麼快速判斷用不用的方法？」**
一個簡單的判斷標準：如果這個概念你以後每天都會用，或者它是某個重要領域的基礎，就值得用第一原理。如果只是一個工具的特定語法，類比式學習完全可以接受。例如：Stack/Heap 值得用第一原理理解（影響所有語言）；某個 pandas 函數的參數格式，類比即可（可以隨時查文件）。

**4. 「SWEBOK 是理論，我在業界用不到吧？」**
SWEBOK 不是考試用的，是認知工具。它告訴你工程的完整面貌，讓你知道自己的能力分佈。很多工程師職涯卡關，不是因為「程式碼寫不好」，而是因為不知道「需求怎麼分析」或「系統怎麼設計」——這些恰好是 SWEBOK 指出的其他領域。用 SWEBOK 做自我盤點，找出自己的薄弱環節，是長期職涯發展的重要練習。
