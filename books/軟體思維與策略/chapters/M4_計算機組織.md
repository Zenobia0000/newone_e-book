## 一、模組定位

你有沒有遇過這些問題：

- 訓練一個簡單的神經網路，GPU 使用率卻只有 30%
- 把 `list` 換成 `numpy array`，程式突然快了 10 倍
- DataLoader 的 `num_workers` 設定不對，訓練就卡在 I/O
- batch size 從 32 改成 256，速度沒有等比例提升

這些不是程式 bug，也不是框架問題——它們是硬體行為在程式層面的投影。

本模組的目標不是讓你成為硬體工程師，而是讓你能夠**從症狀反推根因**：看到效能問題，知道瓶頸在哪一層；設計系統，知道什麼決策會衝擊哪個硬體資源；閱讀 profiling 報告，能夠識別 memory-bound、compute-bound 還是 I/O-bound。

這是 AI 時代工程師的必要素養，因為當你使用 GPU 叢集訓練模型、設計高吞吐資料管線、或者優化推論延遲時，硬體的邊界就是你系統的邊界。

## 二、學習目標（6 項）

完成本模組後，學員能夠：

1. **解釋 CPU 執行指令的基本流程**，包括 pipeline、指令層級平行性（ILP），理解為什麼現代 CPU 的「時脈速度」不是效能的全貌。

2. **描述快取層次結構的作用與代價**，能具體說明 cache miss 對延遲的影響，以及時間局部性與空間局部性如何影響程式碼撰寫方式。

3. **解釋 RAM 的角色與頻寬限制**，理解為什麼記憶體頻寬（而非計算速度）往往才是現代程式的真正瓶頸。

4. **區分 GPU 與 CPU 的架構差異**，能說明 CUDA core 與 Tensor Core 的用途，解釋為什麼深度學習的矩陣運算天然適合 GPU。

5. **比較不同儲存媒介的特性**，能根據 SSD/HDD/NVMe 的延遲與頻寬特性，判斷哪種存取模式會造成 I/O 瓶頸。

6. **追蹤資料從磁碟到 GPU 的完整移動路徑**，理解 DataLoader、memory pinning、batching 等設計決策背後的硬體動機。

### 投影片 1：硬體決定軟體的邊界

**核心訊息：**  
效能問題的根因通常在硬體層，而不在演算法或框架。

**談話要點：**

1. 開場問題：「你有沒有遇過 GPU 使用率很低但訓練還是很慢的情況？」這是 I/O-bound，不是 compute-bound。
2. 三類效能瓶頸：compute-bound（計算受限）、memory-bound（記憶體頻寬受限）、I/O-bound（輸入輸出受限）。大多數工程師預設是第一種，但現實中後兩種更常見。
3. 本模組的診斷框架：從症狀（程式慢）逆向追蹤到根因（哪一層硬體是瓶頸）。
4. 不需要成為硬體工程師，但需要有足夠的心智模型（mental model）來做正確判斷。
5. 預告：我們將跟著「一筆訓練資料從磁碟到 GPU 完成一次前向計算」這條路徑，逐層拆解硬體。

**視覺建議：**

**過渡：**  
「讓我們從 CPU 開始，因為你的程式都在這裡啟動。」

### 投影片 2：CPU — 不只是「計算速度」

**核心訊息：**  
CPU 的效能由 pipeline 效率、指令並行度、以及分支預測準確率共同決定，單純的 GHz 數字遠不足以說明全貌。

**談話要點：**

1. 指令週期的四個階段：fetch（取指令）、decode（解碼）、execute（執行）、writeback（寫回）。現代 CPU 不是一次做完四步，而是四步同時在做不同指令——這就是 pipeline（流水線）。
2. 以洗碗來類比 pipeline：不是洗完一個碗再拿下一個，而是洗、沖、擦、放同時進行。理論吞吐量從 4 步驟 = 4 cycle 降為 1 cycle/指令。
3. Pipeline 的弱點：分支指令（if/else）會導致 pipeline 停頓（stall）。CPU 必須「猜」執行哪條分支——這是分支預測（branch prediction）。猜錯代價是清空 pipeline，損失 10-20 個 cycle。
4. ILP（指令層級平行性）：現代 CPU 有多個執行單元，可以同時執行多條不相依的指令，稱為 superscalar 架構。這是一個 4 GHz CPU 實際上每個 cycle 能完成多於 1 條指令的原因。
5. 工程含義：可預測的控制流（少 if/else、多順序操作）對 CPU 更友好。這也解釋了為什麼向量化操作（numpy）比 Python loop 快很多。

**視覺建議：**  
Pipeline 執行示意圖：橫軸是時間（cycle），縱軸是指令（I1, I2, I3...），每格填入 F/D/E/W 來展示流水線如何重疊執行。旁邊加一個對比：沒有 pipeline vs. 有 pipeline 的 cycle 數差異。

**過渡：**  
「CPU 有了 pipeline 和多核心，但它還有一個更根本的限制——快取。」

### 投影片 3：多核心的真相

**核心訊息：**  
多核心能提升吞吐量，但並行需要程式設計支援，且核心之間的協調本身有代價。

**談話要點：**

1. 現代 CPU 有 8/16/32 個核心，每個核心都有完整的 pipeline。不同的程序（process）和執行緒（thread）可以在不同核心上同時執行。
2. 關鍵問題：你的程式是否在利用多核心？Python 的 GIL（Global Interpreter Lock）讓多執行緒 Python 程式無法真正並行執行 CPU 密集任務。這是 Python 並發的著名限制。
3. 超執行緒（Hyper-Threading）：一個物理核心模擬成兩個邏輯核心。適合 I/O 密集任務，對 CPU 密集任務效益有限——因為底層執行資源（ALU、FPU）仍然共享。
4. 核心間通訊的代價：當多個核心存取同一塊記憶體，快取一致性協定（cache coherence）需要協調哪個核心的快取資料是最新的。這個協調本身有延遲。
5. DataLoader 的 `num_workers`：開多個 worker process（不是 thread），繞過 GIL，讓 I/O 和資料預處理在多個核心上並行。這是 Python 多核心利用的正確姿勢。

**視覺建議：**  
兩欄對比：左欄「多執行緒 Python（受 GIL 限制）」、右欄「多程序 DataLoader workers（真並行）」。用時間軸和 CPU 核心分配圖展示差異。

**過渡：**  
「多核心解決了計算並行的問題，但還有一個更隱蔽的問題在等著你：記憶體存取速度。」

### 投影片 4：快取 — 記憶體系統的關鍵

**核心訊息：**  
快取失誤的延遲可以是快取命中的 100 倍。這個數字決定了程式碼的存取模式有多重要。

**談話要點：**

1. 為什麼需要快取？CPU 的計算速度和 RAM 的存取速度之間有巨大鴻溝。一個現代 CPU core 每秒可執行數十億次操作，但 RAM 存取延遲約 60-80 ns，而 CPU 一個 cycle 約 0.3 ns——相差 200 倍以上。沒有快取，CPU 80% 的時間都在等記憶體。
2. 三層快取的規模與延遲：
   - L1 cache：約 32-64 KB per core，延遲 ~4 cycles（約 1 ns）
   - L2 cache：約 256 KB - 1 MB per core，延遲 ~12 cycles（約 3 ns）
   - L3 cache：約 8-32 MB，多核共享，延遲 ~40 cycles（約 10 ns）
   - RAM：延遲 ~200 cycles（約 60-80 ns）
3. 快取運作原理：CPU 要讀取記憶體地址 X 時，先查 L1，找到就用（cache hit）；沒找到就查 L2，依此類推；最終沒有就從 RAM 載入一整個「快取行」（cache line，64 bytes）。
4. Cache miss 的真實代價：L1 miss 讓你等 ~10 ns，RAM 存取讓你等 ~70 ns。如果你的程式頻繁觸發 cache miss，大部分時間都在「等」，不在「算」。
5. 為什麼這對 Python 工程師重要？`numpy` 陣列是連續記憶體，一次 cache miss 能預載 64 bytes（16 個 float32）。Python list 的元素是指標，每個元素指向一個獨立的 Python 物件，存取下一個元素就是一次新的 cache miss。這是 numpy vs. list 速度差的核心原因。

**視覺建議：**  
一張「記憶體層次金字塔」：從頂（L1，最快最小）到底（磁碟，最慢最大）。每層標示：容量範圍、典型延遲（用 ns 和 cycle 雙標），以及「如果每次 L1 hit = 1 秒，L2 = 3 秒，L3 = 10 秒，RAM = 1 分鐘，磁碟 = 幾天」的類比。

**過渡：**  
「知道了快取失誤的代價，下一步是：你能控制嗎？答案是可以——透過局部性原理。」

### 投影片 5：局部性原理 — 讓快取幫你工作

**核心訊息：**  
程式碼的存取模式決定快取效率。時間局部性和空間局部性是讓快取命中率提升的兩個設計原則。

**談話要點：**

1. 時間局部性（Temporal Locality）：剛被存取的資料，很快會被再次存取。意味著「重複使用同一批資料」的演算法（例如迴圈）能從快取受益。對策：減少資料集的工作集（working set）大小，讓它能放進 L3 甚至 L2。
2. 空間局部性（Spatial Locality）：剛被存取的記憶體地址，附近的地址很快也會被存取。快取行（cache line）的設計正是利用了這個原理——一次 miss 就載入 64 bytes，若下次存取的資料就在旁邊，就是免費的 cache hit。
3. Row-major vs. Column-major 的快取影響：矩陣在記憶體中按列儲存（row-major，C/numpy 預設）。按列走訪（row by row）符合空間局部性；按行走訪（column by column）每次存取跳躍一整列，大量 cache miss。這就是為什麼矩陣乘法的迴圈順序很重要。
4. 資料結構選擇：`numpy array` 的元素緊密排列，空間局部性好。Python `list` 的元素分散在記憶體各處（因為 Python 物件動態分配），空間局部性差。`pandas DataFrame` 的欄（column）是連續記憶體——這解釋了為什麼欄操作比列操作快。
5. 工程師的行動：寫高效能程式碼時，問自己「我的存取模式是循序的嗎？我的資料集能放進快取嗎？」

**視覺建議：**  
兩個並排的程式碼片段對比：一個是 row-major 走訪的矩陣迴圈（快），一個是 column-major 走訪（慢）。下方配合記憶體布局圖，用箭頭顯示兩種走訪模式如何對應到連續 vs. 跳躍存取。

**過渡：**  
「快取層次解決了 CPU 和 RAM 之間的速度鴻溝，但 RAM 本身也有瓶頸。」

### 投影片 6：RAM — 工作記憶體的頻寬牆

**核心訊息：**  
現代計算中，記憶體頻寬（能多快傳輸資料）往往比記憶體容量（能存多少）更先成為瓶頸。

**談話要點：**

1. RAM 的角色：所有執行中的程式、載入的資料集、模型的參數——都住在 RAM 裡。CPU 和 GPU 要用任何資料，都必須先從 RAM 讀取。
2. 頻寬 vs. 延遲：延遲（latency）是單次存取需要多少時間；頻寬（bandwidth）是每秒能傳輸多少資料。對大量資料處理（如 AI 訓練），頻寬才是限制因素，不是延遲。
3. DDR 演進的本質：DDR4 典型頻寬約 50 GB/s（雙通道），DDR5 可達 80+ GB/s。每一代的核心改進都是提升頻寬，而非大幅降低延遲。
4. Memory-bound 的診斷：如果你的程式每次計算都要從 RAM 讀取新資料，而計算量本身很輕（例如矩陣加法、element-wise 操作），那麼效能瓶頸在 RAM 頻寬，提升 CPU 頻率幾乎沒用。解法是：減少記憶體存取（使用快取友善的演算法），或使用有更高頻寬的記憶體（如 GPU 的 HBM）。
5. AI 訓練的記憶體壓力：一個 70B 參數的語言模型，光儲存參數就需要約 140 GB（FP16）RAM。這不是「容量」不足，而是每次前向計算需要把所有參數通過記憶體總線搬運，頻寬是根本瓶頸。

**視覺建議：**  
一個「水管類比」圖：頻寬是水管直徑（每秒能傳多少），延遲是水的流速（從水源到水龍頭要多久）。旁邊列出 DDR4、DDR5、HBM2（GPU 用）的頻寬數字對比，凸顯 GPU 記憶體的頻寬優勢。

**過渡：**  
「正是因為 GPU 配備了超高頻寬的 HBM 記憶體，它才能成為 AI 訓練的主要硬體。讓我們來看 GPU 的架構。」

### 投影片 7：GPU — 為平行計算而生

**核心訊息：**  
GPU 有數千個簡單核心，設計目的是同時對大量資料做相同操作（SIMT），天然適合矩陣運算。

**談話要點：**

1. CPU vs. GPU 的設計哲學：CPU 有幾個（4-32）功能強大的核心，每個核心能做複雜的控制流和分支。GPU 有數千個（CUDA 核心通常 2000-16000+）功能簡單的核心，每個核心的計算能力遠低於 CPU core，但勝在數量。
2. SIMT（Single Instruction, Multiple Threads）：GPU 的執行模型是「一條指令，同時在數千個核心上對不同資料執行」。如果你要對 1000 萬個數字各加 1，GPU 一次就能做到，CPU 需要 loop。
3. Warp 的概念：NVIDIA GPU 中，32 個執行緒組成一個 warp，warp 是最小的排程單位。一個 warp 中所有執行緒執行相同指令，但各自有不同資料（不同的記憶體地址）。
4. 為什麼 GPU 適合深度學習？神經網路的前向和反向傳播本質上是矩陣乘法——對大矩陣的每個元素做相同的數學運算。這完全符合 SIMT 模型。
5. GPU 的弱點：不擅長複雜控制流（if/else 分支讓 warp 內的執行緒走不同路徑，稱為 warp divergence，效率大幅下降）。不擅長隨機記憶體存取（相比連續存取）。

**視覺建議：**  
左右對比圖：左邊 CPU，4 個大核心，標示「擅長複雜任務、控制流、分支」；右邊 GPU，數千個小核心排列成方陣，標示「擅長大量同質計算、矩陣運算、無分支迴圈」。底部加一行：「神經網路訓練 = 大量矩陣乘法 = GPU 的主場」。

**過渡：**  
「在 GPU 的基礎上，NVIDIA 進一步加入了 Tensor Core，專門為 AI 矩陣運算打造。」

### 投影片 8：Tensor Core 與深度學習加速

**核心訊息：**  
Tensor Core 是專門為混合精度矩陣乘法設計的硬體單元，能在單個 cycle 內完成 CUDA core 需要數十個 cycle 才能完成的計算。

**談話要點：**

1. CUDA Core vs. Tensor Core：CUDA Core 是通用浮點計算單元，適合任何操作。Tensor Core（從 Volta 架構引入）是專用矩陣乘法加速器，每個 cycle 能完成 4x4 矩陣乘加（D = A×B + C），相較 CUDA Core 有 8-16 倍的矩陣運算吞吐量提升。
2. 混合精度訓練（Mixed Precision Training）：Tensor Core 最高效的模式是 FP16（半精度）輸入、FP32（單精度）輸出的矩陣乘法。PyTorch 的 `torch.cuda.amp`（Automatic Mixed Precision）自動使用 Tensor Core，訓練速度可提升 2-3 倍，顯存使用量減半。
3. A100 / H100 中的 Tensor Core：現代 NVIDIA GPU 的 Tensor Core 還支援 BF16、TF32、INT8 等精度，每種精度對應不同的速度與精確度取捨。
4. 工程師的注意事項：要讓 Tensor Core 高效工作，矩陣的維度必須是 8 的倍數（對 FP16）或 16 的倍數（對 INT8）。這就是為什麼模型的 hidden size 通常是 256、512、1024 這樣的數字，而不是隨意設置。
5. 不使用混合精度的代價：在 A100 上，純 FP32 訓練只能利用 CUDA Core，Tensor Core 閒置。這相當於買了一輛跑車但只用 2 檔行駛。

**視覺建議：**  
一個簡化的 4x4 矩陣乘法示意圖，標示「Tensor Core 在一個 cycle 完成這個操作」。旁邊對比 CUDA Core 需要多少個 cycle 做同樣的事。底部加入 PyTorch AMP 的程式碼片段（4 行）作為「如何啟用」的直接指引。

### 投影片 9：儲存層次 — 速度與容量的取捨

**核心訊息：**  
每一層儲存媒介的速度差距都是 10-100 倍，選錯存取模式可以讓整個訓練管線慢一個數量級。

**談話要點：**

1. 三種主要儲存媒介的特性比較：
   - HDD（機械硬碟）：循序讀取約 150-200 MB/s，隨機存取延遲約 5-10 ms（磁頭需要物理移動）
   - SATA SSD：循序讀取約 550 MB/s，隨機存取延遲約 0.1 ms
   - NVMe SSD：循序讀取約 3,500-7,000 MB/s，隨機存取延遲約 0.02-0.05 ms
2. 循序存取 vs. 隨機存取：對 HDD 影響最大——因為磁頭移動是機械動作。對 SSD 影響較小，但仍存在（Flash 記憶體的存取顆粒度和平行性問題）。訓練 AI 時，資料集應該以循序存取友善的格式儲存（如 TFRecord、HDF5、WebDataset）。
3. IOPS（每秒輸入輸出操作數）：對於大量小檔案的資料集（如 ImageNet 的 JPG 檔），HDD 和 SSD 的差距主要體現在 IOPS，而非循序讀取速度。HDD 典型 100-200 IOPS，NVMe 可達 500,000+ IOPS。
4. 實際工程場景：在 HDD 上用小 batch 訓練影像模型，80% 的時間可能都在等磁碟 I/O，GPU 幾乎閒置。解法：換 NVMe 或把資料預先載入 RAM。
5. 雲端訓練的特殊性：雲端的物件儲存（S3、GCS）延遲更高（通常 1-100 ms），必須使用專門的資料管線（如 FFCV、WebDataset）做大批次預取，否則 I/O 就是殺手。

**視覺建議：**  
一張三行的速度對比表：HDD / SATA SSD / NVMe，列出循序讀取速度、隨機讀取延遲、IOPS。每格用顏色編碼（紅/黃/綠）顯示效能等級。底部加一行：「訓練資料在 HDD 上 = 把 GPU 的潛力鎖住 90%」。

**過渡：**  
「知道了每層的速度，下一步是理解資料如何從磁碟一路走到 GPU 核心。這條路徑比你想的複雜。」

### 投影片 10：資料移動的真實代價

**核心訊息：**  
「資料搬到哪裡，計算就在哪裡」——每次跨層移動都有頻寬代價，訓練效率由最慢的一段決定。

**談話要點：**

1. 完整的資料路徑：磁碟 → RAM（透過 DMA，直接記憶體存取）→ CPU L3 Cache（部分資料）→ GPU VRAM（透過 PCIe 匯流排）→ SM（Streaming Multiprocessor）→ Tensor Core。
2. 每段的頻寬上限（以 PCIe 4.0 x16 系統為例）：
   - NVMe 讀取：約 7 GB/s
   - PCIe 4.0 x16（CPU↔GPU）：約 32 GB/s（雙向各 16 GB/s）
   - DDR5 RAM 頻寬：約 80 GB/s
   - GPU HBM2e 頻寬：約 900 GB/s - 2 TB/s
3. PCIe 是隱藏的瓶頸：GPU 的 HBM 頻寬高達 TB/s，但 CPU→GPU 的資料傳輸只有約 16 GB/s（PCIe 4.0）。這意味著：如果你在訓練時頻繁地在 CPU 和 GPU 之間移動資料，PCIe 頻寬就是你的上限，無論 GPU 多強大。
4. 工程含義：儘量讓資料「住在」GPU VRAM 裡。避免在每個 mini-batch 之間把資料搬回 CPU 做後處理（如 CPU 上的資料增強），再搬回 GPU。
5. NVLink（補充知識）：高端伺服器（如 DGX）使用 NVLink 讓多個 GPU 直接互連，頻寬可達 600 GB/s——繞過了 PCIe 的瓶頸。這是大規模分散式訓練的基礎設施。

**視覺建議：**  
一條橫向「資料流」圖，從左到右：磁碟 → RAM → GPU VRAM → SM。每個箭頭標示頻寬數字，用箭頭的寬度直觀表示頻寬大小。最後一段（SM 內部）明顯最寬，第一段（磁碟讀取）最窄——讓學員直觀看到瓶頸通常在哪裡。

**過渡：**  
「知道了資料移動的代價，DataLoader 的設計就不再是玄學，而是有明確硬體動機的工程決策。」

### 投影片 11：DataLoader 的硬體哲學

**核心訊息：**  
DataLoader 的每一個參數（`num_workers`、`pin_memory`、`prefetch_factor`）都對應一個硬體優化，不是「調參玄學」。

**談話要點：**

1. 問題出發：一個 GPU 訓練迴圈的時間線。如果沒有最佳化：GPU 做計算 → 計算完成 → 等待 CPU 準備下一個 batch（讀磁碟、解碼、增強）→ 把 batch 搬到 GPU → GPU 繼續計算。GPU 大量時間在等待，使用率低下。
2. `num_workers`：開啟多個 worker process（非 thread，因此不受 GIL 限制），在 CPU 的多個核心上並行做資料讀取和前處理。目標：讓 CPU 的資料準備速度 > GPU 的消耗速度，GPU 永遠有資料可吃。
3. `pin_memory=True`：預設情況下，CPU 的 RAM 是「可分頁記憶體」（pageable），GPU 無法直接存取；需要先複製到「頁面鎖定記憶體」（pinned memory），然後才能 DMA 傳輸到 GPU。`pin_memory=True` 讓 DataLoader 直接把資料放在 pinned memory，省掉這次複製，加快 CPU→GPU 傳輸速度（通常 20-30% 提升）。
4. `prefetch_factor`：每個 worker 預先準備多少個 batch 放在 buffer 裡。讓 GPU 在消費當前 batch 的同時，下一個 batch 已在準備中——這是硬體上的 pipeline 原理在軟體資料管線上的複現。
5. 整體效果：最佳化後的 DataLoader 讓 GPU 訓練從 I/O-bound 變成 compute-bound，這才是你買 GPU 本來期望的狀態。監控工具：`nvidia-smi` 看 GPU 使用率，Pytorch Profiler 看 DataLoader 時間佔比。

**視覺建議：**  
兩條時間軸對比：上方「沒有最佳化」的 GPU 使用率（交替的計算和等待，使用率約 40%），下方「有最佳化」的使用率（接近連續計算，使用率 90%+）。旁邊標示每個最佳化技術對應改善了哪段空白。

**過渡：**  
「把所有硬體層拼在一起，我們可以得出一個結論，它改變了我們對現代計算瓶頸的理解。」

### 投影片 12：記憶體頻寬 — 現代計算的真正瓶頸

**核心訊息：**  
在現代 AI 計算中，把資料搬到計算單元的速度，比計算本身更常成為瓶頸。這個認知改變了系統設計的優先順序。

**談話要點：**

1. 回顧數字的意義：CPU 的 ALU 理論峰值算力遠超 RAM 頻寬所能餵給它的資料量。GPU 更明顯——A100 的 FP32 算力是 19.5 TFLOPS，HBM2 頻寬是 2 TB/s，以 FP32（4 bytes）計算，頻寬只能支撐 500 GFLOPS 的計算——算力的 2.5%。這就是「roofline model」的核心洞察。
2. Roofline Model 的直觀理解：每個演算法都有「算術強度」（Arithmetic Intensity）= FLOP / bytes。如果算術強度低（每搬一次資料只做少量計算），就是 memory-bound；如果算術強度高（每搬一次資料做大量計算），才能是 compute-bound。矩陣乘法算術強度高（O(n³) 計算，O(n²) 記憶體），element-wise 操作算術強度低（O(n) 計算，O(n) 記憶體）。
3. 診斷框架的應用：
   - 程式慢 → 先看 GPU 使用率（`nvidia-smi`）
   - 使用率低 → I/O-bound（磁碟/PCIe），修 DataLoader
   - 使用率高但速度慢 → compute-bound 或 memory-bound，看 profiler
   - memory-bound → 考慮 fusion（把多個 kernel 合併），減少記憶體來回
4. 更廣的系統設計含義：同樣的原則適用於分散式系統——網路頻寬往往是分散式訓練的瓶頸；適用於資料庫——磁碟 I/O 是 OLAP 查詢的瓶頸；適用於微服務——序列化/反序列化的記憶體操作可能比業務邏輯更耗時。
5. 一句話總結：「計算是廉價的，搬資料才是昂貴的。好的系統設計，是讓資料少動，讓計算在資料旁邊發生。」

**視覺建議：**  
一個 Roofline Model 示意圖（雙對數坐標系）：橫軸算術強度，縱軸算力（GFLOPS）。兩條線：水平的「算力上限」和斜向上的「頻寬上限」。在圖上標示幾個典型操作（矩陣乘法在右側 compute-bound 區，element-wise 加法在左側 memory-bound 區）。

**過渡（模組收尾）：**  
「帶著這個硬體心智模型，下一個模組 M5 將討論作業系統如何在這些硬體之上協調資源——process、thread、context switch，以及為什麼你的程式有時候「突然變慢」是 OS 的調度在作怪。」

### 練習一：記憶體布局觀察實驗

**目標：** 直觀體驗 cache 局部性對效能的影響。

**難度：** 基礎  
**預計時間：** 30 分鐘

**說明：**

以下兩個函數做相同的矩陣求和，但存取順序不同：

```python
import numpy as np
import time

def sum_row_major(matrix: np.ndarray) -> float:
    """Row-major access — follows memory layout."""
    total = 0.0
    rows, cols = matrix.shape
    for i in range(rows):
        for j in range(cols):
            total += matrix[i, j]
    return total

def sum_col_major(matrix: np.ndarray) -> float:
    """Column-major access — goes against memory layout."""
    total = 0.0
    rows, cols = matrix.shape
    for j in range(cols):
        for i in range(rows):
            total += matrix[i, j]
    return total

# Setup
N = 2000
matrix = np.random.rand(N, N).astype(np.float64)

# Benchmark
start = time.perf_counter()
sum_row_major(matrix)
row_time = time.perf_counter() - start

start = time.perf_counter()
sum_col_major(matrix)
col_time = time.perf_counter() - start

print(f"Row-major: {row_time:.3f}s")
print(f"Col-major: {col_time:.3f}s")
print(f"Speedup: {col_time / row_time:.1f}x")
```

3. 修改 `N = 100`，重新執行。時間差異是否縮小了？為什麼？（提示：小矩陣可以完全放入快取）
4. （進階）嘗試使用 `np.asfortranarray(matrix)` 把矩陣轉成 column-major，重新執行 `sum_col_major`，觀察結果。

### 練習二：DataLoader 瓶頸診斷

**目標：** 實際測量 DataLoader 的各種設定對訓練吞吐量的影響，理解每個參數的硬體意義。

**難度：** 中階  
**預計時間：** 45 分鐘  
**前置需求：** PyTorch、至少一個 CUDA GPU

**說明：**

```python
import torch
from torch.utils.data import Dataset, DataLoader
import time
import numpy as np

class SyntheticDataset(Dataset):
    """Synthetic image dataset with configurable I/O simulation."""
    
    def __init__(self, num_samples: int = 10000, simulate_io: bool = False):
        self.num_samples = num_samples
        self.simulate_io = simulate_io
        # Pre-allocate data in memory
        self.data = torch.randn(num_samples, 3, 224, 224)
        self.labels = torch.randint(0, 1000, (num_samples,))
    
    def __len__(self) -> int:
        return self.num_samples
    
    def __getitem__(self, idx: int):
        if self.simulate_io:
            # Simulate disk I/O delay (~2ms per sample, like a slow HDD)
            time.sleep(0.002)
        return self.data[idx], self.labels[idx]

def benchmark_dataloader(
    dataset: Dataset,
    num_workers: int,
    pin_memory: bool,
    prefetch_factor: int = 2,
    batch_size: int = 64,
    num_batches: int = 50,
) -> dict:
    loader = DataLoader(
        dataset,
        batch_size=batch_size,
        num_workers=num_workers,
        pin_memory=pin_memory,
        prefetch_factor=prefetch_factor if num_workers > 0 else None,
        shuffle=True,
    )
    
    device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
    
    start = time.perf_counter()
    for i, (data, label) in enumerate(loader):
        if i >= num_batches:
            break
        # Simulate GPU computation
        data = data.to(device, non_blocking=True)
        _ = data.mean()  # Trivial computation
    elapsed = time.perf_counter() - start
    
    throughput = (num_batches * batch_size) / elapsed
    return {
        "num_workers": num_workers,
        "pin_memory": pin_memory,
        "throughput_samples_per_sec": throughput,
        "total_time_s": elapsed,
    }

# Run experiments
dataset = SyntheticDataset(num_samples=5000, simulate_io=True)

configs = [
    (0, False),
    (0, True),
    (2, False),
    (2, True),
    (4, True),
    (8, True),
]

print(f"{'num_workers':>12} | {'pin_memory':>10} | {'throughput':>15} | {'time':>8}")
print("-" * 55)
for num_workers, pin_memory in configs:
    result = benchmark_dataloader(dataset, num_workers, pin_memory)
    print(
        f"{result['num_workers']:>12} | "
        f"{str(result['pin_memory']):>10} | "
        f"{result['throughput_samples_per_sec']:>12.0f}/s | "
        f"{result['total_time_s']:>6.2f}s"
    )
```

**任務：**

1. 執行程式碼，記錄不同設定的吞吐量。
2. 繪製一個簡單的長條圖，比較各設定的 throughput。
3. 回答以下問題：
   - `num_workers=0` 與 `num_workers=2` 的差異說明了什麼？
   - `pin_memory=True` 的效果在什麼設定下最明顯？
   - 繼續增加 `num_workers` 到 16，吞吐量是否持續提升？到什麼點開始飽和或下降？
4. 把 `simulate_io=False`，重新執行。`num_workers` 的效果是否消失了？這說明了什麼？

**預期發現：**  
I/O 模擬開啟時，`num_workers` 的效果顯著（多核並行解決 I/O 瓶頸）；`pin_memory` 在有 GPU 且 `num_workers>0` 時效果最明顯。I/O 模擬關閉時，資料已在記憶體，`num_workers` 反而可能因為 process 啟動開銷而略慢。

### 常見學員誤解與應對

**誤解 1：「GPU 快是因為時脈速度高」**  
澄清：GPU 的單核時脈速度通常比 CPU 低（~1.5 GHz vs. ~3-5 GHz）。GPU 快是因為核心數量多（SIMT 大規模並行），以及 HBM 的極高記憶體頻寬。

**誤解 2：「快取是 OS 或硬體自動管的，程式設計師不用在意」**  
澄清：快取本身確實是硬體自動管理的，但程式設計師的**存取模式**決定了快取的命中率。寫高效能程式碼的關鍵在於讓你的存取模式符合快取的工作方式（局部性）。

**誤解 3：「增加 `num_workers` 越多越好」**  
澄清：`num_workers` 的最佳值取決於 CPU 核心數和 I/O 特性。超過 CPU 核心數後，worker process 之間的上下文切換反而增加開銷。一般建議從 `num_workers = 2-4` 開始，再根據 profiling 結果調整。

**誤解 4：「我的資料全在 RAM，不需要最佳化 I/O」**  
澄清：即使資料在 RAM，從 RAM 搬到 GPU VRAM 仍需通過 PCIe 匯流排（~16 GB/s），`pin_memory` 仍然有意義。更重要的是，如果資料前處理（augmentation）在 CPU 上做，CPU→GPU 的傳輸仍是潛在瓶頸。

## 八、模組小結

**三個關鍵認知：**

1. **計算層次：** CPU 靠 pipeline 和 ILP 提升吞吐量；GPU 靠 SIMT 大規模並行；兩者都受到各自記憶體頻寬的限制。

2. **快取是效能的關鍵槓桿：** 程式碼的存取模式（是否符合時間和空間局部性）直接決定快取命中率，進而決定真實效能。這是「numpy 比 list 快」、「row-major 比 col-major 快」、「連續緩衝區比鏈結串列快」的共同根因。

3. **資料移動是現代系統的主要成本：** 磁碟→RAM→GPU 的每一次搬移都有頻寬代價。DataLoader 的最佳化策略（`num_workers`、`pin_memory`、`prefetch_factor`）本質上都是在減少資料移動的代價，或讓資料移動和計算重疊執行（pipeline）。

**一句話帶走：**

> 計算是廉價的，搬資料才是昂貴的。理解這一點，你就理解了大部分效能問題的根源。
