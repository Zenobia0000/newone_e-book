**M6 是整門課的樞紐。**

這不是技術細節的疊加，而是一次思維方式的根本轉換。

| 這個 SQL query 怎麼優化？ | 這個場景應該用 SQL 還是 NoSQL？ |

## 學習目標

完成本模組後，你將能夠：

1. **解釋思維躍遷**：用具體語言描述從「實作視角」到「設計視角」的差異，並在面對需求時主動切換到系統視角
2. **運用 RESHADED 框架**：依照八個步驟系統化地分析任意規模的系統設計題目，不再漫無目的地畫方塊圖
3. **做出擴展性決策**：判斷一個場景需要垂直擴展還是水平擴展，並說明理由
4. **選擇資料庫**：根據一致性、查詢模式、規模需求，為不同場景選擇 SQL 或 NoSQL，並能說明取捨
5. **設計快取策略**：決定快取應該放在哪一層、用哪種失效策略，理解快取與資料庫一致性的張力
6. **理解負載平衡**：解釋負載平衡器的作用與常見算法，說明無狀態設計為何是水平擴展的前提
7. **設計基礎 API**：遵循 RESTful 慣例設計資源導向的 API，理解冪等性與版本管理的重要性
8. **走完架構七步流程**：用架構師的七步設計流程從零分析一個系統，產出可執行的架構草圖

### Slide 2：思維躍遷——兩種工程師的對話

**版面：左右分欄對話氣泡**

**場景設定：** 產品經理說：「我們需要一個讓用戶上傳照片的功能。」

**左欄 — 工程師思維（技術實作視角）：**
> - 「用 multipart/form-data 上傳？」
> - 「儲存在本地磁碟還是 S3？」
> - 「要做圖片壓縮嗎？」
> - 「資料庫怎麼存路徑？」

**右欄 — 架構師思維（系統設計視角）：**
> - 「預期 DAU 是多少？同時上傳的峰值 QPS？」
> - 「照片的存取模式？上傳多讀少？還是讀寫均衡？」
> - 「全球用戶還是單一地區？需要 CDN 嗎？」
> - 「保留多久？容量怎麼增長？成本預算？」

**結論：**
工程師問「怎麼做」。架構師先問「做什麼、做多大、做給誰、成本是什麼」。

**兩者都需要，但順序不能錯。** 架構決策一旦錯誤，需要改的不是幾行程式碼，而是整個系統。

**講師備注：**
這不是說工程師比架構師差——恰恰相反，好的架構師必然是出色的工程師。差別在於視角的高度和關注的時間尺度。工程師優化今天的實作；架構師設計未來三年的系統邊界。

### Slide 3：RESHADED 框架全覽

**版面：八格圓環，每格一個字母**

**框架說明：** RESHADED 是一套系統設計的思考框架，確保你在設計系統時不遺漏任何關鍵維度。

```
        R
   D         E
  E           S
   S         H
        A
```

| 字母 | 步驟 | 核心問題 | 輸出產物 |
|---|---|---|---|
| **R** | Requirements（需求） | 系統要做什麼？不做什麼？ | 功能清單、NFR 指標 |
| **E** | Estimation（估算） | 規模有多大？ | QPS、儲存量、頻寬數字 |
| **S** | Storage Schema（資料模型） | 數據長什麼樣？存在哪？ | ER 圖、Schema 草圖 |
| **H** | High-level Design（高階設計） | 系統方塊圖是什麼？ | 架構方塊圖 |
| **A** | API Design（介面設計） | 系統的入口和出口？ | REST API 定義 |
| **D** | Detailed Design（詳細設計） | 核心組件如何運作？ | 關鍵算法、快取策略 |
| **E** | Evaluation（評估） | 設計的弱點是什麼？ | 瓶頸分析、Trade-off 清單 |
| **D** | Distinctive Features（特殊功能） | 邊緣案例怎麼處理？ | 特殊場景設計 |

**使用時機：**
- 系統設計面試（30-45 分鐘面試的黃金結構）
- 新功能上線前的設計審查
- 評估現有系統的改進方向

**講師備注：**
RESHADED 不是一個死板的檢查清單，而是確保「不遺漏重要問題」的思考導軌。在面試中，前四步（R、E、S、H）的時間要占 60% 以上——大多數候選人的失敗不是因為不懂技術，而是因為跳過需求澄清，直接開始畫方塊圖，最後發現自己設計了一個不符合題目的系統。

### Slide 4：擴展性——向上 vs 向外

**版面：左右對比 + 底部結論表格**

**問題設定：** 你的服務現在每秒處理 1,000 個請求。兩週後要上線一個行銷活動，預計流量暴增 10 倍。怎麼辦？

**垂直擴展（Scale Up）— 換更強的機器**
```
  [小伺服器 2核4G]  →  [大伺服器 16核64G]
      1,000 QPS    →      10,000 QPS
```
- 優點：簡單，不需要改程式碼，沒有分散式複雜度
- 缺點：有物理上限（單機最大只能這麼大）、停機升級、費用非線性成長
- 適合：早期快速增長、有狀態服務、關聯式資料庫

**水平擴展（Scale Out）— 加更多台機器**
```
              ┌── [伺服器 A] ──┐
客戶端 → LB → ├── [伺服器 B] ──┤ → 資料庫
              └── [伺服器 C] ──┘
```
- 優點：理論上無限擴展、容錯（一台壞了其他繼續跑）、可按需增減機器
- 缺點：需要負載平衡器、Session 管理變複雜、程式必須是無狀態（Stateless）
- 適合：Web 服務層、API 服務、現代雲端架構

**無狀態（Stateless）是水平擴展的前提**

如果你的 API 伺服器把用戶 Session 存在本機記憶體裡，那麼同一個用戶的請求必須打到同一台機器（Sticky Session）。這破壞了水平擴展的靈活性。

解法：把 Session 外移到共享的 Redis 或資料庫，讓每一台伺服器都「無記憶」——請求打到哪台都能正確處理。

**M5 連結：** Process / Thread 的管理知識告訴我們，單機的並發能力有上限。水平擴展是突破這個上限的工程方法。

### Slide 5：資料庫選型地圖

**版面：決策樹 + 對比表格**

**核心問題不是「哪個更好」，而是「哪個更合適」。**

**SQL（關聯式資料庫）**

代表：PostgreSQL、MySQL、SQLite

核心特性：ACID
- **A**tomicity（原子性）：一個交易要嘛全成功，要嘛全失敗
- **C**onsistency（一致性）：交易前後資料符合所有約束
- **I**solation（隔離性）：並行交易互不干擾
- **D**urability（持久性）：已提交的交易永遠保存

適合場景：
- 金融交易（不能有資料遺失或不一致）
- 電商訂單（訂單、庫存、付款的強關聯）
- 任何需要複雜 JOIN 查詢的業務

**NoSQL（非關聯式資料庫）**

| 類型 | 代表 | 最佳場景 |
|---|---|---|
| 文件型 | MongoDB | 用戶個人檔案、產品目錄（Schema 彈性） |
| 鍵值型 | Redis | 快取、Session、計數器（極速讀寫） |
| 列族型 | Cassandra | 時序資料、日誌（高寫入、按時間查詢） |
| 圖型 | Neo4j | 社交關係、推薦系統（關係查詢） |

BASE 原則（NoSQL 的設計哲學）：
- **B**asically Available（基本可用）：系統大部分時間可用，允許局部故障
- **S**oft state（軟狀態）：數據狀態可能在不同節點暫時不一致
- **E**ventually consistent（最終一致）：最終所有節點會達到一致

**選型決策樹：**
```
需要複雜 JOIN？──是──► SQL
        │
        否
        │
數據 Schema 固定？──是──► SQL
        │
        否
        │
需要超高寫入 QPS？──是──► Cassandra / Redis
        │
        否
        │
數據是圖關係？──是──► Neo4j
        │
        否
        └──────────────► MongoDB
```

**黃金準則：** 不要因為 NoSQL「潮流」就放棄 SQL。關聯式資料庫解決的問題，在過去五十年從未消失。選錯資料庫是系統重寫的主要原因之一。

### Slide 6：快取策略三層架構

**版面：分層架構圖 + 各層說明**

**核心原則：** 快取是用記憶體換時間。熱門數據存在離用戶更近、速度更快的位置，減少對慢速資源（磁碟 DB）的訪問。

**三層快取架構：**

```
用戶
  │
  ▼
[第一層：CDN]
  瀏覽器 / CloudFront / Nginx
  快取靜態資源（圖片、CSS、JS）
  延遲：< 10ms，節省頻寬
  │
  ▼
[第二層：應用層快取]
  Redis / Memcached
  快取計算結果、用戶 Session、熱門查詢
  延遲：< 1ms，減少資料庫壓力
  │
  ▼
[第三層：資料庫層快取]
  MySQL Query Cache / PostgreSQL Buffer
  資料庫內建的查詢結果快取
  延遲：10-100ms
  │
  ▼
[磁碟儲存]
  延遲：1-10ms（SSD）/ 5-20ms（HDD）
```

**M4 連結：** 這個分層快取架構完美對應 M4 學到的記憶體階層（L1/L2/L3 Cache → RAM → 磁碟）。原理相同：速度快的容量小，離計算單元近；速度慢的容量大，離計算單元遠。

**快取命中率（Cache Hit Rate）是關鍵指標：**

`命中率 = 從快取成功讀取次數 / 總讀取次數`

命中率 90% 意味著每 10 次讀取，只有 1 次需要打到資料庫。對於高並發系統，這個差距決定了資料庫是否能撐住。

### Slide 7：Cache-Aside 模式與快取失效

**版面：讀寫流程圖 + 失效問題說明**

**最常見的快取模式：Cache-Aside（旁路快取）**

讀取流程：
```
讀請求
  │
  ▼
查 Redis ──命中──► 返回資料 (快)
  │
未命中
  │
  ▼
查資料庫 ──► 返回資料 (慢) ──► 寫入 Redis ──► 返回資料
```

寫入流程：
```
寫請求
  │
  ▼
寫入資料庫 ──► 刪除 Redis 中的舊快取 ──► 完成
```

注意：寫入時是「刪除」快取，而不是「更新」快取。這是為了避免並發寫入時的競態條件（Race Condition）。

**Write-Through 模式（寫穿）：**
```
寫請求 ──► 同時寫入 Redis + DB ──► 完成
```
優點：快取永遠是最新的
缺點：寫入較慢（需要等兩個操作）

**快取失效（Cache Invalidation）——電腦科學的兩大難題之一**

> "There are only two hard things in Computer Science: cache invalidation and naming things."
> — Phil Karlton

快取失效策略：
- **TTL（Time To Live）過期**：設置有效期，到期自動刪除。簡單，但時效性差。
- **主動失效**：資料更新時主動刪除快取。時效性好，但需要程式碼配合，易有 bug。
- **事件驅動失效**：透過消息隊列廣播失效事件。複雜但可靠。

**快取雪崩（Cache Stampede）問題：**
如果大量快取同時過期，所有請求同時打到資料庫，可能導致資料庫崩潰。

解法：設置隨機 TTL 偏移（例如 3600 秒加上 0-300 秒的隨機值），讓過期時間分散。

### Slide 8：負載平衡基礎

**版面：架構圖 + 算法比較表**

**負載平衡器（Load Balancer）的作用：**

```
                    ┌──► [伺服器 A]  ◄── 30% CPU
客戶端請求  ──► LB ─├──► [伺服器 B]  ◄── 25% CPU
                    └──► [伺服器 C]  ◄── 45% CPU
```

1. **分散流量**：把請求均勻分配給後端伺服器，避免單台過載
2. **健康檢查**：定期 ping 後端服務，自動排除不健康的節點
3. **零停機更新**：逐一重啟服務，更新時不中斷服務（Rolling Deploy）
4. **SSL 終止**：統一處理 HTTPS 加解密，後端伺服器用 HTTP 通訊

**常見負載平衡算法：**

| 算法 | 說明 | 適合場景 |
|---|---|---|
| Round-Robin | 依序輪流分配 | 請求處理時間均勻 |
| Weighted Round-Robin | 依照伺服器權重分配（強機器多接） | 伺服器規格不同 |
| Least Connections | 分配給當前連線數最少的伺服器 | 長連線、WebSocket |
| IP Hash | 根據客戶端 IP 固定分配 | 需要 Session 親和性（Sticky Session）|
| Random | 隨機分配 | 簡單場景 |

**Layer 4 vs Layer 7 負載平衡：**

- **L4（傳輸層）**：根據 IP + Port 分流，不看內容，速度快
- **L7（應用層）**：能看 HTTP Header、URL Path，按內容路由（例如 `/api/` 打到 API 伺服器，`/static/` 打到 CDN），更靈活但有額外開銷

常見工具：Nginx（L7）、HAProxy（L4/L7）、AWS ALB（L7）、AWS NLB（L4）

### Slide 9：API 設計原則

**版面：好壞對比 + 六大原則**

**REST（Representational State Transfer）的核心思想：** 把系統中的每樣東西都視為「資源（Resource）」，用統一的介面（HTTP 動詞）對資源進行操作。

**好的 API vs 壞的 API：**

| 操作 | 壞的設計 | 好的設計 |
|---|---|---|
| 取得所有文章 | `GET /getArticles` | `GET /articles` |
| 取得特定文章 | `GET /getArticle?id=123` | `GET /articles/123` |
| 建立文章 | `POST /createArticle` | `POST /articles` |
| 更新文章 | `POST /updateArticle` | `PUT /articles/123` |
| 刪除文章 | `GET /deleteArticle?id=123` | `DELETE /articles/123` |

REST 六大約束（簡化版）：
1. **統一介面**：用名詞命名資源，用 HTTP 動詞表示操作
2. **無狀態**：每個請求包含完整信息，伺服器不儲存 Session 狀態
3. **可快取**：回應標記是否可快取，讓客戶端或代理重用
4. **分層系統**：客戶端不知道自己是直接連伺服器還是透過代理

**HTTP 狀態碼語義（必背）：**

| 狀態碼 | 意義 | 使用場景 |
|---|---|---|
| 200 OK | 成功 | GET、PUT 成功 |
| 201 Created | 已建立 | POST 成功建立資源 |
| 204 No Content | 成功但無內容 | DELETE 成功 |
| 400 Bad Request | 請求格式錯誤 | 參數缺失或格式錯 |
| 401 Unauthorized | 未認證 | 缺少或無效的 Token |
| 403 Forbidden | 無授權 | 有 Token 但無權限 |
| 404 Not Found | 資源不存在 | ID 對應的資源找不到 |
| 429 Too Many Requests | 超過頻率限制 | Rate Limiting |
| 500 Internal Server Error | 伺服器錯誤 | 未預期的異常 |

**冪等性（Idempotency）：**

同一個請求執行一次和執行多次，結果相同。

- `GET` — 冪等（讀取不改變狀態）
- `PUT` — 冪等（設置為特定值，多次設置結果相同）
- `DELETE` — 冪等（刪除後再刪除，資源還是不存在）
- `POST` — 通常不是冪等（每次 POST 可能建立新資源）

**為什麼冪等性重要？** 網路不可靠。客戶端超時後會重試——如果操作不冪等，重試可能導致重複扣款、重複建立訂單等問題。

**API 版本管理：**
- URL 版本：`/v1/articles`（最常見，清晰可見）
- Header 版本：`Accept: application/vnd.api+json;version=1`

### Slide 10：架構師的七步設計流程

**版面：七步驟線性流程 + 每步關鍵問題**

架構設計不是一蹴可幾的靈光乍現，而是一個有紀律的探索過程。以下七步流程是本課程架構設計線（M6-M8）的核心方法論：

```
步驟 1          步驟 2          步驟 3          步驟 4
需求分析  ──►  領域建模  ──►  技術選型  ──►  高階設計
    │                                           │
    └───────────────────────────────────────────┘
                                                │
步驟 7          步驟 6          步驟 5          ▼
演進規劃  ◄──  實施指導  ◄──  風險評估  ◄──  詳細設計
```

**七步驟詳解：**

**步驟 1：需求分析**
- 功能性需求：系統要做什麼？不做什麼？（明確邊界）
- 非功能性需求（NFR）量化：拒絕形容詞，要求數字
  - 壞：「系統要很快」
  - 好：「P99 延遲 < 200ms，P50 延遲 < 50ms」
- 約束條件：預算、Deadline、團隊技能樹

**步驟 2：領域建模**
- 識別核心實體（名詞）：User、Article、Comment
- 定義邊界（DDD 的 Bounded Context）：哪些事物緊耦合？哪些可以分開？
- 畫 ER 圖或 Domain Model

**步驟 3：技術選型**
- 基於 NFR 和團隊能力選擇技術棧
- 關鍵決策：
  - 單體（Monolith）vs 微服務？
  - SQL vs NoSQL？
  - 同步（REST/gRPC）vs 非同步（Message Queue）？
- 每個重要決策寫 ADR（Architecture Decision Record）

**步驟 4：高階設計**
- 畫出系統的方塊圖（C4 模型的 Container 層）
- 定義主要組件及其職責
- 確認數據流向（誰讀誰寫什麼）

**步驟 5：風險評估**
- 攻擊你自己的設計：
  - 如果資料庫掛了怎麼辦？（單點故障 SPOF）
  - 如果流量突然暴增 10 倍？
  - 如果第三方 API 斷線？
- 設計容錯機制：Circuit Breaker、Retry、Fallback

**步驟 6：實施指導**
- 定義開發規範（Git Flow、Linter、測試策略）
- 規劃分階段交付（MVP 先上什麼？）
- 確保架構決策被正確傳達給開發團隊

**步驟 7：演進規劃**
- 架構不是一次性產物，要能隨業務增長演進
- 定義可觀察性策略（Logging / Metrics / Tracing）
- 規劃 MVP 之後的技術路線圖

## 演練一：快速估算練習

**時間：15 分鐘 | 形式：個人作答後小組討論**

**題目：為以下三個場景估算關鍵指標**

### 場景 A：個人部落格
假設你的部落格有 10,000 名訂閱者，每天平均 3,000 次頁面瀏覽，文章每週更新 2-3 篇。

請估算：
1. 平均 QPS（每秒查詢數）
2. 峰值 QPS（假設在發文後的第一個小時流量是平均的 10 倍）
3. 每月儲存成長量（假設每篇文章含圖片平均 2MB）

### 場景 B：中型電商平台
一個台灣中型電商，DAU（日活躍用戶）50,000，每天產生 5,000 筆訂單，商品數量 100,000 件。

請估算：
1. 讀寫比例（Read:Write Ratio）
2. 資料庫每日寫入量（假設每筆訂單記錄 1KB）
3. 是否需要資料庫讀寫分離？為什麼？

### 場景 C：即時通訊服務
一個類似 LINE 的即時通訊服務，MAU（月活躍用戶）1,000 萬，每日平均每人發送 50 則訊息，訊息平均大小 200 bytes。

請估算：
1. 平均訊息 QPS
2. 每年訊息儲存量（GB）
3. 這個場景適合 SQL 還是 NoSQL？為什麼？

**參考答案（計算邏輯）：**

場景 A：
- 平均 QPS = 3,000 / (24 × 3,600) ≈ 0.035 QPS（非常低，一般共享主機即可）
- 峰值 QPS = 0.035 × 10 ≈ 0.35 QPS
- 每月儲存 = 10 篇 × 2MB = 20MB/月

場景 B：
- 電商通常讀:寫 = 10:1 以上（大多數人在瀏覽，少數人在下單）
- 每日寫入 = 5,000 筆 × 1KB = 5MB/天（相當低）
- 需要讀寫分離？此規模可能不需要，但考慮到搜尋和篩選的讀取壓力，可以做 Read Replica

場景 C：
- 平均 QPS = (1,000 萬 × 0.3 × 50) / 86,400 ≈ 1,736 QPS（日活 30% 估算）
- 年儲存 = 1,736 QPS × 200 bytes × 86,400 × 365 ≈ 10.9 TB/年
- 適合 NoSQL（Cassandra）：高寫入、按時間序列查詢、Schema 彈性

## 演練二：用 RESHADED 設計 URL 縮短器

**時間：45 分鐘 | 形式：個人設計後對照參考答案**

**題目說明：**
設計一個 URL 縮短服務，功能類似 TinyURL 或 bit.ly。

---

### R — Requirements（需求分析）

**功能性需求（在範圍內）：**
- 給定一個長 URL，生成一個短 URL
- 訪問短 URL，重定向到原始長 URL
- 可選：自訂短 URL 後綴

**功能性需求（不在範圍內，明確排除）：**
- 用戶帳號系統（為了簡化設計）
- 即時點擊分析儀表板
- 批量 URL 縮短

**非功能性需求（量化）：**
- 可用性：99.9%（允許每年 8.7 小時停機）
- 延遲：生成短 URL < 100ms；重定向 < 10ms
- 規模：每天建立 100 萬個短 URL；每天 1 億次重定向
- 短 URL 有效期：預設 5 年

---

### E — Estimation（容量估算）

**讀寫比例：**
- 寫入（建立短 URL）：100 萬/天
- 讀取（重定向訪問）：1 億/天
- 讀:寫 = 100:1（典型的讀多寫少場景）

**QPS 計算：**
- 寫入 QPS = 1,000,000 / 86,400 ≈ 12 QPS
- 讀取 QPS = 100,000,000 / 86,400 ≈ 1,157 QPS
- 峰值讀取 QPS（假設 3 倍）≈ 3,500 QPS

**儲存估算：**
- 每筆記錄：短 URL (7 bytes) + 長 URL (100 bytes avg) + 建立時間 (8 bytes) ≈ 500 bytes
- 每天新增：100 萬 × 500 bytes = 500 MB/天
- 五年總量：500 MB × 365 × 5 ≈ 912 GB ≈ 1 TB

**頻寬估算：**
- 寫入頻寬：12 QPS × 500 bytes = 6 KB/s
- 讀取頻寬：1,157 QPS × 500 bytes ≈ 578 KB/s ≈ 0.6 MB/s

---

### S — Storage Schema（資料模型）

**表格設計（SQL 方案）：**

```sql
CREATE TABLE url_mappings (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    short_code  VARCHAR(8)   NOT NULL UNIQUE,  -- 短 URL 後綴
    long_url    VARCHAR(2048) NOT NULL,         -- 原始長 URL
    created_at  TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    expires_at  TIMESTAMP    NOT NULL,
    click_count BIGINT       NOT NULL DEFAULT 0
);

CREATE INDEX idx_short_code ON url_mappings(short_code);
```

**為什麼用 SQL？**
- 資料量 1 TB，不算極大，關聯式資料庫可以處理
- 需要唯一性約束（short_code 不能重複）
- 簡單的讀寫模式，不需要 NoSQL 的 Schema 彈性

瀏覽器 / 客戶端
      │

### A — API Design（介面設計）

**建立短 URL：**
```
POST /api/v1/urls
Content-Type: application/json

Request Body:
{
    "long_url": "https://www.example.com/very/long/path",
    "custom_code": "my-link",  // optional
    "expires_days": 365         // optional, default 1825 (5 years)
}

Response 201 Created:
{
    "short_url": "https://short.ly/abc123",
    "short_code": "abc123",
    "long_url": "https://www.example.com/very/long/path",
    "expires_at": "2031-04-12T00:00:00Z"
}
```

**重定向短 URL：**
```
GET /{short_code}

Response 301 Moved Permanently:
Location: https://www.example.com/very/long/path
```

關於 301 vs 302 的選擇：
- 301（永久重定向）：瀏覽器會快取，後續直接跳轉，節省伺服器流量，但失去統計能力
- 302（臨時重定向）：每次都打回伺服器，可統計點擊數，但增加流量

---

### D — Detailed Design（詳細設計）

**短碼生成算法——核心設計問題**

要求：7 位英數字符，62 個可能字符（a-z, A-Z, 0-9），總組合數 62^7 ≈ 3.5 兆，遠超我們的需求。

**方案一：Hash 截斷**
```python
import hashlib

def generate_short_code(long_url: str) -> str:
    hash_value = hashlib.md5(long_url.encode()).hexdigest()
    return hash_value[:7]  # 取前 7 位
```
問題：不同 URL 可能產生相同的前 7 位（Hash Collision）

解法：衝突時嘗試 hash[1:8]、hash[2:9]... 直到找到未使用的短碼

**方案二：Base62 自增 ID**
```python
CHARS = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"

def id_to_short_code(num: int) -> str:
    code = []
    while num:
        code.append(CHARS[num % 62])
        num //= 62
    return ''.join(reversed(code)).zfill(7)

# ID 1 → "0000001", ID 62 → "0000010"
```

優點：無衝突，天然唯一
問題：如果直接用自增 ID，短碼是可預測的（容易被爬蟲遍歷）
解法：在 ID 前加隨機鹽，或使用 Snowflake ID

**Redis 快取策略：**
- Key：`url:{short_code}`
- Value：長 URL 字串
- TTL：與 DB 中的 `expires_at` 一致
- 快取命中率目標：> 95%（熱門連結會被反複訪問）

---

### E — Evaluation（設計評估）

**設計的強項：**
- 讀取路徑完全走快取，延遲 < 10ms 可達成
- 水平擴展 API 伺服器可以應對流量增長
- 讀寫分離減輕主庫壓力

**潛在瓶頸：**

| 瓶頸 | 描述 | 緩解方案 |
|---|---|---|
| 短碼生成競態 | 分散式環境下多台伺服器同時生成可能衝突 | 使用 Redis `INCR` 做分散式計數器，或 Snowflake ID |
| 快取雪崩 | 大量快取同時失效打垮 DB | TTL 加隨機偏移 |
| 熱點 URL | 單一 URL 被瘋狂訪問 | 本地記憶體快取（Guava Cache）+ Redis 雙層 |
| DB 寫入積壓 | 寫入 QPS 較低（12 QPS），此場景不是瓶頸 | 目前不需要處理 |

---

### D — Distinctive Features（特殊功能）

**自訂短碼：**
- 允許用戶指定短碼（如 `short.ly/my-brand`）
- 需要額外的唯一性檢查
- 可能與自動生成的短碼衝突：用前綴區分（`c_` for custom）

**URL 有效期處理：**
- 過期的短碼應該返回 410 Gone，而不是 404 Not Found
- 定期清理過期記錄（Background Job），避免 DB 無限增長
- 過期後短碼可以複用（需要確保 Redis 快取也一起清除）

**點擊統計：**
- 直接在重定向路徑上做計數會增加延遲
- 非同步計數：重定向後把事件發送到 Kafka，由消費者非同步更新 DB
- 結果：重定向延遲不受統計邏輯影響

---

**演練總結：從這道題學到的設計思路**

1. 讀:寫 = 100:1 → 快取是核心設計決策，不是優化細節
2. 重定向延遲要求 < 10ms → 完全依賴 Redis，不能每次打 DB
3. 短碼生成是分散式唯一 ID 問題 → 有多種成熟方案，選哪個取決於你能接受多少複雜度
4. 301 vs 302 是 trade-off → 沒有絕對正確答案，取決於業務需求

- Slide 8-9：負載平衡和 API 設計相對直觀，可以加快節奏。
- Slide 10：架構七步流程是 M7-M8 的預告，花時間建立整體感。
- 演練一：15-20 分鐘的快速估算，讓學員習慣「把問題數字化」。

**第三小時（2:00 - 3:00）：演練實戰**
- 演練二：URL 縮短器設計。給學員 20 分鐘自己嘗試，再帶著走 RESHADED 流程。
- 重點不是「正確答案」，而是「思考過程的紀律性」。

### 常見學員問題

**Q：什麼時候用微服務？什麼時候用單體架構？**
A：M6 的答案是：「先用單體，不要過早拆分。當單一團隊無法獨立部署，或某個組件需要獨立擴展時，再考慮微服務。」詳細討論在 M7。

**Q：Redis 很貴，一定要用嗎？**
A：用 Redis 不一定是因為貴不貴，而是因為記憶體的讀取速度是磁碟的 10 萬倍以上。當重定向要求 < 10ms 時，沒有其他選擇。成本是 trade-off，不是禁止使用的理由。

**Q：SQL 和 NoSQL 能混用嗎？**
A：可以，而且大多數生產系統都是混用的。用 SQL 存結構化的業務數據，用 Redis 做快取，用 Elasticsearch 做全文搜尋，用 S3 存檔案。每種工具做它最擅長的事。

**Q：RESHADED 和七步流程有什麼差別？**
A：RESHADED 是設計「一個系統」的結構化分析框架，側重於單次設計活動的完整性。七步流程是架構師進行架構設計活動的職業流程，涵蓋需求到上線後演進的完整生命週期。RESHADED 可以嵌入在七步流程的第四步（高階設計）中使用。

### 與其他模組的連結

**與 M1 的連結：**
第一原理思考是架構師最重要的能力之一。在估算容量時，不要查公式——從基本量（DAU、每用戶行為頻率、每次操作數據量）推導。

**與 M3 的連結：**
M3 學到的並行模型（Event Loop vs Goroutine vs Thread）直接影響系統設計決策。Node.js 適合高並發 I/O 密集場景；Go 適合需要真正並行計算的場景。這不是個人偏好，而是架構決策。

**與 M4 的連結：**
快取的三層架構（CDN / Redis / DB Cache）和 CPU 的 L1/L2/L3 快取層次是同一個物理原理的不同尺度應用。理解硬體快取，就理解了系統快取的設計動機。

**與 M5 的連結：**
Process 和 Thread 的管理方式決定了單機能承受多少並發。水平擴展是在單機能力達到上限後的解法。Kubernetes（M8 會提到）本質上是在多台機器上管理 Process 的工具。

## 模組總結

M6 完成了你的思維轉換：

從問「怎麼寫這個函數」，到問「這個系統如何在三年後仍然能服務百萬用戶」。

RESHADED 給了你一個不遺漏的分析框架。垂直/水平擴展給了你應對流量成長的工具。資料庫選型和快取策略給了你兩個最常見的架構決策的思路。API 設計給了你定義系統邊界的語言。七步流程給了你架構師工作方式的全景。

架構師最昂貴的錯誤，永遠發生在設計階段。
