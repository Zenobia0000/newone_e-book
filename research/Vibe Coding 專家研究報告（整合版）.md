## 執行摘要

軟體產業正在經歷一場根本性的變革。

2025 年 2 月 6 日，前 Tesla AI 總監、OpenAI 創始團隊成員 Andrej Karpathy 在 X（前 Twitter）上提出了「Vibe Coding」（氛圍編碼）這個概念——開發者不再逐行敲程式碼，而是用自然語言告訴 AI「我想要什麼」，讓 AI 來生成完整的軟體。

這不只是工具升級，而是整個產業的遊戲規則在改寫。

**四大核心發現：**

1. **市場正在重新洗牌**：Replit Agent、Cursor、Windsurf（Codeium 開發）等工具，讓「不會寫程式的人也能做出 App」成為現實。波士頓諮詢公司（BCG）的研究顯示，AI 輔助編碼能力可分為五個等級（L0-L4），最高等級的「系統編排者」正在創造最大的商業價值。

2. **速度提升，但風險也跟著來**：研究顯示，約 **45% 的 AI 生成程式碼含有安全漏洞**。更危險的是，一種叫「Slopsquatting」的新型攻擊手法——駭客利用 AI 容易「瞎掰」不存在的軟體套件這個弱點，預先埋設惡意程式等你上鉤。

3. **寫程式變便宜了，但「知道該寫什麼」變貴了**：OpenAI 執行長 Sam Altman 預測，AI 的使用成本將趨近於零。當程式碼變成廉價品，真正值錢的是：搞清楚要做什麼（規格定義）、讓 AI 不亂來（上下文管理）、確保成品安全（驗證）。

4. **2026-2027 年會出現一波「大掃除」**：現在用 Vibe Coding 快速拼出的產品原型，很快就會需要專業工程師來重構，這將催生全新的服務市場。

## 第一章：遊戲規則改了——從「寫程式」到「說程式」

### 1.1 Vibe Coding 到底是什麼？

簡單說：**你用自然語言描述你想要的軟體，AI 幫你把程式碼寫出來。**

Karpathy 的原始描述是：開發者可以「忘記程式碼的存在」，完全用對話的方式告訴 AI 要做什麼，AI 就會生成完整的應用程式 [1]。

但這只是光譜的一端。在實際的專業環境中，Vibe Coding 是一個從「完全信任 AI」到「嚴格管控 AI」的連續範圍：

| | 探索模式 | 工程模式 |
|---|---|---|
| **適合場景** | 週末副專案、快速原型 | 正式產品、企業系統 |
| **AI 扮演** | 全權創作者 | 精力充沛但需要指導的「初級工程師」|
| **人類扮演** | 驗收者（看成果滿不滿意） | 資深工程師（定規格、審代碼、扛責任）|
| **失敗代價** | 低（大不了重來） | 高（資料外洩、財務損失） |

> **重點**：Vibe Coding 的價值不是消滅工程師，而是**把工程師從打字員升級成指揮官**。Google Cloud [1]、IBM [7] 和 Cloudflare [2] 的分析都支持這個判斷。

### 1.2 你在哪一級？BCG 的五級能力模型

波士頓諮詢公司（BCG）提出了一個實用的分級框架，幫助個人和組織定位自己目前的 AI 協作程度 [11]：

| 等級 | 名稱 | 你在做什麼 | 你的角色 |
|------|------|-----------|----------|
| **L0** | 拒絕者 | 完全不用 AI，靠 Google 和 Stack Overflow | 傳統工程師 |
| **L1** | 問答者 | 把 AI 當高級搜尋引擎，問問題但不讓它寫程式 | 資訊檢索者 |
| **L2** | 補全者 | 用 GitHub Copilot 自動補完程式碼，AI 貢獻 < 20% | 加速版工程師 |
| **L3** | 功能生成者 | 用提示詞讓 AI 寫出整個功能模組，包括重構和測試 | 模組設計者 |
| **L4** | 指揮家 | 管理多個 AI 代理，用規格文件驅動整個開發流程 | **系統編排者** |

**L4 是價值最大化的階段**——你不再關心語法怎麼寫，而是專注在「系統應該長什麼樣」和「AI 的工作範圍在哪裡」。

### 1.3 AI 自主性的五個層級：像自動駕駛一樣理解

軟體開發的 AI 化，正在走一條類似自動駕駛的升級路線 [25]：

| 層級 | 類比 | 代表工具 | 人類的角色 |
|------|------|----------|-----------|
| **Level 1** 自動補完 | 車道維持 | GitHub Copilot, Tabnine | 你在打字，AI 猜你下一行要寫什麼 |
| **Level 2** 對話式協作 | 定速巡航 | ChatGPT, Claude, Gemini | 你問問題，AI 給你程式碼片段 |
| **Level 3** 自主建構 | 有條件自動駕駛 | Replit Agent, Cursor (Agent 模式), Windsurf (Codeium) | 你說「做一個發票 App」，AI 處理所有細節 |
| **Level 4** 多代理協作 | 高度自動駕駛 | Devin (Cognition AI)（早期） | 多個 AI 代理互相協調，你做策略監督 |
| **Level 5** 完全自主 | 無人駕駛 | 尚不存在 | AI 管理整個軟體生命週期 |

**我們現在大多在 Level 3。** 你是「坐在駕駛座上的人」——指揮著一支你不完全理解其運作方式的 AI 樂隊。這個認知很重要：它決定了你需要學什麼、防什麼。

## 第二章：上下文工程——讓 AI 不失控的關鍵技術

### 2.1 AI 為什麼會「忘記」和「亂來」？

AI（大型語言模型）有一個致命弱點：**它沒有長期記憶。**

每次對話就像它的「工作桌面」，桌面空間有限（稱為「上下文視窗」）。對話越長，它就越容易忘記你一開始說的規則，甚至開始胡說八道。這個問題在技術上叫「上下文漂移」[14]，具體表現為三種症狀：

| 症狀 | 發生了什麼 | 實際影響 |
|------|-----------|---------|
| **指令遺忘** | AI 忘記你說過「一律用 TypeScript」 | 程式碼風格不一致，技術債務增加 |
| **事實混淆** | AI 搞混了檔案結構，重複定義已存在的東西 | 功能壞掉，除錯時間倍增 |
| **修復死循環** | AI 修 Bug A 時製造了 Bug B，修 Bug B 又回到 Bug A | **開發完全卡住，API 費用爆表** |

> **真實案例**：有人讓 AI 代理修復一個功能，結果 AI 反覆嘗試了超過 30 次相同的失敗策略，不但沒修好，還把資料儲存層搞壞了 [13]。

### 2.2 解方：用「專案文件」給 AI 裝上長期記憶

對抗 AI 失憶的方法，是把關鍵資訊寫進「規則文件」，每次開新對話時自動灌進去，讓 AI 始終記得「遊戲規則」。

以下是經過實證驗證、在 AI 協作開發中不可或缺的五份文件 [17]：

#### 文件 1：CLAUDE.md / AGENTS.md —— AI 的入職手冊

| 項目 | 說明 |
|------|------|
| **放哪裡** | 專案根目錄 |
| **做什麼** | 告訴 AI「這個專案是做什麼的、用了什麼技術、有什麼規矩」 |
| **為什麼重要** | 沒有這份文件，AI 每次對話都是失憶的，必須從頭摸索，極容易出錯 |

**該寫些什麼：**

- **專案簡介**：2-3 句話說清楚這個產品在做什麼
- **技術版本**：寫死版本號。例如：「Python 3.12, FastAPI 0.109, Pydantic v2（禁止使用 v1 語法）」[19]
- **常用指令**：怎麼啟動、怎麼測試、怎麼部署 [18]
- **架構規範**：程式碼怎麼組織。例如：「資料庫模型放 `src/models`，API 路由放 `src/api`」[20]
- **編碼規矩**：必須用 Type Hints、禁止用 `print` 除錯、必須用 `logger` [19]

> arXiv 上的實證研究 [17] 顯示，有 Agent README 文件的專案，AI 輸出的一致性顯著提升。

#### 文件 2：.cursor/rules —— AI 的行為守則

（注意：舊版 Cursor IDE 使用 `.cursorrules` 單一檔案，新版已改為 `.cursor/rules` 目錄，採用 MDC 格式）

| 項目 | 說明 |
|------|------|
| **放哪裡** | `.cursor/rules/` 目錄 |
| **做什麼** | 控制 AI 在 IDE 裡的行為模式——它該扮演什麼角色、什麼事不能做 [21] |

**四個必填維度：**

| 維度 | 範例 |
|------|------|
| 角色設定 | 「你是有 20 年經驗的資深後端工程師」[23] |
| 禁止事項 | 「不要刪除現有註解」「不要省略程式碼」「不要用過時的套件」[24] |
| 回應格式 | 「先列出你要改哪些檔案，再給 diff 格式的程式碼」[24] |
| 工作流規則 | 「寫新程式碼之前，先搜尋現有程式庫有沒有可以重用的」[24] |

#### 文件 3：llms.txt —— AI 讀外部文件的「導覽手冊」

由 Answer.AI 的 Jeremy Howard 提出的新興標準 [25]，放在文件網站根目錄（如 `docs.example.com/llms.txt`），幫助 AI 快速找到一個套件最重要的文件，而不是盲目爬取整個網站。

**內容包括**：套件簡介 + 精選的高價值文件連結（只放核心概念、API 參考、最佳實踐，過濾掉更新日誌等雜訊）。

#### 文件 4：SPEC.md / WBS.md —— 動態任務清單

| 項目 | 說明 |
|------|------|
| **放哪裡** | 根目錄或 `docs/` |
| **做什麼** | 把複雜需求拆成一步步的小任務，防止 AI 在大工程中迷路 [10] |

範例：

```markdown
# 目標：實作 OAuth2 登入功能

## 工作分解
- [ ] 第一步：資料庫設計（users, oauth_accounts 兩張表）
- [ ] 第二步：後端 API（/auth/login, /auth/callback）
- [x] 第三步：前端狀態管理（authStore）

## 完成標準
登入成功後回傳 JWT，自動跳轉到 Dashboard
```

每完成一步就打勾——這不只是給人看的，也幫助 AI 理解「現在進度到哪了」[28]。

#### 文件 5：AI 優化版 .gitignore —— 幫 AI 減少雜訊

一般的 `.gitignore` 是防垃圾檔案上傳 Git，AI 優化版還要**防止 AI 讀到無用資訊**，浪費它有限的「注意力」[30]：

| 類型 | 該排除什麼 | 為什麼 |
|------|-----------|--------|
| 鎖定檔 | `package-lock.json`、`yarn.lock` | 動輒數萬行，AI 讀了也沒意義 [31] |
| 大型資料 | `.csv`、測試用 `.json` | 灌爆 AI 的工作記憶 |
| 編譯產物 | `dist/`、`build/` | 避免 AI 讀到加工過的程式碼而非原始碼 |

### 2.3 自動化工具：一鍵打包程式碼給 AI

手動複製貼上太沒效率。以下是三個主流的「程式碼庫打包」工具，可以一鍵把整個專案轉成 AI 看得懂的格式：

| 工具 | 技術特色 | 最適合 |
|------|---------|--------|
| **Repomix**（原名 repopack） | XML 結構化輸出、內建 Secretlint 安全掃描、支援遠端 GitHub 倉庫打包。GitHub: `yamadashy/repomix` [31] | **首選推薦**——複雜除錯或架構重構 |
| **code2prompt** | Rust 開發，速度極快；Handlebars 模板系統可自訂輸出格式；內建 Token 計數。GitHub: `mufeedvh/code2prompt` [33] | CI/CD 自動化流程 |
| **repo-to-prompt** | Python 開發，功能簡單直覺，適合快速複製貼上 [30] | 輕量級日常使用 |

**使用技巧**：不要打包整個專案，只打包你正在處理的部分。例如：

```bash
repomix --include "src/auth/**/*.ts"
```

只給 AI 看它需要的東西，保持「訊號清晰、雜訊最低」[35]。

### 2.4 AI 卡在修復死循環怎麼辦？

修復死循環（Fix Loop）是 Vibe Coding 最常見的效率殺手 [6]。原因是：失敗嘗試越堆越多，AI 被這些錯誤資訊帶偏，陷入「怎麼修都修回同一個錯」的死胡同 [40]。

三招破解：

| 策略 | 怎麼做 | 為什麼有效 |
|------|--------|-----------|
| **開新對話** | AI 連續修失敗兩次 → 立刻開一個全新對話 | 等於幫 AI 清除「被汙染的記憶」，重新開始 [40] |
| **把任務拆更小** | 不要說「修復登入功能」→ 改說「寫一個驗證 JWT 簽名的函數，附上測試」 | 縮小問題範圍，AI 更容易找到正確答案 [42] |
| **先寫測試再寫程式** | 讓 AI 先寫一個會失敗的測試案例，再寫程式讓測試通過 | 給 AI 一個明確的「對錯標準」，取代模糊的「修好它」[10] |

## 第三章：風險控制——AI 寫的程式哪裡會出事

### 3.1 三大常見安全漏洞

AI 生成的程式碼和人類寫的不一樣——人類的 Bug 通常是邏輯沒想清楚，AI 的問題是**它不理解「安全」這回事**。研究顯示，大約 **45% 的 AI 生成程式碼含有常見安全漏洞** [6]。

#### 漏洞 1：身份驗證做半套

AI 擅長做「登入功能」，卻常常忘記做「權限控制」——它分不清楚「確認你是誰」和「確認你能做什麼」：

```
你登入成功了（身份驗證 OK）
但只要把網址從 /user/123/profile 改成 /user/124/profile
就能看到別人的資料（權限控制沒做）
```

這種漏洞叫 IDOR（不安全的直接物件引用），在 AI 產出的後端 API 中極為常見 [8]。

更常見的問題：AI 為了讓程式「能動」，直接把資料庫密碼寫死在程式碼裡 [7]：

```javascript
// AI 常犯的錯——絕對不能這樣做
mongoose.connect('mongodb://user:password@host:port/db')
```

#### 漏洞 2：供應鏈攻擊——Slopsquatting

這是 Vibe Coding 帶來的全新攻擊手法，利用 AI 愛「瞎掰」的弱點：

```
第一步：你問 AI「怎麼做反向代理？」
        AI 自信地說：「安裝 starlette-reverse-proxy」
        ——但這個套件根本不存在，是 AI 編的

第二步：駭客早就研究過 AI 常「發明」哪些不存在的套件名稱

第三步：駭客在 PyPI/npm 上搶先註冊這些名字
        上傳看起來功能正常、但暗藏惡意程式碼的套件

第四步：你照著 AI 的建議執行 pip install
        恭喜，你的開發環境被入侵了
```

Trend Micro [36]、HackerOne [38] 和 Raconteur [37] 都對此威脅發出了警告。

#### 漏洞 3：AI 代理的無限循環

當 AI 代理（如 Replit Agent 或 Cursor Agent 模式）被指派「修復 Bug」時，它可能會：

- 反覆嘗試**同一個失敗方案** 30 多次 [13]
- 燒掉大量 API 費用
- 為了繞過錯誤，硬塞一堆品質極差的修補程式碼

結果：程式雖然暫時能動，但變成一團沒人維護得了的「義大利麵條式代碼」[14]。

### 3.2 分層風險管控：不是所有地方都不能用 AI

關鍵不是「能不能用 AI」，而是「在哪裡用、用多少」。以下是一個實用的三級分層框架 [6]：

| 風險等級 | 什麼情境 | AI 的角色 | 人類的把關 |
|----------|---------|----------|-----------|
| **高風險** | 登入驗證、金流處理、個資處理、加密、核心業務邏輯 | **禁區**：AI 只能寫測試或解釋程式碼，不碰核心邏輯 | 2 位以上資深工程師逐行審查 + 自動化安全掃描 |
| **中風險** | 內部工具、資料圖表、不涉及個資的 CRUD、前端 UI | **有條件使用**：AI 可以寫，但要設嚴格護欄 | 1 位安全審查員 + 自動掃描，重點看邏輯漏洞 |
| **低風險** | UI 原型、用完就丟的測試 App、行銷頁面 | **放手讓 AI 發揮**：重點是速度 | 自動化格式檢查 + 抽查功能是否正常 |

### 3.3 「Vibe Check」——四步安全檢查流程

在 AI 時代，程式碼審查的重點要從「語法對不對」（AI 很擅長）轉移到「有沒有安全問題」（AI 很不擅長）：

| 步驟 | 做什麼 | 怎麼做 |
|------|--------|--------|
| **1. 驗證套件** | 確認 AI 推薦的套件真的存在 | 去 PyPI/npm 官網查：有沒有下載量？維護者是誰？是不是新建帳號剛發布的？ |
| **2. 掃描機密** | 確認程式碼裡沒有密碼或 API 金鑰 | 設定 Pre-commit hook，自動攔截含有金鑰模式的程式碼 |
| **3. 測試失敗情境** | 不只測「能不能用」，還要測「壞掉會怎樣」 | 資料庫掛了會怎樣？有人亂改網址參數會怎樣？[9] |
| **4. 隔離執行** | AI 生成的新程式碼，先在隔離環境跑 | 用 Docker 容器或臨時虛擬機，別直接跑在你的開發機上 [36] |

### 3.4 安全悖論：門檻降低，風險升高

Vibe Coding 最弔詭的地方：

> **做出軟體變得超簡單 → 但確保軟體安全變得更難了**

一個沒有技術背景的創業者，現在可以用 Replit Agent 在一個週末做出 SaaS 產品。但要理解為什麼這個產品可能被駭客攻擊——需要的專業知識比以前更深 [4]。

數據佐證：60% 的開發者說 AI 提高了他們的效率 [18]，但同時也承認——軟體的穩定性因此下降了。**程式碼的產出速度，已經超過了理解它的速度。**

## 第四章：人才轉型——新時代需要什麼樣的工程師

### 4.1 寫程式變便宜了，什麼變貴了？

OpenAI 執行長 Sam Altman 預測：AI 和算力的成本將在未來十年趨近於零 [21]。這代表「把邏輯寫成程式碼」這件事，很快就會像用計算機做加法一樣廉價。

那什麼才是新時代的稀缺資源？矽谷投資人 Naval Ravikant 給了答案 [23]：

| 新時代的稀缺能力 | 意思 | 具體來說 |
|-----------------|------|---------|
| **品味 (Taste)** | 知道「該做什麼」的判斷力 | 產品直覺、使用者體驗感、架構美感 |
| **分發 (Distribution)** | 把做好的東西推到對的人面前 | 社群經營、內容行銷、通路建設 |

> **Vibe Coder 不是初級工程師——他們是拿著魔法棒的產品經理。** 學習路線要從「怎麼寫 for 迴圈」轉向「這個系統應該怎麼運作」。

### 4.2 新時代工程師的三大核心能力

#### 能力 1：上下文工程——比「提示工程」更持久的技能

「提示工程」（Prompt Engineering）只是過渡期的技能。真正有長期價值的是**上下文工程（Context Engineering）**——知道該給 AI 看什麼、不該看什麼、怎麼組織這些資訊。

**規格定義變成了一種程式語言：**

| 你給 AI 的輸入 | AI 產出的品質 |
|---------------|-------------|
| 模糊：「做一個網站」 | 模糊的程式碼，什麼都是半成品 |
| 精確：「用 Next.js + Tailwind CSS + 三個 API 端點 + JWT 驗證」 | 可以直接部署的軟體 |

**真實案例：Printing Plus 印刷管理系統**

一位 Replit 使用者從「做一個報價生成器」開始，透過數百次對話逐步添加「資料庫」「響應式設計」「CRM 功能」，最終做出了完整的印刷店管理系統 [28]。成功關鍵不是某一句神奇的提示詞，而是他在持續對話中**維護上下文**的能力。

#### 能力 2：系統素養——看不懂程式碼，至少要看懂「系統」

你不用自己寫 SQL，但你要知道資料庫是什麼；你不用自己寫 API，但你要知道前後端怎麼溝通。

三個必備技能：

- **技術架構認知**：理解資料庫（關聯式 vs. 非關聯式）、API（REST vs. GraphQL）、前後端的資料流向
- **錯誤訊息解讀**：把報錯訊息貼回給 AI 是基本操作，但你要能分辨「哪一段報錯是關鍵、哪些是雜訊」
- **瀏覽器工具除錯**：用 DevTools 的 Network Tab 確認 AI 說的「已修好」是不是真的有送出網路請求

#### 能力 3：品味——AI 做得出一千個版本，你選哪一個？

當 AI 可以無限生成程式碼時，價值在於**選對的那一個**：

- 動畫順不順？載入狀態讓人困惑嗎？——AI 做不好這些細緻的判斷
- 開發者的新角色是**策展人**：拒絕 AI 的 90% 垃圾輸出，只挑與產品願景一致的 10%

Google Chrome 工程負責人 Addy Osmani 在他的文章「Vibe coding is not the same as AI-Assisted engineering」[8] 中強調：AI 是你的工具，不是你的替代品。

### 4.3 最直接的市場影響：Excel 正在被取代

| 過去 | 現在 |
|------|------|
| 小企業用 Excel 管營運，因為請人寫系統太貴（2 萬美元起跳） | 老闆自己用 Replit Agent 週末就做出客製化系統 |

> **產業洞察**：「個人軟體革命」的對手不是 Salesforce，而是 **Excel**。Vibe Coding 讓「一次性軟體」成為可能——這些 App 太特定、太小眾，不值得訂閱 SaaS，但對使用者本人有巨大價值 [4, 28]。

## 第五章：未來市場——軟體產業的下一個十年

### 5.1 「後軟體」時代：軟體從「產品」變成「服務」

| | 現在 | 未來 |
|---|------|------|
| **軟體形態** | 固定的 App，你下載了就得配合它的流程 | AI 即時生成、用完即消的個人化工具 |
| **舉例** | 下載一個卡路里追蹤 App，被迫適應它的操作方式 | 告訴 AI：「我要追蹤生酮飲食的卡路里，連接我的智能冰箱」→ AI 即時生成專屬介面 |
| **壽命** | 長期存在 | 按需產生、用完即棄 |

### 5.2 Vibe Coding 的核心矛盾：做得快 vs. 顧得了

| 表面 | 現實 |
|------|------|
| Vibe Coding 讓開發更快 | Vibe Coding 讓維護更難 |

當「週末做的副專案」變成了公司的核心系統，缺乏架構設計的程式碼就變成了定時炸彈。

**市場預測：**

- **2026-2027 年「大掃除」熱潮**：大量企業會聘請工程師，專門重構創始人用 Vibe Coding 拼出來的 MVP [32]
- **新工具品類崛起**：「AI 重構代理」——專門把品質差的 AI 生成程式碼整理成結構清晰、可維護的工程架構

### 5.3 自然語言是程式設計的終極抽象層

程式設計的歷史就是一部「讓人類離機器越來越遠」的進化史：

```
二進制 (0/1) → 組合語言 → C / Java → Python / JavaScript → 自然語言（Vibe Coding）
```

Karpathy 說「英語是最熱門的新程式語言」[33]。但自然語言天生就是模糊的，所以未來一定是**混合模式**：用自然語言說意圖 + 用工具做精確控制。

> **預測**：未來的開發環境會變成「意圖工作室」——程式碼預設隱藏，只有 AI 搞不定需要人類介入時才會顯示出來。

### 5.4 生產成本歸零後，什麼才是真正的護城河？

當 AI 可以幾乎免費地寫程式，剩下來的競爭壁壘只有三個：

| 護城河 | 說明 | 深度 |
|--------|------|------|
| **獨有數據** | AI 憑空編不出來的「秘方」——你的用戶行為數據、產業 know-how | 深 |
| **品牌與通路** | 跟客戶的信任關係、社群資產 | 中-深 |
| **實體世界接口** | AI 能寫程式，但還不能烤漢堡或送包裹 | 深 |

Naval Ravikant 說得對 [23]：「程式碼」以前是槓桿，因為它難。當它變簡單，槓桿就轉移到**說故事的能力**（媒體）和**擁有受眾**（社群）[24]。

## 第六章：行動建議——現在該做什麼

### 6.1 給技術主管（CTO / VP Engineering）

| 優先級 | 行動 | 預期效果 |
|--------|------|---------|
| **立刻做** | 按高/中/低風險分級所有開發任務，明確哪些地方禁止 AI 碰核心邏輯 | 把風險可視化，防止高風險區域的 AI 濫用 |
| **立刻做** | 在 CI/CD 流程中加入套件驗證（白名單 + 自動查驗），防堵 Slopsquatting | 封鎖新型供應鏈攻擊 |
| **一個月內** | 建立標準化的 CLAUDE.md / `.cursor/rules` 模板，全專案強制採用 | 減少 AI 的上下文漂移，提升輸出一致性 |
| **一個月內** | 設計團隊的 L0→L4 升級路線，投資上下文工程培訓 | 從「個人加速」升級到「組織級轉型」 |
| **一季內** | 評估 AI 重構工具，準備應對歷史技術債務 | 提前為 2026-2027 的「大掃除」做準備 |

### 6.2 給產品團隊與創業者

| 行動 | 為什麼 |
|------|--------|
| 用 Vibe Coding 加速 MVP 驗證（低風險區），但第一天就建好 SPEC.md | 速度和品質兼顧 |
| 核心邏輯（金流、身份驗證）一定要人類工程師寫或逐行審查 | 安全債務的複利效應，會在 B 輪融資時爆發 |
| 投資「品味」和「分發」，而不只是技術技能 | 生產成本歸零後，這才是真正的競爭壁壘 |

### 6.3 給投資人：四個值得關注的投資主題

1. **AI 安全審計平台**：自動化「Vibe Check」的 SaaS——類比傳統的 Snyk/SonarQube，但專門針對 AI 生成程式碼的特有漏洞
2. **上下文工程工具鏈**：Repomix 等工具的企業版（打包 + 安全掃描 + Token 最佳化）
3. **AI 重構代理**：2026-2027「大掃除」熱潮的核心受益品類
4. **垂直化 Vibe Coding 平台**：針對特定行業（法律、醫療、零售）的低程式碼開發環境

**報告結語**

Vibe Coding 不是工程的終結，而是工程的擴展。它讓數百萬「不會寫程式的人」也能做出軟體，創造了一個充滿機會但也充滿風險的新生態。

對專業人士來說，方向很清楚：

> **享受 AI 帶來的速度，同時用專業的紀律守住安全底線。**

程式碼本身正在變成廉價品。真正稀缺的是：**搞清楚該做什麼的判斷力、讓 AI 不亂來的管理能力、確保成品安全的專業功力，以及知道什麼才是好產品的品味。**
