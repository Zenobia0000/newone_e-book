# 第1章：程式到底是什麼？

> 「寫程式很難嗎？」
>
> 這是我最常被問到的問題。每次聽到，我總會微笑著回答：「你知道嗎？你每天都在寫程式，只是用的不是電腦語言而已。」

---

## 一個平凡的早晨

讓我跟你分享一個故事。

小美，35歲，行銷經理。每天早上七點，鬧鐘準時響起。她習慣性地按下貪睡鈕，給自己額外的十分鐘。這個簡單的動作，其實就是一段完美的程式：檢查時間、判斷是否該起床、執行相應的動作。

那天早上，小美一邊刷牙一邊想著昨天同事展示的「魔法」——原本需要她花三個小時整理的報表，同事用幾行程式碼，三分鐘就完成了。

「如果我也會這個，生活會變成什麼樣子？」她看著鏡中的自己，牙刷停在半空中。

但隨即，一連串的疑慮湧上心頭：「我數學不好」、「英文也普通」、「35歲才開始會不會太晚」...

如果你也曾有過這些想法，請讓我告訴你一個秘密——這些都不是問題。真的。

---

## 從蛋炒飯說起

你會做蛋炒飯嗎？或者，你至少能想像怎麼做蛋炒飯吧？

準備材料、打蛋、熱油、炒蛋、加飯、調味、起鍋。每個步驟都有其順序，每個動作都有其目的。如果我告訴你，這就是程式設計的全部，你相信嗎？

程式，說穿了，就是告訴某個東西（可能是電腦、可能是機器、可能是另一個程式）該做什麼、怎麼做、什麼時候做。就像食譜告訴你如何做出一道菜，就像樂譜告訴演奏者如何演奏一首曲子。

當你教朋友做蛋炒飯時，你已經在「編程」了——只是你的執行者是人類，而不是電腦。

---

## 電腦：世界上最聽話的笨蛋

這個標題可能讓你意外。我們總認為電腦很聰明——它能在一秒內計算數百萬次、能記住海量資訊、能打敗世界棋王。但我要告訴你另一個角度的真相。

想像你有一個朋友，他：

- 完全不懂暗示
- 不會舉一反三
- 只認識「是」和「否」
- 需要你把每件事都說得清清楚楚

這就是電腦。

當你告訴人類「去買個漢堡」，他會自動理解需要帶錢、知道去哪裡買、懂得怎麼點餐。但對電腦來說？你需要告訴它什麼是錢、什麼是漢堡、什麼是「買」、如何找到店家、如何與人互動...每一個我們視為理所當然的細節。

有趣的是，正是這種「笨」，讓電腦變得可靠。它不會自作聰明，不會隨意解讀，給它同樣的指令，永遠會得到同樣的結果。這種可預測性，正是我們能夠信任電腦處理重要工作的原因。

就像小美發現的——當她終於理解電腦需要精確指令時，反而鬆了一口氣。「原來不是我不夠聰明，而是需要學會用電腦能懂的方式說話。」

---

## 生活中無所不在的程式思維

每天早上，你的鬧鐘在執行這樣的邏輯：

```
如果現在是設定時間：
    響鈴
    如果使用者按下貪睡：
        等待十分鐘
        重新開始
    否則：
        停止
```

每次使用Google Maps，背後都在進行複雜的運算：收集你的位置、分析所有可能路線、比較時間和距離、選出最佳方案、即時調整。這個我們習以為常的功能，其實是無數個「如果...就...」判斷的組合。

甚至連自動販賣機都在執行程式：等待投幣、計算金額、判斷是否足夠、執行相應動作。那個看似簡單的「投幣→選擇→取貨」流程，背後是嚴謹的邏輯判斷。

你看，程式思維從來都不陌生。它就在我們的日常生活中，只是我們從未用這個角度去看待它。

---

## 關於那些迷思

### 「我數學不好」

讓我問你：買東西算找零需要微積分嗎？做菜調味需要線性代數嗎？

大部分的程式設計，用到的數學不會超過小學程度。加減乘除、比大小、數數字——這就是90%程式工作需要的全部數學。

當然，如果你要開發3D遊戲引擎或研究人工智慧，那是另一回事。但對於想要自動化日常工作、建立網站、開發應用程式的人來說，會用計算機就夠了。

小美後來跟我說：「我發現寫程式的邏輯思考，反而讓我的數學變好了。因為我開始理解數字背後的意義，而不只是死記公式。」

### 「英文不好怎麼辦」

程式語言用到的英文，大概只有50個單字。而且都是最簡單的：if（如果）、for（對於）、while（當）、true（真）、false（假）。

這些單字，你現在就認識了，不是嗎？

更重要的是，理解這些單字背後的概念遠比拼寫它們重要。我認識許多優秀的工程師，他們的英文可能不流利，但這從未阻礙他們寫出優雅的程式。

### 「35歲會不會太晚」

若宮雅子，一位日本奶奶，81歲開始學程式，開發了一款專為銀髮族設計的手機遊戲。她說：「年齡給了我獨特的視角，我知道同齡人需要什麼。」

年齡從來不是限制，反而可能是優勢。你有豐富的人生經驗、明確的學習目標、更好的耐心。你知道自己為什麼要學，這比什麼都重要。

我見過太多30、40甚至50歲轉職成功的例子。他們的共同點不是年輕，而是清楚知道自己要什麼，並願意為此努力。

### 「需要天賦嗎」

如果寫程式需要天賦，那開車、做菜、學語言是不是也都需要天賦？

程式設計是一項技能，就像其他任何技能一樣。一開始會笨拙、會犯錯、會覺得困難。但隨著練習，你會越來越熟練，直到有一天，它變成你的第二天性。

我記得我的第一個程式，就是在螢幕上顯示「Hello World」。那兩個單字出現時，我興奮得像個孩子。現在回想起來，那個簡單到不能再簡單的程式，卻是一切的開始。

### 「需要很貴的電腦」

你現在用來閱讀這本書的設備，就足夠開始學習程式了。

真的，就是這麼簡單。

甚至，你連軟體都不用安裝。打開瀏覽器，進入任何一個線上程式環境，就可以開始寫你的第一行程式碼。我認識的許多工程師，他們的第一個程式就是在十年前的舊筆電上寫的。

---

## 錯誤訊息：最誠實的朋友

新手最害怕看到紅色的錯誤訊息。但讓我告訴你一個秘密——錯誤訊息是你最好的老師。

當電腦說「找不到檔案」，它在告訴你確切的問題所在。當它說「語法錯誤」，它在指出需要修正的地方。不像人類會委婉、會暗示，電腦的錯誤訊息永遠直接而誠實。

小美第一次看到錯誤訊息時嚇了一跳，以為自己做錯了什麼嚴重的事。後來她發現，那只是電腦在說：「我不懂你的意思，能說清楚一點嗎？」

學會閱讀錯誤訊息，就像學會聽懂電腦的語言。它不是在責備你，而是在幫助你。

---

## 程式改變了什麼

回到小美的故事。

三個月後，她寫了第一個自動化程式——自動整理每週的銷售報表。原本需要半天的工作，現在只要按個按鈕。她說：「不是程式有多厲害，而是我終於理解了，重複的工作就該交給電腦。」

六個月後，她開始幫同事解決各種自動化需求。不是因為她變成了工程師，而是因為她學會了用程式思維看待問題。

一年後，她獲得了升遷。老闆說：「不只是因為妳會寫程式，而是妳展現了解決問題的新思維。」

程式改變的，不只是工作效率，更是看待世界的方式。當你開始用程式思維思考，你會發現處處都有可以優化的流程、可以自動化的任務、可以簡化的複雜。

---

## 第一章的最後

如果你讀到這裡，你已經理解了程式的本質——它不是神秘的咒語，不是天才的專利，而是一種表達邏輯的方式，一種與機器溝通的語言。

你不需要成為全職工程師才值得學程式。就像你不需要成為作家才值得學寫字，不需要成為音樂家才值得學樂器。程式，可以只是你工具箱裡的另一個工具，幫助你更好地完成工作、解決問題、創造價值。

還記得開頭小美的疑問嗎？「如果我也會這個，生活會變成什麼樣子？」

現在，我想問你同樣的問題。不是「你能不能學會」——你當然能。而是「當你學會後，你想用它做什麼？」

這個答案，只有你知道。而這個答案，將成為你學習路上最強大的動力。

下一章，我們將一起探索不同程式語言的世界。不是要你全部學會，而是要找到最適合你的那一個。就像選擇一把趁手的工具，關鍵不在於工具本身，而在於你想創造什麼。

準備好了嗎？讓我們繼續這趟旅程。

---

*每一個改變世界的程式，都始於某個人的第一行程式碼。*
*而你的第一行，可能就是改變你自己世界的開始。*

# 第2章：程式語言大觀園

> 「我應該學哪個程式語言？」
>
> 如果說上一章的問題是最常被問的，那這個問題絕對是第二名。而我的答案總是：「你想用它來做什麼？」

---

## 選擇的焦慮

記得小美第一次打開程式學習網站時的表情。

Python、JavaScript、Java、C++、Ruby、Go、Swift...螢幕上密密麻麻的選項讓她眼花撩亂。每個語言旁邊都寫著聽不懂的特色：「物件導向」、「函數式」、「強型別」、「直譯式」...

「這就像站在一家有200種口味的冰淇淋店前，」她苦笑著說，「選擇太多，反而不知道該選什麼。」

她花了整整一週研究「最好的程式語言」，看了無數篇比較文章，每一篇都說得頭頭是道，但看完後她更迷茫了。有人說Python最簡單，有人說JavaScript最實用，有人說Java找工作最容易...

直到有一天，她問了我那個關鍵的問題：「我應該學哪個？」

而我反問她：「你想解決什麼問題？」

---

## 語言，只是工具

讓我分享一個可能會讓你驚訝的事實：對於大部分的程式任務，用哪個語言其實沒那麼重要。

這就像問：「我該學開手排車還是自排車？」答案是——重要的不是車子的類型，而是你要去哪裡。

每個程式語言，本質上都在做同樣的事：

- 儲存資料
- 做判斷
- 重複動作
- 與外界溝通

差別只在於表達方式和擅長的領域。就像中文適合寫詩，英文適合寫科學論文，但兩者都能表達「我愛你」。

讓我用一個簡單的例子說明。假設我們要讓電腦說「Hello, 小美」：

**Python的方式：**

```python
name = "小美"
print(f"Hello, {name}")
```

**JavaScript的方式：**

```javascript
let name = "小美";
console.log(`Hello, ${name}`);
```

**Java的方式：**

```java
String name = "小美";
System.out.println("Hello, " + name);
```

看出來了嗎？概念完全一樣，只是「口音」不同。

---

## Python：像在寫詩的程式語言

如果程式語言是樂器，Python就是鋼琴——優雅、直觀、適合初學者，但也能演奏出大師級的作品。

我永遠記得小美第一次寫Python時的反應：「等等，就這樣？不用分號？不用大括號？這真的是程式嗎？」

```python
# 檢查今天是否該運動
體重 = 70
目標體重 = 65

if 體重 > 目標體重:
    print("該運動了！")
else:
    print("保持下去！")
```

Python的美在於它的簡潔。它去掉了所有不必要的符號，讓你專注在邏輯本身。這就像寫作時，你不需要先學習複雜的文法規則，而是直接開始表達想法。

**Python特別適合：**

- 數據分析（「這個月哪個產品賣最好？」）
- 自動化任務（「幫我整理這1000個檔案」）
- 人工智慧（「訓練一個認識貓的程式」）
- 網路爬蟲（「收集所有競爭對手的價格」）

小美最後選擇了Python，不是因為它「最好」，而是因為她的第一個目標是自動化Excel報表。三週後，她寫出了第一個實用程式——自動從20個分店的銷售檔案中提取數據，生成週報。原本要花她整個Monday早上的工作，現在只要按個按鈕。

---

## JavaScript：網頁的魔法師

如果說Python是鋼琴，那JavaScript就是吉他——隨處可見、容易上手、而且超級酷炫。

「你現在看到的每個網頁，背後都有JavaScript在運作，」我告訴小美的同事阿威，一個想要創業的設計師。

阿威原本是平面設計師，他想要的不是處理數據，而是讓他的設計「活起來」。當他第一次用JavaScript讓按鈕被點擊時改變顏色，他的眼睛亮了起來。

```javascript
// 當按鈕被點擊時
按鈕.addEventListener('click', function() {
    if (心情 === "開心") {
        背景.style.color = "黃色";
        播放音樂("快樂頌");
    } else {
        背景.style.color = "藍色";
        顯示訊息("需要一個擁抱嗎？");
    }
});
```

JavaScript的魔力在於「即時回饋」。你改一行程式碼，重新整理瀏覽器，立刻看到效果。這種即時的視覺回饋，對創意工作者來說特別有吸引力。

**JavaScript特別適合：**

- 網頁互動（「讓這個選單滑出來」）
- 前端開發（「做一個漂亮的購物網站」）
- 後端開發（「處理用戶登入」）——是的，JavaScript前後端通吃
- 手機App（React Native讓你用JavaScript寫App）

六個月後，阿威用JavaScript做出了他的第一個作品集網站。不是靜態的圖片展示，而是會動、會回應、會說故事的互動體驗。他說：「我終於能把腦中的創意完整呈現出來了。」

---

## 其他語言的故事

### Java：企業的可靠夥伴

我有個朋友老陳，在銀行工作。當他決定學程式時，毫不猶豫地選了Java。

「銀行系統不能出錯，」他說，「Java就像穿西裝的管家，規規矩矩，但絕對可靠。」

Java要求你明確定義每個變數的類型，就像銀行要求每筆交易都要有明確的記錄。這看似麻煩，但當系統變大時，這種嚴謹能救你一命。

### Go：速度與簡潔的平衡

小李是個系統工程師，他需要處理每秒數萬筆請求。他選擇了Go。

「Go就像瑞士軍刀，」他形容，「簡單、高效、專注於把事情做好。」

Go的並發處理能力驚人，就像一個超級協調者，能同時處理成千上萬件事而不會搞混。

### Swift：蘋果生態系的語言

如果你的夢想是做iOS App，Swift是唯一的選擇。就像要在法國當廚師，你最好會說法語。

---

## 如何選擇你的第一個語言

經過這麼多故事，你可能還是在想：「所以我該選哪個？」

讓我給你一個簡單的指南：

**如果你想要：**

**快速看到成果** → JavaScript
打開瀏覽器就能開始，改了就能看到效果。

**處理數據、自動化工作** → Python
簡潔優雅，大量現成工具，適合解決實際問題。

**找傳統大公司的工作** → Java
企業愛Java，工作機會多，薪水穩定。

**做iOS App** → Swift
別無選擇，但也是個優雅的語言。

**打好基礎、理解底層** → C
困難但值得，像學武功要先蹲馬步。

但最重要的建議是：**不要糾結太久**。

記住，程式語言不是終身伴侶，你不需要「從一而終」。大部分工程師都會3-5種語言，而且學了第一個之後，學第二個會快很多。

就像小美，她從Python開始，一年後因為想做網頁，很自然地學了JavaScript。她說：「第二個語言只花了一個月就上手，因為概念都是相通的。」

---

## 語言的偏見與真相

在程式世界裡，有很多偏見和迷思。讓我來打破幾個：

### 「Python太慢了」

是的，Python執行速度比C慢。但你知道嗎？Instagram用Python處理數億用戶的照片。速度很重要，但開發速度更重要。用Python兩天寫完的程式，C可能要兩週。

### 「JavaScript不是真正的程式語言」

Netflix、PayPal、Uber的後端都大量使用JavaScript。如果它不是「真正」的程式語言，這些公司怎麼敢把幾億美金的生意交給它？

### 「Java過時了」

全世界有900萬個Java開發者，Android App大部分用Java寫。過時？我看是穩如泰山。

### 「C++太難了」

遊戲產業愛死C++了。你玩的3A大作，底層引擎幾乎都是C++。難？是的。但值得？看你想做什麼。

---

## 超越語言的思考

讓我告訴你一個秘密：真正的程式高手，不是精通某個語言的人，而是能夠超越語言思考的人。

當你理解了「變數」的概念，你就能在任何語言中使用它。
當你學會了「迴圈」的邏輯，你就能用任何語法表達它。
當你掌握了「問題分解」的能力，你就能用任何工具解決它。

這就像一個真正的廚師，給他任何廚房、任何食材，他都能做出美味的料理。工具只是工具，創造力才是關鍵。

小美現在已經能用Python和JavaScript工作了。但更重要的是，她學會了「程式思維」。她看問題的角度變了，解決問題的方法變了，甚至思考的方式都變了。

「語言只是表達方式，」她說，「重要的是你想表達什麼。」

---

## 你的第一步

如果你讀到這裡還在猶豫，讓我給你一個最簡單的建議：

閉上眼睛，想像一個你想解決的問題。

是想要自動化無聊的工作？選Python。
是想要做個人網站？選JavaScript。
是想要開發App？看你用iPhone還是Android。

然後，今天就開始。

不要再研究了，不要再比較了，不要再等「最佳時機」了。

打開電腦，選一個語言，寫下你的第一行程式碼。哪怕只是讓電腦說「Hello」，那也是你改變的開始。

記住阿威的話：「我糾結了三個月該學什麼，但真正改變我的，是我終於開始寫的那一天。」

---

## 寫在第二章的最後

程式語言就像世界各地的語言，每種都有its own beauty，都有它的文化和哲學。

選擇哪一種開始，真的沒那麼重要。重要的是開始。

下一章，我們要談談更基礎的東西——如何像電腦一樣思考。因為無論你選擇什麼語言，邏輯思維都是相同的。

但在那之前，我想留給你一個思考：

如果程式語言是工具，那你想用這個工具創造什麼？

這個答案，比選擇哪個語言重要一萬倍。

---

*「The best programming language is the one you actually use.」*
*最好的程式語言，是你真正在用的那一個。*

# 第3章：像電腦一樣思考

> 「我的大腦就是轉不過來。」
>
> 小美盯著螢幕上的程式碼，挫折地說。她已經學會了Python的基本語法，但當要解決一個實際問題時，她不知道從何下手。

---

## 思維的轉換

那是一個週五下午，小美想寫一個程式來整理客戶名單。需求很簡單：把重複的刪除，按照消費金額排序，然後標記VIP客戶。

她知道Python有排序功能，也知道怎麼刪除重複，但就是不知道怎麼把這些組合起來。

「我知道要做什麼，」她說，「但不知道怎麼告訴電腦。」

這就是問題所在。學會語法就像學會了字母，但要寫出文章，你需要學會思考的方式。

人類的思考是跳躍的、直覺的、模糊的。
電腦的思考是線性的、邏輯的、精確的。

要寫程式，你需要學會在這兩種思維模式之間切換。

---

## 分解：大象要一口一口吃

還記得那個古老的問題嗎？「如何吃掉一頭大象？」答案是：「一口一口吃。」

程式設計的第一個思維模式就是：分解。

看看小美的需求：「整理客戶名單」。這對人類來說是一個任務，但對電腦來說太模糊了。

我引導她把任務分解：

```
整理客戶名單
├── 讀取客戶資料
├── 處理資料
│   ├── 刪除重複
│   ├── 計算消費總額
│   └── 排序
├── 標記VIP
│   └── 如果消費 > 10000，標記為VIP
└── 輸出結果
```

「哦！」小美恍然大悟，「原來要這樣一步一步來。」

這就像做菜。你不會說「做一頓飯」，而是：

1. 決定菜單
2. 準備食材
3. 處理食材（洗、切、醃）
4. 烹煮
5. 擺盤

每個步驟都是具體的、可執行的。

---

## 順序思維：第一件事永遠是第一件事

電腦很笨，它不會「同時」做很多事（看起來同時，其實是快速切換）。它只會一步一步，從上到下執行。

小美一開始寫的程式是這樣的：

```python
排序客戶名單()
刪除重複()
讀取檔案()
標記VIP()
```

「為什麼不能運作？」她困惑地問。

問題很明顯——她在讀取檔案之前就想要排序。這就像還沒買菜就想炒菜。

人類可以在腦中跳來跳去：「我要排序...啊對了要先讀檔...然後刪除重複...」但電腦不行。

正確的順序應該是：

```python
# 1. 先有資料
客戶資料 = 讀取檔案()

# 2. 再處理資料
客戶資料 = 刪除重複(客戶資料)

# 3. 然後排序
客戶資料 = 排序(客戶資料)

# 4. 最後標記
標記VIP(客戶資料)
```

這種「順序思維」一開始很不自然，但慢慢地，你會發現它其實讓事情變簡單了。因為你被迫把複雜的任務整理成清晰的步驟。

---

## 條件思維：如果...就...否則...

生活充滿了選擇。程式也是。

但電腦的選擇必須是明確的、沒有模糊地帶的。

人類的思考：「如果客戶買得夠多，就給他VIP。」
電腦需要：「如果客戶消費金額大於10000元，就標記為VIP，否則標記為一般會員。」

看出差別了嗎？

- 「買得夠多」→「大於10000元」（明確的數字）
- 暗示的「不是VIP」→明確的「否則標記為一般會員」

小美在處理促銷邏輯時遇到了困難：

「滿千送百，但如果是VIP再打9折，如果又是生日月份再送生日禮...」她的頭都暈了。

我教她畫決策樹：

```
消費金額
├── 大於1000？
│   ├── 是 → 送100元折價券
│   │   └── 是VIP？
│   │       ├── 是 → 再打9折
│   │       └── 否 → 維持原價
│   └── 否 → 無優惠
└── 是生日月份？
    ├── 是 → 送生日禮
    └── 否 → 無生日禮
```

「原來可以這樣整理！」她興奮地說。把模糊的規則變成清晰的決策樹，程式碼就自然而然地寫出來了。

---

## 迴圈思維：重複，但要知道何時停止

如果要你數到100，你不會說：「1、2、3、4、5...」一直說到100。你會說：「從1數到100」。

這就是迴圈思維。

但電腦的迴圈需要三個要素：

1. 起點（從哪裡開始）
2. 終點（什麼時候停）
3. 步進（每次變化多少）

小美要處理1000個客戶資料，一開始她想：

```python
處理第1個客戶
處理第2個客戶
處理第3個客戶
# ...天啊要寫1000行？
```

當她學會迴圈思維後：

```python
for 客戶 in 所有客戶:
    處理(客戶)
```

兩行搞定。

但迴圈最難的是知道何時停止。無限迴圈是新手最常犯的錯誤。就像小美第一次寫的：

```python
while 還有客戶:
    處理下一個客戶
    # 忘記更新"還有客戶"的狀態
```

程式永遠不會停，因為它永遠認為「還有客戶」。

這就像在跑步機上跑步，卻忘記設定要跑多久。你會一直跑到累垮。

---

## 資料思維：一切都是資料

這個思維轉變最抽象，但也最重要。

在電腦的世界裡，一切都是資料：

- 你的名字是資料
- 你的年齡是資料
- 你的照片是資料
- 甚至程式本身也是資料

小美一開始很困惑：「照片怎麼會是資料？」

我打開一張照片的原始碼給她看——密密麻麻的數字。每個數字代表一個像素的顏色。對電腦來說，蒙娜麗莎的微笑不過是幾百萬個數字的組合。

這個認知很重要，因為一旦你理解「一切都是資料」，你就能理解為什麼電腦可以：

- 壓縮檔案（改變資料的排列方式）
- 加密訊息（把資料變成另一種形式）
- 傳送影片（把資料切成小塊傳送）

更重要的是，你會開始用「資料」的角度看問題。

客戶不滿意？那是一個資料點。
銷售下滑？查看資料趨勢。
要不要推出新產品？讓資料說話。

---

## 抽象思維：見樹又見林

程式設計的最高境界是抽象思維——能夠在細節和全局之間自如切換。

還記得小美的客戶整理程式嗎？一開始她寫的是：

```python
# 處理A公司的客戶
讀取A公司檔案()
刪除重複()
排序()
標記VIP()
輸出()

# 處理B公司的客戶
讀取B公司檔案()
刪除重複()
排序()
標記VIP()
輸出()

# 處理C公司的客戶...
```

她發現每個公司的處理流程都一樣，只是檔案名稱不同。這時候就需要抽象思維：

```python
def 處理客戶資料(公司名稱):
    讀取檔案(公司名稱)
    刪除重複()
    排序()
    標記VIP()
    輸出()

# 現在處理任何公司都只要一行
處理客戶資料("A公司")
處理客戶資料("B公司")
處理客戶資料("C公司")
```

抽象就是找出共同模式，把變化的部分參數化。

這就像食譜。你不會為每種蛋糕寫一個完全不同的食譜，而是有一個基本蛋糕食譜，然後說：「如果要巧克力口味，加可可粉；如果要檸檬口味，加檸檬汁。」

---

## 除錯思維：錯誤是最好的老師

小美最怕看到錯誤訊息。紅色的文字讓她覺得自己做錯了什麼嚴重的事。

直到有一天，她的程式出現了這個錯誤：

```
IndexError: list index out of range
```

她panic了。但我要她深呼吸，然後一起分析：

「Index是索引的意思，out of range是超出範圍。所以電腦在告訴你：你要找的東西超出了清單的範圍。」

就像你有一個10格的置物櫃，卻要打開第11格。

她檢查程式碼，果然發現她的客戶清單只有100個，但她卻想要處理第101個。

從那天起，她對錯誤訊息的態度變了。她開始把它們當作線索，而不是指責。

錯誤訊息就像醫生的診斷：

- 它告訴你哪裡出問題
- 它縮小了搜尋範圍
- 它給你改正的方向

學會「除錯思維」的人，會主動製造錯誤來測試程式的邊界。就像疫苗一樣，小劑量的錯誤能讓程式更強壯。

---

## 模式識別：發現重複的力量

人類大腦擅長模式識別，但在寫程式時，我們常常忘記運用這個能力。

小美有一天要處理公司的班表。她發現一個規律：

- 週一到週五是一個模式
- 週末是另一個模式
- 國定假日又是一個模式

與其為365天寫365個規則，她學會了識別和利用模式：

```python
if 是平日():
    套用平日班表()
elif 是週末():
    套用週末班表()
elif 是國定假日():
    套用假日班表()
```

這種思維不只在程式設計有用。當你開始注意模式，你會發現：

- 客訴總是集中在某幾個問題
- 銷售高峰總是出現在特定時段
- 成功的專案總是有某些共同特徵

程式思維訓練你成為模式偵探。

---

## 寫在第三章的最後

「像電腦一樣思考」聽起來很冰冷，但其實是在學習一種新的表達方式。

就像學習繪畫改變你看世界的方式——你開始注意光影、比例、色彩。學習程式思維也會改變你看問題的方式——你開始注意順序、條件、重複、模式。

小美現在不只會寫程式了。她的工作方式完全改變了：

- 開會時，她會把討論內容整理成決策樹
- 做專案時，她會先分解成小任務
- 遇到問題時，她會系統性地除錯

她說：「我沒有變成電腦，我只是學會了另一種思考方式。需要的時候可以切換，就像會說兩種語言的人。」

這就是程式思維的真正價值。不是要你變成機器，而是要你掌握機器的語言，讓機器為你工作。

下一章，我們要談談程式的基礎材料——資料。就像廚師需要了解食材，程式設計師需要了解資料。

但在那之前，試著用今天學到的思維方式，重新看看你的日常工作。你會發現，很多事情都可以被分解、被排序、被自動化。

而這，正是改變的開始。

---

*「學習程式，不是學習成為電腦，*
*而是學習如何與電腦合作。」*

# 第4章：資料與變數

> 「變數到底是什麼？為什麼要叫變數？它會變嗎？」
>
> 阿威第一次聽到「變數」這個詞時，滿臉疑惑。作為一個設計師，他習慣視覺化的東西，但變數...看不見也摸不著。

---

## 盒子的比喻

我給阿威看了我桌上的收納盒。

「你看，這個盒子上貼著標籤『筆』，裡面放著各種筆。這個貼著『名片』，裡面放著收集來的名片。」

他點點頭。

「變數就是這樣的盒子。標籤是變數的名字，裡面的東西是變數的值。」

```python
筆 = "原子筆"
名片 = "王小明的名片"
```

「但為什麼叫『變』數？」他追問。

我把筆盒裡的原子筆換成鉛筆。

「盒子還是同一個盒子，標籤還是『筆』，但內容變了。這就是為什麼叫變數——內容可以變。」

```python
筆 = "原子筆"
# 過了一會兒...
筆 = "鉛筆"
# 盒子（變數）沒變，但內容（值）變了
```

阿威恍然大悟：「所以變數就像Photoshop的圖層名稱，名稱不變，但圖層內容可以隨時更換！」

沒錯，每個領域都有類似的概念，只是叫法不同。

---

## 資料的形態

如果變數是盒子，那資料就是放進盒子的東西。而不同的東西，需要不同的存放方式。

### 數字：精確的朋友

數字是最單純的資料。

```python
年齡 = 35
薪水 = 50000
溫度 = 23.5
```

小美發現了一個有趣的事：「為什麼有時候數字有小數點，有時候沒有？」

這就像問：為什麼有時候我們說「3個人」而不是「3.5個人」？

- 整數（Integer）：完整的數字，用來數東西
- 浮點數（Float）：有小數點的數字，用來測量

你不會說你有2.5個小孩，但你會說今天氣溫25.5度。

程式也是這樣。庫存數量用整數，價格用浮點數。選對了，程式更清晰；選錯了，可能出現奇怪的結果。

### 文字：有故事的資料

```python
姓名 = "小美"
公司 = "快樂行銷有限公司"
座右銘 = "每天進步1%"
```

文字（String）是用引號包起來的任何東西。

阿威問：「為什麼要用引號？」

「因為要告訴電腦：這是文字，不是指令。」

看看這個差別：

```python
print(Hello)    # 電腦會找一個叫Hello的變數
print("Hello")  # 電腦知道要顯示Hello這個字
```

就像在對話中，我們用引號標示別人說的話：
小美說：「我要學程式。」

引號告訴讀者：這是原話，不是我的詮釋。

### 布林：是與否的世界

這是最簡單卻最強大的資料型態。

```python
已婚 = True
有駕照 = False
是VIP = True
```

## 第一個自動化：批次改名

## Excel自動化：解放小美

```python
平均客單價 = 總表["銷售額"].mean()
```

## 網路爬蟲：資訊自動收集

小李的工作是市場研究，需要每天查看競爭對手的價格。

自動化解決方案：

```python
import requests
from bs4 import BeautifulSoup
import schedule

def 檢查競品價格():
    競品網址 = [
        "https://competitor1.com/product",
        "https://competitor2.com/product",
        # ...
    ]

    今日價格 = {}

    for 網址 in 競品網址:
        回應 = requests.get(網址)
        網頁 = BeautifulSoup(回應.text, 'html.parser')

        # 找出價格（每個網站結構不同）
        價格元素 = 網頁.find('span', class_='price')
        價格 = float(價格元素.text.replace('$', ''))

        商家名稱 = 網址.split('/')[2]
        今日價格[商家名稱] = 價格

    # 儲存到資料庫或Excel
    儲存價格(今日價格)

    # 如果有異常變動，發出警報
    for 商家, 價格 in 今日價格.items():
        if 價格變動超過10%:
            發送警報(f"{商家}價格異常變動！")

# 每天早上9點自動執行
schedule.every().day.at("09:00").do(檢查競品價格)
```

## 檔案整理：數位斷捨離

阿威的電腦桌面是一場災難。幾百個檔案散落各處，要找東西全靠搜尋。

我們寫了一個自動整理程式：

```python
import os
import shutil
from datetime import datetime

def 整理桌面():
    桌面路徑 = os.path.expanduser("~/Desktop")

    # 建立分類資料夾
    分類 = {
        "圖片": ['.jpg', '.png', '.gif', '.psd'],
        "文件": ['.doc', '.docx', '.pdf', '.txt'],
        "試算表": ['.xls', '.xlsx', '.csv'],
        "影片": ['.mp4', '.mov', '.avi'],
        "其他": []
    }

    # 建立資料夾
    for 資料夾名稱 in 分類.keys():
        資料夾路徑 = os.path.join(桌面路徑, 資料夾名稱)
        if not os.path.exists(資料夾路徑):
            os.makedirs(資料夾路徑)

    # 整理檔案
    for 檔案名稱 in os.listdir(桌面路徑):
        if os.path.isfile(os.path.join(桌面路徑, 檔案名稱)):
            檔案分類 = "其他"

            for 類別, 副檔名列表 in 分類.items():
                if any(檔案名稱.endswith(ext) for ext in 副檔名列表):
                    檔案分類 = 類別
                    break

            # 移動檔案
            來源 = os.path.join(桌面路徑, 檔案名稱)
            目標 = os.path.join(桌面路徑, 檔案分類, 檔案名稱)
            shutil.move(來源, 目標)
            print(f"已移動：{檔案名稱} → {檔案分類}/")
```

執行後，混亂的桌面瞬間井然有序。圖片在圖片資料夾，文件在文件資料夾。

「這就像請了一個數位管家，」阿威說。

```python
def 本月統計(self):
    """顯示本月收支統計"""
    本月 = datetime.datetime.now().strftime("%Y-%m")

    本月帳目 = [
        帳目 for 帳目 in self.帳目
        if 帳目["日期"].startswith(本月)
    ]

    總收入 = sum(
        帳目["金額"] for 帳目 in 本月帳目
        if 帳目["類型"] == "收入"
    )

    總支出 = sum(
        帳目["金額"] for 帳目 in 本月帳目
        if 帳目["類型"] == "支出"
    )

    餘額 = 總收入 - 總支出

    print(f"\n📊 {本月} 月統計")
    print("=" * 40)
    print(f"總收入：${總收入:,.0f}")
    print(f"總支出：${總支出:,.0f}")
    print(f"餘額：  ${餘額:,.0f}")

    # 支出分類統計
    分類統計 = {}
    for 帳目 in 本月帳目:
        if 帳目["類型"] == "支出":
            類別 = 帳目["類別"]
            分類統計[類別] = 分類統計.get(類別, 0) + 帳目["金額"]

    if 分類統計:
        print("\n📈 支出分類：")
        for 類別, 金額 in sorted(分類統計.items(), key=lambda x: x[1], reverse=True):
            百分比 = (金額 / 總支出) * 100
            進度條 = "■" * int(百分比 / 5)
            print(f"  {類別:　<4} ${金額:>8,.0f} {進度條} {百分比:.1f}%")
```

```python
def 檢查預算(self):
    """檢查是否超出預算"""
    本月 = datetime.datetime.now().strftime("%Y-%m")

    本月支出 = sum(
        帳目["金額"] for 帳目 in self.帳目
        if 帳目["日期"].startswith(本月) and 帳目["類型"] == "支出"

    使用率 = (本月支出 / self.每月預算) * 100
    剩餘預算 = self.每月預算 - 本月支出

        print(f"⚠️ 警告：本月已超支 ${-剩餘預算:,.0f}！")
    elif 使用率 > 80:
        print(f"⚠️ 注意：預算已使用 {使用率:.0f}%，剩餘 ${剩餘預算:,.0f}")
```

## 資訊焦慮的時代

我們生活在一個矛盾的時代。

從來沒有一個時代，學習資源如此豐富——YouTube上有無數免費教學、GitHub上有海量開源專案、Medium上有數不清的技術文章。

但也從來沒有一個時代，學習者如此焦慮——怕選錯方向、怕落後潮流、怕浪費時間。

「我光是選要看哪個教學，就花了一個禮拜。」阿威苦笑。

這就是選擇的詛咒。當選項太多，我們反而無法行動。

讓我告訴你一個秘密：你不需要看完所有資源。你只需要找到適合你的那一個，然後堅持下去。

## 免費資源的寶庫

### YouTube：視覺學習者的天堂

小李是視覺學習者，他需要看到畫面才能理解。

**推薦頻道：**

**freeCodeCamp**

- 特色：超長但完整的教學（動輒4-8小時）
- 適合：想要一次學透徹的人
- 小李的體驗：「雖然很長，但講得很細，適合週末慢慢看。」

**Traversy Media**

- 特色：實作導向，跟著做就會
- 適合：想快速上手的人
- 阿威的最愛：「Brad的聲音很舒服，而且專案都很實用。」

**彭彭的課程**（中文）

- 特色：中文教學，適合初學者
- 適合：英文不太好的朋友
- 小美的救星：「終於有人用中文解釋那些概念了！」

但YouTube有個陷阱——你很容易陷入「教學地獄」，一直看影片卻不動手寫。

記住：看一小時，練習兩小時。

### GitHub：程式碼的圖書館

老陳第一次上GitHub時嚇到了：「這麼多程式碼，都是免費的？」

是的，這就是開源的力量。

**如何利用GitHub學習：**

**1. 讀優秀的程式碼**

```
搜尋技巧：
- 加上 "awesome" 找精選列表
- 按星星數排序找熱門專案
- 看 Trending 了解潮流
```

**2. 從README開始**
好的專案都有詳細的README，告訴你：

- 這個專案做什麼
- 如何安裝
- 如何使用
- 如何貢獻

**3. 追蹤學習路徑**
很多人整理了學習路徑：

- developer-roadmap（最完整的技能樹）
- free-programming-books（免費書籍大全）
- project-based-learning（用專案學習）

老陳現在每天早上都會看GitHub Trending：「就像看報紙，了解技術圈在關注什麼。」

### 線上練習平台

```

缺點：
- 免費版內容有限
- 太過引導，缺乏思考

小美的評價：
```

```
時機：
不要太早開始刷題！
先把基礎打好，
準備面試時再來。

策略：
- 從Easy開始
- 重質不重量
- 理解思路比背答案重要
```

Google了無數解法，Stack Overflow看了幾十篇，程式碼改了又改，還是不行。

「為什麼別人的程式碼都能跑，就我的不行？」

他關上筆電，走到陽台。城市的夜景很美，但他沒有心情欣賞。

「也許我真的沒有寫程式的天份。」

這個念頭不是第一次出現。每次遇到解不開的bug，每次看到別人輕鬆寫出複雜功能，每次面對新技術的學習曲線，這個聲音就會在耳邊響起。

如果你也曾經有過這樣的時刻，我想告訴你——這很正常。

每個工程師都經歷過。包括那些你以為的「天才」。

## bug是最好的老師

隔天早上，阿威帶著黑眼圈來找我。

「我想放棄了。」他說。

我請他把程式碼打開，一起看看。

就這樣。三個小時的痛苦，源於兩個字母的大小寫。

阿威愣住了，然後苦笑：「就這樣？」

「就這樣。」

「我覺得自己好蠢。」

「不，」我說，「你剛剛學到了一課——JavaScript區分大小寫。而且我保證，你這輩子都不會再犯這個錯誤。」

這就是bug的價值。每個bug都是一個偽裝的老師：

```
Bug教會我們的事：

邏輯錯誤 → 思考的嚴謹性
效能問題 → 優化的必要性
安全漏洞 → 防禦的意識

每解決一個bug，
你就升級一點。
```

凱文分享了他的經驗：「我有一個『bug日記』，記錄每個遇到的問題和解法。現在回頭看，那些曾經讓我崩潰的bug，都成了我的養分。」

## 冒牌者症候群

小美最近加入了一個程式學習群組。

「裡面的人都好厲害，」她說，「討論的東西我都聽不懂。我是不是太弱了？」

這叫「冒牌者症候群」（Impostor Syndrome）——覺得自己不夠格，隨時會被發現是「冒牌貨」。

真相是什麼？

我打開了一個調查結果：

- 70%的工程師經常感到自己能力不足
- 58%的資深工程師仍然有冒牌者症候群
- 即使是技術主管，也常常懷疑自己

「連資深工程師都這樣？」小美驚訝。

是的。因為技術永遠在進步，永遠有你不知道的東西。

### 如何面對冒牌者症候群

**1. 承認你的成長**

小美三個月前連變數是什麼都不知道，現在已經能寫自動化程式了。

「列出你學會的東西，」我建議，「你會驚訝自己進步了多少。」

```python
三個月前的我：
- 不知道變數是什麼
- 沒寫過一行程式碼
- 看到終端機就害怕

現在的我：
- 能寫Python程式
- 完成了5個小專案
- 幫公司自動化了3個流程
```

**2. 理解知識的層次**

```
程式知識像一座冰山：

水面上（看得見的）：
- 別人展示的成果
- 技術文章的精華
- 開源專案的成品

水面下（看不見的）：
- 無數的失敗嘗試
- 數不清的bug除錯
- 漫長的學習過程
```

你看到的「厲害」，都是冰山一角。

**3. 專注於自己的進度**

「不要和別人比，要和昨天的自己比。」

老陳深有體會：「我50歲才開始學，當然比不過20歲的年輕人。但那又怎樣？我只要比昨天的我進步就好。」

## 學習高原期

學習曲線不是一條斜直線，而是階梯狀的。

```
理想的學習曲線：
能力 ↑
     ╱╱╱╱╱
    ╱╱╱╱╱
   ╱╱╱╱╱
  ╱╱╱╱╱
時間 →

實際的學習曲線：
能力 ↑
       ＿＿＿＿＿╱
      │
   ＿＿╱
  │
時間 →
```

那些平坦的部分，就是「高原期」。

小李最近就在高原期：「我感覺自己沒在進步，每天寫的程式碼都差不多。」

這是最危險的時期，很多人在這裡放棄。

### 突破高原期的方法

**1. 改變學習內容**

如果你一直在寫CRUD（增刪改查），試試：

- 學習新的框架
- 挑戰不同類型的專案
- 深入研究原理

**2. 提高難度**

阿威突破高原期的方法：「我給自己設限制，比如不用框架實現功能，或者優化程式效能到極致。」

**3. 教學相長**

「開始寫部落格後，我發現自己理解更深了。」小美說。

教別人是最好的學習方式。

## 完美主義的陷阱

阿威是完美主義者。

他的第一個專案，改了三個月還沒上線。

「還不夠好，」他總是這麼說，「再改改。」

直到有一天，他發現類似的網站已經有五個了，而他的還在本機裡。

### Done is better than perfect

Facebook辦公室有句標語：「Done is better than perfect」（完成比完美更重要）。

```
完美主義的代價：

拖延上線 → 錯失機會
過度優化 → 浪費時間
害怕批評 → 不敢分享
追求完美 → 永遠無法完成
```

**正確的做法：**

1. 先求有，再求好
2. 快速迭代，持續改進
3. 接受「夠好」的標準
4. 從回饋中學習

凱文的建議：「設定期限。時間到了就發布，不管完不完美。你會發現，用戶在意的往往不是你糾結的細節。」

## 持續學習的智慧

「學程式最累的是，永遠學不完。」小美嘆氣。

確實，新技術層出不窮：

- 昨天流行React
- 今天流行Vue
- 明天可能是Svelte

但這裡有個秘密：你不需要學所有東西。

### 學習的優先順序

```
核心知識（必學）：
- 程式邏輯
- 資料結構基礎
- 問題解決能力
├─ 這些10年都不會變

工作需要（選學）：
- 公司使用的技術
- 產業標準工具
├─ 這些能幫你賺錢

興趣導向（想學）：
- 新奇的技術
- 好玩的專案
├─ 這些讓你保持熱情
```

### FOMO vs JOMO

FOMO（Fear of Missing Out）：害怕錯過
JOMO（Joy of Missing Out）：享受錯過

老陳的哲學：「我不追新技術，我追穩定技術。當大家都在學最新框架時，我在深入研究JavaScript基礎。結果？框架換了三輪，我的基礎知識還是有用。」

- 建立心理韌性
- 累積除錯經驗
