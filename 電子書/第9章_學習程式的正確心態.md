# 第9章：學習程式的正確心態

> 「我已經卡在這個bug三個小時了，是不是我真的不適合寫程式？」
>
> 深夜兩點，阿威盯著螢幕，眼睛布滿血絲。錯誤訊息還是一樣，程式還是跑不動。

---

## 深夜的崩潰

那是一個星期三的深夜。

阿威想要完成一個作品集網站，展示他的設計作品。本來以為很簡單的功能——點擊圖片放大顯示——就是做不出來。

Google了無數解法，Stack Overflow看了幾十篇，程式碼改了又改，還是不行。

「為什麼別人的程式碼都能跑，就我的不行？」

他關上筆電，走到陽台。城市的夜景很美，但他沒有心情欣賞。

「也許我真的沒有寫程式的天份。」

這個念頭不是第一次出現。每次遇到解不開的bug，每次看到別人輕鬆寫出複雜功能，每次面對新技術的學習曲線，這個聲音就會在耳邊響起。

如果你也曾經有過這樣的時刻，我想告訴你——這很正常。

每個工程師都經歷過。包括那些你以為的「天才」。

---

## bug是最好的老師

隔天早上，阿威帶著黑眼圈來找我。

「我想放棄了。」他說。

我請他把程式碼打開，一起看看。

五分鐘後，我們找到了問題——一個拼寫錯誤。`getElementById` 寫成了 `getElementByID`（ID大寫了）。

就這樣。三個小時的痛苦，源於兩個字母的大小寫。

阿威愣住了，然後苦笑：「就這樣？」

「就這樣。」

「我覺得自己好蠢。」

「不，」我說，「你剛剛學到了一課——JavaScript區分大小寫。而且我保證，你這輩子都不會再犯這個錯誤。」

這就是bug的價值。每個bug都是一個偽裝的老師：

```
Bug教會我們的事：

拼寫錯誤 → 細心的重要性
邏輯錯誤 → 思考的嚴謹性
效能問題 → 優化的必要性
安全漏洞 → 防禦的意識

每解決一個bug，
你就升級一點。
```

凱文分享了他的經驗：「我有一個『bug日記』，記錄每個遇到的問題和解法。現在回頭看，那些曾經讓我崩潰的bug，都成了我的養分。」

## 冒牌者症候群

小美最近加入了一個程式學習群組。

「裡面的人都好厲害，」她說，「討論的東西我都聽不懂。我是不是太弱了？」

這叫「冒牌者症候群」（Impostor Syndrome）——覺得自己不夠格，隨時會被發現是「冒牌貨」。

真相是什麼？

我打開了一個調查結果：
- 70%的工程師經常感到自己能力不足
- 58%的資深工程師仍然有冒牌者症候群
- 即使是技術主管，也常常懷疑自己

「連資深工程師都這樣？」小美驚訝。

是的。因為技術永遠在進步，永遠有你不知道的東西。

### 如何面對冒牌者症候群

**1. 承認你的成長**

小美三個月前連變數是什麼都不知道，現在已經能寫自動化程式了。

「列出你學會的東西，」我建議，「你會驚訝自己進步了多少。」

```python
三個月前的我：
- 不知道變數是什麼
- 沒寫過一行程式碼
- 看到終端機就害怕

現在的我：
- 能寫Python程式
- 完成了5個小專案
- 幫公司自動化了3個流程
```

**2. 理解知識的層次**

```
程式知識像一座冰山：

水面上（看得見的）：
- 別人展示的成果
- 技術文章的精華
- 開源專案的成品

水面下（看不見的）：
- 無數的失敗嘗試
- 數不清的bug除錯
- 漫長的學習過程
```

你看到的「厲害」，都是冰山一角。

**3. 專注於自己的進度**

「不要和別人比，要和昨天的自己比。」

老陳深有體會：「我50歲才開始學，當然比不過20歲的年輕人。但那又怎樣？我只要比昨天的我進步就好。」

## 學習高原期

學習曲線不是一條斜直線，而是階梯狀的。

```
理想的學習曲線：
能力 ↑
     ╱╱╱╱╱
    ╱╱╱╱╱
   ╱╱╱╱╱
  ╱╱╱╱╱
時間 →

實際的學習曲線：
能力 ↑
       ＿＿＿＿＿╱
      │
   ＿＿╱
  │
時間 →
```

那些平坦的部分，就是「高原期」。

小李最近就在高原期：「我感覺自己沒在進步，每天寫的程式碼都差不多。」

這是最危險的時期，很多人在這裡放棄。

### 突破高原期的方法

**1. 改變學習內容**

如果你一直在寫CRUD（增刪改查），試試：
- 學習新的框架
- 挑戰不同類型的專案
- 深入研究原理

**2. 提高難度**

阿威突破高原期的方法：「我給自己設限制，比如不用框架實現功能，或者優化程式效能到極致。」

**3. 教學相長**

「開始寫部落格後，我發現自己理解更深了。」小美說。

教別人是最好的學習方式。

## 完美主義的陷阱

阿威是完美主義者。

他的第一個專案，改了三個月還沒上線。

「還不夠好，」他總是這麼說，「再改改。」

直到有一天，他發現類似的網站已經有五個了，而他的還在本機裡。

### Done is better than perfect

Facebook辦公室有句標語：「Done is better than perfect」（完成比完美更重要）。

```
完美主義的代價：

拖延上線 → 錯失機會
過度優化 → 浪費時間
害怕批評 → 不敢分享
追求完美 → 永遠無法完成
```

**正確的做法：**

1. 先求有，再求好
2. 快速迭代，持續改進
3. 接受「夠好」的標準
4. 從回饋中學習

凱文的建議：「設定期限。時間到了就發布，不管完不完美。你會發現，用戶在意的往往不是你糾結的細節。」

## 持續學習的智慧

「學程式最累的是，永遠學不完。」小美嘆氣。

確實，新技術層出不窮：
- 昨天流行React
- 今天流行Vue
- 明天可能是Svelte

但這裡有個秘密：你不需要學所有東西。

### 學習的優先順序

```
核心知識（必學）：
- 程式邏輯
- 資料結構基礎
- 問題解決能力
├─ 這些10年都不會變

工作需要（選學）：
- 公司使用的技術
- 產業標準工具
├─ 這些能幫你賺錢

興趣導向（想學）：
- 新奇的技術
- 好玩的專案
├─ 這些讓你保持熱情
```

### FOMO vs JOMO

FOMO（Fear of Missing Out）：害怕錯過
JOMO（Joy of Missing Out）：享受錯過

老陳的哲學：「我不追新技術，我追穩定技術。當大家都在學最新框架時，我在深入研究JavaScript基礎。結果？框架換了三輪，我的基礎知識還是有用。」

## 失敗是功能，不是bug

每個成功的工程師都有失敗的故事。

凱文分享了他的：
- 第一個專案，資料庫設計錯誤，重寫三次
- 第一次面試，連FizzBuzz都寫不出來
- 第一個上線產品，被駭客攻擊

「但每次失敗，都讓我變強。」

### 重新定義失敗

```python
def 學習程式(嘗試):
    if 成功:
        return 獲得經驗
    else:
        return 獲得教訓

# 注意：沒有真正的失敗
# 只有不同形式的學習
```

失敗的價值：
- 知道什麼不可行
- 理解問題的複雜性
- 建立心理韌性
- 累積除錯經驗

---

## 找到你的為什麼

當學習變得困難，當bug解不開，當新技術讓你頭痛，記得你的「為什麼」。

小美的為什麼：「我想要更多時間陪家人，不想把時間浪費在重複工作上。」

阿威的為什麼：「我想創造美的東西，而且是能動的、有互動的。」

老陳的為什麼：「我想證明50歲也能學新東西。」

小李的為什麼：「我想要自由，能在任何地方工作。」

你的為什麼是什麼？

寫下來，貼在螢幕旁。當你想放棄時，看看它。

## 慶祝小勝利

我們太習慣關注還不會的，忘記慶祝已經會的。

建立慶祝儀式：
- 解決一個bug → 伸個懶腰，smile
- 完成一個功能 → 喝杯好咖啡
- 上線一個專案 → 和朋友分享

「我開始記錄『成就日記』，」阿威說，「每天寫下一件進步的事，哪怕很小。」

```
今天的小勝利：
✓ 理解了閉包的概念
✓ 修復了一個困擾兩天的bug
✓ 幫同事解決了問題
✓ 程式碼比昨天寫得更簡潔
```

累積小勝利，就能堆出大成就。

## 寫在第九章的最後

學程式不只是學技術，更是一場心理修煉。

你會遇到：
- 讓你懷疑自己的bug
- 讓你焦慮的新技術
- 讓你沮喪的面試
- 讓你迷茫的選擇

但記住，這些都是過程，不是終點。

每個你admire的工程師，都走過同樣的路。他們不是沒有挫折，而是學會了與挫折共處。

正確的心態不是永遠正面，而是：
- 接受困難是常態
- 相信自己能成長
- 享受解決問題的過程
- 保持好奇心

最後，我想分享一句話，這是一位資深工程師告訴我的：

「寫程式就像種樹。你不會因為樹長得慢就懷疑自己不會種樹。你只是每天澆水，總有一天，你會看到它開花結果。」

你已經在澆水了。

繼續澆，保持耐心。

花會開的。
